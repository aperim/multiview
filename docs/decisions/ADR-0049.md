# ADR-0049: WebRTC program outputs ŌĆö WHEP serve (`webrtc`) and WHIP push (`whip_push`)

- **Status:** Proposed
- **Area:** Core Engine
- **Date:** 2026-06-10
- **Source brief:** [webrtc.md](../research/webrtc.md)

## Decision

Add two WebRTC variants to the internally-tagged `Output` enum (`crates/multiview-config/src/schema.rs`, `#[serde(tag = "kind", rename_all = "snake_case")]`), matching the established variant shape (optional `id` per [ADR-0034](ADR-0034.md), `gpu_pin: Option<DevicePin>` per [ADR-0018](ADR-0018.md) ┬¦2.1, `audio: Option<OutputAudio>` validated by the `Output::audio_capability` matrix at `crates/multiview-config/src/schema.rs:869`). **`Output::Webrtc { id, label, max_viewers, token, codec, gpu_pin, audio }`** serves the program to N browser viewers over WHEP: `label` is explicit (like `Aes67`, there is no mount/path/url to derive one from), `max_viewers` is a serde-defaulted `u32` (default **8**), `token: Option<String>` is a per-output Bearer secret ŌĆö plaintext in v1: tokens follow the existing config-secret posture (returned to authorized readers, present in config export, exactly like the stream keys embedded in `rtmp`/`srt` URLs) and migrate together with those url-embedded keys if/when a `secret_ref` indirection lands ŌĆö and `codec` defaults to `"h264"` (the only v1 value ŌĆö the variant names a **program rendition to consume**, not an encode to spawn). **`Output::WhipPush { id, url, token, codec, gpu_pin, audio }`** publishes the program to a remote WHIP ([RFC 9725](https://www.rfc-editor.org/rfc/rfc9725)) endpoint ŌĆö the WebRTC sibling of the `Rtmp`/`Srt` push variants, IPv6-first in every example (`url = "https://[2001:db8::15]:8443/whip/pgm1"`, [ADR-0042](ADR-0042.md)). For both, `gpu_pin` pins the encode stage of the rendition they consume, and the audio capability is declared **single-track** (one Opus m-line): multitrack selections are rejected at config time and degrade explicitly to the mixed program bus, never silently ([ADR-R005](ADR-R005.md) ┬¦4.2 doctrine).

**Encode-once fan-out (invariant #7).** Neither variant ever encodes video. Each is a `PacketSink` registered on the `multiview-output` fan-out `PacketRouter` (`crates/multiview-output/src/fanout.rs`): enabling a WebRTC output is `route()+1` on an already-encoded program rendition, and the per-viewer marginal cost is **packetization only** ŌĆö the same access-unit bytes are sample-written into each viewer's str0m session over the shared transport endpoint of [ADR-0048](ADR-0048.md) (RTP header work + SRTP per viewer, one encode total; the encode-once proof in `crates/multiview-output/tests/encode_once_sink_mover.rs` covers these sinks unchanged). The sink caches the rendition's SPS/PPS and prepends them at every IDR so a late joiner decodes from the next keyframe ŌĆö in-band parameter sets as [RFC 7742](https://www.rfc-editor.org/rfc/rfc7742) ┬¦6.2 requires (`sprop-parameter-sets` MUST NOT be relied on), the unit-remuxer pattern proven in open-source gateways. RTP timestamps derive from the packet's tick-stamped PTS (invariant #3) at 90 kHz video / 48 kHz audio; raw input time never reaches a viewer.

**Rendition constraints are a config-time error, and keyframes are demand-driven but rate-limited.** The consumed rendition must be H.264 with **B-frames off** and **repeat-headers on**: RTP carries no DTS, so WebRTC receivers require decode order == transmission order, and mid-stream joiners need headers in-band. `MultiviewConfig::validate` ŌåÆ `validate_outputs` (`crates/multiview-config/src/lib.rs:477`) validates `bf=0` + repeat-headers on any rendition a `webrtc`/`whip_push` output consumes and **fails the config-file load otherwise** (a hard validation error, never a silent rewrite); a **live** apply that would force an existing rendition to `bf=0` is surfaced as a Class-2 plan (encoder reset) per the live-apply doctrine ([ADR-R004](ADR-R004.md)) ŌĆö never a runtime surprise. Viewer-join and incoming PLI/FIR (str0m `Event::KeyframeRequest`) drive a new **force-keyframe seam** on the program encoder, rate-limited to a **Ōēź2 s floor and coalesced across viewers**, so a PLI storm cannot inflate the program bitrate (worst case one extra IDR per 2 s; a forced IDR is a valid segment boundary for HLS sinks sharing the rendition, [ADR-0007](ADR-0007.md)).

**One Opus audio rendition (encode-once extends to audio).** The post-loudnorm program `AudioBlock` stream in `consumer_main` (`crates/multiview-cli/src/pipeline.rs`; `LoudnormProcessor`, `crates/multiview-audio/src/loudnorm.rs`) feeds a bounded drop-oldest ring into **one** libopus encode per program (48 kHz / 20 ms frames), and those same coded packets fan to every program-scope WHEP preview session ([ADR-P006](ADR-P006.md)), every `webrtc` viewer, and the `whip_push` session ŌĆö the AUD-4b shape in [`docs/development/work-schedule.md`](../development/work-schedule.md): audio encoder count == number of distinct audio renditions, not number of sinks. The encoder is **lazy with refcount semantics** mirroring the [ADR-P003](ADR-P003.md) taps: it starts when the first audio-consuming WebRTC session or output attaches and stops with the last ŌĆö a configured `webrtc`/`whip_push` output with audio counts as a standing consumer, so it is always-on while that output runs. Each session carries it as one sendonly Opus `m=audio` line ([RFC 7587](https://www.rfc-editor.org/rfc/rfc7587) payload).

**WHEP serve surface.** `POST /api/v1/whep/{output_id}` with `application/sdp` ŌåÆ `201 Created` + `Location: /api/v1/whep/{output_id}/sessions/{session_id}` + SDP answer; OPTIONS preflight with `Accept-Post: application/sdp`; `DELETE` on the session URL releases it (idempotent; it requires the **same credential class as the creating POST** ŌĆö the per-output Bearer or an API key ŌĆö and session ids are **Ōēź128-bit random** from the OS RNG, hex/base64url-encoded, never sequential); `PATCH` ŌåÆ **405 Method Not Allowed** with `Allow: DELETE, OPTIONS` (vanilla ICE ŌĆö trickle/restart are unimplemented and serverŌåÆclient trickle is forbidden; RFC 9725 defines only **422** for a server that supports one kind of PATCH and receives the other and leaves the supports-neither case unspecified, so the rejection rides [RFC 9110](https://www.rfc-editor.org/rfc/rfc9110)'s generic method semantics, matching deployed WHEP servers); 415 wrong content type, 400 malformed offer, **406** for a codec-incompatible offer (ecosystem practice, no RFC behind it; the existing preview WHEP routes keep their shipped mapping), 401/403 missing/bad credentials: with `token` set the per-output Bearer alone suffices ([RFC 6750](https://www.rfc-editor.org/rfc/rfc6750)), with `token = None` viewing requires a control-plane API key with **View** scope (read-shaped ŌĆö View suffices, not Write) ŌĆö never anonymous. **Capacity:** a viewer beyond `max_viewers` ŌĆö or beyond the endpoint-global `webrtc.max_sessions` of [ADR-0048](ADR-0048.md), which caps **preview + output-viewer sessions only**: WHIP-ingest sessions (bounded by the count of configured `webrtc` sources) and the `whip_push` client session (bounded by configured outputs) are admitted **outside** that pool, so a viewer flood can never starve a publisher or the push client's reconnect ŌĆö receives **503 + `Retry-After`**; the **409** conflict status remains the WHIP-ingest second-publisher answer ([ADR-T014](ADR-T014.md)) and has no WHEP analogue (viewer slots are capacity, not exclusivity). Errors are RFC 9457 problem+json, matching the preview WHEP handlers (`crates/multiview-control/src/routes/preview.rs`); these routes are a **real-output** surface distinct from `/api/v1/preview/ŌĆ”/whep`, registered in OpenAPI with the utoipa triple registration and covered by the route-completeness assertion (`ApiDoc::rest_routes` in `crates/multiview-control/src/openapi.rs` vs the mounted router, `tests/openapi.rs`). Signaling rides the control listener but media rides the shared `[::]` UDP endpoint: a control-plane outage stops *new* joins, never existing sessions, never the engine.

**WHIP push client.** Multiview is the RFC 9725 client: sendonly offer with **host candidates only** ŌĆö we initiate the outbound UDP flow, so our own connectivity checks create the NAT mapping and the remote's responses return on the established 5-tuple; no STUN/TURN client is needed, the same reachability argument that lets RTMP/SRT push work from behind NAT. The client POSTs the offer with Bearer `token`, follows **307/308 redirects preserving method and headers** (depth-capped at 3, **https-only** ŌĆö an httpsŌåÆhttp downgrade aborts; 308 and relative-`Location` resolution are generic HTTP, [RFC 9110](https://www.rfc-editor.org/rfc/rfc9110)/[RFC 3986](https://www.rfc-editor.org/rfc/rfc3986), not RFC 9725 provisions), applies the answer, and offers **`a=setup:actpass`**: the answerer chooses its DTLS role ([RFC 5763](https://www.rfc-editor.org/rfc/rfc5763) answerers typically pick `active`), so Multiview commonly runs the DTLS **server** role. The client is **supervised with backoff reconnect** exactly like the RTMP/SRT push clients, riding the same output state machine.

**Both are REAL outputs (invariants #10/#11).** They are engine-protected sinks, **never shed by the preview degradation rungs** (the PRV-4 ladder and `FocusGate` caps of [ADR-P003](ADR-P003.md) govern preview sessions only); shedding them follows output policy like any RTSP/HLS sink. Every viewer/session owns a **bounded drop-oldest send ring** behind the non-blocking `PacketSink::deliver` contract ŌĆö a slow viewer loses *its* packets, never stalls the fan-out or the output clock (the invariant-#10 chaos-gate shape; the engine never awaits a client). Adding/removing a viewer or the whole output is a pure routing-table change (`PacketRouter::register`/`deregister`) applied at a frame boundary; forcing a previously non-compliant rendition to `bf=0` is an encoder reset and is surfaced as Class-2 per [ADR-R004](ADR-R004.md).

## Rationale

Sub-second, browser-native program distribution is the one delivery rung HLS/LL-HLS cannot reach, and serving it as a fan-out consumer keeps the cost honest: the video encode already exists, the encode-once contract is already proven by call-count tests on the very router these sinks register with, and the per-viewer cost collapses to RTP headers + SRTP. The B-frame/repeat-header constraint usually costs nothing because [ADR-E003](ADR-E003.md) already mandates minimal/zero B-frames and zerolatency-class encodes for the program renditions, so in the common case "validation" is a no-op confirmation rather than a quality trade. The single Opus encode is the entire marginal media cost of audio and is exactly the encode-once-per-distinct-rendition rule ([ADR-E004](ADR-E004.md)) applied to audio. WHIP gives push parity with RTMP/SRT against the growing set of WHIP-terminating origins; WHEP's draft status (draft-ietf-wish-whep, expired) is acknowledged in the [brief](../research/webrtc.md) ŌĆö it is nonetheless the de-facto ecosystem playback handshake, and our surface sticks to its stable core (POST/Location/DELETE).

## Alternatives considered

**Per-viewer (or per-output) re-encode / ABR per viewer** (rejected: an explicit product non-goal ŌĆö [core-engine ┬¦2](../research/core-engine.md) ŌĆö and a direct violation of invariant #7; viewers consume the program rendition or nothing). **Serving viewers from the preview encoder ladder** (rejected: preview encodes are FocusGate-capped, deliberately sheddable, and canvas-approximate in fidelity ŌĆö an `Output` must survive degradation and carry the *real* coded program; the real-vs-approx fidelity labels live in [ADR-P006](ADR-P006.md)). **A multi-protocol sidecar gateway fed by RTSP/SRT** (rejected as *the* WebRTC path: an extra hop adds latency and an ops surface, and it severs the PLIŌåÆforce-IDR coupling to our encoder and the in-process encode-once fan; [ADR-0006](ADR-0006.md) keeps a sidecar as an optional interop story, which stands). **Embedded STUN/TURN client for `whip_push`** (rejected: the initiator-with-host-candidates argument covers NAT egress; a relay, where genuinely needed, belongs at the remote origin ŌĆö boundary stated in the brief). **Trickle ICE (`PATCH`) and ICE restart** (out of scope, answered 405 + `Allow` per RFC 9110 generic method semantics: both are OPTIONAL in the specs, and vanilla-ICE clients ŌĆö every browser, OBS Ōēź30 ŌĆö complete without them).

## Consequences

Enabling a `webrtc`/`whip_push` output **pins the consumed rendition's encoder settings** (`bf=0`, repeat-headers): a config that pairs such an output with a B-frame rendition fails validation, and retrofitting compliance onto a running rendition is a Class-2 encoder reset the operator sees before applying (invariant #11). The program gains one Opus encode (small, refcounted-lazy per the lifecycle above) and a new exposure surface: signaling routes with Bearer handling, plus the shared UDP media port ŌĆö bounded by `max_viewers`, `webrtc.max_sessions`, and per-session memory caps ([ADR-0048](ADR-0048.md)); session counts, ring drops, and PLI rates become Prometheus metrics in `multiview-telemetry`. The Ōēź2 s force-IDR floor means a worst-case joiner waits up to ~2 s for first decodable video (SPS/PPS are prepended, decode starts at the next IDR) ŌĆö accepted as the cost of keeping keyframe demand from becoming a bitrate amplifier. The push client deliberately carries `Authorization` across cross-origin **https** 307/308 hops: the bearer is already entrusted to the configured endpoint's operator, who could forward it server-side regardless ŌĆö the real leak channel is a plaintext downgrade, which is forbidden (httpsŌåÆhttp aborts); depth is capped at 3 and each hop is logged. The pinned str0m 0.16.2 carries a known Firefox RTX-SSRC caveat (fix landed upstream in 0.18.0): Firefox playback is hardware-validated before ship, and any fix pickup beyond 0.16.x is a tracked follow-on requiring the MSRV decision ŌĆö the MSRV 1.85 + Edition 2024 requirement begins at str0m 0.17.0 ([ADR-0048](ADR-0048.md)). Downstream doc/UI surfaces must track the new kinds: the capability matrix row, OpenAPI registration, and the SPA output forms + in-app docs ([ADR-W020](ADR-W020.md)).
