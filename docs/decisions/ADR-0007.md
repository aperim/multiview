# ADR-0007: CMAF-first HLS with custom-built Apple LL-HLS

- **Status:** Proposed
- **Area:** Core Engine
- **Date:** 2026-06-02
- **Source brief:** [core-engine.md](../research/core-engine.md)

## Decision

Encode once into CMAF/fMP4 via libav movenc and derive HLS, LL-HLS, and optional DASH from one in-memory store. Reuse the hls-playlist crate for LL-HLS tags; build the CMAF segmenter (sub-second parts on IDR boundaries) and the LL-HLS HTTP server (axum, blocking playlist reload, preload-hint byte-range, chunked part push) in-house.

## Rationale

FFmpeg's hls muxer cannot emit Apple LL-HLS (its -lhls is the legacy EXT-X-PREFETCH DASH variant). Verification refuted 'no Rust LL-HLS crate exists' — hls-playlist provides the tag serialization; only the segmenter and blocking-reload server are genuinely custom. CMAF lets one fMP4 set serve HLS+LL-HLS+DASH, halving packaging.

## Alternatives considered

FFmpeg hls muxer for LL-HLS (rejected: impossible). gohlslib via cgo (rejected: Go dependency, defeats pure-Rust serving). Hand-roll all playlist tags (rejected: hls-playlist already does the tag layer). LL-DASH only (acceptable interim, but misses the LL-HLS goal).

## Consequences

Must engineer the segmenter + blocking-reload HTTP server (HTTP/2, held requests via tokio Notify keyed on msn/part). GOP must align to part cadence for independent parts. Keep a standard HLS rendition for compatibility/DVR; validate with mediastreamvalidator; CDN/proxy must support chunked transfer.
