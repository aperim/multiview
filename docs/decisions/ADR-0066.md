# ADR-0066: The inference seam — a read-only frame-tap + a provider-agnostic `Detector` trait (embedded ONNX vs external model service vs future multimodal API), structurally isolated from the engine

- **Status:** Proposed
- **Area:** Detection / Preview / Engine / HAL / Config / Control
- **Date:** 2026-06-13
- **Source brief:** [object-detection-ai.md](../research/object-detection-ai.md)
- **Builds on:** [ADR-P001](ADR-P001.md) (preview taps are read-only / drop-oldest / Tier-A / shed-first — the pattern this reuses), [preview-subsystem.md](../research/preview-subsystem.md) (the tap registry + isolation guarantee), [efficiency.md](../research/efficiency.md) (decode-at-display-res, bounded queues, admission ladder), [self-aware-placement.md](../research/self-aware-placement.md) (DETECT capability report; inference as a GPU co-tenant).
- **Relates to:** invariant #1 (output clock never gated on detection), invariant #6 (decode-at-display-resolution), invariant #10 (best-effort consumers cannot back-pressure the engine), [ADR-0042](ADR-0042.md) (IPv6-first for the external-service transport).

> **Scope note.** The operator asked for **preparation** — a seam to connect image/video models for object detection *later* — not a shipped detector. This ADR fixes the *seam*: how a detector gets frames, what interface it satisfies, and how it is isolated. It bundles **no** model weights and **no** vendor SDK; the default build compiles with a deterministic stub detector only.

---

## Context

Multiview will need to connect to image and/or video object-detection models (embedded, or running on a separate box, or a future multimodal API). The risk is coupling the protected output core to a fast-moving, fault-prone, resource-hungry subsystem. Committing the engine to one runtime/model would (a) age badly as the ONNX execution-provider / accelerator / multimodal landscape churns, and (b) risk a hot-path coupling that violates invariant #1 or #10.

The codebase already has the exact isolation machinery a detector needs — it was built for preview:

- **Read-only last-good frame slots.** `TileStore::read`/`read_at` over a lock-free `arc_swap::ArcSwap` ring (`crates/multiview-framestore/src/tile.rs:371,421,191`) and the simpler `LatestSlot` (`crates/multiview-framestore/src/latest.rs:33`) — a reader never blocks a producer.
- **A tap registry with lazy-start / refcount / auto-stop / drop-oldest.** `multiview-preview`'s `TapRegistry` (`crates/multiview-preview/src/tap.rs:61`) and the program downscale tap `ProgramTap`/`ProgramFrame` (`crates/multiview-preview/src/whep/program.rs`), reading from bounded drop-oldest rings (`crates/multiview-preview/src/lib.rs:11,22`). Cost is ~zero when nobody subscribes.
- **Per-vendor feature islands** (`cuda`/`vaapi`/`videotoolbox`) that map onto ONNX Runtime execution providers, and the `*sys`-crate precedent for native code behind a feature while consumers stay `forbid(unsafe_code)` ([iso-program-recording.md §D](../research/iso-program-recording.md)).
- A grep confirms **no inference/ONNX/detector seam exists yet** (`rg -i 'onnx|inference|detector' crates/` → 0 hits in code), so this is genuinely net-new and unconstrained by prior art in-tree.

The decision is *what interface to commit to* and *how to wire the frame source* — not which model to ship.

---

## Decision

**1. Detection reads frames through a read-only tap, exactly like preview (invariant #10).** A new `DetectionTap` is the moral twin of `multiview-preview`'s `TapRegistry`: it subscribes (lazy-start on first detection binding, refcounted, auto-stop on last) to an existing last-good slot — a per-input `TileStore`, the program `ProgramTap` downscale ring, or a cued off-air source — and reads via a bounded **drop-oldest** ring. A slow/stalled/crashed/absent detector gets a stale frame or nothing and can **never** hold a ref the producer awaits. No second decode (invariant #6): the detector samples the already-display-res frame and downscales once to model geometry.

**2. The output clock is never gated on detection (invariant #1).** No detection result is read on the output-clock thread or by the compositor. Results flow strictly *outward* (to events + the trigger of [ADR-0067](ADR-0067.md)). `out_pts = f(tick)` is independent of detection latency, success, or failure. The detector *pulls* at its own cadence; it never paces the engine.

**3. One provider-agnostic `Detector` trait; three sanctioned backings.** A new pure-Rust crate **`multiview-detect`** owns `trait Detector` (dyn-dispatchable via a boxed-future method — `fn detect(frame, ctx) -> Pin<Box<dyn Future<Output = Result<Detections, _>> + Send + '_>>`, the `async-trait`-style shape; native `async fn` in a trait is **not** object-safe and the registry stores `dyn Detector`, so the boxed future is required — normalized coordinates), a `DetectorProvider` capability surface (`labels()`/`input_geometry()`/`is_remote()`/`max_fps()`), and a **deterministic stub** detector that is always compiled (the default build + TDD use it). The trait signature carries **no** runtime/tensor types, so a backend swap is a feature flip, not an API change. The three backings, each off-by-default:
   - **(a) Embedded ONNX Runtime** via the `ort` crate (`onnx` feature), execution provider chosen per vendor island; **operator-supplied model weights** (we ship none).
   - **(b) External model service** over a small detector protocol (HTTP/gRPC + frame-push), **IPv6-first** ([ADR-0042](ADR-0042.md)), as a process-isolated worker with timeouts + circuit breaker + supervised backoff — for Coral/Hailo/Triton/Frigate-style detector boxes, spoken to **nominatively** over their published protocol (the ZowieTek-driver posture, [managed-devices.md](../research/managed-devices.md)).
   - **(c) Future multimodal / vision-model API** — an image/short-clip-in, structured-regions-out backing satisfying the same async trait at a slower cadence.

**4. Structural isolation, proven by a chaos gate.** All inference runs in the supervised **Tier-A** task tier behind depth-1–3 drop-oldest hand-offs, admitted against the planner's per-engine budget at **below-program, below-preview** priority, and is the **first** rung shed by the degradation ladder. Isolation strength differs by backing: the **external** service/sidecar is a separate **process** (or remote box), genuinely SIGKILL-able; the **embedded** ONNX session is an in-process thread/pool, supervised by panic-catch + watchdog timeout + bounded memory/concurrency caps — a thread is *not* SIGKILL-able independently of the host, so a hard native segfault/OOM is a residual risk the operator accepts when enabling `onnx` (operators needing hard crash isolation run the external-service backing). A held-out soak test stalls and (for the external worker) SIGKILLs the detector under load and asserts the program output is byte-for-byte unaffected with zero added frame-interval jitter — the same hard gate ADR-P001 mandates for preview.

**5. Capability advertisement + fail-safe.** The build/host advertises which providers are actually available (embedded only if `onnx` compiled and a model loads; service/multimodal only if a URL is configured), the same "report what this build can do" discipline as preview and the capability matrix. A missing/mismatched model or a dead service is a **`HealthWarning`**, never a crash.

## Consequences

**Positive.**
- The engine commits to a *durable* interface (trait + tap + capability surface) that survives every runtime/model/accelerator swap and the eventual multimodal future — exactly the "preparation" asked for.
- Maximum reuse: the tap, the isolation tier, the drop-oldest rings, the admission ladder, and the warning model are all existing preview/placement machinery, not new hot-path code.
- The default build stays pure-Rust, native-dep-free, and model-free; ONNX/service are opt-in features, mirroring `gpl-codecs`/NDI discipline. Licensing stays the operator's choice.

**Negative / costs.**
- A new crate + a real (if small) trait, the `ort` dependency behind the `onnx` feature (native ONNX Runtime + its accelerator providers, `cargo deny`-clean), and a network detector protocol to define and version.
- Program-scope detections need a coordinate mapping back to tiles (open question in the brief §7).
- Operator-supplied weights bring a licensing/governance surface (AGPL-3.0 weights, cloud-egress for multimodal) we must surface but cannot fully control.

**Neutral.**
- The stub detector means the seam is exercised in CI with no model — the interface is real, not aspirational.
- Sharing the program downscale ring between preview and detection is the leaning default (brief §7 Q2), reversible.

## Alternatives considered

1. **Feed the detector inside the compositor for "free" frames.** *Rejected* — couples a best-effort fault to invariant #1, exactly the coupling ADR-P001 forbids for preview. The detector reads the *published* store, off the clock thread.
2. **A bounded blocking queue the producer pushes frames into.** *Rejected* — any full queue back-pressures the protected path. Read-only latest-wins drop-oldest only.
3. **Commit the engine to one runtime/model (e.g. bundle YOLOv8 + ONNX Runtime).** *Rejected* — ages badly, couples the engine to a fast-moving dependency, and (for Ultralytics weights) drags AGPL-3.0 into the default build. A trait + opt-in backings keeps the engine clean and the choice the operator's.
4. **Put the ONNX `unsafe`/FFI directly in `multiview-core` or the engine.** *Rejected* — `core` is native-dep-free and the engine is the protected core. FFI lives behind the `onnx` feature in `multiview-detect` (or a `*sys` sub-crate), `// SAFETY:`-documented, never on the hot path.
5. **A soft priority hint for inference instead of hard admission.** *Rejected* — cannot guarantee inference never starves program/preview under contention. Hard admission + shed-first, below preview.
6. **External service over IPv4-first / `0.0.0.0`.** *Rejected* ([ADR-0042](ADR-0042.md)) — IPv6-first, dual-stack, bracketed literals.

---

## How this honors the invariants

- **#1 (output clock never falters).** No detection result is read on the output-clock thread or by the compositor; the detector pulls at its own cadence and never paces the engine. `out_pts = f(tick)` is independent of detection entirely. A dead detector changes nothing about output.
- **#6 (decode-at-display-resolution).** The detector samples the frame already decoded near display size and downscales **once** to model geometry — no second full-res decode, no per-tile RGBA blow-up.
- **#10 (isolation — no back-pressure).** Read-only `arc_swap` last-good slots + depth-1–3 drop-oldest hand-offs + Tier-A SIGKILL-able workers + hard admission at below-preview priority + shed-first + a held-out chaos/soak gate. The engine never awaits the detector; a slow/crashed detector degrades to "paused, warned," never to "output faltered."
