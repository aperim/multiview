## Context

[ADR-0059](ADR-0059.md) pins the load-bearing audio nucleus the switcher needs: the consumer-owned
`AudioControlHandle` seam, AFV from the state machine, the master/FTB envelope, gain-preserving mute,
the **`AudioReader` multi-cursor view** (one ring, many wait-free cursors), live meters, and
**per-bus audio = a `ProgramBus` per audio-bearing bus over the same decoded stores**. It explicitly
defers, "post-MVP … with no design yet": **per-strip dynamics, ducking, mix-minus buses, and
per-source audio delay** (ADR-0059 Consequences §Deferred). [ADR-R005](ADR-R005.md)/[ADR-M004](ADR-M004.md)
give the discrete-source + per-output mapping substrate; [ADR-0034](ADR-0034.md) gives the
audio-track→bus crosspoints (with BREAKAWAY + the RT-9 equal-power crossfade).

What is missing is the **operator-facing mixer model** that turns "sources reach the program bus"
into a small live audio console: input strips with the full per-channel control set, a set of busses
(Program / aux / monitor / record / language / clean), and a **send matrix** wiring strips→busses —
the structure familiar to ATEM/vMix-style operators, implemented Multiview-style.

## Decision

Adopt the **input-strip + bus + send-matrix** model from [switcher-audio.md](../research/switcher-audio.md)
as the audio mixer layer **on top of** ADR-0059, with no new invariant risk and no new DSP primitives
beyond ADR-0059's.

1. **Input strip** (`AudioInputStrip`, [switcher-audio.md §4](../research/switcher-audio.md)) is the
   operator channel for a source (or selected source channels): source + channel select; gain/trim;
   mute (the ADR-0059 gain-preserving `mute_env`); fader; pan/balance; **per-input delay (ms)**;
   phase invert; Program mode `off | on | afv`; level meter + clip; PFL/AFL; per-bus sends.
   Per-strip EQ/dynamics/gate/de-esser/sidechain are **post-MVP** strip processing (closing the
   ADR-0059 "no design yet" gap with a designed-but-phased path), each a property-testable insert in
   the per-strip mix loop.
2. **Bus** (`AudioBus` with `AudioBusKind ∈ {Program, Aux, Monitor, Pfl, Afl, MixMinus, Ifb, Record,
   Stream, Clean, Language}`) — each audio-bearing bus is an ADR-0059 `ProgramBus` (own sample
   cursor/state **derived from the single master output clock**, route table, master stage, optional
   loudnorm) ticked by its own bake consumer over the **same** stores via `AudioReader` cursors. **No
   bus owns time or may pace output** — every bus's sample budget comes from the same master
   running-time/output tick (inv #1), so all output tracks stay sample-aligned; a bus is a consumer,
   never a clock. **No second decode, no store duplication** (ADR-0059 §5).
3. **Send matrix.** Each strip sends to each bus via `AudioSend { bus_id, enabled, level_db, pan,
   tap ∈ {PreFader, PostFader, PostProcessing, PostMute} }`. The matrix is the **operator-facing
   expression of the [ADR-0034](ADR-0034.md) crosspoints** — a send maps to a crosspoint route; a
   send level/enable change is a Class-1 gain/route apply (ADR-0059 §1/§4). MVP may surface a reduced
   matrix (`Program: off/on/afv + fader`, `Aux A/B: off/on + level`) but the **state model is the
   full matrix** from the start.
4. **Per-strip delay (lip-sync trim).** A bounded per-strip delay line (≈ −250 ms … +2000 ms, 1 ms
   fine) realized by pre-buffering / delaying other paths — **never** by violating the output
   invariant. Negative delay is delay-of-others, not a clock change.
5. **Config & surface.** Additive `[audio]` config (strips / buses / sends) under the switcher config
   blocks ([ADR-M012](ADR-M012.md)); REST under `/api/v1/audio/*` consistent with
   [ADR-W021](ADR-W021.md); meters + strip/bus changes on the [ADR-RT008](ADR-RT008.md) topic,
   conflated per [ADR-RT004](ADR-RT004.md).

## Rationale

- **Reuse, don't rebuild** — every mechanism (consumer-owned seam, `AudioReader`, master stage,
  gain-preserving mute, R128 meters, crossfade) is ADR-0059's; this ADR adds *structure* (strips,
  buses, a matrix), not DSP.
- **The send matrix is the crosspoints** — there is one routing truth ([ADR-0034](ADR-0034.md)); the
  mixer is its operator UI, so a send and a crosspoint can never diverge.
- **Invariant posture by construction** — buses are consumer-side `ProgramBus`es; the engine never
  awaits them; meters/UI are drop-oldest (inv #1/#10), exactly as ADR-0059 established.
- **Integer time everywhere** — send ramps and delays use exact-rational sample budgets
  ([ADR-T003](ADR-T003.md)/[ADR-T015](ADR-T015.md)); only gains/pans are floats.

## Alternatives considered

- **A — Flat "include_in_program_bus" booleans only** (no matrix). Rejected: cannot express aux /
  language / clean / record sends or PFL/AFL; the operator asked for a mixer, not a program toggle.
- **B — A second decode per bus.** Rejected by ADR-0059 §5 (store-per-(source,bus)) — N× publish +
  ring memory; `AudioReader` cursors are the minimal-entropy answer.
- **C — A new "audio mixer" crate/actor.** Rejected: [ADR-0054](ADR-0054.md) pins no-new-crates;
  `multiview-audio` owns the DSP and the bake consumer owns execution.

## Consequences

- **Positive.** Multiview gains a real, routable, meterable small-studio audio console with a bounded,
  designed growth path (more buses = more `ProgramBus`+readers; more sends = more crosspoints); the
  ADR-0059 "no design yet" deferral for dynamics/ducking/delay/mix-minus is closed (this ADR + 0078).
- **Negative / cost.** `multiview-audio` grows per-strip processing inserts + a bus registry + a send
  matrix; each bus adds a bake consumer (bounded, shed-with-consumer). The send matrix's UI is a new
  SPA surface ([ADR-W021](ADR-W021.md)/[ADR-W023](ADR-W023.md)).
- **Risks.** Matrix size × bus count is the efficiency variable — each aux bus is a full mix + encode;
  admission/degradation ([ADR-E007](ADR-E007.md)) sheds aux buses before Program. **The Program bus is
  never shed:** it stays in the protected output path (inv #1) and under overload degrades only by
  simplifying its processing or silence-filling a failed contributor — never by stopping or dropping a
  Program audio frame. Chaos/soak per the ADR-0059 MP-1 pattern (wedge a bus consumer; the engine keeps
  ticking) gates the lane.
- **Deferred (named, scheduled).** Full per-strip dynamics, ducking, and the full N-aux UI are
  post-MVP per [switcher-audio.md §19–§20](../research/switcher-audio.md); MVP ships one Program +
  one monitor + two aux busses.
