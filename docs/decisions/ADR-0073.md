# ADR-0073: GPI/GPO/GPIO/relay ingress — the Raspberry Pi GPIO node: edges, levels, min-pulse, debounce, retrigger-lockout

- **Status:** Proposed
- **Area:** Control plane / Engine / Config
- **Date:** 2026-06-13
- **Source brief:** [broadcast-cues.md](../research/broadcast-cues.md) (§2)
- **Extends:** [ADR-0072](ADR-0072.md) (the cue bus + action vocabulary),
  [managed-devices.md](../research/managed-devices.md) device/driver model (closed driver enum,
  poller/actor isolation), [display-out.md](../research/display-out.md) (the `multiview node`
  display-node tier this mirrors)
- **Relates to:** the BUILT virtual GPI/GPO value machine
  (`crates/multiview-engine/src/tally/gpio.rs`), [ADR-T001](ADR-T001.md)/[ADR-RT004](ADR-RT004.md)
  (invariants #1/#10)

## Context

The operator wants a local physical trigger: a relay/opto closure on a Raspberry Pi running
Multiview "in a GPIO mode" must fire a cue, with the full broadcast trigger model — **rising edge,
falling edge, high level, low level, minimum pulse width, debounce window, retrigger lockout** —
and symmetric **GPO** assertion (Multiview drives a relay/lamp out).

The pure logical layer already exists and is verified in-tree: `crates/multiview-engine/src/tally/
gpio.rs` provides `GpiPoint::sample(raw_high) -> Edge` (`:123`, one-sample edge memory), `Polarity`
(`:28`, `ActiveHigh`/`ActiveLow` for normally-open/normally-closed wiring), `Edge`
(`:54`, `Rising`/`Falling`/`None`), and `GpoPoint::request(active) -> raw_high` (`:174`). The module
docstring already names physical relay/opto inputs and AMWA NMOS IS-07 as the *transports* over this
protocol-agnostic logic. What is missing is (a) the **physical transport** (kernel GPIO → `sample`,
`raw_high()` → pin write), (b) the **debounce/min-pulse/lockout timing layer**, and (c) the **node
identity** in the device model.

External grounding: GPI inputs are opto-isolated, contact-bounce-prone closures (a debounce window
is mandatory); the `rppal` Rust crate accesses Pi GPIO via the `gpiochip` character device and
supports `InputPin::set_async_interrupt` with an **optional debounce** argument and edge reporting
([rppal docs](https://docs.rs/rppal/latest/rppal/gpio/struct.Gpio.html)). **(unverified)** `rppal`
was retired for new features as of 2025-07-01.

## Decision

1. **A `multiview gpio-node` mode is a `displaynode`-family control-plane driver actor**, the
   structural twin of `multiview node` ([display-out.md](../research/display-out.md)): a registry
   entry + a driver actor + projections, in the **closed `#[non_exhaustive]` driver enum** — no
   plugin system, no `dyn Any` ([managed-devices.md](../research/managed-devices.md) §2.3). It owns
   the physical pins and posts normalized GPI cues to its controller, and asserts GPO levels the
   controller requests. The engine only ever sees a `Cue` and a `Command`.

2. **The GPIO HAL is an injectable trait; the hardware backend is feature-gated.** A `gpio`
   off-by-default feature pulls the hardware backend (`rppal` by default, or the kernel
   `libgpiod`/`gpiod` char-device interface — deferred to the implementer, brief §8 Q6); the default
   build compiles + tests the pure value machine against an **in-memory fake pin source**
   (the `tower::oneshot`/socket-free CI pattern, [managed-devices.md](../research/managed-devices.md)
   §2.3). The cue logic never depends on the crate, so `rppal`'s retirement is not load-bearing.

3. **The trigger model is a pure value machine over `GpiPoint`** (brief §2.3), all timing as
   integer monotonic durations (never float seconds, safety rule #6):
   - **Rising/falling edge** — `GpiPoint::sample` reports the `Edge`; a rule matches `Edge(Rising)`
     or `Edge(Falling)`.
   - **High/low level** — a steady post-polarity level (`GpiPoint::is_active()`); a *level* matcher
     fires while held (rate-limited by `min_interval`), distinct from an edge matcher.
   - **Minimum pulse width** — an edge is accepted only if the new level persists for `min_pulse`
     (a pending-confirm timer; glitches shorter than the width are rejected).
   - **Debounce window** — after an accepted edge, raw transitions inside `debounce` are ignored.
     Hardware debounce (`rppal`) is the first line; the value-machine debounce is a second,
     transport-independent line so an IS-07/network GPI (no hardware debounce) behaves identically.
   - **Retrigger lockout** — after a cue fires, the point is locked out for `lockout` (rate-limits
     intentional repeats, distinct from debounce's bounce filtering); the point still tracks level
     for accurate edge detection on unlock.
   The model is property-tested: a generated bounce/pulse/repeat sequence yields a deterministic
   accepted-cue stream.

4. **GPO is the inverse.** A `fire GPO` cue action ([ADR-0072](ADR-0072.md) §6) calls
   `GpoPoint::request(active)` (`gpio.rs:174`) and the node drives the pin honouring polarity,
   optionally pulsed for a configured duration.

5. **The node↔controller channel is IPv6-first** (dual-stack `[::]`, bracketed literals,
   conventions §10), one small cue/ack message per accepted edge; the node **cannot back-pressure
   the engine** — a hung node stalls only its own driver task
   ([managed-devices.md](../research/managed-devices.md) §2's inv-#10 proof).

6. **Config** is additive: GPIO points (`{pin, polarity, debounce, min_pulse, lockout, mode:
   edge|level}`) on a node device, and the cue rules that match them live in the
   [ADR-0072](ADR-0072.md) `cues` block.

## Consequences

**Positive.** The operator's exact trigger semantics land on a tested value machine; the node is a
device-model citizen (enrollment, isolation, reconnect-converge) for free; GPO closes the loop so a
cue can drive a physical relay/lamp. CI stays GPU-free and hardware-free (fake pin source).

**Negative / cost.** A new feature-gated hardware backend with real footguns (gpiochip permissions,
`/dev/gpiomem` access, async-interrupt task lifecycle) and the debounce/min-pulse/lockout timing
layer. The `rppal`-retirement risk is mitigated by the HAL trait but a `libgpiod` backend is more
code if chosen.

**Neutral.** GPIO state is a handful of `Copy` bytes per pin; no allocation per sample; interrupt-
driven (no busy poll).

## Alternatives considered

1. **Poll the pins on a timer.** Rejected — `gpiochip` async interrupts are lower-latency and
   lower-CPU; polling would miss sub-poll-interval pulses.
2. **Do debounce/min-pulse/lockout only in hardware (`rppal` debounce).** Rejected — network/IS-07
   GPIs have no hardware debounce; the value-machine layer gives identical semantics across
   transports and is property-testable.
3. **Make GPIO a plugin.** Rejected — [managed-devices.md](../research/managed-devices.md) §2.3
   mandates a closed driver enum (no plugin loading, no reflection); a new pin source is an enum
   variant + module.
4. **Re-implement the GPI value machine for cues.** Rejected — the BUILT `tally/gpio.rs` value
   machine is exactly the right primitive; cues add the timing layer + transport on top.

## How this honors the invariants

- **#1 (output clock):** the GPIO node and its driver run entirely on control-plane tasks; no pin
  read, debounce timer, or cue post touches the output-clock thread.
- **#10 (isolation):** the node↔controller channel is bounded best-effort; a hung/disconnected node
  stalls only its own driver task and its cues are dropped, never the engine. GPO assertion is a
  fire-and-forget request the engine never awaits.
