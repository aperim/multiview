## Context — what is BUILT, what is HELD

The pop-free switching nucleus **exists and is tested**; what is missing is the
*seam* that lets anything reach it live, plus four envelope stages a production
switcher needs. As-built (every path verified in-tree):

The architectural reason `RouteAudio` is held: the `ProgramBus` is **moved into
and owned by the bake-consumer thread** (`pipeline.rs:1955-1997`) — correctly,
per invariants #1/#10 (audio mix + AAC encode never run on the output-clock
loop) — and, unlike subtitles (which have the wired `SubtitleRouteHandle`
ArcSwap-pending seam the consumer drains at the start of each sample,
`crates/multiview-cli/src/captions.rs:118-155`, threaded in by
`command_drain_with_seams`, `control.rs:574`), **no audio re-point seam exists**.

The switcher ([ADR-0054](ADR-0054.md)) makes all of this load-bearing: a video
take without audio-follow-video (AFV) is not a switcher, fade-to-black (FTB,
[ADR-0056](ADR-0056.md)) must fade audio with it, and an operator panel without
live meters is flying blind.

## Decision

### 1. The audio control seam — the subtitle seam's twin (first slice)

Add an **`AudioControlHandle`**: an `Arc<arc_swap::ArcSwap<Vec<AudioCommand>>>`
RCU slot (bounded drop-oldest backlog, exactly the
`SubtitleRouteHandle`/`MAX_SUBTITLE_REPOINT_BACKLOG` pattern,
`captions.rs:130-155`), created in `drive_streaming` (`pipeline.rs:1779`)
alongside `audio_bus`, threaded into `StreamEgress::spawn` →
`consumer_main` (`pipeline.rs:2768`, `:3015`), and **drained once per
`StreamItem` before `drive_audio_for_item`** (`pipeline.rs:3054`) — the
sample-boundary apply, the audio analogue of the video command drain at the
frame boundary.

- **Command set** (resolved, idempotent, latest-wins-safe):
  `Repoint { point, store, ramp_frames }` (carries the resolved
  `Arc<AudioStore>` reader — see §5), `SetGain { point, linear, ramp_frames }`,
  `SetMute { point, muted }`, `MasterEnvelope { to_linear, ramp_frames }`,
  and the AFV batch (§2). Resolution (StreamRef → store/`RoutePoint`) happens
  **drain-side** via the existing `RouteResolution` maps (`route.rs:165-167`),
  which the pipeline finally populates at build via
  `set_audio_store`/`set_audio_channel` (`route.rs:244`, `:254` — BUILT,
  today test-only). `RouteApplier::apply_audio` keeps its coalescing
  (last-per-target, `route.rs:364-372`) and re-targets its output at the seam
  instead of a `&mut ProgramBus` it can no longer reach.
- **Invariant posture.** Publishing is wait-free (one ArcSwap store; a small
  bounded Vec clone at operator rate — the accepted subtitle-seam precedent);
  the consumer polls without locks; **no lock is ever shared with the
  output-clock loop, and the engine never awaits the consumer** (inv #1/#10
  hold by construction, as they do for subtitles). A command lands on the next
  surviving `StreamItem`; under `DropOnOverload` the slot persists until
  drained, so commands are delayed, never lost, and never pace anything.
- This single seam un-holds `Command::RouteAudio` (`control.rs:933-953`) with
  zero new invariant risk and is the prerequisite for everything below.

### 2. Audio-follow-video (AFV)

AFV is **derived from the same switcher state machine output that drives
video** ([ADR-0054](ADR-0054.md)): when the machine opens a transition window
(start tick `s`, duration `T` frames, [ADR-0055](ADR-0055.md)), it emits the
matching audio batch through the seam in the same drain pass. Per-input mode
(schema placement in [ADR-M012](ADR-M012.md)):

- **`fade_with_transition` (default):** an equal-power A→B crossfade via the
  BUILT `ProgramBus::repoint_crossfade` (`program.rs:217-269`) with

  ```text
  ramp_frames = SampleClock::total_at(s + T) − SampleClock::total_at(s)
  ```

  — an **integer** sample count from exact-rational budgets
  (`cadence.rs:80`), never `T × samples_per_tick` as a float multiply
  (at 30000/1001 fps and 48 kHz a tick is 1601.6 samples; a 12-frame mix is
  19 219 or 19 220 samples depending on phase — `total_at` deltas get this
  exact, [ADR-T015](ADR-T015.md)). Because `mix` advances ramps by each tick's
  actual budget and `tick_to` catches up shed ticks (`program.rs:306-354`),
  the fade **completes on the same tick the video transition completes, even
  across overload gaps**. MVP contour: full-window A→B for every transition
  type; a dip-shaped contour (A down to the switch point, B up after) is a
  post-MVP refinement.
- **`hard_cut`:** a zero-ramp re-point (`ramp_frames == 0` degrades to the
  hard `repoint`, `program.rs:223-227`) at the transition's **switch point**:
  the cut tick for a cut, else the first tick where progress ≥ 1/2 (exact
  rational comparison `2·elapsed ≥ T`, matching the dip `switch_point`
  default of [ADR-0055](ADR-0055.md)). Honestly badged `SwitchTier::SoftStep`
  (not guaranteed pop-free — that is what the operator asked for).
- **Manual (T-bar) transitions** have no pre-known `T`. AFV follows the
  conflated absolute-progress samples ([ADR-0055](ADR-0055.md)/
  [ADR-T015](ADR-T015.md)): each update `u` sets paired strip ramps from the
  current gain to `cos/sin(u·π/2)` targets over roughly one tick's budget —
  constructible today because `GainRamp` has public arbitrary endpoints
  (`mixer.rs:35-45`, `:89-110`); a seam-side policy, not a new DSP primitive.
  Latch at 1.0 completes the re-point; flip-flop (PGM↔PVW swap) needs no
  special audio handling because the route table follows the same machine.

### 3. Master/program gain stage — FTB audio fade + master gain

Add **one bus-level envelope** to `ProgramBus`: a steady `master_gain`
(linear `f64`; dB→linear `10^(dB/20)` at the seam — floats are fine for
*gain*, never for *time*) plus an optional master `GainRamp`, applied as **one
multiply pass over the mixed block inside `ProgramBus::mix`**
(`program.rs:316-354`), per-sample while ramping, scalar when steady — the
exact per-strip pattern `mix_program` already uses (`mixer.rs:330-391`). The
engine-side seam command stays this ADR's `MasterEnvelope`/`SetMute`; the
operator REST surface for it is pinned in [ADR-W021](ADR-W021.md):
`PATCH /api/v1/switcher/audio {master_gain_db, master_mute}`, converted at
the boundary (Hot, [ADR-M012](ADR-M012.md) §9).

- **FTB audio-follow** ([ADR-0056](ADR-0056.md)'s optional flag): master ramp
  to 0 over `total_at(s + ftb_frames) − total_at(s)` samples; release ramps
  back up. Held-at-black still emits exact-budget zeroed blocks — the AAC
  stream never stops (the audio continuity guarantee of `mix` is untouched).
  Gain-preserving by construction: the **master envelope must never be folded
  into `master_gain` at settle** (the per-strip endpoint-settle at
  `mixer.rs:261` is explicitly *not* replicated for the master stage).
- **Loudness-normaliser interaction** (efficiency/correctness review): the
  R128 `LoudnormProcessor` runs downstream in `consumer_main`
  (`pipeline.rs:3063-3066`, [ADR-R006](ADR-R006.md)) and would push makeup
  gain *against* an intentional fade. The seam therefore asserts a **hold** on
  loudnorm adaptation while a master ramp is in flight (a small
  `LoudnormProcessor` extension; the existing −70 LUFS gate already covers
  held-at-black). Documented operator note: with loudnorm enabled, the master
  stage is for *envelopes* (FTB, fades, master mute); steady offsets are, by
  design, re-normalised toward the target.

### 4. Gain-preserving mute; honor `gain_db`/`mute` end-to-end

- Add a per-strip **`muted` flag as a separate multiplicative envelope**
  (its own short anti-click ramp, default 480 frames @ 48 kHz ≈ 10 ms):
  effective gain = `program_gain × crossfade_ramp × mute_env`. Mute **never
  touches `program_gain`** — required because both existing paths are lossy:
  unroute forces the caller to re-supply gain on re-entry
  (`mixer.rs:208-220`) and a ramp-to-zero zeroes the steady gain at settle
  (`mixer.rs:261`). Unmute restores the exact prior gain, click-free.
- `RouteApplier::apply_audio` stops dropping `gain_db`/`mute`
  (`route.rs:367`) and applies them via `SetGain`/`SetMute`; the REST take
  stops hardcoding `gain_db: 0.0, mute: false`
  (`routes/routing.rs:223-236`). Gain/mute changes are **Class-1** per
  [ADR-M004](ADR-M004.md) ("bitrate/gain/mute/labels are hot").
- The pipeline finally **consumes the BUILT `AudioRouting` config** at build
  (`audio.rs:75-167`): initial route gains from `gain_db` (replacing the
  pinned unity at `pipeline.rs:1968-1971`), initial mute state from `mute` —
  and the gain/mute fields of `/api/v1/audio-routing` upgrade from
  `X-Multiview-Apply: restart` to live ([ADR-M012](ADR-M012.md) /
  [ADR-W021](ADR-W021.md) carry the surface change).

### 5. Per-bus audio — and THE data-structure change: a multi-cursor reader view

Per [ADR-0054](ADR-0054.md), buses that need their own encode are ProgramSet
programs; each **audio-bearing bus gets its own `ProgramBus`** (own
`SampleClock`, route table, master stage, optional loudnorm), ticked by its
own bake consumer, over the **same** decoded stores — never a second decode.

The blocker is the store contract: `AudioStore` is single-producer /
**single-consumer** — one `AtomicI64` read cursor that only the one consumer
may advance (`store.rs:34-36`, `:82-83`). A second bus reading the same store
would tear the cursor. Two candidate fixes; **pinned: the multi-cursor reader
view.**

- **Pinned — `AudioReader`:** split cursor from data. The ring window is
  already an immutable `ArcSwap<RingSnapshot>` (`store.rs:81`) and a read is a
  pure function of `(window, cursor)`, so an `AudioReader { store:
  Arc<AudioStore>, cursor: AtomicI64 }` carrying `read`/`seek_to_live_edge`/
  `read_cursor` gives each bus its own wait-free cursor with **zero new
  synchronisation** — every (producer, reader) pair stays SPSC. The store's
  embedded cursor becomes the default reader (back-compat; existing call
  sites unchanged); `ProgramBus` routes become `(RoutePoint, AudioReader)`.
  Cost: one `AtomicI64` per (source, bus). A stalled/shed bus's cursor falls
  behind and reads silence-filled evicted spans on resume — the existing
  honest `read` semantics — without ever perturbing the program bus's cursor
  (inv #10 across buses).
- **Rejected — store-per-(source, bus):** the decode thread must publish
  every block N times (N window swaps + N× ring memory per source — the ring
  is 96 000 frames = 2 s, `crates/multiview-cli/src/audio.rs:47`; at 48 kHz
  stereo f32 that is ~750 KiB *per source per bus* of pure duplication), and
  ingest learns bus topology (a layering violation: adding
  an aux bus would require a decode-thread rendezvous instead of a
  control-seam reader creation).

Bus taxonomy mapping (vocabulary per [ADR-0054](ADR-0054.md)/
[ADR-P007](ADR-P007.md)): **program** = the existing bus + master stage
(post-FTB); the **clean feed reuses the program audio mix unchanged** (clean
is a *video* tap — pre-DSK — and de-facto industry practice carries program
audio on it; no second mix, no second encode); **aux buses** = own
`ProgramBus` + readers (post-MVP, with RT-12); the **preview bus carries no
encoded audio in MVP** (meters only); monitor/PFL buses are out of scope
(roadmap M4).

### 6. Live meters — wire `Event::AudioMeter` (prerequisite for the switcher UI)

Tap the post-loudnorm program block and the per-strip contributions **on the
bake consumer** with the BUILT `Ballistics`/`LoudnessMeter` DSP, conflate via
`meterdata::Conflator` at ~30 Hz (`meterdata.rs:108`, `:206`), and publish
`Event::AudioMeter` on the existing drop-oldest publisher — every piece
exists; only the glue is missing. The event already rides the conflated lane
(excluded from replay: `event.rs:1309`, `topic.rs:122`), satisfying inv #10
and the 10–30 Hz telemetry rule; cadence + connect-time snapshot frames are
[ADR-RT008](ADR-RT008.md)'s (publisher-side 30 Hz conflation). The event's
`track: u32` (`event.rs:111-126`) addresses the program bus as a reserved
index with a published strip→index map (wire shape in
[ADR-RT008](ADR-RT008.md)). This also retires the build-time-timeline meters
for live sources (which today read the −90 dB floor forever). Example client:
`wss://[2001:db8::10]/api/v1/ws` (topic `audio.meters`).

### 7. Resolve the Class-1/Class-2 labelling tension

**The control classifier is authoritative** (inv #11, [ADR-R004](ADR-R004.md)):
a program-bus re-point/crossfade is **Class-1** — on this path the bus is
*always* decode→mix→re-encode when program audio is on, so the crossfade adds
no reset and no new transcode (`crates/multiview-control/src/routing.rs:20`).
`SwitchTier::ClickFree.class() == Class2` (`program.rs:77-82`) describes the
**unbuilt coded-passthrough alternative**, where choosing the decoded
crossfade would *force* a transcode that was not otherwise happening. Pinned
doc fix (Lane D item, no behaviour change): re-document `SwitchTier` —
the apply class is a property of the **path**, not the tier; `class()` is
re-described as the tier's worst-case *cost floor* on a passthrough path, and
its doc points at the control classifier for the live answer. AFV crossfades,
gain, mute, master envelopes, FTB audio: **all Class-1**. (Bus *layout*
changes — count/identity/per-track codec — stay pinned Class-2 per
[ADR-M004](ADR-M004.md).)

### 8. Program audio becomes config-declared

Today program audio exists only behind `multiview run --program-audio`
(`main.rs:163-165`, ignored without `ffmpeg` at `main.rs:511-512`). Decision:
program audio is **declared in config** — enabled by the presence of the
ADR-R005 `[audio]` block, and **default-on when a `switcher` block is
declared** (a switcher without program audio is not a switcher; refusal when
no AAC encoder opens is surfaced as a typed config/capability error, the
`AudioCapability` precedent). `--program-audio` remains as a legacy override.
The byte-identical-when-off guarantee is already test-proven and retained.
Schema placement: [ADR-M012](ADR-M012.md).

## Rationale

- **Reuse, don't rebuild.** The equal-power per-sample crossfade, the
  exact-rational sample clock, the lip-sync-across-drops `tick_to`, and the
  engine apply layer are BUILT and tested — the switcher needs a seam, two
  envelope stages, one flag, a cursor split, and glue. No new DSP.
- **The seam shape is proven.** The subtitle seam (`captions.rs:118-155`) is
  the in-tree demonstration that a consumer-owned resource can take
  frame-coordinated control input wait-free without touching inv #1/#10; the
  audio seam is its literal twin, drained at the same place in the same loop.
- **AFV from the state machine, not from events.** Deriving audio from the
  same per-tick switcher output that drives video gives sample-exact window
  alignment (`total_at` deltas) with one source of truth — any event-driven
  follower (macro, control-plane rule) re-introduces scheduling jitter on a
  bounded bus and cannot be sample-accurate.
- **Integer time everywhere** (inv #3, safety rule #6): durations are frame
  counts; sample budgets are `total_at` deltas; the only floats are gains and
  envelope shapes, which are genuinely continuous quantities.
- **One ring, many cursors** is the minimal-entropy concurrency change: it
  preserves every existing wait-free property, adds no copy, and keeps ingest
  ignorant of bus topology — squarely the efficiency standing review's answer.
- **Honest classification** (inv #11): the operator must see "hot" for an
  operation that is hot on the only built path; keeping a worst-case label on
  the live path would force pointless Class-2 ceremony onto every AFV take.

## Alternatives considered

- **A — Engine-thread audio mutation** (drain mutates the `ProgramBus`
  directly via a shared `Mutex`, or the bus moves onto the engine thread).
  Rejected: a lock shared between the output-clock loop and the bake consumer
  is a lock a slow consumer can hold across a tick (inv #1/#10 violation by
  construction — the exact failure mode the consumer-owned design exists to
  prevent); moving the bus engine-side puts mix + loudnorm + AAC encode on
  the hot loop.
- **B — AFV as a control-plane macro** (a sequencer emits `RouteAudio`
  whenever a video crosspoint changes). Rejected: not sample-accurate — macro
  steps ride the bounded command bus with no tick guarantee, so the audio
  window drifts from the video window under load; two sources of truth for
  one transition. (Macros remain the right tool for *operator-authored*
  sequences — [production-switcher §10.2](../research/production-switcher.md) —
  just not for AFV.)
- **C — Per-bus full decode duplication** (each bus gets its own
  decode→store chain). Rejected: multiplies decoder CPU and store memory per
  bus for zero benefit — the stores are already immutable-snapshot shared and
  only the *cursor* needs to multiply.
- **D — Store-per-(source, bus)** (the other cursor fix). Rejected in §5:
  N× publish work and ring memory, and ingest learns bus topology.
- **E — Mute via `GainRamp` to zero / via unroute.** Rejected in §4: both are
  gain-state-lossy on the verified code (`mixer.rs:208-220`, `:261`), and
  unroute clicks.
- **F — Master fade post-loudnorm in `consumer_main`.** Considered for the
  normaliser-fights-fade problem; rejected in favour of the §3 multiply-in-
  `mix` + loudnorm-hold: it keeps the bus the single owner of all bus math
  (one testable pure unit), and the hold rule solves the interaction at its
  cause instead of relocating the symptom.
- **G — A new "audio engine" crate/actor.** Rejected: [ADR-0054](ADR-0054.md)
  pins no-new-crates; `multiview-audio` already owns the DSP and the
  consumer already owns the execution context.

## Consequences

- **Positive.** Cut/auto/FTB get sample-aligned, genuinely pop-free audio out
  of primitives that already pass tests; mute/gain become live, lossless, and
  Class-1; the switcher UI gets real 30 Hz meters; aux-bus audio has a
  designed, bounded growth path (readers, not copies); `RouteAudio` stops
  being a warn.
- **Negative / cost.** `multiview-audio` grows: `AudioReader` (cursor split),
  per-strip `muted` envelope, master stage, loudnorm hold — each a small,
  property-testable pure extension, but the crate's SPSC story becomes
  SP-multi-reader and must be re-documented (the crate has no `CLAUDE.md`;
  write one in this pass's lane). The seam adds one RCU slot and one
  drain-per-item branch to `consumer_main` (bounded, off-hot-path). Meter DSP
  adds O(sources × block) work per item on the bake consumer — bounded,
  conflated to 30 Hz on the wire, and shed with the consumer under overload
  (never the engine).
- **Risks.** The seam carries `Arc` store/reader handles across threads —
  drop of a retired reader must stay trivial (it is: an atomic + an Arc).
  Transition-under-source-loss: a dying source's store silence-fills, so an
  in-flight crossfade **completes over silence and never stalls**
  ([ADR-R011](ADR-R011.md)); chaos/soak gates per the MP-1 pattern (wedge the
  consumer; the engine keeps ticking) are required for the seam PR.
- **Deferred (named, scheduled — not silent).** Discrete-track carriage to
  sinks (AUD-4b) and per-output `OutputAudio` consumption stay open
  work-schedule items; aux/PFL buses and the dip-shaped AFV contour are
  post-MVP per the brief's phasing; per-strip dynamics (compressor/gate/EQ),
  ducking, mix-minus buses, and per-source audio delay (lip-sync trim) are
  post-MVP mixer items with no design yet; the coded-passthrough `SoftStep`
