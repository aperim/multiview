# ADR-0063: ONVIF server — present qualifying Multiview outputs as ONVIF/RTSP devices, with PTZ passthrough for single-camera layer/transcode outputs

- **Status:** Proposed
- **Area:** Management (Devices domain) / Output / Control plane
- **Date:** 2026-06-13
- **Source brief:** [onvif-and-ptz.md](../research/onvif-and-ptz.md)
- **Extends:** [ADR-0006](ADR-0006.md) (RTSP serving — the transport this facet advertises), [ADR-M008](ADR-M008.md) (Devices domain feature-gating), [ADR-0042](ADR-0042.md) (IPv6-first), [ADR-W017](ADR-W017.md) (route style)
- **Relates to:** [ADR-0062](ADR-0062.md) (the ONVIF client — the inverse direction, shared SOAP/WS-Discovery codec), [ADR-0064](ADR-0064.md) (the PTZ command model the passthrough relays), [decoupled-routing.md](../research/decoupled-routing.md) (the per-stream crosspoint that resolves "which camera is behind this output"), invariants #1/#7/#10

## Context

The operator wants Multiview's **own outputs** to be **discoverable and consumable by third-party
ONVIF/RTSP clients** (NVRs/VMS) where the transport and codec permit — and, where an output is a
**passthrough/transcode of an upstream camera through a layer (not the program/preview)**, to allow
**some ONVIF passthrough, e.g. PTZ**, so the downstream NVR can steer the relayed camera.

Multiview already serves RTSP in-process ([ADR-0006](ADR-0006.md)); the `Output::RtspServer` variant
carries a mount + codec (`crates/multiview-config/src/schema.rs:685`). ONVIF Profile-S streaming **is**
RTSP/RTP H.264 (H.265 in Profile-T), discovered via WS-Discovery (`NetworkVideoTransmitter`) and
described via Media2 `GetProfiles`/`GetStreamUri` (web-verified). So "outputs-as-ONVIF" is **metadata +
control bolted onto an already-served RTSP stream** — not a new media path. The hard part is being
**honest about which outputs qualify** and about **when PTZ is meaningful**.

## Decision

Add an **opt-in ONVIF-server facet** to `multiview-control` that presents **qualifying** Multiview
outputs as ONVIF `NetworkVideoTransmitter` devices, built clean-room from the ONVIF/WS-Discovery
specs (sharing the SOAP/WS-Discovery codec with [ADR-0062](ADR-0062.md)), behind an off-by-default
feature, auth-required.

1. **The qualification gate (hard, typed).** Only an output a **generic ONVIF/RTSP client can actually
   consume** may be advertised: an **`Output::RtspServer` (`schema.rs:685`) carrying H.264 or H.265**.
   The advertised profile **scope is gated by codec**: **H.264** outputs advertise a
   **Profile-S-compatible** streaming scope, **H.265** outputs a **Profile-T-compatible** scope (H.265
   is Profile T, not Profile S) — so a Profile-S-only NVR is never offered an H.265 profile.
   `LlHls`/`Hls` (`:708`/`:734`), `Rtmp` (`:770`), `Srt` (`:798`), and `Ndi` (`:754`) are **not**
   ONVIF transports and are **never** advertised. The facet is enabled **per output** via a new
   additive `onvif_server: Option<OnvifServerConfig>` block (`#[serde]`-defaulted, `#[non_exhaustive]`,
   the established schema-evolution pattern); enabling it on a non-qualifying output is a **typed
   config-validation error at admission** — never a silent no-op (the honesty discipline of
   [ADR-0037](ADR-0037.md) §6 rejecting `ByteExact` on non-byte transports).

2. **The service set (minimal but real).**
   - **WS-Discovery responder** — answer `Probe` for `NetworkVideoTransmitter` on **`ff02::c` first**
     / `239.255.255.250:3702` ([ADR-0042](ADR-0042.md)), advertising the streaming-profile scope.
   - **Device service** — `GetDeviceInformation` (our identity: Multiview + version),
     `GetSystemDateAndTime` (read-only), `GetCapabilities`/`GetServices`,
     `GetNetworkInterfaces`. **`SetSystemDateAndTime` is *not* honoured on the server facet** — a
     third-party NVR setting the Multiview host clock would affect auth, logs, scheduling, and the
     output timeline; an incoming set is rejected (or no-op-acknowledged), and host time is managed
     only via the deliberate privileged control-plane time action, never the ONVIF surface.
   - **Media2 service** — `GetProfiles` exposing **one profile per advertised output** with its real
     codec/resolution/bitrate (mostly read-only), and `GetStreamUri` returning the output's existing
     RTSP URL (`rtsp://[…]:8554<mount>`). The RTSP serve is the **existing** path
     ([ADR-0006](ADR-0006.md)); a third-party client is just another consumer of the existing
     rendition — **encode-once-mux-many preserved (invariant #7), zero new encode**.
   - **PTZ service** — present **iff** the output is a single-camera passthrough (§3); absent otherwise.
   - **Imaging service** — generally absent (no physical imager); thin passthrough only where §3 applies.
   - Auth: WS-Security UsernameToken / HTTP Digest backed by the control-plane auth/secret model
     (`crates/multiview-control/src/auth.rs`). Off by default; feature-gated.

3. **PTZ passthrough = a relay, present only when there is exactly one steerable camera.** When the
   advertised output is a **passthrough/transcode of a single upstream ONVIF/VISCA camera through a
   layer** — its video resolves, at the relevant crosspoint, 1:1 to one camera
   ([decoupled-routing.md](../research/decoupled-routing.md)) — the ONVIF-server PTZ service is real
   but **relays** each incoming move to that camera's driver via the shared `PtzCommand` model
   ([ADR-0064](ADR-0064.md)). For a **multi-tile composite, program, or preview** output, **no PTZ
   node is offered** (no single camera to steer); for a single tile bound to a **non-PTZ** source, no
   PTZ node. The PTZ node is present **iff** exactly one steerable camera sits behind the output —
   honest by construction. The relay is a control-plane SOAP-in → driver-call-out hop; it adds **no
   engine path** and is authorised against control-plane authz (an NVR cannot steer a camera it was
   not granted) and rate-limited per device.

## Consequences

- **Multiview becomes a discoverable ONVIF source** for generic NVRs/VMS over the universal Profile-S
  RTSP/H.264 (Profile-T H.265) baseline — and a relayed PTZ controller for single-camera passthrough
  outputs — at **zero extra encode** (invariant #7) and **zero engine surface**. The claim is
  **best-effort interoperability, gated on hardware validation against ≥2 real NVRs/VMS** (§Consequences
  below), not generic-NVR compatibility asserted up front.
- **Honest scope.** We claim **interoperability with generic ONVIF clients, not ONVIF certification**
  (a paid, hardware-gated logo program); the conformance gap to full Profile-S/T is documented, not
  closed, and gated on hardware validation against ≥2 real NVRs/VMS.
- **The qualification gate prevents a half-feature:** an HLS/RTMP/SRT/NDI output is never falsely
  advertised as ONVIF; enabling the facet on it fails loudly at config time.
- **PTZ-passthrough edges are real** (live crosspoint swap changes which camera is steered) — resolved
  in the brief's open questions (§8) and the routing seam; the conservative default drops the PTZ node
  briefly during a swap rather than steer the wrong camera.
- **A new responder attack surface** — the WS-Discovery responder and SOAP endpoints are
  feature-gated, auth-required (beyond identity scopes), and SOAP-body size-capped; off by default.

## How it honors the invariants

- **#1 (output-clock untouchable):** the ONVIF-server facet is pure control plane. Advertising an
  output and answering SOAP/WS-Discovery never touches the output-clock loop; the served video is the
  **existing** RTSP rendition at existing cost. A third-party client that is slow or floods SOAP
  affects only the control-plane handler, never the engine. PTZ passthrough is a control-plane relay
  with no data-plane hop.
- **#7 (encode-once-mux-many):** an ONVIF/RTSP client is just one more consumer of the already-encoded
  `RtspServer` rendition — **no second encode**.
- **#10 (isolation):** all SOAP handling and the PTZ relay run on control-plane tasks publishing into
  bounded channels; the engine never awaits the facet. A hostile or slow downstream NVR cannot
  back-pressure the engine.

## Alternatives considered

- **Advertise all outputs as ONVIF (best-effort).** Rejected: an ONVIF client cannot consume
  HLS/RTMP/SRT/NDI; advertising them is a silent half-feature. The gate is hard and typed.
- **Build a new ONVIF-specific RTSP server.** Rejected: violates encode-once-mux-many and duplicates
  the [ADR-0006](ADR-0006.md) RTSP path; the facet is metadata+control over the existing serve.
- **Offer PTZ on any output containing a PTZ tile** (composite included, steering "the active tile").
  Rejected: ambiguous and surprising — there is no single camera; PTZ is offered **iff** exactly one
  steerable camera is behind the output. Operators steer composited cameras from the Multiview UI or
  the camera's own endpoint.
- **Re-implement PTZ on the server side (don't relay).** Rejected: Multiview has no imager; PTZ on a
  passthrough output is *definitionally* the upstream camera's PTZ — relaying through that camera's
  driver is the only correct semantics.
- **Pursue ONVIF certification now.** Rejected for this pass: certification is paid and hardware-gated
  (the no-paid-services posture); we ship interoperable wire behaviour and document the gap.
