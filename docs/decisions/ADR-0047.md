# ADR-0047: Clock & timer synthetic sources — dual analogue+digital face, IANA-timezone/label/offset metadata, and countdown/countup `Timer`s (extends ADR-0027; tz via `chrono-tz`)

- **Status:** Proposed
- **Area:** Core engine / Input / Config (synthetic source model)
- **Date:** 2026-06-10
- **Source brief:** [clock-timer-sources.md](../research/clock-timer-sources.md)
- **Extends:** [ADR-0027](ADR-0027.md) (synthetic sources are first-class
  in-process `SourceKind`s)
- **Relates to:** [ADR-0010](ADR-0010.md) (declarative config schema),
  [ADR-T001](ADR-T001.md) / [ADR-R001](ADR-R001.md) (output clock = sole pacer,
  inv #1), [ADR-T003](ADR-T003.md) (integer/exact-rational time, never float fps),
  [ADR-T012](ADR-T012.md) (disciplined reference is a media-clock reference only,
  never a pacer), [ADR-T002](ADR-T002.md) (last-good-frame), [ADR-E007](ADR-E007.md)
  (resource-adaptive degradation, inv #9)

## Context

ADR-0027 made colour **bars**, a **solid** slate, and a full-frame **clock**
first-class, in-process `SourceKind`s on the one uniform
ingest→`TileStore`→compositor path. The shipped clock is minimal: one face at a
time (`analog` **or** `digital`, `ClockFaceConfig` at
`crates/multiview-config/src/schema.rs:197`), a single **fixed** offset
(`tz_offset_minutes`, `schema.rs:238`), no label, no countdown.

The operator has asked for two concrete extensions:

1. **A dual analogue+digital clock as one source**, plus richer per-clock
   metadata — a **location/label**, an **IANA timezone** (e.g. `Australia/Sydney`),
   and a visible **UTC-offset** indicator — so several clocks can sit in a layout,
   each labelled with its zone (a "world-clock wall").
2. **A digital countdown / countup to a target** — count **down** to a target or
   **up** from it, where the target is either a **time-of-day** (optionally
   recurring daily) or a **specific date+time** (absolute, with a timezone), with
   configurable behaviour **at and after** the target, a configurable display
   format, and overrun styling.

The as-built code is well-shaped for both:

- `render_clock` (`crates/multiview-cli/src/synth.rs:150`) already composes the
  **analogue face** (`clock_face(...)` vector primitives,
  `crates/multiview-compositor/src/overlay/subpass.rs:342`) **and** the **digital
  readout** (`TextEngine::rasterize_run` centred, `synth.rs:204`) onto a near-black
  slate, baked NV12/linear (`apply_overlays_to_nv12`). A `dual` face is a
  *placement* of both — **no new renderer**.
- The clock **model** is pure integer arithmetic
  (`crates/multiview-overlay/src/clock.rs`): `WallTime` (whole Unix seconds),
  `LocalTime`, `AnalogHands::for_dial` (12h/24h dial math). The only float is hand
  angles, from one division (`clock.rs:17`).
- `TimeZoneOffset` is a **fixed whole-minute shift** that *explicitly delegates
  DST to "resolved upstream to a concrete offset"* (`clock.rs:57`) — but that
  upstream resolver does not exist yet.

The two open design problems are therefore: **(a)** how to resolve an IANA zone to
a DST-correct offset for the displayed instant (the missing upstream resolver),
and **(b)** how to model the timer's target/direction/at-target/format without
breaking the source model, the integer-time rule (inv #3, safety rule #6), or the
output-clock invariant (inv #1).

## Decision

Extend the synthetic-source model **additively** (`schema_version` stays `1`, as
ADR-0027 set the precedent), reusing the existing renderer and model:

1. **Dual face.** Add `ClockFaceConfig::Dual` (alongside `Analog`/`Digital`) — an
   analogue face with a digital readout beneath it. This is a new *placement*
   path in `render_clock` that calls the existing `clock_face(...)` (with
   `ClockFaceStyle::at`, `subpass.rs:328`) for the upper region and the existing
   centred-text logic for the lower region. **No new `OverlayPrimitive`, no new
   shader.**

2. **Clock metadata.** Add to `SourceKind::Clock`:
   `timezone: Option<String>` (IANA id, preferred over `tz_offset_minutes`),
   `label: Option<String>` (location/title), `show_offset: bool` (draw a
   `UTC±HH:MM` badge for the displayed instant), `show_reference: bool` (draw the
   ADR-T012 reference lock badge — **display only**), and `numerals: bool` (hour
   numerals on the analogue/dual face). All drawn as additional `TextEngine` runs.

3. **IANA timezone → concrete offset via `chrono-tz`.** Add the missing upstream
   resolver: parse `timezone` to a `chrono_tz::Tz`, and **per render** compute the
   resolved offset for the displayed instant
   (`tz.offset_from_utc_datetime(utc).fix()`), feeding the existing model as a
   `TimeZoneOffset::from_minutes(resolved)`. This is DST-correct (Sydney shows
   `UTC+11:00` in January, `UTC+10:00` in July) and the `show_offset` badge
   reflects it. `tz_offset_minutes` is retained as a legacy fixed-offset field;
   if both are set, `timezone` wins (config warning). The resolver is a pure
   helper `resolve_offset(Tz, WallTime) -> TimeZoneOffset` (unit-testable with
   injected instants), reused by the control plane for the form preview.

4. **New `SourceKind::Timer`** (a synthetic source like the others — silent, pure
   pixels, in-process):
   - `target: TimerTarget` — internally tagged on `target` and `#[serde(flatten)]`ed:
     `TimeOfDay { at, timezone|tz_offset_minutes, recur_daily }` or
     `DateTime { at, timezone|tz_offset_minutes }`.
   - `direction: TimerDirection` — `Down` (default) | `Up`.
   - `on_target: TimerOnTarget` — `Hold` (freeze at `00:00:00`, default) |
     `Continue` | `ZeroThenUp` | `Recur` (re-arm; `time_of_day`+`recur_daily` only).
   - `format: TimerFormat` — `DHhMmSs` (default, day field dropped when zero) |
     `HhMmSs` | `MmSs` | `HhMmSsFf` (frames from the canvas `Rational` cadence) |
     `Auto` (drop leading zero units).
   - `label`, `overrun_prefix`, `overrun_badge` (an `OVER`/`ELAPSED` a11y word —
     state read without colour, mirroring `RefStatus`, `clock.rs:277`).
   The displayed duration is **integer** seconds (`target_unix − now_unix`, or the
   reverse); frames come from the sub-second part against the cadence rational —
   never float seconds.

5. **Timing is unchanged and load-bearing.** The **output clock paces**
   (`out_pts=f(tick)`, inv #1); the **displayed time is wall-clock sampled at
   render** and stamped as the generator's monotonic elapsed (`synth.rs:339`). A
   clock/timer source can never pace or stall the engine; a slow bake republishes
   last-good (inv #2). The timer target is an absolute instant; neither it nor
   "now" feeds the encoder PTS (safety rule #6). The disciplined reference
   (ADR-T012) is **display only** when `show_reference` is set — never a pacer.

6. **Efficiency.** A clock/timer re-renders only when its displayed **field**
   changes — `render_key` (`synth.rs:138`) generalises from "second" to a field
   index: once/second for `hh:mm:ss`, once/**frame** only for the opt-in `_ff`
   format. Each bake is a few SDF primitives + short text runs over one tile —
   bounded, and shed cheapest-impact-first by the degradation loop (inv #9).

## Rationale

- **Maximum reuse, minimum surface.** The analogue face, the text engine, the
  linear NV12 bake, and the integer clock model already exist and are tested; this
  ADR adds *placement* + *config* + a *tz resolver*, not a new rendering or timing
  path. The "a source is a source" principle (ADR-0027) is preserved — nothing
  downstream of ingest learns about clocks or timers.
- **IANA over fixed offset is what the operator asked for** and is the only
  correct answer across DST. The model already anticipated an upstream offset
  resolver (`clock.rs:57`); this ADR supplies it without changing the model.
- **Integer time throughout** (inv #3, safety rule #6): durations and offsets are
  `i64`/`chrono` values; frames are an integer division against the cadence
  `Rational`; the only float remains hand angles.
- **Invariants hold:** #1 (never paces), #2 (holds last-good), #5/#8 (NV12/linear
  bake unchanged), #9 (bounded per-field bake), ADR-T012 (reference display-only).

## Decision — timezone dependency: `chrono` + `chrono-tz`

`chrono-tz` is **dual-licensed `MIT OR Apache-2.0`**
([crates.io](https://crates.io/crates/chrono-tz),
[LICENSE](https://docs.rs/crate/chrono-tz/latest/source/LICENSE)) — both already
on the `deny.toml` `[licenses].allow` list (`deny.toml:65`). It **bundles the IANA
tz database at build time** via a `parse-zoneinfo` build script
([lib.rs](https://lib.rs/crates/chrono-tz)); the tz database is **public domain**,
so there is **no GPL** and **no runtime network**. `chrono` is also
`MIT OR Apache-2.0`. The pair is pure-Rust and lands in the **default build** (no
`ffmpeg`, no `gpl-codecs`). Build `chrono-tz` with `default-features = false`
(+ optional `filter-by-regex` to bound binary size). `cargo deny check` stays
green (the whole closure is `MIT`/`Apache-2.0`/`Unicode-3.0`, all allowed); the
implementing PR commits the updated `Cargo.lock`.

## Alternatives considered

- **A — `jiff`** (`MIT OR Apache-2.0`, first-class IANA, reads the *system* tzdb
  on Unix). Rejected for v1 only: reading host `/usr/share/zoneinfo` makes output
  host-dependent and the container may ship without tzdata, breaking ADR-0027's
  "same config, same picture across builds." `chrono-tz`'s *bundled* db is
  reproducible and self-contained. Revisit `jiff` if live tzdata updates without a
  rebuild become operationally required.
- **B — `time` + `tzdb`/`tz-rs`.** Permissive and workable, but `chrono` is the
  ecosystem default and `chrono-tz` is the lowest-friction IANA layer over the
  integer `WallTime` model already in `multiview-overlay`.
- **C — keep fixed-offset only (no new dep).** Rejected: the operator asked for
  IANA zones + offset display, and a fixed offset is an hour wrong for half the
  year wherever DST applies.
- **D — model the dual clock as two stacked single-clock sources.** Rejected:
  conflates layout with content, doubles the generator threads, and cannot share
  one resolved instant/offset between the two faces (they could tick a frame
  apart). One source, one sampled instant, both faces.
- **E — a `dual`/timer field on the existing `Clock` variant instead of a new
  `Timer` variant.** Rejected for the timer: a countdown has a fundamentally
  different field set (target/direction/at-target/format) and overloading `Clock`
  would force a pile of `Option`s and runtime validation. A distinct `Timer`
  variant is cleaner and round-trips unambiguously. `dual` *is* added to the clock
  (it is genuinely a clock face), per Decision 1.
- **F — float seconds for sub-second/frames.** Rejected outright (inv #3, safety
  rule #6 — float fps drifts ~3.6 s/hour). Frames are an integer division against
  the cadence `Rational`.

## Consequences

- **Positive**
  - A single source renders a labelled, DST-correct, multi-zone analogue+digital
    clock; a layout of them is a world-clock wall. A `timer` source gives live
    production a countdown/countup ("ON AIR IN", "we're N over") with no external
    media — pure-Rust, default build.
  - The analogue face, text engine, and NV12/linear bake are reused verbatim; the
    only new drawing is text placement.
  - DST correctness arrives for free across the product wherever a `timezone` id
    is used, via the new pure `resolve_offset` helper.
- **Negative / cost**
  - New `chrono` + `chrono-tz` dependencies (permissive, deny-clean) and a small
    binary-size increase from the bundled tz database (bounded with
    `filter-by-regex` / `default-features = false`).
  - New `SourceKind::Timer` variant + extended `Clock` fields ripple to config
    validation, the WebUI source-kind picker (a new zone dropdown from
    `chrono_tz::TZ_VARIANTS`), the OpenAPI/JSON-schema, and docs/examples.
    Additive — existing configs keep parsing (`schema_version` unchanged).
  - The `HhMmSsFf` timer format bakes at the canvas cadence (one bake/frame); it
    is opt-in and the degradation loop sheds it like any tile.
  - tz-rule changes need a `chrono-tz` bump + rebuild (bundled db). Acceptable;
    `jiff`/system-tzdb is the documented escape hatch if needed.

## Implementation plan (TDD, sliced) — see `SYN-CLOCK-*` / `SYN-TIMER-*` in the work schedule

1. Pure tz resolver + timer math (`resolve_offset`, duration decompose/format,
   next-occurrence, at-target clamp) with golden + property tests; inject all
   instants. No I/O.
2. Config: `ClockFaceConfig::Dual`, the new `Clock` fields, the `Timer` variant +
   `TimerTarget`/`Direction`/`OnTarget`/`Format` enums, validation (unknown IANA
   id ⇒ typed `ConfigError`; `timezone` wins over `tz_offset_minutes` with a
   warning); round-trip TOML/JSON tests; `examples/*.toml` parse.
3. Renderer: `dual` placement + metadata strip in `render_clock`; the timer render
   (centred mono run + overrun badge); content-aware golden tests extending the
   as-built clock goldens (`synth.rs:464`).
4. Generator: `SyntheticKind::{Clock,Timer}` resolved descriptors, per-render tz
   resolution (follows DST), `render_key` → field index; wire into the existing
   `generator_loop`.
5. Surface: WebUI source-kind picker + clock/timer forms + the IANA zone dropdown,
   OpenAPI/JSON-schema regen, docs + a `examples/clock-timer.toml`.
