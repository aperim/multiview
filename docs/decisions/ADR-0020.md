# ADR-0020: Layered timing — monotonic pacing + optional reference-lock + per-input frame-sync

- **Status:** Proposed
- **Area:** Streaming/Timing
- **Date:** 2026-06-03
- **Source brief:** [timing-architecture.md](../research/timing-architecture.md)
- **Builds on:** [ADR-T001](ADR-T001.md) (monotonic output clock = Layer A), [ADR-T003](ADR-T003.md) (per-input PTS normalisation = Layer C input), [ADR-T006](ADR-T006.md) (drift loop = Layer C), [ADR-T002](ADR-T002.md)/[ADR-T008](ADR-T008.md) (frame-sync + A/V); relates to [ADR-0018](ADR-0018.md) (which uses `mosaic-engine/src/ptp.rs` for placement, not timing)

## Decision

Model all Mosaic timing as **five strictly separated layers**, of which **only Layer A paces the output and Layer A is never paced by an input**:

- **Layer A — Pacing clock (unchanged, invariant #1):** one fixed-cadence **monotonic** clock, `out_pts = f(tick)`, exact rational cadence, `CLOCK_MONOTONIC`/`mach_continuous_time`, absolute deadlines. (`mosaic-engine/src/clock.rs`; ADR-T001.)
- **Layer B — Optional reference-lock:** the pacing clock's *rate* (and, at relock points, *phase*) MAY be **disciplined** to a trusted external **reference** — a PTP/**SMPTE ST 2059-2** grandmaster — via the existing `mosaic-engine/src/ptp.rs` `PtpServo` (**slew for small offsets, step only for large discontinuities/grandmaster change**). The output clock origin may optionally be seeded to the **ST 2059-1 SMPTE-Epoch** alignment grid at start/relock. On reference loss the clock enters **holdover** (last frequency estimate on the local oscillator) then **free-run** (undisciplined monotonic). The servo lives *beside* the pacer; it changes only the *rate of tick advance*, never `out_pts = f(tick)` and never which input is sampled. A reference is NOT a source.
- **Layer C — Per-input frame-sync (always on):** each input is a **frame synchroniser** — normalise its PTS (ADR-T003), store last-good, and at each output tick **sample** the frame `nearest_at_or_before(t_N)`, holding last-good on starvation. A source's timestamp is *used* to phase-align it, carry A/V together, and absorb drift by drop/repeat (+ audio soft-resample), **never to pace the output**. When a source is verifiably PTP-locked to the output's grandmaster (ST 2110-10), its RTP timestamp maps onto the **common** internal timeline for true cross-source phase alignment; otherwise its timestamp is treated as untrusted per-source ordering. Either way the source is sampled, never pacing.
- **Layer D — Wall-clock reference (time-of-day; task #30):** the NTP-disciplined (default) or PTP-disciplined (pro) **system** clock, read **for labels only** (HLS `EXT-X-PROGRAM-DATE-TIME`, TOD overlays, logs, NDI timestamps), anchored to the media timeline via a monotonic→UTC map. It can step, so it **never** paces.
- **Layer E — Timecode (label, not clock):** embedded ATC/VITC/LTC for per-tile display; output TC generated from Layer A (free-run / TOD-run sourced from Layer D / jammed to the facility). A per-frame label, never a pacer (`mosaic-overlay/src/timecode.rs`).

**Default mode (commodity, shipping default):** A free-runs; B off; C on with untrusted per-source PTS; D = NTP system clock; E = embedded-or-generated. **Pro mode (opt-in, feature `ptp`):** A disciplined by B; C maps PTP-locked sources to the common timeline; D shares the grandmaster; E jammed to the facility. "Tying the output to timed sources" = locking **both** output and sources to a **common reference (Layer B)**, never slaving the output to a source.

## Rationale

Professional video has separated **reference** (a stable, disciplined, low-jitter clock carrying rate/phase — genlock, then PTP/ST 2059) from **source** (arbitrary, flaky program content) for decades, and never paces a device's output from a source. PTP "carries both absolute epoch time (replacing timecode) and microsecond phase alignment (replacing genlock)" (JEM Productions; B&H eXplora, 2026-06-03). ST 2059-1:2015 fixes the SMPTE Epoch (1970-01-01 TAI = PTP epoch) so any frame boundary is computable from time-of-day modulo the frame period (Artel; SMPTE ST 2059-1). ST 2059-2 mandates 5 s fast-lock / 1 µs inter-slave (Tektronix; SMPTE). ST 2110-10:2022 derives RTP timestamps from the common PTP clock so PTP-locked senders/receivers are frame-aligned, while receivers still realign for path delay (SMPTE ST 2110-10; Riedel). Servos slew for small offsets and step only for large ones (linuxptp `phc2sys(8)`), and reference-locked devices hold over then free-run on loss (TimeTools; Microchip) — with NDI Genlock as the direct software precedent: on lock failure it "fall[s] back to using the system clock and… continue[s] to work reasonably" (docs.ndi.video).

This layering **satisfies the owner's wish to tie the output to timed sources while honouring invariant #1**: the output is disciplined to *trusted infrastructure* (a reference), and sources are frame-synced *onto* the output; sources and output become frame-aligned because both lock to one reference — not because any source paces the output. Invariant #1 (output never stalls/falters) and invariant #10 (isolation) are preserved exactly: the servo only adjusts the cadence *rate* (slew) and degrades to holdover→free-run on reference loss, the picture never stalling or stepping; a dying source still holds-last-good. `mosaic-engine/src/ptp.rs` already documents and tests that "PTP does not pace the output clock." The model also *unifies* the existing timing docs by naming layers the briefs already implied (the "one PTP-locked feed" hybrid and "loosely disciplined to wall-clock" notes).

## Alternatives considered

- **Pace the output from a (master) source's PTS/timecode.** *Rejected — breaks invariant #1.* A source can stall (dead camera freezes the mosaic), burst (time-warp), wrap (33-bit TS ~26.5 h / 32-bit RTP ~13.25 h), or reset; source-level verification (ADR-T001) shows a shared graph is gated by its slowest input. This is exactly the failure invariant #1 exists to prevent, and the opposite of industry practice (a source is never a reference).
- **Always require PTP (no free-run mode).** *Rejected for commodity.* PTP needs a PTP-capable NIC/PHC and an ST 2059 fabric, which commodity Linux/macOS deployments do not have. Forcing it would make the default build unusable. PTP is therefore the opt-in pro mode (feature `ptp`); the default free-runs.
- **Free-run only (never reference-lock).** *Rejected as the sole design — insufficient for IP broadcast.* Free-run cannot frame-align the mosaic to a wider ST 2110 facility or eliminate facility-wide drift; the owner explicitly wants the *option* to tie output to timed sources. Free-run is kept as the **default** and as the **holdover/loss fallback**, with reference-lock layered on top as optional.
- **Wall-clock as the pacing clock.** *Rejected — it can step (NTP/leap/operator), is non-monotonic, and would inject jitter and non-monotonic output PTS* (cf. `use_wallclock_as_timestamps`, streaming-gotchas §2). Wall-clock is Layer D, for labels only.

## Consequences

- **No change to the commodity default.** Layers A/C/D/E are the current design; this ADR *names* them and *adds* B (and the explicit D for task #30) as opt-in/explicit. Existing code (`clock.rs`, `ptp.rs`, `timecode.rs`, the ADR-T003 normaliser) needs no behavioural change to ship the default.
- **Pro mode (feature `ptp`) requires real hardware to validate.** The `ptp.rs` `phc` module is compile-verified only here (no PTP NIC); lock/holdover/phase behaviour MUST be validated on a real ST 2059 fabric before any pro-mode promise. The servo math is unit/property tested.
- **Commodity holdover = TCXO/`CLOCK_MONOTONIC`-grade only.** Do not promise sub-µs phase holdover without dedicated timing hardware (OCXO/rubidium). For never-falter this is sufficient (it is the same clock we already free-run on); for facility-grade phase it is not.
- **A mid-run phase re-jam is a Class-2 (controlled-reset) change**, not hot, because it shifts frame boundaries (cf. ADR-R004 / ADR-M005). Syntonization (rate-only discipline) and origin-seed-at-startup are the hot/clean operations.
- **A common-timeline mapping must be gated.** A source's timestamp is trusted as a *common* reference only when verifiably ST 2110/PTP-locked to the output's grandmaster (SDP/NMOS signal or operator assertion); otherwise it is untrusted per-source ordering. The frame synchroniser is correct in both cases.
- **Layer D needs a monotonic→UTC anchor** (captured at start, re-anchored on large wall-clock steps) so HLS PDT / TOD overlays are smooth and never summed from segment durations.
- **New normative source to track:** SMPTE ST 2059-2:2021 for exact PTP message intervals; the representative log2 values in the brief are deployment-tunable, not fixed.
