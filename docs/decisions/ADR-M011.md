# ADR-M011: Google Cast output driver: protocol stance, sessions, legacy-IPv4 interop

- **Status:** Proposed
- **Area:** Management
- **Date:** 2026-06-10
- **Source brief:** [managed-devices.md](../research/managed-devices.md)

## Decision

Ship Google Cast playback of the program as a **Devices-domain driver `cast`** ([ADR-M008](ADR-M008.md)) behind an off-by-default `cast` feature — **not** a new `Output` enum variant. The media path is an existing HLS rendition already served by `multiview-output` (encode-once-mux-many preserved); a Cast session is pure control plane, so invariants #1/#10 are satisfied by construction: the engine never sees the session and nothing in it can pace or back-pressure the output clock. Multiview acts as a **server-initiated sender** (the proven Home Assistant/pychromecast pattern): discover devices via mDNS `_googlecast._tcp` and connect to the **advertised** port (Cast groups advertise non-8009 ports; manual `address:port` entry is the cross-VLAN fallback), open a TLS socket (default 8009) speaking the CASTV2 framing (4-byte big-endian length prefix + protobuf `CastMessage`; the device presents a self-signed certificate, so the sender uses a permissive verifier by design), `LAUNCH` Google's hosted **Default Media Receiver (app id `CC1AD845`)** — which requires **no developer registration, no public URL, and no cloud account** — then `LOAD` with `contentId` = the rendition URL, `contentType: application/vnd.apple.mpegurl`, `streamType: LIVE`, plus `hlsVideoSegmentFormat: "fmp4"` when the rendition is CMAF (our HLS is CMAF-first per [ADR-0007](ADR-0007.md); receivers assume TS unless told otherwise).

Each connected device gets a **session actor on a dedicated thread**: implementation starts with a spike of `rust_cast` 0.21 (MIT; blocking API fits one-OS-thread-per-session; pinned `protobuf =3.7.2`, rustls 0.23; edition-2024 dependency is acceptable under our pinned toolchain), with the fallback being a hand-rolled ~2k-LOC async channel on `prost` + `tokio-rustls` implemented from the BSD-3-Clause Chromium Open Screen protocol sources (the clean-room-friendly reference). Lifecycle: sender PING every 10 s, session declared dead after 20 s without PONG, reconnect retry every 5 s **re-resolving by mDNS UUID, not IP** (DHCP changes addresses), `GET_STATUS` on reconnect → re-`LAUNCH` + re-`LOAD` if the session is gone, and a supervisor that re-`LOAD`s on `MEDIA_STATUS` IDLE (keyed on `idleReason`). Sessions are **ephemeral by default** (runtime-only, never exported in config-as-code) with a "save as device" promotion into the registry. Prerequisites shipped with the driver: **CORS headers on the HLS endpoints** (the receiver is a browser app on a Google origin fetching playlists/segments cross-origin; currently absent — zero hits in `crates/multiview-output/src` and `crates/multiview-control/src`), a **guaranteed H.264 High@≤4.1 + AAC rendition** for universal device coverage, and media URLs as **IP literals** (or publicly resolvable names) — the devices ignore DHCP-supplied DNS and resolve via hardcoded public resolvers, so `.local` and LAN-DNS names never resolve on the device. Cast is **sync Tier D** ([ADR-M010](ADR-M010.md)): device-buffered latency in seconds with no controllable knobs; a Cast device is never a synchronized-canvas participant — the UI refuses sync-group membership (a group containing one would honestly report tier "none"). Custom/styled receiver apps (one-time Cast Developer Console fee + public valid-CA HTTPS hosting + publication) are **out of scope** as a product default and exist only as a documented operator opt-in. Trademark posture: nominative use only ("works with Google Cast devices"), a "Google Cast and Chromecast are trademarks of Google LLC" attribution line, and **no** "Works with Google Cast" badge or any implied certification.

## Rationale

The Default-Media-Receiver sender path delivers "put the composited program on a TV/smart display" with zero engine contact, zero new encode, no paid services, and no cloud dependency — and it is the same path Home Assistant, pychromecast, catt, and VLC have shipped for a decade without challenge. The wire protocol is proprietary and Google-undocumented, but Google's own open-sourced Open Screen code (BSD-3-Clause) publishes the protobuf schema and message surface, making implementation-from-published-sources the defensible stance rather than reverse engineering. Modelling Cast as a device + session (not an Output) matches its real lifecycle: nothing in the media graph changes when a TV starts or stops watching an existing rendition.

## Alternatives considered

**Register a custom or styled receiver app** (rejected as default: requires Cast Developer Console registration with a one-time fee plus permanently hosted public valid-CA HTTPS receiver pages — paid external services for a feature that needs neither branding nor `disableIdleTimeout` to play a LIVE stream; retained as operator opt-in only). **Cast Streaming / tab-mirroring** (rejected: the real-time RTP mirroring path exists only in Chrome senders and Chromium's C++ `libcast` — a heavy FFI burden — and would re-encode the canvas, breaking encode-once, without escaping the seconds-class buffering). **DIAL** (rejected: app-launch-only, no media namespace, no heartbeat/session control, and deprecated on Cast devices in favour of CASTV2 launch). **A new `Output::Cast` config variant** (rejected: it would churn the exhaustive `Output` matches and pipeline build for something that moves no media; sessions are control-plane state over an existing HLS output). **No Cast support** (rejected: large installed base, cheap bounded scope once the Devices domain and CORS exist, and CORS benefits every browser consumer anyway).

## Consequences

We accept and document **proprietary-protocol drift risk**: Google can change the protocol or receiver behaviour at any time (the receiver's default HLS engine already flipped to Shaka in 2025–2026, materially changing HLS handling), so Cast is a best-effort convenience output, never a guaranteed transport, and hardware validation is firmware-pinned across **≥2 device generations** including a multi-hour LIVE-session soak on a smart-display-class device (these reportedly kill long LIVE sessions early). **LL-HLS does not engage**: the shipped receiver player runs with low-latency mode off and the Default Media Receiver exposes no configuration hook, so Cast consumes a standard-latency rendition at an observed 6–30 s glass-to-glass. The media URL handed to the device is the one deliberate **IPv4/dual-stack legacy-interop carve-out** under conventions §10 / [ADR-0042](ADR-0042.md) — Cast devices are effectively IPv4-only in practice; this is documented as a legacy accommodation, never a counterexample to IPv6-first. There is **no sender authentication** in the protocol (device-auth proves the device to the sender, not vice versa): any LAN client can cast or preempt our session; we observe the receiver-status app change and surface "preempted" rather than fighting for the device. mDNS discovery is link-local, so routed/VLAN'd plants need the manual-address escape hatch. Finally, validation requires acquiring real Cast hardware; until that soak passes, the feature ships clearly labelled best-effort.
