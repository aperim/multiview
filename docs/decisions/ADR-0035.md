# ADR-0035: Self-aware placement — detect capability mismatches, warn actionably, then wire the affinity-first planner into the run

- **Status:** Proposed
- **Area:** Efficiency / HAL / Engine / Control / Web
- **Date:** 2026-06-08

- **Builds on:** [ADR-0018](ADR-0018.md) (adaptive affinity-first GPU work-placement —

## Context

The product promise — *"even commodity hardware should EASILY do a 2×2 4K multiview with
subtitles"* — fails **silently** today. On the GPU test box the wgpu compositor fell back
to the CPU reference because the host had no usable Vulkan adapter (missing Vulkan loader
/ the NVIDIA `graphics` driver capability not granted). NVML reported the RTX 4060 the
whole time; compositing ran on ~5 CPU cores with the GPU ~80% idle, and **there was no
warning anywhere** — not in the UI, not on a health endpoint, not even at `warn` level.

The pieces to fix this mostly exist but are disconnected:

- **SENSE is built.** An off-engine poller (`multiview-cli/src/system_metrics.rs`)
  publishes `Event::SystemMetrics` carrying per-GPU `DeviceLoad`
  (`multiview-hal/src/load.rs`, every field `Option`) and ours-vs-total `SelfShare`, via
  the engine's **drop-oldest** `EnginePublisher::publish_event` (inv #10 holds).
- **DETECT is presence-only.** `EnvProbe` (`multiview-hal/src/probe.rs`) checks device
  nodes; ADR-0003 defers *usability* queries to the backend crates — unbuilt. No
  `CapabilityReport` exists (`rg CapabilityReport crates/` → 0). The compositor retains
  the adapter (`GpuContext::adapter()`) but `adapter().get_info()` is **never called**
  (`rg get_info crates/` → 0) — so a software adapter (llvmpipe, `device_type == Cpu`) is
  accepted as a "GPU" with no warning.
- **No actionable health-warning model exists**, though the transport, severity ladder
  (`AlertSeverity`), dedupe (`Alert.key`), raise/clear coalescing (`Alert.active`),
  topic (`Topic::Alerts`), ingest pattern (`alarm_ingest`), store (`AlarmRepository`),
  and REST list pattern (`routes/alarms.rs`) are all built.
- **PLAN is built and tested but inert.** `select_device()`, `plan_split()`,
  `PlacementController`, `ControlLoop` are referenced only by `pub use` re-exports;
  `rg select_device|PlacementController|ControlLoop crates/multiview-cli/src` → 0. SENSE's
  signal never reaches them.
- **APPLY is the silent fallback.** The composite backend is chosen by one line —
  `RunBackend::select(true)` (`multiview-cli/src/pipeline.rs`) — which catches every
  GPU-init error, logs at **`info`**, returns `Self::Cpu`, and is documented to *"never
  return an error."* `kind()`/`is_gpu()` are read by nothing in run code.

## Decision

Build **one coherent self-aware-placement subsystem — SENSE → DETECT → WARN → PLAN →
APPLY** — on top of the existing (built) SENSE and the existing (built, unwired) ADR-0018
planner, shipped smallest-first.

1. **DETECT — `multiview_hal::capability::CapabilityReport` (the keystone, NEW).** A
   per-`(Stage, DeviceId)` table of records `{stage, backend, device {id, name, vendor},
   present, usable, in_use, reason}`, assembled at **pipeline build time** (off the
   output-clock thread, refreshed on re-plan), aligned with the ADR-M007 shape. Three
   probe sites **record** their outcome instead of swallowing it: composite reads
   `adapter().get_info()` (`usable = adapter present AND device_type != wgpu::DeviceType::Cpu`;
   `device_type` is the discriminator, **not** "exclude `Gl`"; `{VirtualGpu, Other}` warn
   at lower severity, never silently accept); encode/decode do a **throwaway**
   `ctx.open_as`/`*_cuvid` open at target res/format (the real check, beyond
   `find_by_name` presence). The **MISMATCH cross-check** = *(hardware discovered:
   NVML/`DeviceLoad` ≥1 GPU OR `EnvProbe` Present, independently of which feature is
   compiled) AND (resolved backend is software/CPU-tier)*. This is the single DETECT
   output consumed by **both** WARN and PLAN.

2. **WARN — `HealthWarning` as a richer *sibling* of `Alert` (NEW shapes, reused
   machinery).** Reuse `Alert`'s `key` dedupe + `active` coalescing; **add** `code:
   WarningCode` (`#[non_exhaustive]`), `subsystem`, `remediation`, `since`, optional
   `current`/`expected`. New `Event::HealthWarningRaised`/`HealthWarningCleared` variants
   (mirroring `Event::AlertRaised`/`AlertCleared`), routed on `Topic::Alerts`, emitted
   through the **identical drop-oldest publisher** as `SystemMetrics`, and registered in
   the hand-listed `asyncapi.rs` + `openapi_schemas.rs`. Add a **new** read-only
   `GET /api/v1/health` (active warnings + worst severity; `?severity`/`?active`),
   modeled on `routes/alarms.rs::list_alarms` — **not** an extension of `/livez`/`/readyz`
   (ADR-R009: a capability warning must not restart-loop the container). Mirror engine
   warnings into a control-side `WarningRepository` via a `warning_ingest` task copying
   `alarm_ingest`. Capability codes are **latched** (build-time facts); metric codes are
   **dwelled** (raise-after-N / clear-after-M via `degradation::Hysteresis`);
   `degradation-active` is **controller-gated** with Info→Critical escalation at the
   program-affecting rung (`DegradationAction::affects_program()`).

3. **PLAN — wire ADR-0018's `select_device`, unchanged.** Affinity stays a hard gate;
   the device-wide `nvenc_session_count` gate (a co-tenant NVR near the ceiling gates that
   GPU out); `self_*` is the ours-vs-total *attribution* discriminator (UI/scoring), not
   the gate term. `plan_split` remains a gain-gated last resort
   (`NoCandidateFitsWholePipeline | AllOverHeadroomCeiling`).

4. **APPLY — Increment 1 (build-time, pure, synchronous).** Replace the
   `RunBackend::select(true)` line: assemble candidates + `PipelineDemand` from the
   `CapabilityReport` + one `DeviceLoad` snapshot → `select_device(...)` → choose GPU
   compositing **only** when the chosen device is *usable* per the report, else CPU →
   stash the chosen `DeviceId` → **emit a `HealthWarning` on any mismatch and log at
   `warn`**. Keep the CPU-fallback path (inv #1): the fix makes the decision **loud and
   reported**, never makes GPU init fatal.

5. **APPLY — phase 2 (runtime re-plan, off the clock thread).** Decode placement
   (`IngestPlan` + `hwdecode.rs::select_decoder` with fused `-resize` per inv #6) and
   encode placement (`resolve_encoder` NVENC-vs-x264 by report + ceiling, pin GPU) behind
   the same `Selection`. Wire the per-system NVENC ceiling into
   `PlacementPolicy::with_nvenc_ceiling()`. Build the **missing slow control tick**: a
   separate task/thread owning `PlacementController` + `ControlLoop`, reading `DeviceLoad`
   from a `watch` latest-slot fed by the **existing** poller, running `observe()`/`step()`
   at ~1 Hz, publishing a small decision into a `watch` slot. The per-tick
   `FnMut(&mut CompositorDrive)` hook only **reads** that decision and applies O(1),
   non-blocking, frame-boundary-safe changes (ladder/backend), mirroring `command_drain`.
   A `Migrate` is a whole-island Class-2 make-before-break on the supervisor path, never
   on the clock thread.

6. **UI.** `useHealth` + `HealthBanner` (renders nothing when clean) + `HealthPanel`
   (severity glyph **+ text**, mono `code`, callout `remediation`) + a nav text-count
   badge + a `placement.snapshot` event and "What runs where" section joining each
   stage's `{backend, device_id}` to `GpuMetrics.id`. Populate `GpuMetrics.name` and
   `encoder_session_ceiling` (display, orthogonal to the gate's policy ceiling).

## Consequences

**Positive.**
- The exact silent fallback that burned 5 CPU cores becomes a **clear, actionable UI
  message** with a concrete remediation, plus a `warn` log and a `GET /api/v1/health`
  entry. Increment 1 delivers this alone.
- One DETECT artifact (`CapabilityReport`) feeds **both** the operator warning and the
  placement gate — no second source of truth (aligned with ADR-M007).
- The built-but-inert ADR-0018 planner becomes live with a tested target to call into;
  co-tenant headroom (the ours-vs-total signal) finally informs placement.
- No invariant regresses: Increment 1 is build-time only; phase 2's loop is off the
  output-clock thread and emits only through the drop-oldest publisher.

**Negative / cost.**
- New public surface in `multiview-hal` (`CapabilityReport`), `multiview-events`
  (`HealthWarning`, `WarningCode`, two `Event` variants, `PlacementSnapshot`),
  `multiview-control` (`/api/v1/health`, `WarningRepository`, `warning_ingest`), and the
  web app. The new event variants must be added to the hand-listed AsyncAPI/OpenAPI
  registrations or codegen drifts.
- Throwaway encode/decode `open_as` probes add a small, bounded build-time cost (one open
  + drop per HW backend at target res); acceptable because it is the only way to tell
  "registered" from "session-openable" (ADR-0003).
- Phase 2 requires building the **slow control tick** that does not exist today — a prior
  note wrongly assumed the per-tick `run_with_control` hook was off-thread; it runs **on**
  the output-clock loop every tick, so `observe()`/`poll()`/`select_device` must never be
  called from it. Runtime affinity-preservation is *latent* (unit-proven) until that loop
  + a chaos/soak test land.

**Risks + mitigations.**
- *False positives.* Mitigated structurally: a capability warning needs *both* a
  discovered GPU *and* a software-tier resolution; a GPU-free or intentional
  software-only host trips neither half. The first conjunct ORs in `EnvProbe` presence
  independently of which feature is compiled, so an asymmetric build on a GPU box still
  trips. Capability codes are latched so they cannot flap.
- *False negatives.* `device_type != Cpu` (plus the `driver` string for
  llvmpipe/softpipe/swiftshader) avoids excluding a legitimate GL-only real GPU.
- *Back-pressure (inv #10).* `warning_ingest` swallows-and-skips; `Lagged` resubscribes
  at head; the phase-2 `watch` load-slot is read with `borrow()`/`try_recv`, never
  `.changed().await` on the clock loop. A CI chaos/soak test asserts flooding warnings +
  placement snapshots while subscribers stall delays no output tick.
- *Affinity / no-fragment.* Preserved by `select_device`'s whole-island gate; phase-2 adds
  a chaos test asserting no proposal fragments a live island and a pinned/in-flight
  pipeline never migrates.

## Alternatives considered

1. **Make GPU init fatal (crash if no usable adapter).** Rejected — violates inv #1
   (output must proceed on commodity / degraded hosts). The product must run on CPU and
   *report*, not refuse to start.
2. **Surface the mismatch via `/readyz` / `/livez`.** Rejected per ADR-R009 — a
   capability warning would flip liveness and restart-loop the container. Health warnings
   get a dedicated read-only `/api/v1/health`.
3. **A brand-new parallel warning envelope unrelated to `Alert`.** Rejected — `Alert`
   already provides dedupe + coalescing + topic + drop-oldest transport + AsyncAPI
   plumbing; a richer *sibling* (new type + new event variants, **not** mutating `Alert`)
   reuses all of it at lower risk. Mutating `Alert` in place is a breaking wire change
   (live producers + AsyncAPI), so it is explicitly out.
4. **Drive runtime re-plan from the per-tick `run_with_control` hook.** Rejected — that
   hook runs on the output-clock loop; `observe()`/`poll()`/`select_device` there risks
   inv #1/#10. The planner gets its own slow control tick off-thread; the hook only reads
   the latest decision.
5. **Gate placement on `self_encoder_sessions` (ours-only).** Rejected — the NVENC
   hardware ceiling is shared system-wide, so the gate correctly uses the **device-wide**
   `nvenc_session_count`; `self_*` is for attribution/UI ("is the pressure ours or the
   NVR's"), not gating.
6. **Presence-only DETECT (keep `EnvProbe` as the sole signal).** Rejected — presence
   ≠ usability (ADR-0003); the verified bug had an NVML-present GPU with an *unusable*
   compositor. DETECT must cross-check intent against *actual* usability.
