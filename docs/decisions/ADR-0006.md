# ADR-0006: RTSP serving via in-process gst-rtsp-server, with MediaMTX as optional sidecar

- **Status:** Proposed
- **Area:** Core Engine
- **Date:** 2026-06-02
- **Source brief:** [core-engine.md](../research/core-engine.md)

## Decision

Serve RTSP in-process via gst-rtsp-server (gstreamer-rtsp-server crate), feeding pre-encoded NAL units through appsrc ! h264parse ! rtph264pay (h265 equivalent) with no re-encode and a shared live factory. Offer MediaMTX as an optional sidecar for unified RTSP/HLS/SRT/WebRTC/RTMP fan-out.

## Rationale

There is no production-grade pure-Rust embeddable RTSP server (retina is client-only; rtsp-rs is a hobby project; xiu is standalone). gst-rtsp-server is mature, LGPL-2.1 (dynamic-link friendly), and rtph264pay accepts encoded H.264 directly so engine output is packetized without transcode. MediaMTX covers multi-protocol egress with zero custom code.

## Alternatives considered

Pure-Rust RTSP on rtsp-types (rejected for v1: months of work re-implementing RTSP/RTP/RTCP/SDP). retina/rtsp-rs/xiu as embedded server (rejected: client-only / not production / not embed-friendly). MediaMTX-only (viable but adds a publish hop and a Go runtime).

## Consequences

Pulls the GStreamer/GLib C stack and a GLib main loop (run on its own thread bridged to Tokio); conflicts with a lean static single binary. Must prototype exact appsrc caps (stream-format/alignment/SPS-PPS) against real NVENC/VideoToolbox output. For minimal-dependency builds, fall back to MediaMTX sidecar or native-Rust RTSP.
