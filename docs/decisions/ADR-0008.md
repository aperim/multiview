# ADR-0008: NDI first-class but feature-gated, dynamically loaded, with attribution

- **Status:** Proposed
- **Area:** Core Engine
- **Date:** 2026-06-02
- **Source brief:** [core-engine.md](../research/core-engine.md)

## Decision

Integrate NDI via grafton-ndi behind an ndi Cargo feature, provide a NDIlib_v6_load() dynamic-load backend so the default build needs no SDK, bundle the unmodified NDI runtime (or direct users to NDIRedistV5), and enforce mandatory ndi.video link + trademark attribution. NDI Advanced (HX) is a separate paid feature. Additionally, NDI I/O is gated at RUNTIME behind an explicit, audited operator license acceptance (a `[system.ndi] accept_license` setting / one-time UI consent): no NDI source or output starts until accepted.

## Rationale

grafton-ndi links the NDI dylib and bindgens headers at build time (panics without SDK), and the NDI SDK is proprietary/royalty-free with redistribution/branding obligations and no static-vendoring rights. Feature-gating + dynamic loading keeps the default open-source build free of proprietary code and obligations while delivering first-class NDI when present.

## Alternatives considered

Bundle NDI in the default image/binary (rejected: triggers full EULA/branding/version-currency obligations on all users). Statically vendor the SDK (rejected: license-prohibited). VideoLAN libndi (rejected for output: receive-only, not production-ready). GStreamer gst-plugin-ndi (possible alternative plumbing).

## Consequences

CI/Docker for the ndi feature must fetch the SDK; runtime needs the resolvable dylib (or the dynamic-load path). NDI frames are host-memory: always one host<->GPU copy at the NDI boundary. Must carry attribution in UI/About/docs and avoid 'NDI' in the product name without approval. NDI stays inert until the operator confirms NDI license acceptance at runtime (audited who/when); unaccepted NDI I/O is refused with a clear status and never started.
