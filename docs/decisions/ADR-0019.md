# ADR-0019: Native multi-form caption ingest — unified cue model decoded in-demux into a per-tile sampled cue store

- **Status:** Proposed
- **Area:** Core Engine (Caption / Subtitle ingest)
- **Date:** 2026-06-03
- **Source brief:** [io/captions.md](../io/captions.md), [resilience-and-av.md](../research/resilience-and-av.md), [streaming-gotchas.md §5, §7](../research/streaming-gotchas.md)
- **Realizes / governs:** [ADR-R007](ADR-R007.md) (burn-in + passthrough), [ADR-R008](ADR-R008.md) (overlay stack), [ADR-T003](ADR-T003.md) (timestamp normalisation)

## Decision

Mosaic ingests captions **natively from the source stream**, in **all** the forms they arrive, and
treats a caption stream as a **`CaptionTrack` discovered like an audio track** — not as an external
sidecar (the sidecar stays as **one** of the forms). Six input forms are supported, each mapped to the
libav decoder this FFmpeg 7.1 build already links (libzvbi + libass + dvbsub):

1. **DVB teletext** (broadcast TS PID) → `libzvbi_teletextdec` (page-selectable + auto) → **text** cue.
2. **DVB subtitle** (broadcast TS PID) → `dvbsub` → **bitmap** cue (premultiplied RGBA + rect).
3. **CEA-608/708** (H.264/MPEG-2 video SEI / user-data — **not a PID**, surfaced as
   `AV_FRAME_DATA_A53_CC` side data / `lavfi subcc`) → `cc_dec` → **text** cue.
4. **HLS WebVTT rendition** (resolve the master playlist's `#EXT-X-MEDIA:TYPE=SUBTITLES` group, then
   decode that rendition's own WebVTT segments) → `webvtt` → **text** cue.
5. **Sidecar SRT/WebVTT file** (the existing `--subtitles` path) → pure-Rust parse → **text** cue.
6. **(Bonus) mov_text/tx3g · ASS/SSA** → `mov_text` / `ass` (+ libass) → **text** (or bitmap via libass).

All forms decode to **one unified cue model**: `CaptionCue = Text { start_ns, end_ns, lines, region? }`
**or** `Bitmap { start_ns, end_ns, rgba, rect }` (a `#[non_exhaustive]`, adjacently-tagged enum; never
`untagged`), extending the existing pure `mosaic_overlay::subtitle::Cue`. Caption discovery (TS PMT walk
for teletext/DVB-sub; A53 side-data sniff for embedded CC; HLS master `SUBTITLES` group for WebVTT;
moov box for mov_text; sidecar file) and a per-source **selector** (`auto | off | teletext_page N |
track id | embedded_cc field/service | sidecar path`, owned by the Source) live in **`mosaic-input`**.
The raw libav decode FFI lives in **`mosaic-ffmpeg`** behind the existing **`ffmpeg`** feature (the
decoders are already linked — **no new `cargo deny`-relevant dependency**); burn-in uses the existing
**`mosaic-overlay`** renderer behind `overlay` (styled ASS behind `libass`).

Stream captions are decoded **in the same demux pass** that already feeds the tile's video — **one
open, no second pass, no sidecar staging** (the only exception is the HLS WebVTT rendition, which is a
separate media playlist by HLS design and needs an isolated second reader for that track). Cue PTS is
normalised/unwrapped/rebased onto the source's nanosecond timeline (invariant #3), then **sampled** by
the compositor at `out_pts` from a per-tile **last-good cue store** that runs **off** the output-clock
hot path. A stalled, absent, wrong-page, or flooding caption stream degrades to *"no cue on this tile
right now"* and can **never** stall the output clock or back-pressure the engine (invariants #1, #10).
Caption decoders are instantiated **only when a tile actually shows captions** (selector ≠ `off` and the
tile is on-canvas); the cue store is bounded (drop-oldest, freed on expiry).

## Rationale

The owner's requirement is captions *from the source*, every way they arrive — broadcast TS (teletext +
DVB-sub), embedded 608/708, and HLS WebVTT renditions — so the design must discover and decode each at
its real point of origin, not assume an out-of-band file. Modelling a caption as a `CaptionTrack`
discovered like an audio track gives a uniform selector and a clean read-only discovery surface for the
UI, and matches how these streams genuinely present: teletext/DVB-sub are subtitle PIDs in the PMT,
608/708 are A53 side data on *video* frames (so they cost ~nothing extra during the video decode that
already runs), and HLS captions are a `SUBTITLES`-group rendition reachable only from the **master**
(opening the default video variant misses them — a verified footgun). Collapsing all forms to one
two-shape cue model keeps everything downstream format-agnostic and reuses the already-committed pure
`Cue`/`CueTrack` and the linear-light premultiplied-alpha overlay pipeline. Decoding in the existing
demux honours **invariant #6** (one open per source, no wasteful second pass) and the engine's
isolation guarantees: caption work sits on input threads with a non-blocking sampled store, so the
intermittent/absent/wrong-page nature of real broadcast captions is handled by design rather than
risking an output stall. Gating only the FFI decoders (already-linked libraries) keeps the default
build pure-Rust, GPU-free, and `cargo deny`-clean, and keeps the cue-store state machine, selector
schema, SRT/VTT parser, and HLS master-playlist parser property/golden-vector testable on synthesised
fixtures with no live broadcast.

## Alternatives considered

- **Sidecar-only / sidecar-primary** — rejected: the owner explicitly wants captions decoded from the
  source stream as the primary path; sidecars remain one supported form (#5).
- **A second demux/decode pass for stream captions** — rejected for TS/embedded forms: it doubles
  `av_read_frame` work and risks timeline skew between the caption pass and the video pass; in-demux
  decode keeps one timeline and one open. (The HLS WebVTT rendition is the unavoidable exception, since
  it is a distinct media playlist.)
- **One materialised text type for everything** — rejected: DVB-sub is inherently a pre-rendered
  bitmap; forcing it to text loses fidelity. The two-shape cue model keeps both first-class.
- **Caption decode on the output/compositor thread** — rejected outright: would couple an intermittent,
  possibly-stalling decoder to the output clock, violating invariants #1 and #10.
- **`untagged` serde enums for cue/selector** — rejected per conventions §9 (ambiguous across TOML/JSON);
  adjacently/internally-tagged only.

## Consequences

- One open per source for stream captions; HLS adds a single isolated reader for the WebVTT rendition.
- The compositor's caption read is a non-blocking sample of a bounded per-tile store; "no cue", "wrong
  page", and "track absent" are graceful, expected states, optionally surfaced by the existing
  caption-presence probe ([resilience-and-av](../research/resilience-and-av.md)) without affecting output.
- `mosaic-ffmpeg` gains feature-gated wrappers for `libzvbi_teletextdec`, `dvbsub`, `cc_dec`, `webvtt`,
  `mov_text`, and `ass`; `mosaic-input` gains caption discovery + the per-tile cue store + selector;
  the unified cue type extends `mosaic_overlay::subtitle::Cue`. No new crate is added.
- Open-ended cues (608 roll-up without an explicit clear) must be bounded by the next cue or a max
  on-screen duration so a stale caption never sticks — a known correctness detail to test.
- Decode-only-when-shown means caption cost scales with the number of caption-bearing **tiles**, not
  sources, preserving the efficiency budget.
- Discrete caption **passthrough** to outputs remains governed by [ADR-R007](ADR-R007.md); this ADR is
  the ingest side.
