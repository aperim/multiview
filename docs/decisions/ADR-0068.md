# ADR-0068: Configurable motion / scene-change detection — algorithms, zones/masks, sensitivity/debounce; feeds record + cues

- **Status:** Proposed
- **Area:** Detection / Engine / Events / Config / Output / Control / Web
- **Date:** 2026-06-13
- **Source brief:** [motion-scene-detection.md](../research/motion-scene-detection.md)
- **Builds on:** [ADR-MV001](ADR-MV001.md) + [broadcast-multiviewer-features.md §4](../research/broadcast-multiviewer-features.md) (the **existing** content-aware probe engine — `LumaView`, `DetectionZone`, `BlackProbe`/`FreezeProbe`, the dwell/hysteresis `AlarmStateMachine` — that this ADR generalizes), [ADR-0066](ADR-0066.md) (the read-only detection frame-tap this reuses), [ADR-0067](ADR-0067.md) (the detection-event + record-trigger pattern this mirrors), [ADR-0037](ADR-0037.md) + [iso-program-recording.md](../research/iso-program-recording.md) (the recorder arm/disarm the trigger reuses), [ADR-R004](ADR-R004.md) (Class-1 hot live-apply, invariant #11).
- **Relates to:** invariant #1 (output clock never gated on detection), invariant #5/#6 (luma-only, decode-at-display-res), invariant #10 (best-effort consumers cannot back-pressure the engine), [broadcast-cues.md](../research/broadcast-cues.md) (the GPIO/cue debounce + retrigger-lockout vocabulary this mirrors and the cue engine this can feed), [efficiency.md](../research/efficiency.md) (admission ladder, bounded queues).

> **Scope note.** The operator asked for **finely configurable motion / scene-change detection** and **record-on-motion**. Multiview already ships the load-bearing kernel and lifecycle for this: the [ADR-MV001](ADR-MV001.md) probe engine computes `LumaView::changed_fraction` (frame-differencing / SAD) and runs **freeze** detection — *motion below a threshold* — through a per-zone, dwell-debounced X.733 `AlarmStateMachine`. This ADR does **not** re-build that. It fixes the *delta*: add **motion** (changed-fraction above a threshold) and **scene-change/cut** (histogram/SSD) probes as siblings of black/freeze, add a **retrigger-lockout** field that mirrors the GPIO-cue semantics, and make **record-on-motion** a thin rule arming the existing recorder. No new subsystem; additive config/event/engine variants only.

---

## Context

The existing probe engine ([ADR-MV001](ADR-MV001.md)) already inspects decoded essence per tile and raises debounced X.733 alarms for **black** and **freeze**. The relevant code is all in-tree and tested:

- `LumaView::changed_fraction(prev, zone, tolerance)` (`crates/multiview-engine/src/probe/luma.rs:286`) — the fraction of luma samples that changed by more than a per-sample tolerance over a normalized zone. This **is** frame-differencing / SAD; FFmpeg's own `freezedetect` uses the same kind of "mean average absolute difference vs noise floor" statistic (*to verify against the FFmpeg filter docs before implementation*).
- `FreezeProbe::detect` (`crates/multiview-engine/src/probe/freeze.rs:103`) — raises when that fraction is **at or below** a threshold.
- `DetectionZone` (config `crates/multiview-config/src/probe.rs:30`; runtime `crates/multiview-engine/src/probe/luma.rs:67`) — the normalized detection zone / active-picture window.
- `Dwell { up_ms, down_ms }` + `AlarmStateMachine::observe` (`crates/multiview-config/src/probe.rs:113`, `crates/multiview-engine/src/alarm/state.rs:321`) — dwell-up (min-duration) / dwell-down (debounce) hysteresis as a pure function of an injected `MediaTime`, with latch + ack + a non-monotonic-clock guard.
- `AlarmKind` + `Event::AlarmRaised/Updated/Cleared(AlarmTransition)` (`crates/multiview-core/src/alarm.rs:83`, `crates/multiview-events/src/event.rs:1201`) — the `#[non_exhaustive]` taxonomy and the realtime lane.

**Motion is freeze with the comparison flipped** (changed-fraction *at or above* a threshold) — the same primitive under separately configured thresholds/dwell, not a strict logical inverse — so it should reuse `changed_fraction`, `DetectionZone`, `Dwell`, and `AlarmStateMachine` rather than fork them. A **scene-change / cut** is a *single-frame discontinuity* (not sustained motion) needing a different statistic (histogram distance / SSD — the in-house analogue of FFmpeg `scdet`, whose `threshold` is a percent-of-max-change in `[0,100]`, default 10, and `select='gt(scene,T)'`, whose `scene` is a normalized `[0,1]` scene-change **score** — not a calibrated probability — *to verify against the FFmpeg filter docs before implementation*) and an *edge/pulse* lifecycle with a refractory window.

The operator's two asks map to: (1) configurability — zones/masks, sensitivity, min-duration, debounce, **retrigger-lockout**; (2) **record-on-motion** — a rule arming the *already-designed* ISO/Program recorder ([ADR-0037](ADR-0037.md)), exactly as the AI record-trigger ([ADR-0067](ADR-0067.md)) does for named objects. The decision is *how to add motion/scene without breaking the probe engine's isolation or the output clock*, not how to build a detector from scratch.

---

## Decision

**1. Add `MotionProbe` + `SceneChangeProbe` as siblings of black/freeze, reusing the kernel.** `MotionProbe` calls the *same* `LumaView::changed_fraction` and reports `condition_present` when the changed fraction (or an added cheap `mean_abs_diff`/MAFD scalar) is **at or above** a `motion_threshold` — the same kernel as `FreezeProbe` with the opposite threshold comparison (under separately configured thresholds/dwell). `SceneChangeProbe` computes a per-frame **scene score** in `[0,1]` (normalized histogram distance or SSD over the downscaled luma plane) and fires on a single-frame crossing of `scene_threshold`. New `AlarmKind::Motion` and `AlarmKind::SceneChange` variants (additive, `#[non_exhaustive]`), new `ProbeKind::{Motion, SceneChange}` config variants (internally tagged, never `untagged`), and a `MotionAlgorithm #[non_exhaustive] = FrameDiff | BackgroundSubtraction(future)` / `SceneMetric = Histogram | Ssd` enum. v1 ships `FrameDiff` only.

**2. Fine configurability reuses `DetectionZone` + `Dwell`; adds exactly one new field.** Per probe: a normalized **zone** (`DetectionZone`, rectangle in v1; polygon/bitmap mask noted as a future `#[serde(tag="kind")]` variant), a **sensitivity/threshold** (`motion_threshold`/`scene_threshold`, `u16` per-mille to match the existing freeze threshold), a **min-duration** (`Dwell::up_ms`), a **debounce** (`Dwell::down_ms`), and a **retrigger-lockout** (`retrigger_lockout_ms: Option<u32>`, additive on `Probe`). The lockout is a refractory window *after a fire* during which the probe will not re-fire — its name, units (ms), and semantics are **deliberately identical to the GPIO-cue debounce + retrigger-lockout** in [broadcast-cues.md](../research/broadcast-cues.md), so operators learn one model. It is implemented as **one optional field + two guard clauses** extending the existing total `AlarmStateMachine` transition function (a `locked_until` instant that blocks re-entry to `Pending`/re-raise), fully property-testable with the existing injected clock — not a new subsystem.

**3. Detection shares the read-only frame-tap (ADR-0066), luma-only.** Motion/scene read frames the same non-blocking way preview and AI detection do: the per-tile `TileStore` last-good slot (the existing probe path) or the shared `DetectionTap` downscale ring for program/off-air scope ([ADR-0066](ADR-0066.md); `crates/multiview-preview/src/tap.rs:61`, drop-oldest ring `crates/multiview-preview/src/lib.rs:11`). One tap, N subscribers — the cheap motion probe and the expensive AI detector **share the same ring**, and motion **motion-gates** the AI detector (run object detection only on moving regions). Analysis runs on the **downscaled luma plane only** (`plane[0]`, zero-copy `LumaView`) — never RGBA (invariant #5), never a second decode (invariant #6) — with one reusable previous-frame buffer per source (pooled, never per-frame).

**4. Results are advisory; record-on-motion arms the existing recorder.** Motion/black/freeze ride the existing X.733 alarm lane; scene-change is a dedicated `Event::SceneChange` *pulse*. Both are conflated drop-oldest on the realtime stream, never polled, the engine never awaiting a subscriber. A `MotionRecordRule` (additive config) arms an **ISO** or **Program** recorder ([ADR-0037](ADR-0037.md)) via the *same* `Arm{Iso,Program}Recording` command (`202` + op-id, shed-to-`503`) the AI trigger uses, with `pre_roll` + `min_duration` + `hold`/`cooldown` + the `retrigger_lockout` as the inter-clip gap. The rule is evaluated against the detector's `AlarmStateMachine` transitions / scene-cut edges **before** realtime conflation (so a busy scene never drops a trigger on the best-effort lane), and the `202`/`503` shape is REST vocabulary reused for the internal command-bus enqueue: idempotency is keyed **deterministically** off `{rule_id, scope, detection_epoch}` (the dwell-confirmed fire window), not a client-supplied `Idempotency-Key`, so a sustained detection arms once per fire window rather than once per tick. Arm/disarm is **Class-1 hot** ([ADR-R004](ADR-R004.md), invariant #11). The same events are an advisory automation source the cue engine ([broadcast-cues.md](../research/broadcast-cues.md)) may consume; driving *program* (auto-director) is explicitly out of scope.

**5. Admitted + shed like any analyser, and cheap by construction.** Per-source fps cap (default ≤ a few fps); on the degradation ladder detection is shed **first**, above preview/program ([efficiency.md §3.5](../research/efficiency.md), ADR-MV001 *"individually disableable to protect the output-clock invariant"*). Frame-differencing on a 320×180 luma zone is a few tens of thousands of integer ops per sampled frame: negligible CPU, **zero GPU**, **zero extra decode**, bounded memory.

## Consequences

**Positive.**
- Maximum reuse: the differencing kernel, the zone model, the dwell/hysteresis state machine, the alarm/event lane, the northbound delivery (SNMP/syslog/webhook), the frame-tap, and the recorder are all **existing** machinery — the net-new surface is two probe structs, two `AlarmKind`/`ProbeKind` variants, a scene-score statistic, one state-machine field, and a record rule.
- The operator gets *finely configurable* motion/scene with cue-grade debounce + retrigger-lockout, and *record-on-motion* with pre-roll/hold, without any new hot-path code or any risk to invariants #1/#10.
- Motion detection **pays for itself** by motion-gating the expensive AI detector — the biggest GPU/VRAM saving in the detection stack.
- Default build stays pure-Rust, native-dep-free, GPU-free: motion/scene are CPU luma passes; background-subtraction is a future opt-in.

**Negative / costs.**
- Two new probe kinds + a scene-score statistic + a `mean_abs_diff` kernel + the `retrigger_lockout` extension to `AlarmStateMachine` (each property-tested; the state-machine change touches a load-bearing total function and needs a careful diff).
- Scene-change's edge/pulse lifecycle is a slightly different shape from the level-based alarms (resolved by the extended state machine, but a real design choice — brief §6 Q1).
- Background subtraction (MOG2-style), polygon masks, and program-scope→tile coordinate mapping are deferred surfaces operators may want.

**Neutral.**
- "Motion" may read as *activity*, not *fault* — surfacing it as an activity event vs an X.733 alarm is an events-model nuance (brief §6 Q2), not an architectural change.
- Per-mille `u16` thresholds match the existing freeze config; a float is the reversible alternative if 0.1 % steps are too coarse.

## Alternatives considered

1. **A brand-new motion/scene subsystem separate from the probe engine.** *Rejected* — duplicates `changed_fraction`, `DetectionZone`, and the dwell/hysteresis machine that already exist and are tested. Motion is freeze inverted; it belongs in the same engine.
2. **Shell out to FFmpeg `scdet` / `select=scene` / `freezedetect` on the data plane.** *Rejected* — couples a filtergraph to the protected pipeline (the house rule: compositing/analysis is ours, not an FFmpeg filtergraph dependency) and would re-decode/re-process frames. We compute the equivalent score in-house over the already-sampled luma plane.
3. **Model the cut as a level alarm with zero dwell, no lockout.** *Rejected* — a dissolve/montage emits a burst of cuts. The refractory **retrigger-lockout** is required, and mirroring the GPIO-cue lockout keeps the operator model consistent.
4. **Ship MOG2 background subtraction in v1.** *Rejected for v1* — materially heavier (per-pixel Gaussian-mixture state), false-fires avoided by frame-diff for the common "did something move" case, and it would drag a heavier analyser into the default build. Gated behind an off-by-default `MotionAlgorithm` variant; built only if frame-diff's limits bite.
5. **A new event type + new recording path for record-on-motion.** *Rejected* — the recorder ([ADR-0037](ADR-0037.md)) and its arm/disarm command + the detection record-trigger pattern ([ADR-0067](ADR-0067.md)) already exist. Record-on-motion is the *same* command with a motion match; reusing it gives pre-roll, retention, disk-pressure gating, and isolation for free.
6. **Read motion results on the compositor/clock thread to drive layout.** *Rejected* — couples a best-effort signal to invariant #1. Results flow strictly outward; any program action goes through the command bus at a frame boundary.

---

## How this honors the invariants

- **#1 (output clock never falters).** No motion/scene result is read on the output-clock thread or by the compositor; probes are *sampled* analysers over already-published last-good frames (the existing probe contract, `probe/mod.rs:10`). `out_pts = f(tick)` is independent of detection latency, success, or failure. A stalled or crashed detector changes nothing about output. The record trigger never calls the engine on the clock thread — it enqueues a command the existing drain applies at a frame boundary.
- **#5 / #6 (luma-only, decode-at-display-res).** Detection runs on the **first NV12 plane (Y)** of the already-decoded, display-res frame — read through its stride/pitch metadata, not assumed tightly packed (hardware frames are often row-padded) — downscaled **once** to a tiny analysis plane — zero-copy `LumaView`, never RGBA, never a second full-res decode.
- **#10 (isolation — no back-pressure).** Frames arrive via read-only `ArcSwap` last-good slots / depth-1–3 drop-oldest tap rings (shared with preview/AI, [ADR-0066](ADR-0066.md)); events ride conflated drop-oldest lanes; the engine never awaits a subscriber. A busy scene cannot back-pressure — conflation collapses it. The recorder the trigger arms has its own bounded drop-oldest ring + disk-pressure gating ([ADR-0037 §E](../research/iso-program-recording.md)), so even a storm of triggers cannot stall output or grow RAM. Detection is admitted at below-program/below-preview priority and is shed **first**.
