# ADR-0081: Templating / data-binding — window-variable injection API + optional template language; n-source graphics (wipe / PIP+scorecard)

- **Status:** Proposed
- **Area:** Compositor / Overlays / Control (motion-graphics data + composition)
- **Date:** 2026-06-13
- **Source brief:** [motion-graphics-html.md](../research/motion-graphics-html.md)
- **Builds on:** [ADR-0080](ADR-0080.md) (the graphics engine — producer/sampler split, the surface contract)
- **Extends:** [ADR-0056](ADR-0056.md) (keyer multi-box composition — N source slots reuse it), [ADR-0058](ADR-0058.md) (NV12+A fill payload), [ADR-0059](ADR-0059.md) (the bounded RCU control-seam shape the data path reuses)
- **Relates to:** invariant #10 / [ADR-RT004](ADR-RT004.md) (the data seam never back-pressures the engine), invariant #5 / [ADR-E002](ADR-E002.md) (program video stays out of the page), [ADR-0055](ADR-0055.md) (transition engine drives a web wipe's progress), [ADR-M012](ADR-M012.md) (desired-state resource + cycle check + apply class), [ADR-R004](ADR-R004.md) (Class-1/Class-2), [ADR-W021](ADR-W021.md) (control surface), [ADR-0010](ADR-0010.md) (internally-tagged config unions), [ADR-0042](ADR-0042.md) (IPv6-first asset/data fetch)

> **Vocabulary.** *Injection API* = Multiview sets named `window` variables in the page and calls an author-implemented update hook. *Template language* = an author convenience (Mustache/Handlebars class — Mustache is the strictly logic-less one; Handlebars adds restricted built-in helpers) that renders DOM from a data record. *N-source graphic* = a graphic that declares named source slots Multiview binds to buses/sources, used as a wipe or a PIP+scorecard. All generic industry terms; no vendor model is copied.

## Context

The operator asked **which templating approach is best** — *define "window" variables the app sets (names in lower-thirds, live scorecards), or handlebars?* — and required that a graphic **support N source definitions** (use as a wipe `Source1 → Source2`; use as a PIP with a scorecard). [ADR-0080](ADR-0080.md) establishes the graphic as an isolated producer publishing a premultiplied surface; this ADR pins **how live data reaches it** and **how it samples N sources**.

Two distinct jobs exist: **live per-frame data binding** (a changing name, a ticking score) and **authoring-time structure** (build the DOM from a record). They have different cadences and different correctness needs. Verified externally (2026-06): **Mustache** is the strictly logic-less template language; **Handlebars** is a **superset of Mustache** that adds a restricted set of built-in helpers (`if`/`unless`/`each`/`with`) and can register arbitrary custom helpers (so it is *not* logic-less), and precompiles for runtime cheapness. Neither built-in helper set can express animation/easing/canvas — those are CSS/JS regardless.

For N-source composition the repo already has the answer: [ADR-0056](ADR-0056.md) §5 defines a **multi-box composition** — "a background plus overlapping, z-ordered, sized/cropped boxes, usable as one source on any bus, rendered as a drive-internal pre-pass in the same tick," with a recursive PROGRAM tally and a config cycle check. An N-source HTML graphic is that pattern with web-authored geometry/chrome.

## Decision

### 1. Primary live-data contract: the `window`-injection API

Multiview injects a stable namespace and calls a hook the author implements:

- `window.mvData` — the validated JSON data record (re-set on every update).
- `window.mvConfig` — canvas size + exact-rational fps + tier.
- `window.mvUpdate(data)` — author-implemented; called after every injection.
- Animation verbs the switcher can call on a take: `window.mvPlay()` / `mvNext()` / `mvStop()` (in/out animations, frame-coordinated where the tier is on-clock).

This is the **lowest-level, always-available contract**: no template engine required, engine-agnostic (CEF or Servo, [ADR-0080](ADR-0080.md)), and it maps directly onto the bounded data seam (§3). It is the recommended primary path.

### 2. Per-graphic manifest (the desired-state resource)

Each bundle declares a manifest validated like every other config ([ADR-M012](ADR-M012.md), internally-tagged unions per [ADR-0010](ADR-0010.md)): `id`, `entry` (the HTML file), `canvas` (native authoring w×h), `fps` (exact rational, never float), an **optional JSON `data` schema** (validated before injection), a `[sources]` block (§4), and an **optional `[template]` block** (§5). Probed bundle facts are read-only; declared fields are author intent — the [media-playout §3](../research/media-playout.md) probed/declared split.

### 3. The data seam (invariant #10)

Live data rides the **one bounded apply path** everything else uses: REST verb → a `Command` on the bounded bus (202 + operation id, [conventions §6](../architecture/conventions.md)) → the frame-boundary drain → a **bounded, conflated-latest-wins RCU slot** the graphic's producer reads between BeginFrames — the `AudioControlHandle`/subtitle-seam shape ([ADR-0059](ADR-0059.md) §1). The engine never awaits the page; a slow page sees the **latest** data on its next paint, intermediate updates conflated. Data is validated against the manifest schema **before** injection, so a malformed update is a typed control-plane rejection (RFC 9457 `problem+json`), never a mid-air page crash. Data updates are **Class-1 hot** ([ADR-M012](ADR-M012.md)) — a store write + a JS call, no engine reconfiguration.

### 4. N-source graphics — reuse the [ADR-0056](ADR-0056.md) multi-box composition

The manifest `[sources]` block declares named slots (`source1`, `source2`, …) with roles (`fill`/`background`/`inset`). At use time the operator/switcher **binds each slot to a bus or source** (PGM/PVW/camera/media player) — the same bus-selection the switcher does for keyer fills. Two delivery shapes:

- **Engine-side composition (the production live path, recommended):** the graphic's DOM declares **placeholder boxes** with stable ids/CSS rects; Multiview composites the bound source textures **into those rects in our compositor**, on-clock, as the [ADR-0056](ADR-0056.md) §5 multi-box pre-pass. The graphic supplies geometry/animation/masks; the engine supplies pixels. **Program video never enters the page** — no per-frame video upload, no NV12→RGBA→NV12 round trip (inv #5 honored).
  - **The extraction contract (bounded, deterministic — not arbitrary layout readback).** The producer publishes, alongside each surface, a **bounded per-frame manifest**: each placeholder rect's transform, a static/baked **alpha-mask reference**, and scalar **transition progress**. Multiview evaluates *that manifest* on the output clock — it never reads back arbitrary browser CSS-mask/layout per tick into compositor primitives. **v1 is intentionally narrow**: rectangular placeholder transforms + a baked/static alpha mask + scalar progress. If the manifest/surface for tick `k` is late the engine samples the **last-good** pair (inv #2/#10), so a per-frame web-authored *look* can be one tick stale; frame-critical web-authored masks/geometry go to render-to-VT ([ADR-0082](ADR-0082.md)). Arbitrary per-tick CSS-mask → primitive extraction is **out of scope for v1** (brief §7.9).
- **In-page sampling (monitoring tier or render-to-VT only):** the bound texture is delivered to the page as a drawable frame. Heavier (per-frame upload into the browser); **not** the live production path.

**Wipe (`Source1 → Source2`):** two `fill` slots bound to PGM/PVW, the reveal mask/geometry authored in CSS/JS (animated clip-path, logo-shaped reveal), with **progress driven by the transition engine as `f(tick)`** ([ADR-0055](ADR-0055.md)). The graphic provides the *look*; the engine provides the *timing*.

**PIP + scorecard:** one `inset` slot (the PIP feed, engine-composited into the inset rect) plus a DOM scorecard data-bound through §1/§3. The whole graphic is one bus-selectable source, **recursively PROGRAM-tallied** (sources inside it contribute to program, [ADR-0056](ADR-0056.md) §5). Config adds the **cycle check** (a slot cannot bind to a bus that contains the graphic, [ADR-M012](ADR-M012.md)).

### 5. Optional template language: Handlebars (recommended sugar, not mandated)

The manifest may declare `[template] engine = "handlebars"`. When present, the author may `{{handlebars}}`-render DOM fragments **from the same injected `window.mvData`** instead of writing imperative `mvUpdate` code. **Handlebars** is recommended (Mustache superset; restricted built-in `if`/`each`/`unless`/`with` only; precompiled for runtime cheapness). Handlebars is *not* logic-less and can register arbitrary custom helpers — those are **banned** unless separately sandboxed and reviewed. It is **optional and reversible**: injection-only is the default; the template engine is an author convenience layered on the same data, never a requirement, and cannot express animation/canvas (always CSS/JS).

### 6. Recommendation (do not mandate)

Ship the **`window`-injection API as the primary, always-available live-data contract**; offer **Handlebars as opt-in authoring sugar** declared per graphic. The choice is per-graphic, reversible, defaulted to injection-only. MVP may ship **injection-only** and add Handlebars when authors demand it (brief §7.5).

## Consequences

**Positive.**
- The operator's question is answered with a recommendation *and* options: a clean, engine-agnostic injection contract for everyone, with Handlebars sugar for declarative authors — neither forced.
- N-source graphics fall out of the existing multi-box composition; no second compositor, no program video in the page (inv #5), recursive tally + cycle check inherited.
- Wipes and PIP+scorecard are one mechanism (named slots + the transition engine for timing), not bespoke features.

**Negative / cost.**
- A JSON data schema per graphic + validation is new control-plane surface; the conflated data seam adds one RCU slot and a drain branch ([ADR-0059](ADR-0059.md) precedent, bounded, off-hot-path).
- The optional Handlebars dependency (if shipped) needs a `cargo deny` pass; it runs inside the already-heavy browser, so it adds little marginal weight but is still strictly opt-in.
- Authoring two delivery shapes (engine-side vs in-page sampling) is a real distinction docs must keep clear.

**Neutral / deferred.**
- Whether to ship Handlebars in v1 at all (injection-only may suffice — brief §7.5).
- A "bake the chrome, inject live numbers over the played-back clip" middle path couples a media player + a thin live graphic — deferred ([ADR-0082](ADR-0082.md) §5.4 / brief §7.6).

## Alternatives considered

- **Mandate a template engine (Handlebars/Mustache as the only data path).** Rejected: restricted-helper templates cannot express animation/easing/canvas; forcing one constrains authors and adds a dependency graphics without tables do not need. Injection is the contract; templates are sugar.
- **A heavy logic-ful framework (React/Vue runtime) as the binding layer.** Rejected: unnecessary weight inside an already-heavy browser; re-implements what the DOM + `mvUpdate` already do.
- **Deliver program video into the page per frame for N-source (in-page sampling as the live path).** Rejected for production: a per-frame video upload into the browser + an NV12→RGBA→NV12 round trip violates NV12-throughout (inv #5) and burns bandwidth; engine-side composition into placeholder rects keeps video out of the page. In-page sampling is monitoring/baked-only.
- **A new N-source composition model distinct from [ADR-0056](ADR-0056.md).** Rejected: the multi-box composition already *is* "a background plus z-ordered boxes as one bus source"; a second model would duplicate schema, validation, tally, and the render path. We extend it to accept a web chrome layer + web-authored geometry.
- **Event-stream data push (the page subscribes to a WebSocket itself).** Rejected as the primary: a page opening its own socket bypasses the bounded conflated seam and the validation gate, re-introducing an unbounded, unvalidated path into a graphic; data flows through Multiview's seam, validated, conflated.

## How this honors the invariants

- **#1 (output clock):** data injection and source binding never pace the engine; the transition engine drives wipe progress as `f(tick)`; the graphic is still sampled, never awaited ([ADR-0080](ADR-0080.md)).
- **#10 (isolation):** the data seam is bounded, conflated latest-wins, drop-oldest ([ADR-0059](ADR-0059.md) shape); the page reads the latest snapshot between BeginFrames; the engine never blocks on a data update or a source bind. Program video is composited engine-side, so a slow page cannot stall the video path.
