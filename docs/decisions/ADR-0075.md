# ADR-0075: HLS/DASH manifest cues — EXT-X-DATERANGE / CUE-OUT/IN, DASH inband emsg + MPD events

- **Status:** Proposed
- **Area:** Input / Output / Control plane
- **Date:** 2026-06-13
- **Source brief:** [broadcast-cues.md](../research/broadcast-cues.md) (§4)
- **Extends:** [ADR-0072](ADR-0072.md) (the cue bus + action vocabulary),
  [ADR-0074](ADR-0074.md) (the SCTE-35 bytes a manifest cue carries)
- **Relates to:** [ADR-0037](ADR-0037.md)/ADR-0032 (the HLS/LL-HLS segmenter the emit path extends),
  [ADR-T001](ADR-T001.md)/[ADR-RT004](ADR-RT004.md) (invariants #1/#10)

## Context

The operator wants HLS/DASH **manifest cues**: react to `EXT-X-DATERANGE` / `EXT-X-CUE-OUT` /
`EXT-X-CUE-IN` (HLS) and DASH inband/MPD events on ingested streams, and emit them into Multiview's
own output packagers.

External grounding (to verify against the specs before implementation): per
[RFC 8216](https://datatracker.ietf.org/doc/html/rfc8216) and
[AWS Elemental](https://docs.aws.amazon.com/elemental-live/latest/ug/scte-35-ad-marker-ext-x-daterange.html)
/ [Flussonic markers](https://flussonic.com/doc/iptv/markers/), RFC 8216 §4.3.2.7.1 maps SCTE-35
into HLS via `EXT-X-DATERANGE` with `SCTE35-OUT`/`SCTE35-IN`/`SCTE35-CMD` attributes (hex-encoded
raw SCTE-35 bytes + `ID`/`START-DATE`, UTC-anchored); per the RFC the break-out tag carries
`PLANNED-DURATION` (expected length) while the actual `DURATION`/`END-DATE` belongs to the matching
`SCTE35-IN` and must not appear on the out tag when the true end is unknown. The older
`EXT-X-CUE-OUT:DURATION=…` / `EXT-X-CUE-IN` (playlist-position-anchored) are **de-facto/vendor tags,
not RFC 8216 tags**; both forms are widely deployed with differing downstream support. DASH carries cues as inband **`emsg`** boxes (signalled by an MPD
`InbandEventStream`) and **MPD `EventStream`** elements, with SCTE-35 binary in the message data
([dash.js event handling](https://dashif.org/dash.js/pages/usage/event-handling.html)).

In-tree: Multiview ingests HLS and parses playlists (`crates/multiview-input/src/hls.rs`) and parses
MPDs (`crates/multiview-input/src/dash/xml.rs`); the SCTE-35 bytes in a `SCTE35-OUT`/`emsg` decode
through the BUILT `splice35.rs` parser ([ADR-0074](ADR-0074.md)). The output HLS/LL-HLS segmenter
exists ([ADR-0037](ADR-0037.md)/ADR-0032 lineage). What is missing: the cue tag read on ingest and
the cue write on the output packagers.

## Decision

1. **React on ingest.** The cue decoder reads `EXT-X-DATERANGE`/`CUE-OUT`/`CUE-IN` during HLS
   manifest refresh (`hls.rs`) and `emsg`/MPD `EventStream` during DASH refresh (`dash/xml.rs`),
   emitting a `Cue` ([ADR-0072](ADR-0072.md)): a `SCTE35-OUT` hex decodes through the existing
   `splice35.rs` parser → the same `CueEvent`; a bare `CUE-OUT` yields a `SpliceOut` with the carried
   `DURATION`. Reacting is pure observation of an already-sampled input (inv #1/#10), identical
   posture to SCTE-35 ingest ([ADR-0074](ADR-0074.md) §3).

2. **Emit into the output manifests.** An `emit HLS manifest cue` / `emit DASH event` action
   ([ADR-0072](ADR-0072.md)) writes the cue into Multiview's packagers (`multiview-output`), on the
   packager's own task (off the output clock):
   - **HLS** — insert `EXT-X-DATERANGE` (carrying the hex of the emitted SCTE-35 from
     [ADR-0074](ADR-0074.md) §4) **and/or** `EXT-X-CUE-OUT`/`-IN` (operator-selectable form) at the
     current segment boundary.
   - **DASH** — emit an inband `emsg` box on the segment and/or an MPD `EventStream` entry. To
     verify before implementation: the concrete `schemeIdUri` (the SCTE-35 binary scheme
     `urn:scte:scte35:2013:bin`, with the XML-cue scheme `urn:scte:scte35:2014:xml+bin` as the
     alternate form), the `emsg` payload encoding (raw `splice_info_section` bytes), the event
     `timescale`/`presentation_time` derivation from the segment timeline, and the inband-vs-MPD
     placement, are pinned at implementation against the DASH-IF SCTE-35 carriage guidance.

3. **One logical cue, every transport.** The emitted SCTE-35 bytes (TS PID, [ADR-0074](ADR-0074.md)),
   the HLS `DATERANGE`, and the DASH `emsg` are **one cue rendered into every active transport** —
   the multi-transport analogue of encode-once-mux-many: build the splice once, render it per
   packager, so an ad-break is signalled consistently to a TS affiliate, an HLS OTT feed, and a DASH
   OTT feed from one cue.

4. **Bad cues warn + recover, never falter.** A malformed/oversized cue tag from a bad upstream HLS
   source is warned + skipped (the "manage the log + recover" posture of
   [iso-program-recording.md](../research/iso-program-recording.md)); it must not stall the sibling
   renditions (the ingest-all-streams-isolated principle). An emit-write failure degrades that one
   rendition's cue, never the program (inv #1).

5. **HLS form default (open question Q3).** Default to the **RFC 8216-compliant `EXT-X-DATERANGE`**
   form (`SCTE35-OUT`/`SCTE35-IN`, with planned-vs-actual duration per §4.1); the de-facto/vendor
   `EXT-X-CUE-OUT`/`-IN` tags are **optional, operator-selectable per rendition/profile** for
   downstream consumers that require them. This stays an explicit open question (brief §8 Q3): the
   default is RFC-clean, legacy tags are opt-in rather than emitted-both unconditionally.

## Consequences

**Positive.** Manifest cues react and emit on the existing HLS/DASH parsers and the existing
segmenter; combined with [ADR-0074](ADR-0074.md), one cue is signalled consistently across TS/HLS/
DASH. Reacting reuses the SCTE-35 parser verbatim.

**Negative / cost.** A cue-tag reader on ingest and a cue-writer on each packager (HLS playlist +
DASH `emsg`/MPD); the dual-form HLS emit (DATERANGE + CUE-OUT/IN) and its selectability. DASH `emsg`
box authoring is net-new.

**Neutral.** Reacting costs one tag-scan per manifest refresh / one `emsg` parse per segment, on the
existing ingest task; emitting adds a few bytes per cue to a manifest/segment already written.

## Alternatives considered

1. **Emit `EXT-X-CUE-OUT`/`-IN` by default.** Rejected — `CUE-OUT`/`CUE-IN` are de-facto/vendor
   tags, not RFC 8216; the RFC-compliant `EXT-X-DATERANGE` is the correct default, with the legacy
   tags available as an opt-in per rendition/profile for consumers that need them (brief §8 Q3).
2. **React to manifest cues by converting to the engine on a fast path.** Rejected — react is pure
   observation; the cue enters the standard dispatcher, no privileged path (inv #10).
3. **Write manifest cues on the output-clock thread for tighter timing.** Rejected — the packager
   owns its own task; a manifest write must never be on the output clock (inv #1). Cue placement is
   at the segment boundary, which the packager already controls.
4. **DASH MPD `EventStream` only (no inband `emsg`).** Rejected — inband `emsg` reaches players that
   do not re-fetch the MPD; both are emitted for coverage.

## How this honors the invariants

- **#1 (output clock):** ingest cue-reads are on the input task; emit cue-writes are on the packager
  task at segment boundaries — never the output-clock thread. A failing emit degrades one
  rendition's cue, never the program.
- **#10 (isolation):** reacting is observation of an already-sampled input; the emitted cue rides the
  packager's existing bounded write path; the engine never awaits a manifest consumer. A bad upstream
  cue tag warns + recovers without touching sibling renditions.
