# ADR-0013: Deadline-driven compositor with per-tile FrameSync and continuous drift correction

- **Status:** Proposed
- **Area:** Core Engine
- **Date:** 2026-06-02
- **Source brief:** [core-engine.md](../research/core-engine.md)

## Decision

Run the compositor on a fixed rational output cadence driven by a single master clock; at each deadline composite whatever each tile's per-tile FrameSync currently provides (hold-last-frame for stalled tiles, drop for over-rate, placeholder after stale timeout). Each source is rebased onto the master timeline and continuously corrected via video drop/repeat and adaptive audio resampling.

## Rationale

Waiting for all inputs freezes the whole multiview on any single dead RTSP/HLS source (very common). Independent sources drift (e.g. 48000 vs 48001 Hz) and there is no genlock/PTP over arbitrary transports, so align-once breaks within minutes. This mirrors GstAggregator get_next_time()+timeout and NDI framesync.

## Alternatives considered

Wait-for-all-pads aggregation (rejected: stalls on one source). Event-driven recomposite per input frame (rejected: non-deterministic output rate). Align-once-at-startup (rejected: drifts/glitches). Lock to one source's clock (rejected: fragile if that source dies).

## Consequences

Each tile keeps a retained last-good GPU surface (VRAM per tile). Must own PTS rebasing across wrap/discontinuity, master-clock selection, and adaptive resampling. Stale tiles must visibly indicate 'no signal' to distinguish dead from static sources.
