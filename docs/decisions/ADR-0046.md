# ADR-0046: Skip the heavy CI matrix on prose-docs-only PRs via a fail-safe paths filter

- **Status:** Accepted
- **Area:** Engineering Guardrails — CI
- **Date:** 2026-06-10
- **Affects:** [`.github/workflows/ci.yml`](../../.github/workflows/ci.yml)

## Context

Every pull request runs the full `CI` workflow matrix: `fmt + clippy`,
`check + test` on **ubuntu** and **macos**, nine feature-gated clippy/test legs,
the AsyncAPI spec validation, `cargo-deny`, the web build, and the
inclusive-language check — on the order of a dozen jobs. That is the right cost
for any change that can affect the build/test/lint/spec surface.

It is pure waste for a PR whose entire diff is prose documentation — editing a
research brief, an ADR, a README — which cannot change a single compiled byte,
test outcome, lint result, or generated spec. On a constrained GitHub Actions
runner pool those wasted jobs back up the queue, wedging unrelated runs and
adding merge latency. The repository's `main` branch is **not** protected today,
so a skipped job does not block a merge; but the design must also stay safe if
branch protection is added later.

The hard constraint is asymmetric: a **false skip** (letting code/spec changes
merge without their checks) is far worse than a **wasted run**. The mechanism
must therefore *fail safe* — when there is any doubt about whether a file can
affect the build, it must run everything.

## Decision

Add a first `changes` job to `ci.yml` that classifies the PR diff with
[`dorny/paths-filter@v3`](https://github.com/dorny/paths-filter) (free OSS), and
gate the heavy jobs on its `code` output:

```yaml
if: ${{ github.event_name != 'pull_request' || needs.changes.outputs.code == 'true' }}
```

### Fail-safe classification

The `code` filter is expressed as **`'**'` minus a strict prose allowlist**,
evaluated with `predicate-quantifier: every`. Under `every`, a changed file
counts toward `code` only if it matches `**` **and** matches *none* of the prose
negations — i.e. only a file that is **not** prose makes `code` true. The
consequence is that any file type **not** explicitly allowlisted as prose (a new
extension, a dotfile, a config, a lockfile, anything under `crates/`, `web/`,
`xtask/`, `deploy/`, `.github/`, or a spec under `docs/api/`) keeps
`code = true` and runs the full matrix. "When in doubt, run everything" is the
default, not the exception.

The prose allowlist (the only things that can be skipped) is:

- Markdown anywhere **except** under `docs/api/`: `docs/!(api)/**/*.md`,
  `docs/*.md`, `*.md` (root), `crates/**/*.md`, `web/**/*.md`,
  `examples/**/*.md`, `.github/**/*.md`. No `*.md` is embedded via
  `include_str!`/`include_bytes!` anywhere in the repo (verified), so markdown
  is pure prose.
- Plain-text docs and licence/notice files in the docs tree (not `docs/api/`)
  or at the repo root: `docs/!(api)/**/*.txt`, `docs/*.txt`, `*.txt`, `LICENSE`,
  `LICENSE-COMMERCIAL`, `NOTICE`, `docs/**/LICENSE*`, `docs/**/NOTICE`. Text
  files that are build/test inputs — `crates/**/proptest-regressions/*.txt`,
  bundled font `OFL.txt` under `crates/**/assets/` — are deliberately **not**
  allowlisted and stay `code`.
- Documentation images in the docs tree (not `docs/api/`) or at the repo root:
  `*.png|jpg|jpeg|gif|svg|webp`. Crate/web asset images stay `code`.

`docs/api/**` (the generated `openapi.json` / `asyncapi.json` specs **and** the
`*.md` spec companion pages) is excluded **nowhere** — a change there keeps
`code = true`, so AsyncAPI spec validation always runs. `asyncapi.json` is also
`include_str!`'d into the `multiview-control` binary, so it is doubly code.

### Always-run light jobs

`inclusive-language` and a new `docs-sanity` (a dependency-free relative-link
checker, [`scripts/check-markdown-links.sh`](../../scripts/check-markdown-links.sh))
are **not** gated on `changes`. They run on every push and PR, so a docs-only PR
still gets a real green signal — and, if branch protection is added later, a
docs-only PR's required checks are satisfied by these light jobs while every code
PR still runs the full set.

### Push to `main`

On `push` (no PR base to diff against) the gate is bypassed: the `if` keys on
`github.event_name == 'pull_request'`, so a push to `main` always runs the full
matrix.

## Truth table

| PR diff | `code` | Heavy matrix | Light jobs |
|---|---|---|---|
| `crates/**/foo.rs` only | `true` | **runs** | run |
| `Cargo.lock` / `*.toml` only | `true` | **runs** | run |
| `web/**` only | `true` | **runs** | run |
| `.github/workflows/ci.yml` only | `true` | **runs** | run |
| `docs/api/openapi.json` (or any `docs/api/**`) | `true` | **runs** | run |
| `docs/decisions/ADR-XXXX.md` only | `false` | skipped | run |
| `docs/research/*.md` / `README.md` only | `false` | skipped | run |
| `*.md` + `*.rs` (mixed) | `true` | **runs** | run |
| any unrecognised extension | `true` | **runs** | run |

## Consequences

- **No coverage is reduced for any code/config/spec/web PR.** The heavy jobs'
  steps are unchanged; only *when* they run changes, and only prose-docs-only
  PRs skip them. The classification was verified empirically against all tracked
  files: the only files that classify docs-only are `.md` / `.txt` / images /
  `LICENSE*` / `NOTICE` outside `docs/api/**`; zero non-prose files were
  classified docs-only.
- Docs-only PRs return in seconds (two light jobs) instead of occupying ~12
  runners, freeing the queue.
- The filter is centralized in one job; adding a new prose type is an additive,
  reviewable negation, and forgetting to add one only ever costs a CI run (it
  stays `code`), never a skipped check.
- A new `docs-sanity` link check guards against the most common docs-PR breakage
  (a renamed/moved brief or ADR) now that docs PRs no longer trigger the broader
  build that would have surfaced some breakage indirectly.
