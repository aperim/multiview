# ADR-0032 — HLS / LL-HLS delivery: tiered static-vs-dynamic serving, CMAF-first segments, isolated blocking-reload origin

- Status: Proposed
- Date: 2026-06-07
- Deciders: Multiview engine + output
- Area: streaming / output (T-series)
- Related: ADR-0006, ADR-0007, ADR-T005; brief
  [hls-delivery](../research/hls-delivery.md); core-engine §9.2;
  streaming-gotchas §4
- Supersedes / refines: extends ADR-0007 (CMAF-first) with the serving model

## Context

The HLS text layer (`hls/media.rs`) is RFC 8216bis-correct and already renders
every LL-HLS tag, but the **behaviour layer is ~0%**: the live playlist is
written **once at finalize, non-atomically** (`pipeline.rs:1795`
`std::fs::write`), segments are **MPEG-TS** (`sink.rs:840`/`:1516`) accumulated
in an **unbounded** `done: Vec` with **no disk pruning**, there is **no fMP4
producer, no rolling driver, no byte-range parts, no blocking-reload origin, no
delta production**, and **no config knobs** for container, locations, base URL,
DVR, serving, or headers (`schema.rs:551`/`:573`). Serving today is a stock
`nginx:1-alpine` sidecar that sets **no Cache-Control, no CORS, no
Accept-Ranges** (`deploy/compose.yaml`).

Operators need to front delivery with nginx / traefik / a CDN, with correct
cache/CORS headers and configurable file locations/origins — without putting the
engine hot path (inv #1/#7/#10) on any request path.

## Decision

1. **Bifurcate the serving boundary by latency tier, not file type.** Master
   playlist + closed segments + `init.mp4` are **static-frontable** (any CDN /
   nginx / traefik / object store with correct headers). The **LL-HLS live
   playlist + blocking-reload + preload-hint parts + deltas** are served by
   **Multiview's own async axum/hyper origin** (a static server has no "wait
   until it exists" primitive). **Dual-write:** the segmenter always writes static
   files; the dynamic handler reads those *same* files and only adds "wait until
   it exists" — it is not a second packager or encoder.

2. **CMAF/fMP4 is the default segment container** (`.m4s` + once-served
   EXT-X-MAP `init.mp4`). **MPEG-TS becomes opt-in legacy-reach**, restricted to
   the plain-HLS tier. **LL-HLS requires fMP4** (Safari native LL-HLS plays fMP4,
   fails on TS livestreams). One CMAF store backs plain HLS + LL-HLS (+ future
   additive DASH) — encode-once at the packaging layer. **The `.mpd` builder is
   deferred**; the store is designed CMAF-clean.

3. **LL-HLS uses byte-range parts** (EXT-X-PART …,BYTERANGE into one growing
   `.m4s`, contiguous offsets) — **not** discrete per-part files — to deliver the
   "fewer, larger CDN objects" property.

4. **A rolling-playlist + DVR foundation precedes everything**: convert the batch
   `SegmentState`/finalize model into a rolling driver (push_segment +
   set_window + re-render per closed segment/part, bounded `VecDeque` window),
   with **atomic publish split by artifact class** (whole files = temp+fsync+
   rename(2) same-dir; growing parts = durable-write-then-announce ordering),
   **deferred-unlink disk pruning** honouring the §6.2.2 grace period (grace =
   longest *distributed* playlist + segment + 1×target slack), PROGRAM-DATE-TIME
   from a single `utc0@tick0` anchor, and `program.ts` made opt-in.

5. **The LL-HLS origin upholds inv #10** as a client of an engine-published
   drop-oldest watch/broadcast snapshot. Held GETs await a `watch` (source of
   truth; re-check satisfiability after a `Notify` wakeup to avoid lost wakeups),
   are **hard-capped** (→ `503` shed) and **time-bounded** by the HTTP task's own
   timeout. Unsatisfiable msn/part → `400`. **Hold-then-burst** part delivery; no
   chunked-transfer trickle. A CI chaos/soak gate proves a held-request flood +
   abandoned + slow client cannot delay one output tick.

6. **Headers are config + reference-config**: explicit Content-Type per
   extension, the AWS-MediaPackage-style Cache-Control tiers (live playlist
   `max-age=1, must-revalidate`; segments/init `immutable` 1y as our deliberate
   choice), Origin-reflecting CORS with `Vary: Origin`, `Accept-Ranges: bytes` +
   206 on part-bearing segments. Wire `tower-http CorsLayer` (already a dep) in
   our origin; ship reference nginx/traefik configs (adding the `.m4s` MIME type
   absent from stock nginx) for static fronting.

7. **New config is serde-default + typed** (`container`, `segment_dir`,
   `base_url`, `init_name`, `segment_template`, `dvr_mode`, `dvr_seconds`,
   `delete_grace_segments`, `program_date_time`, `serve`, `cache`,
   `cors_allowed_origins`) with validation; defaults preserve relative-URI,
   static-only, LGPL-clean behaviour. `base_url` is a no-op when unset.

## Consequences

**Positive.** Plain HLS + DVR are CDN/object-store frontable with zero Multiview
HTTP. The protected output core is never on a request path; isolation is the
existing, proven pattern. One encode and one CMAF store serve all HLS tiers (+
future DASH) with no extra encode. Operators get correct headers and a reference
fronting config they only *honor*, not invent.

**Negative / cost.** The fMP4 switch is **more than a muxer-name swap**: it needs
new FFI plumbing (`movflags=cmaf+frag_custom+empty_moov+delay_moov` via an
AVDictionary passed to `avformat_write_header` — `mux.rs` has no option-setting
API today), an explicit `flush_fragment()` wrapping `av_write_frame(ctx, NULL)`
at each part/segment boundary (without it `frag_custom` emits one un-parted
fragment), a once-written shared init replacing the current open-fresh-muxer-
per-segment model (`sink.rs:1517`), and atomic closed-segment muxing (mux to
`.tmp`, rename after finish). The blocking-reload origin is genuinely custom and
inv-#10-sensitive. Tier 2 depends on a correctly-configured LL-capable CDN
(cache-key + coalescing + timeout) — a documented operator contract, not code.

**Risk controls.** `base_url`/serde-default keep existing configs parsing;
advertised capabilities (CAN-BLOCK-RELOAD / CAN-SKIP-UNTIL) are gated on the
behaviour being wired; a TDD test asserts a two-GOP CMAF mux produces exactly one
`ftyp`+`moov` init plus N `moof`+`mdat` fragments with monotonic sequence
numbers; the chaos/soak gate guards inv #10.

## Alternatives considered

- **Keep MPEG-TS as default.** Rejected: Safari native LL-HLS fails on TS; byte-
  range parts and DASH reuse need CMAF; ADR-0007 already chose CMAF-first.
- **Lean entirely on an external LL-capable CDN (no own origin).** Rejected: a
  static/object-store front physically cannot hold GETs for `_HLS_msn`/`_HLS_part`
  — the dynamic origin is irreducible; the CDN can only sit *in front of* it.
- **Discrete per-part `.m4s` files.** Rejected: yields *more* CDN objects, losing
  the CDN-friendliness; byte-range parts into one growing file is the spec's
  intended model.
- **Chunked-transfer trickle of in-progress parts.** Rejected: many CDNs reject
  CTE for live; hold-then-burst is the CDN-safe, spec-permitted method.
- **HTTP/2 mandatory / HTTP/2 server push.** Rejected: HTTP/2 is *preferred*
  (multiplexing held GETs), not required; Apple *removed* server push.
- **Unlink-on-eviction pruning.** Rejected: violates §6.2.2 (removed segment must
  stay available); a one-refresh-behind / mid-download client 404s.
- **Per-part temp+rename.** Rejected: incompatible with the growing-file
  byte-range model; integrity comes from durable-write-then-announce ordering.
