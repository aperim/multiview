# ADR-0071: Iceberg / S3-Tables event+media metadata catalog via SeaweedFS — keeping metadata and events with the media

- **Status:** Proposed
- **Area:** Storage / Events / Control / Config / Web
- **Date:** 2026-06-13
- **Source brief:** [recording-storage-offload.md](../research/recording-storage-offload.md) §4
- **Extends:** [ADR-0070](ADR-0070.md) (the S3 offload that lands the media bytes the catalog indexes), [ADR-0037](ADR-0037.md) (the recordings being catalogued)
- **Builds on:** [ADR-R003](ADR-R003.md) (`backon` back-off for commit retries), [ADR-0035](ADR-0035.md) (`HealthWarning`), [ADR-0042](ADR-0042.md) (IPv6-first)
- **Relates to:** [ADR-0052](ADR-0052.md) / [ADR-0053](ADR-0053.md) (consent, data-minimisation, retention/erasure), [realtime-api.md](../research/realtime-api.md) (the event envelope the `events` table mirrors), invariants #1/#10

## Context

The operator asked for **S3-Tables (Iceberg) buckets** ([github.com/seaweedfs/seaweedfs](https://github.com/seaweedfs/seaweedfs)) to keep **metadata/event data with the video, frame-cap, or ISO-audio**. A pile of media objects in a bucket ([ADR-0070](ADR-0070.md)) has no queryable index — "show me every clip on camera 3 where a person was detected last Tuesday" requires a scan.

**Apache Iceberg** is an open table format that layers a queryable, ACID, time-travelling table over object storage: a catalog maps a table name → a metadata JSON → a manifest list → manifest Avro files → data files; commits are atomic catalog-pointer swaps, giving snapshot isolation and time travel (per the Iceberg spec). **SeaweedFS ships a built-in Iceberg REST Catalog** alongside its S3 gateway — no external Hive/Glue metastore — on a dedicated port (per its README, default **8181**), with S3 data access on the S3 port (default **8333**), auth via SigV4 / OAuth2 / unsigned-REST + S3 keys, and native integrations (Spark/Trino/DuckDB/Dremio/RisingWave) plus automated table maintenance (compaction, snapshot expiry, orphan removal) ([github.com/seaweedfs/seaweedfs](https://github.com/seaweedfs/seaweedfs) — exact port/auth specifics *to verify against SeaweedFS docs before implementation*). Both the media and its metadata can therefore live in the *same* SeaweedFS deployment — "metadata with the media," exactly the ask. No Iceberg/SeaweedFS/catalog code exists in the tree today (net-new).

## Decision

1. **Write an Iceberg sidecar table set, keyed to objects.** Two tables whose **media artefacts are immutable** once closed and whose normal operation is **append-only** (corrections are new snapshots) — but they are created as **Iceberg v2 tables with delete-file support**, because erasure (Decision 5) requires row-level deletes (a tombstone-plus-filter / delete-snapshot model), not a strictly append-only v1 table:
   - **`recordings`** — one row per closed media artefact: `recorder_id`, `media_kind ∈ {video, frame_cap, iso_audio}` (from [ADR-0069](ADR-0069.md)), `source_id`, `object_key` + `bucket` (the [ADR-0070](ADR-0070.md) S3 key), `segment_start_utc`, `segment_end_utc`, `duration_ns`, `byte_size`, `codec`, `container`, `pinned_until` (nullable), `checksum`. Partitioned by `media_kind` + day.
   - **`events`** — one durable row per persisted realtime event ([realtime-api.md](../research/realtime-api.md) envelope content: `DetectionTrigger`, cue events, `RecordingStatus`, `HealthWarning`), keyed to time and optionally `recorder_id`/`object_key`. This is the queryable, durable twin of the conflated-drop-oldest live stream.
   A join on `events.object_key = recordings.object_key` (or overlapping time ranges) answers "give me the clip for this event."

2. **Commit off the data plane, media-first / index-eventually.** The catalog writer runs on a dedicated task, batches rows, and commits to Iceberg **after** the segment is durably uploaded ([ADR-0070](ADR-0070.md)). An Iceberg commit is an atomic catalog-pointer swap, so a batch lands as one snapshot; a failed commit retries with `backon`. **Retry idempotency is Multiview's responsibility, not Iceberg's**: Iceberg identifier fields do *not* enforce row uniqueness (the spec assigns dedup to the writing engine), so a re-tried append could duplicate rows. We make retries idempotent with a durable commit/outbox id and a deterministic data-file path per `(recorder_id, media_kind, object_key)`, so re-committing the same batch overwrites the same data file rather than appending duplicates — *the exact write protocol to verify against the chosen Iceberg writer*. The catalog is **never** read/written on the output-clock or recorder-writer thread. A wedged catalog endpoint loses *queryability of recent rows*, never a frame and never a recording — and because local prune requires *both* upload-confirmed *and* row-written ([ADR-0070](ADR-0070.md) delete-after-confirm), a wedged catalog *keeps* local copies, never loses data.

3. **Write via SeaweedFS's REST catalog; data files in the bucket.** Target the Iceberg REST Catalog API (SeaweedFS default port 8181) for namespace/table/commit; data files live in the S3 bucket (port 8333). Default Rust writer: **`iceberg-rust`** — *but its write/REST-commit maturity must be verified at build time* (it has historically trailed reads); fallback is to emit Iceberg-compatible Parquet data files + commit via the REST catalog HTTP API directly, or stage rows for SeaweedFS's own maintenance/compaction. We lean on SeaweedFS's automated table maintenance rather than reimplementing compaction/expiry.

4. **Data-minimisation in the catalog (consent boundary, [ADR-0052](ADR-0052.md)).** The tables carry operational keys only — source ids, object keys, timestamps, coarse detection labels — and **never** raw serials/MACs/credentials/stream-URLs-with-secrets/PII. The catalog is the **operator's own data** in the **operator's own** SeaweedFS; it **never** phones home to Aperim and is categorically separate from the telemetry/heartbeat pipes ([ADR-0052](ADR-0052.md) §1/§4). A test asserts no catalog field appears in a telemetry or heartbeat payload.

5. **Retention/erasure are operator-driven and enforceable.** A consent-revocation/erasure request maps to: stop offload for the source, lifecycle-expire or hard-delete the affected object keys ([ADR-0070](ADR-0070.md)), and tombstone the catalog rows (an Iceberg delete snapshot). Because both the bucket and the catalog are the operator's, erasure is theirs to enforce; Multiview provides the levers (a `purge` operation + the lifecycle config) and documents them ([ADR-0053](ADR-0053.md) explicit-approval-egress posture).

6. **Config/realtime/UI are additive.** A `[storage]` `catalog: Option<CatalogPolicy{rest_uri, namespace, tables, auth}>` (`rest_uri` accepts bracketed IPv6 literals, [ADR-0042](ADR-0042.md)); the `Event::StorageStatus` carries `catalog_last_commit`; an additive `#[non_exhaustive]` `WarningCode` `catalog-commit-failing` at `GET /api/v1/health`; the `RecordingsPage`/storage panel shows catalog status. Vendor-neutral, nominative (Iceberg/S3-Tables are open; SeaweedFS spoken to over its published REST/S3 APIs — no bundled SDK, no redistributed spec, the [managed-devices.md](../research/managed-devices.md) posture).

## Consequences

**Positive.**
- "Metadata/event data with the video/frame-cap/ISO-audio" is met literally: media + an Iceberg catalog of media-rows + event-rows in one SeaweedFS deployment, joinable by any open query engine (DuckDB/Trino/Spark).
- Open formats end-to-end (Iceberg + S3-Tables) — no proprietary lock-in; SeaweedFS provides the catalog + maintenance for free, so we write rows, not a metastore.
- The catalog is the durable twin of the conflated live event stream — analysts get history the realtime lane intentionally drops.

**Negative / costs.**
- `iceberg-rust` write maturity is the key risk; the fallback (Parquet + direct REST commit) is more code. Verified by a spike before pinning.
- A second metadata domain (Iceberg) alongside the existing SQLite control store — must be clearly scoped as *archival/queryable analytics*, not the live config/control DB.
- Catalog schema evolution must track the versioned realtime envelope ([realtime-api.md](../research/realtime-api.md)); coupling/migration needs a small design pass (Open question).

**Neutral.**
- Catalog is optional ([off] by default); media offload ([ADR-0070](ADR-0070.md)) works without it (delete-after-confirm degrades to upload-confirmed-only when no catalog is configured).

## Alternatives considered

1. **Put metadata in the existing SQLite control store.** *Rejected for the archive* — SQLite is the live control/config DB, not a horizontally-scalable, object-co-located, time-travelling analytics store. Iceberg keeps metadata *with the media* in the same object store, queryable by external engines — the operator's explicit ask.
2. **A bespoke JSON sidecar per object (no table format).** *Rejected* — unqueryable at scale (the scan problem); Iceberg exists precisely to index objects. (A per-segment JSON index already exists locally for retrieval, [iso-program-recording.md §A](../research/iso-program-recording.md); the Iceberg table is the *aggregate, queryable* layer, not a replacement.)
3. **An external Hive/Glue metastore.** *Rejected* — SeaweedFS's built-in REST catalog removes the need; an external metastore adds an operational dependency for no benefit here.
4. **Write the catalog on the recorder/engine thread for "consistency."** *Rejected* — violates #1/#10; commit off the data plane, media-first/index-eventually, idempotent retry.
5. **Index raw identifiers/PII for "richer search."** *Rejected* — data-minimisation ([ADR-0052](ADR-0052.md) §4); operational keys + coarse labels only.
6. **Commit per row (no batching).** *Rejected* — Iceberg commits are catalog-swaps; per-row commits thrash snapshots. Batch + atomic snapshot, retried idempotently.

## How this honors the invariants

- **#1 (output-clock untouchable).** The catalog writer runs on a dedicated task over already-uploaded objects; it never runs on or awaits the output-clock thread. Commit latency/failure cannot affect `out_pts = f(tick)`.
- **#10 (isolation).** The catalog is written after offload, off the data plane, via a bounded batch + `backon`-retried idempotent commit. A wedged catalog endpoint loses recent queryability only — it back-pressures nothing (and, via delete-after-confirm, simply *keeps* local copies until it recovers). The engine and recorder share no channel or lock with the catalog writer.
