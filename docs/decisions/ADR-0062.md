# ADR-0062: ONVIF client — WS-Discovery, manual cross-subnet endpoint add, and full managed-camera control (Device/Media/Imaging/PTZ)

- **Status:** Proposed
- **Area:** Management (Devices domain) / Control plane
- **Date:** 2026-06-13
- **Source brief:** [onvif-and-ptz.md](../research/onvif-and-ptz.md)
- **Extends:** [ADR-M008](ADR-M008.md) (registry + closed driver enum), [ADR-M009](ADR-M009.md) (source/output/display projections), [ADR-0041](ADR-0041.md) (untrusted-inventory confirm-adopt), [ADR-RT007](ADR-RT007.md) (devices realtime topic), [ADR-W017](ADR-W017.md) (bare-verb action routes), [ADR-0042](ADR-0042.md) (IPv6-first)
- **Relates to:** [ADR-0064](ADR-0064.md) (the PTZ command model this driver implements), [ADR-0063](ADR-0063.md) (the ONVIF-server facet — the inverse direction)

## Context

The operator wants **full ONVIF client control** of cameras — not a bare-bones viewer: discover
ONVIF devices on the LAN, **manually add an ONVIF endpoint across a subnet boundary** (test, then
adopt), and then drive the camera the way a real NVR does — set its **time**, read and set its
**codecs / resolution / profiles**, drive **Imaging**, and drive **PTZ**.

ONVIF is an **open standard**: ONVIF Core (WS-Discovery), Device, Media, Media2, Imaging, and PTZ are
SOAP/WSDL web services. Discovery is OASIS **WS-Discovery** — SOAP-over-UDP multicast to
`239.255.255.250:3702` (IPv6 `ff02::c`), Probe type `dn:NetworkVideoTransmitter` (web-verified: ONVIF
APG, OASIS WS-Discovery). The Device service exposes the unauthenticated `GetSystemDateAndTime` (used
for the clock-skew correction WS-Security UsernameToken requires) and **`SetSystemDateAndTime`**
(required for Profile-S conformance, but probed-not-assumed on fielded devices — web-verified). Media2 `GetProfiles` + `GetStreamUri` enumerate
streamable profiles with their encoder config (web-verified). PTZ is its own service ([ADR-0064](ADR-0064.md)).

The Devices domain is **already built** and fits this exactly: a closed `#[non_exhaustive]`
`DeviceDriver` enum (`crates/multiview-config/src/device.rs:35`, today `Zowietek`/`Displaynode`/`Cast`);
control-plane driver actors (`crates/multiview-control/src/devices/zowietek/`); a lifecycle state
machine `ADOPTING → ONLINE / DEGRADED / AUTH_FAILED / UNREACHABLE`
(`crates/multiview-control/src/devices/state_machine.rs:14`); a bounded, single-flight, untrusted
discovery layer with a closed `DiscoveryDriverKind` enum
(`crates/multiview-control/src/devices/discovery.rs:113`); device routes including a confirm-adopt
`POST /devices/{id}` and bare-verb actions (`crates/multiview-control/src/routes/mod.rs:672`); and the
`Topic::Devices` realtime surface ([ADR-RT007](ADR-RT007.md)). WS-Discovery is **not** mDNS, so it is
a new browse transport, but it plugs into this existing pipeline.

## Decision

Add an **`onvif` client driver** as a new variant of the closed driver enum, implemented as a
control-plane poller/actor exactly like `zowietek`, with a **clean-room SOAP client** built from the
published ONVIF/WS-Discovery specifications — **no gSOAP, no vendor ONVIF toolkit, no camera SDK**,
`unsafe_code = forbid`, behind the off-by-default device-network feature.

1. **WS-Discovery as a new browse transport into the existing untrusted inventory.** Send a Probe for
   `NetworkVideoTransmitter` on **IPv6 `ff02::c` first**, IPv4 `239.255.255.250:3702` as legacy peer
   ([ADR-0042](ADR-0042.md)); parse ProbeMatches into the **existing** bounded discovery inventory
   (`discovery.rs`) with a new `DiscoveryDriverKind::Onvif` (`discovery.rs:113`). The same hardening
   the shipped discovery enforces applies unchanged: **single-flight, time-budgeted, hard service
   cap, every responder-controlled string truncated, SOAP body size-capped** (a WS-Discovery
   responder is unauthenticated and spoofable). Results are `device.discovered` rows
   ([ADR-RT007](ADR-RT007.md)) — **candidates requiring explicit confirm-adopt**
   ([ADR-0041](ADR-0041.md)), never auto-ingested.

2. **Manual cross-subnet test + add as a first-class path** (multicast WS-Discovery is link-local).
   `POST /devices/probe-onvif {address}` performs a **read-only** two-step test — unauthenticated
   `GetSystemDateAndTime` (proves "ONVIF here" + clock skew), then authenticated
   `GetDeviceInformation` + `GetCapabilities`/`GetServices` (model/firmware + the per-service endpoint
   map) — and returns a typed result distinguishing **unreachable** vs **not-ONVIF** vs **auth-failed**
   (RFC 9457 problems; the `AUTH_FAILED` lifecycle state already exists, `state_machine.rs:14`). Adopt
   is the **existing** idempotent confirm-bind (`POST /devices/{id}`, `routes/mod.rs:672`) creating a
   `Device` record with `driver = "onvif"`.

3. **Full managed-camera control via the ADR-M009 projections plus a PTZ facet.**
   - **Source candidates:** `GetProfiles` → each profile's `GetStreamUri` becomes a source candidate;
     picking one creates an ordinary `SourceKind::Rtsp` Source (`crates/multiview-config/src/schema.rs:296`)
     with a `device_ref` — ingest path untouched. Prefer a sub-stream profile near the tile's pixel
     size (invariant #6, decode-at-display-resolution).
   - **Device management:** `SetSystemDateAndTime` ("sync camera clock"), network/NTP/hostname, users
     (opt-in), reboot — bounded management actions; clock/encoder changes are **`DEV`-class**
     (device restarts its stream; a bound Source rides the tile ladder; **no Multiview output is
     affected**) with the impact declared **before** apply ([ADR-M009](ADR-M009.md) §mode, the
     `DEV` capability class of [managed-devices.md §10](../research/managed-devices.md)).
   - **Media management:** read encoder config per profile; `SetVideoEncoderConfiguration` (the write
     op is singular in both Media1 `ver10` and Media2 `ver20` — Media2 differs by namespace/binding,
     not op name) to set **codec/resolution/bitrate/GOP/FPS** — one
     internal encoder-config model, two SOAP bindings negotiated from `GetServices`.
   - **Imaging:** `Get·SetImagingSettings` + focus Move/Stop as bounded management actions.
   - **PTZ:** a new fixed `ptz` capability facet implementing the [ADR-0064](ADR-0064.md) `PtzCommand`
     model via the ONVIF-PTZ profile.

4. **Auth + clock discipline.** WS-Security UsernameToken digest (or HTTP Digest), correcting
   local↔device skew from the unauthenticated `GetSystemDateAndTime` before stamping `Created`.
   Credentials are write-only `secret_ref`s (the existing secret model), never echoed; plain-HTTP is a
   management-VLAN recommendation with a warning, never the default; HTTPS where the camera supports it.

5. **Driver shape = the `zowietek` precedent.** Serialise requests per device; poll status ≤1 Hz;
   ride the lifecycle + supervised reconnect; re-converge desired state (e.g. a pending encoder
   change) on reconnect. IPv4-only cameras are legacy-interop islands under
   [conventions §10](../architecture/conventions.md); Multiview-side surfaces stay IPv6-first.

6. **API + realtime** (under `/api/v1`, OpenAPI-first, bare-verb routes [ADR-W017](ADR-W017.md)):
   `POST /devices/probe-onvif`; `GET /devices/{id}/ptz`; `POST /devices/{id}/ptz/move|stop`;
   `GET/POST /devices/{id}/ptz/presets` + `…/presets/{p}/goto`; `POST /devices/{id}/onvif/set-time`;
   `POST /devices/{id}/onvif/encoder`. Long-running/device-reset ops return `202` + op-id (outcome on
   the stream); PTZ ContinuousMove/Stop are fast `200`s; status (incl. a latest-wins PTZ position)
   rides `device.status` on `Topic::Devices` — **no new topic** ([ADR-RT007](ADR-RT007.md)).

## Consequences

- **The operator gets "full ONVIF", honestly scoped.** Discover, cross-subnet add, set time, set
  codecs/res/profiles, Imaging, PTZ — a deep but **bounded, typed** surface (the [ADR-M008](ADR-M008.md)
  YAGNI/closed-enum guard), not an open re-export of every camera knob.
- **A new browse transport with its own attack surface.** WS-Discovery is unauthenticated/spoofable;
  it stays feature-gated, bounded, hardened, and confirm-adopt-only — the discovered list never
  steers the engine.
- **Camera-vs-spec drift is real.** ONVIF cameras are widely non-conformant (Media1-only devices,
  vendor-specific PTZ spaces). Socket-free tests cannot catch it; each surface carries a mandatory
  on-hardware validation item before its typed client is trusted (the `zowietek` lesson).
- **A new family always costs a variant + module + ADR** — no plugins, per [ADR-M008](ADR-M008.md).
- **Default build unchanged.** Pure-Rust, socket-free, CI-green by default; ONVIF I/O is opt-in.

## How it honors the invariants

- **#1 (output-clock untouchable):** the `onvif` driver is a control-plane actor; **no** ONVIF/PTZ
  call, poll, or SOAP parse runs on the output-clock loop, and program output is never contingent on a
  camera. A camera bound as a Source is **sampled** through the framestore last-good ladder (it never
  paces output); a device-side encoder/clock change restarts that camera's stream only — Multiview
  outputs are unaffected.
- **#10 (isolation):** every driver task publishes into watch/broadcast channels and is **never
  awaited by the engine**; a slow/hung/hostile camera can stall only its own driver task. Discovery is
  bounded drop-oldest and confirm-adopt-only. This is the same structural proof the shipped
  `zowietek`/`cast` drivers carry.

## Alternatives considered

- **Bundle a vendor ONVIF SDK / gSOAP toolkit.** Rejected: contradicts the open-standard, no-vendor-SDK,
  `unsafe_code = forbid`, LGPL-clean-default posture; a clean-room SOAP client from the published
  specs is the established `zowietek` clean-room pattern and keeps `cargo deny` clean.
- **Auto-adopt WS-Discovery hits.** Rejected: discovery is untrusted/spoofable; auto-binding lets any
  LAN actor inject a Source into the program ([ADR-0041](ADR-0041.md) doctrine).
- **mDNS-only discovery (reuse the existing daemon as-is).** Rejected: ONVIF announces via
  WS-Discovery, not mDNS — a real new transport is required; it reuses the inventory/confirm-adopt
  *pipeline*, not the mDNS *wire*.
- **A generic "IP camera" property-bag device covering ONVIF+VISCA+vendor-HTTP.** Rejected: untyped
  capability bags cannot gate the UI/validator exhaustively and drift from real behaviour
  ([ADR-M008](ADR-M008.md) — fixed probed flags); `onvif` is a typed variant with fixed facets.
- **Skip manual cross-subnet add (rely on discovery).** Rejected: WS-Discovery is link-local and does
  not cross routers; the operator explicitly requires cross-subnet add — it is a must-have, not a
  fallback.
