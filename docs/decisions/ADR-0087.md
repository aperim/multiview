# ADR-0087: Online-service URL ingest — generalise the `yt-dlp` resolver (ADR-0015) to Vimeo + a general resolver, add download-to-VT and record-to-S3/Iceberg

- **Status:** Proposed
- **Area:** Input / Config / Control / Output·Storage
- **Date:** 2026-06-13
- **Source brief:** [online-services-input.md](../research/online-services-input.md)
- **Extends:** [ADR-0015](ADR-0015.md) + [io/youtube-live.md](../io/youtube-live.md) (the shipped runtime-discovered `yt-dlp` YouTube-live resolver — **the foundation this ADR generalises, never re-litigates**), [ADR-T004](ADR-T004.md) (HLS ingest pacing — PTS-to-wall-clock, not `-re`), [ADR-T003](ADR-T003.md) (PTS normalisation)
- **Builds on:** [ADR-0057](ADR-0057.md) / [media-playout.md](../research/media-playout.md) (media-library import pipeline — *download-to-VT* feeds this), [ADR-0037](ADR-0037.md) (the recorder), [ADR-0070](ADR-0070.md) (S3 offload) + [ADR-0071](ADR-0071.md) (Iceberg catalog) — *record-to-S3* reuses these verbatim
- **Relates to:** [ADR-0008](ADR-0008.md) (NDI runtime-load precedent), [ADR-0012](ADR-0012.md) (LGPL-clean default build), [ADR-R003](ADR-R003.md) (`backon` capped-jitter back-off), [ADR-R004](ADR-R004.md) (Class-1/Class-2, inv #11), [ADR-M006](ADR-M006.md) (reference-only secrets), [ADR-M007](ADR-M007.md) (`CapabilityReport` gate), [ADR-0052](ADR-0052.md) (data-minimisation), [ADR-0042](ADR-0042.md) (IPv6-first), invariants #1/#2/#4/#7/#10

## Context

The operator asked for **native Online-Service URL support — YouTube and Vimeo — with ISO-record, download-to-VT, and S3 / S3-Tables (Iceberg)**.

Multiview **already ships** the load-bearing half. [ADR-0015](ADR-0015.md) added a `youtube` source kind implemented as a **resolver**: an external, **runtime-discovered, never-vendored** `yt-dlp` binary turns a YouTube live URL into an FFmpeg-readable HLS master, which feeds the **existing HLS ingest path** (libav demux/decode + the custom HLS input pacer + PTS normalisation + supervised reconnect). This is built and tested in-tree: a pure `yt-dlp -J` info-dict parser (`crates/multiview-input/src/youtube/resolve.rs:1`), a thin subprocess shell with a hard timeout and no shell (`crates/multiview-input/src/youtube/process.rs:122`), a pure re-resolution policy + supervised loop for the ~6 h signed-URL expiry (`crates/multiview-input/src/youtube/reresolve.rs:1`), and a `Youtube { url }` source variant in the schema (`crates/multiview-control/src/openapi_schemas.rs:882`). The module lives behind the off-by-default `youtube` feature (`crates/multiview-input/src/lib.rs:82`,`:123`).

`yt-dlp` carries a dedicated **Vimeo** extractor and a large extractor set (reported as 1800+) plus a generic embedded-player fallback (to verify against the pinned `yt-dlp` version before implementation), and handles both live and VOD. So Vimeo and "the rest" are **the same resolver with a wider extractor surface** — not a new ingest mechanism.

The recording/storage tail the operator also asked for is **already designed**: the per-source ISO recorder ([ADR-0037](ADR-0037.md)), the bounded never-back-pressuring S3 offload ([ADR-0070](ADR-0070.md)), and the SeaweedFS Iceberg catalog ([ADR-0071](ADR-0071.md)). The media library + import pipeline that a *VT* (media player) plays from is designed in [ADR-0057](ADR-0057.md) (Proposed, unbuilt).

The risk to avoid is **duplicating shipped/designed work** by treating "online services" as a new subsystem. It is not. The genuinely-new code is a resolver *generalisation* and a *download-to-file* fetcher; everything else is reuse.

## Decision

1. **Generalise the resolver from `youtube`-only to an online-service resolver, reusing the built core.** Keep the three-part split — pure parse core (`resolve.rs`), subprocess shell with hard timeout + no shell (`process.rs`), re-resolution policy + supervised loop (`reresolve.rs`) — and widen only (a) **URL recognition / extractor surface** and (b) **format-selection policy** (live HLS master vs VOD/progressive URL). Multiview never hard-codes per-service extraction; it runs `yt-dlp -J`, parses the info-dict, and consumes the processed transport. `yt-dlp` stays an **optional, runtime-discovered, off-by-default-feature** dependency, probed via `probe_version` (`process.rs:87`) / `CapabilityReport` ([ADR-M007](ADR-M007.md)); absent → the kinds report *unavailable*, never crash.

2. **Add source kinds `Vimeo { url }` and `Online { url, mode? }`** as adjacently-tagged siblings of the built `Youtube { url }` ([conventions §5](../architecture/conventions.md) — **never `untagged`**). `mode ∈ { live, vod_as_live, download }`, defaulted **deterministically** from the parsed `live_status`: a `live` result defaults to `live`; any non-live (VOD) result defaults to `download` (the safer transport-rich choice — `vod_as_live` is opt-in). `youtube`/`vimeo` may be thin presets over `online` (e.g. YouTube pins the `web_safari` player client, `process.rs:22`).

3. **Workflow 1 — live ingest is the existing HLS-paced path.** A resolved **live** source becomes an `hls` (or generic-URL) input, paced **wall-clock-by-PTS via the custom pacer**, **never `-re`** ([ADR-T004](ADR-T004.md), invariant #4), PTS-normalised ([ADR-T003](ADR-T003.md), invariant #3). The re-resolution loop refreshes the signed URL ahead of expiry (make-before-break) and on a 403 burst. `vod_as_live` is the same pacer, applied to a VOD. **v1 acceptance is conditional for live Vimeo:** Vimeo VOD is supported via download-to-VT / VOD-as-live; live Vimeo is supported only after its live transport (HLS, or defined DASH pacing/discontinuity behaviour) is confirmed against a real event (brief Open question 2).

4. **Workflow 2 — download-to-VT is a fetch-to-file bridge into the media library.** A new **download fetcher** (`yt-dlp` download mode, or a libav stream-copy when the resolved VOD URL is directly readable) produces a local file in staging; the **existing import pipeline** ([ADR-0057](ADR-0057.md) §4) probes/validates/reject-or-transcodes it into an ordinary library asset; a media-player (VT) channel then cues/plays/loops it. All fetch + probe + transcode is a control-plane long-running op (`202` + op-id, progress on the stream), **off the data plane**. This depends on [ADR-0057](ADR-0057.md) landing and ships with (or after) it.

5. **Workflow 3 — ISO-record and record-to-S3/Iceberg reuse the recorder/offload/catalog verbatim.** A resolved live online source is an ordinary recorder input: the copy-remux ISO recorder ([ADR-0037](ADR-0037.md), zero re-encode, invariant #7), the bounded multipart S3 offload ([ADR-0070](ADR-0070.md), paths-not-bytes, delete-after-confirm, `object_store`, IPv6-first), the SeaweedFS Iceberg catalog ([ADR-0071](ADR-0071.md)). This catalog tail targets **generic Iceberg-over-object-storage** (an Iceberg REST catalog); Amazon **S3 Tables** is a distinct managed catalog with its own endpoint and semantics, so a dedicated S3 Tables adapter (and its compatibility limits) is scoped separately as an Open question in the brief (§9, q9), not assumed satisfied by the SeaweedFS path. The **only** online-specific addition is a **secret-scrubbed `source_origin` provenance field** on the catalog rows (e.g. `online:vimeo` + a stable opaque source id — **never** a signed URL with embedded tokens, [ADR-M006](ADR-M006.md)/[ADR-0052](ADR-0052.md)). No new storage code.

6. **Licensing & ToS posture, stated neutrally.** `yt-dlp` is public-domain (Unlicense), subprocess-only, runtime-discovered, never vendored/linked → no copyleft/linking obligation, default build LGPL-clean ([ADR-0012](ADR-0012.md), mirroring [ADR-0008](ADR-0008.md)). Each service's ToS and the content's rights are **the operator's responsibility**, surfaced as an explicit UI acknowledgement (generalising [ADR-0015](ADR-0015.md) §7 across services and across *download*). Multiview bundles/claims **no DRM circumvention or signature solver** and names services nominatively only (the [managed-devices.md](../research/managed-devices.md) clean-room posture); DRM-protected content is out of scope. Cookies/auth are **reference-only** ([ADR-M006](ADR-M006.md)); `yt-dlp` is spawned argv-only with bounded/redacted stderr (`process.rs:63`).

7. **Surfaces are additive.** New source variants on the existing source union; a `[resolver]` config block; `POST /api/v1/library/import-from-url`; reuse of [ADR-0070](ADR-0070.md)'s `/storage` + offload endpoints; conflated drop-oldest resolver-health events; additive `#[non_exhaustive]` `WarningCode` variants (`resolver-unavailable` / `resolver-extraction-failing` / `resolver-stale`) at `GET /api/v1/health` (**not** `/livez`/`/readyz`). Existing configs keep parsing.

## Consequences

**Positive.**
- The operator's full ask (YouTube + Vimeo + general, ISO-record, download-to-VT, S3/Iceberg) is met by **generalising one proven seam + adding a fetch-to-file stage** — almost entirely reuse, vendor-neutral.
- Fragile per-service extraction stays in an **operator-updatable external process**, off the bulletproof engine; a service rotation degrades a tile + raises a warning, never the output.
- ISO/record-to-S3/Iceberg inherit fully-designed isolation; an online source is just another recorder input.

**Negative / costs.**
- Net-new: a resolver generalisation (extractor surface + VOD format pick + per-service TTL signal) and a download fetcher. Each service's live transport type (HLS vs DASH-only) and signed-URL expiry encoding must be verified per service (Open questions in the brief).
- Download-to-VT **depends on the unbuilt media-library import pipeline** ([ADR-0057](ADR-0057.md)); sequencing is explicit (live ingest + record-to-S3 ship first on built/designed machinery; download-to-VT with the library).
- A new ToS-acknowledgement surface and a provenance field add small operator-facing + privacy-review surface.

**Neutral.**
- Resolver health is a warning surface, never a liveness signal — a broken `yt-dlp` must not restart-loop the container ([ADR-R009](ADR-R009.md) precedent).

## Alternatives considered

1. **Hard-code per-service extraction (Vimeo signature logic) in Multiview.** *Rejected* — the highest-maintenance, most-breakage-prone code on a continuous-output engine; this is precisely what the subprocess boundary externalises ([ADR-0015](ADR-0015.md) rationale).
2. **A new "online services" ingest subsystem/crate.** *Rejected* — duplicates the shipped resolver + HLS path + recorder + offload + library. The right shape is *generalise the one seam + reuse the tails*.
3. **VOD-as-live for everything (skip download).** *Rejected as the default for a VT* — a media player wants seek/scrub on a finite local asset; perpetual re-pacing gives no transport. `vod_as_live` is offered explicitly for the "tile forever" case.
4. **Stream live online packets directly to S3 (skip the recorder).** *Rejected* — would couple the engine to network latency and re-implement the recorder; reuse [ADR-0037](ADR-0037.md)/[ADR-0070](ADR-0070.md) (offload closed files only).
5. **Vendor/bundle `yt-dlp` (or a Python runtime).** *Rejected* — pulls a Python dependency tree + licenses into the product, defeats the LGPL-clean default, and still needs updates for player churn. Runtime-discovery only ([ADR-0008](ADR-0008.md) precedent).
6. **A pure-Rust in-process service client.** *Deferred* (as in [ADR-0015](ADR-0015.md)) — moves each service's churn onto Multiview's release cadence; a future `resolver = yt_dlp | <in-process>` knob once one tracks services as reliably as `yt-dlp`.

## How this honors the invariants

- **#1 (output-clock untouchable).** Resolution, re-resolution, and download all run **off the data plane** with hard timeouts; a hung `yt-dlp` is killed, not awaited. A resolved live source is an ordinary PTS-paced input ([ADR-T004](ADR-T004.md)) — it is *sampled*, never pacing. The recorder/offload/catalog operate on **closed files**, sharing no channel/lock with the clock thread. `out_pts = f(tick)` is independent of any service's reachability, a resolver failure, a URL expiry, a download, a bucket, or a catalog.
- **#4 (HLS ingest pacing).** Live/VOD-as-live online ingest uses the **custom wall-clock-by-PTS pacer**, **never `-re`** ([ADR-T004](ADR-T004.md)).
- **#10 (isolation).** Resolver-health and import/offload status ride **conflated drop-oldest** channels; the engine never `.await`s the resolver, the fetcher, the uploader, or the catalog. A slow/failed service, download, bucket, or catalog degrades a tile / delays a job / leaves local copies pending + a warning — **never** a stall, never RAM growth.
