# ADR-0082: Render-to-VT — bake an HTML graphic to a media asset (alpha NV12+A clip)

- **Status:** Proposed
- **Area:** Compositor / Media (offline bake of a web graphic to a playable asset)
- **Date:** 2026-06-13
- **Source brief:** [motion-graphics-html.md](../research/motion-graphics-html.md)
- **Builds on:** [ADR-0080](ADR-0080.md) (the graphics engine — external BeginFrame, the surface contract), [ADR-0081](ADR-0081.md) (the data record / N-source contract a bake captures)
- **Extends:** [ADR-0058](ADR-0058.md) (NV12+A — the bake output payload), [media-playout](../research/media-playout.md) / [ADR-0057](ADR-0057.md) (media library import + players — the bake target and playout path)
- **Relates to:** invariant #1 / [ADR-T001](ADR-T001.md) (playout is sampled, never browser-paced), invariant #3 / [ADR-T015](ADR-T015.md) (exact-rational frame counts), invariant #8 / [ADR-C006](ADR-C006.md) (color order at convert), [ADR-0055](ADR-0055.md) (a baked full-canvas alpha clip can be a stinger), [ADR-E007](ADR-E007.md) (the bake is bounded control-plane work), [ADR-W021](ADR-W021.md) (the bake operation surface), [conventions §6](../architecture/conventions.md) (202 + operation id)

> **Vocabulary.** "VT" is the operator's term for a video-tape/clip slot; here it is a **media-player asset** ([media-playout](../research/media-playout.md)). "Render-to-VT" = render an HTML graphic off-line, frame-by-frame, into an alpha media clip and import it into the media library, so playout uses no browser.

## Context

The operator: *"maybe allow render-to-video so it can go into a VT."* A live HTML graphic ([ADR-0080](ADR-0080.md)) carries a recurring per-frame browser cost (layout/style/paint/JS/GC + a GPU surface + readback) and a determinism risk on air (a font-load race, a slow data fetch, a JS error). For any **deterministic** graphic — no live external data, or a **captured data take** — that cost and risk are avoidable by rendering **once**.

The substrate already exists: [ADR-0080](ADR-0080.md) drives the graphic by **external BeginFrame** (host-controlled frame stepping, verified via CEF `SendExternalBeginFrame`); [ADR-0058](ADR-0058.md) defines an **NV12+A** alpha payload and a bounded mezzanine; [media-playout](../research/media-playout.md)/[ADR-0057](ADR-0057.md) define an **import pipeline** (probe/policy/decode-verify, intra-alpha format allowlist, stinger conform) and **frame-accurate alpha playout** (output-anchored timeline, play-on-take, mezzanine-resident, zero decode on the hot path). Render-to-VT is the bridge between them.

## Decision

1. **Render-to-VT bakes a deterministic graphic to an NV12+A clip and imports it as a media asset.** A graphic + a **frozen data snapshot** (or "no live data," [ADR-0081](ADR-0081.md)) + a duration in **integer frames** (exact rationals, never float fps — inv #3 / [ADR-T015](ADR-T015.md)) is rendered off-line, frame-by-frame, into an NV12+A sequence and imported into the **media library** ([media-playout §4](../research/media-playout.md)). Playout is thereafter an **ordinary alpha media player** — frame-accurate, mezzanine-resident, **zero browser on the hot path**.

2. **The bake is a control-plane operation; it never touches the engine.** Like every import it returns **202 + operation id** with progress on the realtime stream ([conventions §6](../architecture/conventions.md), [ADR-W021](ADR-W021.md)); it runs on a control-plane/bake task with bounded parallelism, off every engine thread ([media-playout §4](../research/media-playout.md) import posture).

3. **The bake loop is paced by paint completion, not wall clock.** For frame `k`: issue BeginFrame `k`, **await paint `k`**, capture the premultiplied RGBA surface, convert to **NV12+A straight alpha** ([ADR-0058](ADR-0058.md) — the kernel premultiplies once in linear light at composite; the asset carries straight alpha), repeat. Because rendering is off-line, a slow frame simply takes longer to bake — there is **no real-time pressure** and no last-good substitution; every frame is the exact rendered frame. Color conversion follows the fixed order (sRGB EOTF → linear → encode space → quantize, inv #8 / [ADR-C006](ADR-C006.md)).

4. **Output is an intra-alpha mezzanine the import pipeline accepts.** Encode to `qtrle` (FFmpeg-native, intra, RLE, LGPL-clean — the [media-playout §4.3](../research/media-playout.md) default transcode target) / ProRes 4444 / PNG sequence; **never HEVC-with-alpha** (FFmpeg decodes it opaque — [ADR-0058](ADR-0058.md) §5/Alternative E). The import then runs the standard **probe/policy/decode-verify** ([media-playout §4](../research/media-playout.md)) and **stinger conform** (canvas-resolution + output-cadence equality), so a baked transition is take-ready and a wrong-cadence bake is rejected at import, not discovered at take.

5. **Playout reuses the alpha media player wholesale.** No new playout path: the baked asset is a still (one frame), a clip (a lower-third loop), or a full-canvas alpha clip usable as a **stinger transition** ([ADR-0055](ADR-0055.md)) — all already specified in [media-playout](../research/media-playout.md). Frame-accurate start, loop, EOF policy, and play-on-take are inherited unchanged.

6. **Scope boundary.** Render-to-VT is for the **deterministic** case. A graphic with genuinely live, un-bakeable data (a real-time score that updates on air) stays a **live** graphic ([ADR-0080](ADR-0080.md) monitoring/production tiers). A "bake the chrome/animation, inject live numbers over the played-back clip" middle path is attractive but couples a media player and a thin live graphic on one region — **deferred** (brief §7.6).

## Consequences

**Positive.**
- The browser cost is paid **once**, off-line; recurring on-air graphics that do not change between shows run with **zero browser, zero JS, zero GC, and no decode** on the hot path — a pooled `Arc` publish per frame ([ADR-0058](ADR-0058.md) §6). (Decoding the encoded mezzanine — `qtrle`/ProRes/PNG sequence, Decision 4 — happens at *import/preload*, not per tick; oversized mezzanines use the decode-ahead ring, never a synchronous hot-path decode.)
- **Determinism on air:** a baked take is byte-identical every play; no font-load race, no slow fetch, no on-air JS error.
- **Frame-accuracy for free** from the media player (output-anchored timeline + play-on-take, [media-playout §7](../research/media-playout.md)); a baked full-canvas alpha clip drops straight into the stinger transition path.
- Reuses the entire import + alpha-playout stack; the only new code is the off-line render loop + RGBA→NV12+A capture.

**Negative / cost.**
- The browser dependency is still required **at bake time** ([ADR-0080](ADR-0080.md)'s `html-graphics` feature) — render-to-VT does not remove the engine, it removes it from *playout*.
- Baked clips consume mezzanine RAM ([ADR-0058](ADR-0058.md) §6 budgets: a 5 s 1080p60 alpha clip is on the order of 1.5 GB un-trimmed) — admission-accounted and capped like any alpha media; UHD bakes need the decode-ahead ring.
- A re-bake is required whenever the graphic or its frozen data changes — the determinism is the point, but it means baked content is not live-editable.

**Neutral / deferred.**
- The "live numbers over a baked clip" hybrid (brief §7.6) is deferred.
- HDR-native baked graphics are out of scope for v1 (the NV12+A mezzanine is 8-bit, [ADR-0058](ADR-0058.md)).

## Alternatives considered

- **Always run the graphic live (no bake).** Rejected as the default for deterministic graphics: pays the browser cost every frame, every show, and carries the on-air determinism risk [ADR-0016](ADR-0016.md) flags for broadcast graphics. Live is correct only for genuinely live data.
- **Bake to RGBA frames instead of NV12+A.** Rejected: 4 B/px vs 2.5 B/px (+60% RAM at the resident mezzanine) and an RGB side-path through a YUV-centric pipeline — the [ADR-0058](ADR-0058.md) Alternative A rejection; NV12+A keeps the canonical currency.
- **Bake to HEVC-with-alpha (small files).** Rejected: FFmpeg decodes HEVC alpha **opaque** — a silently opaque stinger ([ADR-0058](ADR-0058.md) §5/Alternative E). Intra-alpha mezzanine only.
- **A bespoke render-to-VT playout path separate from the media player.** Rejected: the alpha media player already does frame-accurate, mezzanine-resident, loopable alpha playout ([media-playout](../research/media-playout.md)); a second path duplicates timeline, transport, and resilience for no gain.
- **Real-time (wall-clock-paced) bake.** Rejected: an off-line bake should be paced by **paint completion**, not the clock — a slow frame must not be dropped or substituted with last-good (that would corrupt the deterministic asset). Real-time pacing belongs to *live* playout, not to baking.

## How this honors the invariants

- **#1 (output clock):** the bake is entirely off-line on a control-plane task and never touches the engine; **playout** of the baked asset is the media player's sampled, `f(tick)` path — never browser-paced ([ADR-0080](ADR-0080.md)/[media-playout](../research/media-playout.md)). The output clock is untouched at both bake and playout.
- **#10 (isolation):** bake progress rides the realtime stream drop-oldest (202 + operation id); a failing/slow bake is a typed control-plane error, never an engine effect; nothing engine-side ever depends on a bake in flight ([media-playout §4/§14](../research/media-playout.md)).
