# ADR-0079: Switcher audio — monitor/PFL/AFL, AFV via the visibility graph, and output audio-track routing

- **Status:** Proposed
- **Area:** Audio / Production switcher
- **Date:** 2026-06-13
- **Source brief:** [switcher-audio.md](../research/switcher-audio.md)
- **Extends:** [ADR-0059](ADR-0059.md) (AFV from the state machine; master/FTB; per-bus audio),
  [ADR-0077](ADR-0077.md) (strips + buses + send matrix), [ADR-M004](ADR-M004.md) (per-output audio
  mapping + capability projection), [ADR-R005](ADR-R005.md) (per-output capability matrix)
- **Relates to:** [ADR-0054](ADR-0054.md)/[ADR-MV006](ADR-MV006.md) (the program composition graph
  AFV reads), [ADR-0055](ADR-0055.md) (transition windows), [ADR-0034](ADR-0034.md) (RT-9
  equal-power crossfade), [ADR-P007](ADR-P007.md) (PVW ≠ preview subsystem),
  [ADR-W021](ADR-W021.md)/[ADR-RT008](ADR-RT008.md) (REST + realtime), [ADR-R010](ADR-R010.md)
  (Class-2 make-before-break for track-layout changes), [ADR-T001](ADR-T001.md)/[ADR-RT004](ADR-RT004.md)
  (invariants #1/#10)

## Context

Three operator needs sit on top of [ADR-0059](ADR-0059.md)/[ADR-0077](ADR-0077.md): (a) **monitoring**
— check any audio (PFL/AFL/solo) **without changing Program**; (b) **AFV correctness** — audio must
follow what is actually *visible* in Program (PiP/SuperSource/keyers/stingers), not merely the
selected Program source id; and (c) **output audio-track routing** — Program/ISO/clean/language/return
tracks routed per output, validated against each transport's real audio capability. [ADR-0059](ADR-0059.md)
§2 gives AFV from the state machine but with a "direct Program source equality" MVP and a noted
evolution to the visibility graph; §5 carries program audio on the clean feed; per-output `OutputAudio`
consumption + discrete-track carriage are named-deferred work-schedule items.

## Decision

1. **Monitor / PFL / AFL** ([switcher-audio.md §10](../research/switcher-audio.md)). Control-room +
   headphone monitor busses (`AudioBusKind::{Monitor, Pfl, Afl}`); PFL (pre-fader) and AFL
   (after-fader) listen; solo clear; multi-solo (summed into the monitor bus only); monitor dim/mute;
   optional mono check. **Non-destructive by construction:** PFL/AFL/solo only ever route into the
   *monitor* bus and **never** touch Program/stream/record/return busses (inv #10 — the monitor bus
   is a consumer-side `ProgramBus` like any other). PVW/preview selection may *optionally* cue the
   **monitor** bus (`follow_preview_audio_to_monitor`) but **never** Program audio (the
   [ADR-P007](ADR-P007.md) PVW-vs-preview vocabulary holds: selecting on PVW is not on-air).
2. **AFV via the program visibility graph.** AFV gain is derived from whether a strip's linked video
   is *visible in Program*, using the same per-tick composition graph [ADR-MV006](ADR-MV006.md)
   derives tally from (contributes-to-program), not source-id equality — so PiP/SuperSource/DVE/keyer/
   stinger sources are handled correctly. The MVP "direct equality" of [ADR-0059](ADR-0059.md) §2 is
   the degenerate single-box case; this ADR pins the graph-derived general case as the target.
   Transitions ramp audio even for cuts (20–50 ms de-click) and crossfade on mix/dissolve via the
   BUILT [ADR-0034](ADR-0034.md) RT-9 equal-power crossfade / [ADR-0059](ADR-0059.md)
   `repoint_crossfade`, with sample-exact window alignment from `total_at` deltas
   ([ADR-T015](ADR-T015.md)). PVW/preview selection produces **no** Program audio change.
3. **Output audio-track routing** ([switcher-audio.md §14](../research/switcher-audio.md)). Video and
   audio output routing are **independent**: `AudioTrackRoute { track_id, source_bus, codec,
   channel_layout, language, label }`. Routes are **validated against the transport's audio
   capability** before going live — extending the [ADR-R005](ADR-R005.md)/[ADR-M004](ADR-M004.md)
   per-output capability matrix with the per-transport rules (TS = multiple PIDs; HLS = alternate audio
   renditions (RFC 8216 `EXT-X-MEDIA` groups) of which a typical client selects one at playback, so
   simultaneous multi-track is not portable; RTMP = one track unless the endpoint supports more; NDI =
   one multichannel stream; AES67/ST2110-30 = separate PCM busses; WebRTC/SIP = per-participant
   returns).
   The **clean feed reuses the program audio mix** unchanged ([ADR-0059](ADR-0059.md) §5).
   **Changing the number/identity of encoded audio tracks is Class-2** (make-before-break,
   [ADR-R010](ADR-R010.md), classified by [ADR-M012](ADR-M012.md)); selecting which bus a track
   carries is Class-1.

## Rationale

- **AFV must follow visibility** — visibility *gates/weights* AFV by policy, not by a fixed rule: by
  default a source that is only a small PiP need not be full-Program-on, but the visibility→level
  policy (threshold, and whether coverage/opacity weight the AFV gain or it is full-on once visible) is
  **configurable per strip**, since many shows keep a PiP participant at full level. Reading the same
  composition graph as tally gives one source of truth and sample-exact alignment (the
  [ADR-0059](ADR-0059.md) §2 rationale, generalized).
- **Monitoring must be inert** — making PFL/AFL/solo physically route only into the monitor bus makes
  "solo changed Program" unrepresentable (inv #10).
- **Routing validated, not hoped** — the transport-capability matrix already exists
  ([ADR-R005](ADR-R005.md)); audio-track routes plug into it so an impossible route (8 selectable
  RTMP tracks) is refused before air, with an honest warning.

## Alternatives considered

- **A — AFV by selected-Program-source-id only.** Kept as the MVP degenerate case but rejected as the
  target: wrong for PiP/SuperSource/keyers/stingers — the operator asked for everything-on-air audio.
- **B — Solo that mutes Program / swaps the program source.** Rejected: monitoring must never be
  on-air; PFL/AFL route only into the monitor bus.
- **C — Track routing without capability validation.** Rejected: silently-dropped or
  endpoint-rejected tracks are exactly the [ADR-R005](ADR-R005.md) class of failure the matrix exists
  to prevent.

## Consequences

- **Positive.** Operators can audition anything safely; AFV is correct under real switcher
  composition; output audio tracks are routed per output and validated per transport; closes the
  [ADR-0059](ADR-0059.md) PFL/monitor "out of scope (roadmap M4)" and per-output `OutputAudio`
  deferrals with a designed path.
- **Negative / cost.** The monitor bus is another consumer-side bus; AFV graph derivation adds a
  per-tick read of the composition graph (already computed for tally, [ADR-MV006](ADR-MV006.md) —
  shared, not recomputed); the capability matrix grows audio-track rows.
- **Risks.** Visibility-graph AFV under heavy unions must not add hot-loop cost — it consumes the
  already-derived graph, never recomputes it. Track-layout Class-2 changes use the
  [ADR-R010](ADR-R010.md) make-before-break primitive so an output session is never silently broken.
- **Deferred (named).** Phase/correlation metering and full multi-output language/clean variants are
  post-MVP ([switcher-audio.md §20](../research/switcher-audio.md)); MVP ships one monitor bus +
  graph-aware AFV (falling back to direct-equality where no composition graph is active) + Program/ISO
  output routing.
