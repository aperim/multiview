# ADR-0061: Per-process vs system-wide resource attribution (encoder/decoder/GPU/CPU) + a stats UI

- **Status:** Proposed
- **Area:** Telemetry / HAL / Control / Web
- **Date:** 2026-06-13
- **Source brief:** [system-stats-attribution.md](../research/system-stats-attribution.md)
- **Builds on:** [ADR-0035](ADR-0035.md) (self-aware placement — the `HealthWarning` model,
  `GET /api/v1/health`, the ours-vs-total `self_*` signals, the silent GPU→CPU-fallback
  detector), [ADR-0017](ADR-0017.md) (the per-device live-load probe + per-GPU gauges +
  the verified vendor-asymmetry of per-engine metrics), [ADR-0018](ADR-0018.md) (placement
  reasons net-of-co-tenant load — this is its operator-facing mirror).
- **Relates to:** invariant #1 (output clock untouchable), invariant #10 (no back-pressure),
  invariant #6 (decode-at-display-res — unaffected), invariant #8 (color order — unaffected),
  [ADR-RT004](ADR-RT004.md) (conflated realtime exposure), [ADR-W011](ADR-W011.md)
  (WCAG value+shape, never colour-alone).

> **Scope guard (load-bearing).** This ADR adds an **attribution/stats surface**, not a new
> warning model and not a placement change. [ADR-0035](ADR-0035.md) owns the `HealthWarning`
> catalog and emission; [ADR-0018](ADR-0018.md) owns placement decisions. This ADR consumes
> both, adds a read-only stats page + the per-co-tenant attribution data behind it, and
> *links* warnings to the charts that explain them. It must not re-define `WarningCode` nor gate
> any engine decision.

## Context

The operator's ask: **system stats must visually distinguish *this* instance's load from
everyone else's — who is on the encoder (us or a co-tenant NVR?), why is the CPU pegged (is that
us?), and is the idle GPU idle because we *can't* use it or because we *aren't*?** The motivating
incident is the [ADR-0035](ADR-0035.md) one: the wgpu compositor silently fell back to CPU, ~5
cores burned, and the RTX 4060 sat ~80 % idle — with no way to see, at a glance, that the CPU
load was *ours* and the GPU was *available*.

Most of the attribution **plumbing already ships** and is verified in-tree:

- `GpuMetrics` (`crates/multiview-events/src/event.rs:148`) carries, per GPU, the **device-wide**
  totals *and* our share: `self_compute_util` (`:185`), `self_encoder_util` (`:188`),
  `self_decoder_util` (`:191`), `self_encoder_sessions` (`:197`), alongside `encoder_sessions`
  (`:172`) and `encoder_session_ceiling` (`:175`).
- `SystemMetrics` (`:206`) carries whole-host `cpu_util` (`:209`) **and** our `self_cpu_util`
  (`:220`).
- The probe that fills them ships: `DeviceLoad` (`crates/multiview-hal/src/load.rs:220`),
  `SelfShare` (`:320`), the NVML per-process pass (`:893`–`:956`), and the DRM fdinfo media
  tracker (`:1196`). The off-engine poller (`crates/multiview-cli/src/system_metrics.rs`,
  `SAMPLE_PERIOD:35`) publishes `SystemMetrics` through the drop-oldest publisher.
- The alert surface ships too: `HealthWarning`/`WarningCode` (`event.rs:319`,`:357`),
  `GET /api/v1/health` (`crates/multiview-control/src/routes/health.rs:67`), `warning_ingest` +
  `WarningRepository`.

What is **missing** is the surface that turns those ours-vs-total numbers into the operator's
answer, plus two attribution gaps:

1. **No stats page.** The numbers are on the wire but there is no Frigate-style page that renders
   the ours/co-tenant/free split per resource (CPU, GPU compute, NVENC, NVDEC, VRAM).
2. **No per-co-tenant breakdown.** We know *our* share and the *host total*, but not *which other
   process* is the culprit (the "that's not us — so who?" answer).
3. **No cgroup-v2 CPU denominator.** Inside a container "host CPU 95 %" misleads when our cgroup is
   limited and at its own cap but the host is busy with *other* cgroups.

External attribution sources verified (web, 2026-06-13): NVML `nvmlDeviceGetProcessUtilization`
(`smUtil`/`encUtil`/`decUtil` per PID) + `nvmlDeviceGetMemoryInfo` + `nvmlDeviceGetEncoderStats`;
Linux DRM fdinfo `/proc/<pid>/fdinfo/<drm_fd>` `drm-engine-*` per-process counters (i915 + amdgpu,
amdgpu media merged from VCN4); Linux `/proc/<pid>/stat` CPU + cgroup-v2 `cpu.stat` `usage_usec`;
the Frigate per-process + per-GPU + detector stats model as a **conceptual reference only**.

## Decision

Add a **read-only resource-attribution stats surface** that renders, for every resource, the
unified triple **`(ours, co_tenant, free)`** — and the per-co-tenant attribution data behind it —
consuming the already-shipped `self_*` signals and the existing off-engine poller. Five parts:

1. **The unifying model (pure).** For each resource compute `ours` (the `self_*` share),
   `device_tot` (the authoritative device/host total), `co_tenant = max(device_tot − ours, 0)`,
   and `free = max(capacity − device_tot, 0)`. **`co_tenant` is derived**, so it is **Unknown**
   whenever either operand is unknown — it renders as an "un-attributable" band, never a fabricated
   zero. This is a pure function of one sample, unit-testable with injected metrics (mirroring the
   existing `map_gpu`/`map_system`).

2. **The per-co-tenant roll-up (NEW, sampled, bounded).** Beyond the scalar split, emit a bounded
   **top-N** roll-up naming the biggest *other* consumers: CPU top-N from `/proc/<pid>/stat`
   deltas; NVIDIA per-PID from `nvmlDeviceGetProcessUtilization`/running-process VRAM (where the
   GeForce caveat permits, else one "other (un-attributable)" entry); Linux GPU media from the
   non-self fdinfo walk. **"Ours" is the whole instance defined as a PID set** (our process group /
   cgroup membership via `/proc/self/cgroup`), not just `/proc/self` — the walk excludes that entire set
   so a child/helper we spawn is never mis-named as a co-tenant (the as-built `self_cpu_util` covers only
   the current process). Top-N bounds **output and memory** (a bounded heap), not the per-PID scan, so
   cost is bounded by cadence, not N. Carried **conflated, latest-only, drop-oldest** through the existing
   engine publisher alongside `SystemMetrics`; default N≈5; computed on the existing off-engine
   poller, never on the clock loop.

3. **The cgroup-v2 CPU denominator (NEW, Linux).** Read our cgroup's `cpu.stat` `usage_usec` +
   `cpu.max` (via `/proc/self/cgroup`) so the page can state "**our cgroup** 1.9/2.0 cores;
   **host** 95 % — the saturation is **outside our limit**." The "outside our limit" verdict is asserted
   **only** when our usage is below quota and `cpu.stat` `nr_throttled`/`throttled_usec` is *not* climbing;
   if our cgroup is at/near its `cpu.max` we are saturated *within our own quota* and contributing to host
   load, so the page surfaces four distinct quantities — our-cgroup utilization, quota pressure/throttling,
   host utilization, inferred other-host remainder — and never claims "not us" while our quota is the
   binding constraint. Read-only std file I/O; `None` off-Linux / outside a cgroup.

4. **Three additive wire fields (non-breaking).** `self_vram_used_bytes: Option<u64>` on
   `GpuMetrics` (map the already-sampled `SelfShare.mem_used_bytes`, `load.rs:333`, onto the wire);
   the co-tenant roll-up field/event; `cgroup_cpu_usage_frac`/`cgroup_cpu_quota_frac: Option<f32>`
   on `SystemMetrics`. All `Option`/`Vec`, registered in the hand-listed `asyncapi.rs` +
   `openapi_schemas.rs`.

5. **The stats page (NEW UI).** A Frigate-style **data/graphics** page (not the layout editor):
   a CPU band (ours/co-tenant/free + named top-N + the cgroup line), per-GPU cards with four
   stacked ours/co-tenant/free bars (compute, NVENC, NVDEC, VRAM) — **collapsed to a single combined
   "media" bar (three bars total) on merged-media parts (AMD VCN4) so a merged figure is never
   double-presented as separate encoder/decoder bars** — plus the `self/total/ceiling` session counter
   (ceiling shown as a discovered/estimated `Option`, never an authoritative cap), the "what runs where"
   placement join (consuming [ADR-0035](ADR-0035.md) §8's
   `placement.snapshot`), and a per-tile cost ("detector-style") chart. Renders at the existing
   ~1 Hz cadence from the realtime stream (no REST polling); colour is never the sole signal
   (value+shape+text per [ADR-W011](ADR-W011.md)); a blind vendor greys to "n/a", never a false
   zero. Each active `HealthWarning` links to the chart that explains it (the
   `gpu-present-no-vulkan-adapter` warning *is* the idle-GPU card; `sustained-cpu-saturation` *is*
   the CPU band, with the roll-up naming the culprit).

## Consequences

**Positive.**
- The operator's "is that us?" question is answered at a glance, across CPU/GPU/encoder/decoder/VRAM,
  on Multiview's own page — no SSH + `nvidia-smi` + `htop` PID-correlation.
- It reuses the shipped attribution model (one source of truth) and the net-of-co-tenant quantity the
  placement engine already reasons on — the page is the planner's operator-facing mirror.
- The silent GPU→CPU fallback becomes a *visible chart* (the idle-GPU card) wired to its remediation
  warning; the encoder-ownership question becomes a literal `self/total/ceiling` counter.
- Read-only and additive: no engine decision changes, no breaking wire change.

**Negative / cost.**
- New public surface: three additive `multiview-events` fields/an event, a cgroup + top-N `/proc`
  reader on the poller, control-plane wiring for the roll-up, and a new web page. The new event
  fields must be added to the hand-listed AsyncAPI/OpenAPI registries or codegen drifts.
- The per-co-tenant `/proc` (and optional all-PID fdinfo) walk is a real cost on big multi-tenant
  boxes — mitigated by a bounded single-pass top-N, a config-exposed N/cadence, and a coarsened
  co-tenant tick (the headline ours-vs-total split, already on the wire, costs nothing new).
- On consumer GeForce, `nvmlDeviceGetProcessUtilization` is often `NOT_SUPPORTED`, so the per-co-tenant
  NVENC breakdown degrades to "other (un-attributable)" — honest, but less specific.

**Risks + mitigations.**
- *Fabricated co-tenant numbers.* Prevented structurally: `co_tenant` is derived and inherits
  Unknown; a blind per-process source yields an "un-attributable" band, never a fake per-PID list.
- *Back-pressure (inv #10).* The roll-up is bounded + conflated and rides the existing drop-oldest
  publisher; the page consumes the conflated channel and the engine never awaits it. A chaos/soak
  test asserts flooding the stats/roll-up events while subscribers stall delays no output tick.
- *Probe cost on tiny/huge boxes.* Config-gated N + cadence; default to cheap paths (i915-PMU device
  total + our-fdinfo) and make the all-PID GPU walk opt-in.
- *cgroup mount-path variance (unverified).* Resolve via `/proc/self/cgroup`; default to `None`
  when unresolved (the host-total CPU answer still works); validate on the deploy images.
- *DRM fdinfo correctness (Linux GPU media).* Per [L1], the busy fraction must divide the
  `drm-engine-<class>` ns delta by `wall-clock × drm-engine-capacity-<class>` where a capacity counter
  is present (a class may back several physical engines), and the counters are **not guaranteed
  monotonic** — a negative delta clamps to zero / drops to Unknown, never wraps into a false spike.

## Alternatives considered

1. **No dedicated page — rely on Prometheus + Grafana.** [ADR-0017](ADR-0017.md) already exposes
   `multiview_gpu_*` gauges; a Grafana board could show them. Rejected as the *primary* answer: it
   needs external infra, does not ship in-app, and per-PID co-tenant detail blows up label
   cardinality. A bounded scalar gauge stays available; the *attribution* detail is realtime-only.
2. **Re-derive attribution in the UI from raw totals.** Rejected — the engine already computes the
   `self_*` split honestly (Option-typed, unknown-preserving); re-deriving client-side would
   duplicate logic and risk disagreeing with the placement engine's net-of-co-tenant quantity.
3. **Mutate the `HealthWarning` model to carry the numbers.** Rejected — warnings are the *alert*
   surface (latched/dwelled events), not a continuous metric stream. The page is the continuous
   surface; it *links* to warnings rather than overloading them (mirrors the ADR-0035 "sibling, not
   mutation" posture).
4. **Gate placement on the new per-co-tenant attribution.** Rejected — placement is
   [ADR-0018](ADR-0018.md)'s, already net-of-co-tenant via device-wide totals; per-PID co-tenant
   detail is for the *operator's* understanding, not a new gate term. This surface is read-only.
5. **Whole-`/proc` co-tenant table (full Frigate-style process list).** Rejected as default — a
   full sorted table at ~1 Hz on a busy host is costly and high-cardinality. Bounded top-N answers
   "who's the biggest other consumer?" at a fraction of the cost.
6. **Skip cgroup, use host `/proc` only.** Rejected — inside a container the host total misattributes
   co-tenant load; the cgroup denominator is exactly the "that's not us, that's outside our limit"
   signal the operator asked for. It is cheap (one file) and `None`-safe off-Linux.

## How this honors the invariants

- **Invariant #1 (output clock untouchable).** Nothing here runs on the output-clock loop. Sampling
  (the new cgroup read + top-N `/proc` walk) is on the existing off-engine poller; derivation is a
  pure function of one sample; the page is a *read* of the realtime stream. A dead poller leaves the
  engine ticking on its last sample. No path blocks, paces, or de-paces program output.
- **Invariant #10 (no back-pressure).** Every number is emitted through the **same drop-oldest
  `EnginePublisher::publish_event`** as `SystemMetrics`; the co-tenant roll-up is bounded (top-N) and
  conflated (latest-only); the web page consumes the conflated realtime channel and the engine never
  awaits it. A CI chaos/soak test asserts that flooding the stats + roll-up events while every
  subscriber is stalled delays no output tick.
