## Context

The production switcher ([ADR-0054](ADR-0054.md)) needs bus-selectable media:
stills (slates, logos-with-alpha for DSK fills per [ADR-0056](ADR-0056.md)),
clips (openers, loops, VT rolls), and stinger media
([ADR-0055](ADR-0055.md) Wave 3). Nothing like a media library exists in the
repo today — no still `SourceKind`, no asset storage, no import pipeline, no
transport control (all verified by grep across `crates/` and `docs/`).

What **is** built is about a third of a media player, on the production CLI
ingest path (all paths verified in this worktree):

- **File playout, paced and held** (BUILT). `SourceKind::File { path }` carries
  *only* a path (`crates/multiview-config/src/schema.rs:341`) and maps to a
  non-live ingest plan (`crates/multiview-cli/src/pipeline.rs:4925`,
  `live: false`). The per-source decode thread (`ingest_loop`,
  `pipeline.rs:5533` → `open_and_stream`, `pipeline.rs:5812`) paces each decoded
  frame to wall-clock by PTS (`PtsWallClock`, `pipeline.rs:6189` — inv #4,
  [ADR-T004](ADR-T004.md)) and publishes it stamped with **source-relative media
  time** (`publish_time`, `pipeline.rs:6241`), which the output clock latches
  per tick (`TileStore::read_at`,
  `crates/multiview-framestore/src/tile.rs:421`). On EOF the thread returns
  (`pipeline.rs:5575-5579`) and the tile **holds its last frame forever** — every
  source store is built `NoSignalPolicy::HoldForever` (`pipeline.rs:1187`).
- **A seek primitive with zero callers** (BUILT, unwired). `Demuxer::seek`
  exists (`crates/multiview-ffmpeg/src/demux.rs:549`, a keyframe-bracketed
  `Input::seek`) and is called from **no** production code path.
- **Prime gating** (BUILT). `TileStore::is_primed`
  (`crates/multiview-framestore/src/tile.rs:311`) + the bounded startup
  prime-wait (`wait_for_prime`, `pipeline.rs:4062`, hard-capped at
  `PRIME_WAIT_BUDGET` = 1 500 ms, `pipeline.rs:5249`) already express
  "cued/ready" as a queryable state.
- **Live source lifecycle** (BUILT). [ADR-W018](ADR-W018.md)'s `LiveSourceHub`
  (`crates/multiview-cli/src/live_sources.rs:189`) spawns/tears down producer
  threads off-thread while the frame-boundary drain does only O(1) store-binding
  mutations.
- **A parallel-implementation trap** (SCAFFOLD — named so nobody builds on it).
  `multiview-input`'s `IngestPump`/`FrameProducer`/`Pacer`
  (`crates/multiview-input/src/source.rs:169`/`:110`) is a built-and-tested
  ingest core that the production binary **does not run** — the CLI pipeline's
  `ingest_loop` is a divergent twin that carries the real behaviour (captions,
  reconnect supervision, prime gating, audio). Media players **must** extend the
  production path; extending the pump would fork transport semantics across two
  ingest engines.

Missing entirely: cue/play/pause/seek/loop transport, EOF policy, frame-accurate
start, play-on-take, stills, asset management, and import validation.

## Decision

1. **Library ≠ players.** The **media library** is desired-state asset storage —
   stills, clips, audio files, plus the import/validation/transcode pipeline —
   a config/control-plane resource set ([ADR-M012](ADR-M012.md)
   `media_library`). **Media players** are a bounded set of player *channels*:
   each is an ordinary bus-selectable source that publishes into its own
   `TileStore` (and `AudioStore` when the loaded asset has audio), rides the
   existing crosspoint/rebind/slate machinery, and is tallied like any source
   ([ADR-MV006](ADR-MV006.md)). Player **live state** (loaded asset, transport
   state, position) is engine-owned and mirrored read-only — never persisted in
   config. Vocabulary is pinned generic: *media library*, *media player*,
   *still store* (the still-asset subset). This separation is de-facto industry
   practice across production switchers and playout servers.

2. **Asset model.** An asset = `{ id, name, kind: still|clip|audio, path }` plus
   **probed** metadata recorded at import: container/codec, resolution, frame
   rate (exact `Rational`), duration in **integer frames** at the asset's
   cadence, `has_alpha` (per the [ADR-0058](ADR-0058.md) format table), embedded
   audio (channels/sample rate), optional in/out loop points (frames), and — for
   stinger-eligible clips — a **mandatory** `trigger_point` (frames), declared at
   import; the `trigger_point + mix_rate ≤ duration` window is validated **at arm
   time** per [ADR-0055](ADR-0055.md) (`mix_rate` is a per-transition parameter
   that does not exist at import). A content
   checksum pins the probe to the bytes; a changed file re-probes on next load.
   No float fps/seconds anywhere ([ADR-T015](ADR-T015.md), inv #3).

3. **Probe at import; reject-or-transcode.** Import is a control-plane
   long-running operation (`202` + operation id, [ADR-W021](ADR-W021.md)) that
   runs **off the engine** (inv #10): probe via the existing demux inventory
   (`Demuxer::inventory`, `crates/multiview-ffmpeg/src/demux.rs:375`), then
   either **accept** (decodes natively on the production path), **transcode**
   into a normalized on-disk mezzanine (e.g. alpha formats FFmpeg decodes as
   opaque — [ADR-0058](ADR-0058.md)), or **reject** with a typed validation
   error. Stinger eligibility is validated **at import, never at take time**:
   exact canvas resolution, exact output frame rate, alpha per ADR-0058, and an
   explicit `trigger_point`. A take can therefore never discover an unusable
   asset. Phasing: **import-transcode is post-MVP** — the MVP import pipeline
   rejects with remediation guidance only
   ([media-playout §16](../research/media-playout.md)); the transcode lane lands
   later behind the same `202` operation surface.

4. **Player transport extends the production ingest thread** (the CLI
   `ingest_loop`/`open_and_stream` path — explicitly **not** the unwired
   `multiview-input` `IngestPump` scaffold). The ingest thread already polls an
   atomic per frame (`stop`, `pipeline.rs:5966`); transport adds a per-player
   wait-free command slot (latest-wins, written by the frame-boundary command
   drain, read between frames) — the engine never awaits the player and the
   player never paces the engine (inv #1/#10). Verbs:
   - **load(asset)** — bind an asset to the channel; spawn/replace the producer
     via the `LiveSourceHub` pattern ([ADR-W018](ADR-W018.md)); Class-1.
   - **cue([frame])** — open + seek + decode the target frame + pause: the first
     frame is published (so PVW shows it) and the channel reports *cued* once
     `is_primed` — the WARM-ON-ARM posture of [ADR-P004](ADR-P004.md), making a
     take honest rather than cold-starting a decoder.
   - **play / pause** — gate `PtsWallClock`: pause stops releasing frames;
     resume shifts the pacer anchor by the paused duration so playback continues
     without a catch-up burst (the pacer's re-anchor discipline,
     `pipeline.rs:6205-6208`, already proves anchor moves are safe). While
     paused, the channel **heartbeat-republishes** the held frame at cadence
     with advancing stamps (`publish_arc`,
     `crates/multiview-framestore/src/tile.rs:253` — one `Arc` clone per
     period) so it reads LIVE, never STALE, on the freshness ladder
     (media-playout §7.3).
   - **stop** — halt playout and re-cue to the in-point (the cued first frame
     republishes, so PVW shows the top of the clip); distinct from `pause`,
     which holds the current frame.
   - **seek(frame)** — the first production caller of `Demuxer::seek`
     (`demux.rs:549`): keyframe-bracketed seek, decode-forward to the exact
     frame (frame-accurate), publish it, re-anchor normalizer + pacer.
   - **loop** — EOF with `loop` policy seeks to the in-point in place
     (`Demuxer::seek` + decoder flush) and keeps decoding; a failed seek falls
     back to a clean re-open via the existing supervised reconnect bracket in
     `ingest_loop` (`pipeline.rs:5561-5590`) — the proven recovery shape.
   - **EOF policy** `hold_last_frame` (default — the as-built `HoldForever`
     behaviour) | `loop` | `black` (publish one **terminal frame** — opaque
     black for opaque assets, **fully transparent** for alpha assets, so a
     keyed fill cleanly vanishes instead of painting a black card over program
     — and hold) | `auto_off` (publish the same terminal frame, report
     `Ended`, release decoder/mezzanine, channel idle; the **switcher state
     machine — not the player — applies the bus/keyer consequence**: a DSK
     fed by the channel drops off-air-safe per [ADR-R011](ADR-R011.md), a bus
     selection falls to the cell's `on_loss` slate). Terminal-frame alpha
     semantics per media-playout §7.4.
   - **play_on_take** — roll-clip semantics: a cued player armed on a bus
     transition starts playing at the take tick (the transition state machine
     issues the play command at its start tick — [ADR-0055](ADR-0055.md)).
   - **Frame-accurate start** — cue at frame *N*; on play the pacer anchors at
     the current tick, so frame *N* is on air at the take tick and subsequent
     frames advance 1:1 with output media time via latch-on-tick. Player frames
     are stamped **output-anchored** — `anchor_pts + k × frame_period`, where
     `anchor_pts` is the output media time of the start tick, monotonic across
     loop laps (media-playout §7.2). This deliberately differs from the boot
     path's source-relative `publish_time` (`pts − first_pts`,
     `pipeline.rs:6241`), which is correct only because boot ingest starts at
     run start — it would be wrong for a player cued mid-asset (e.g. minute 40).
     Stingers are stronger still: mezzanine frames are indexed directly by tick
     offset ([ADR-0058](ADR-0058.md)).
5. **Stills = decode-once + HoldForever + heartbeat.** A still asset loads by
   decoding one frame via libav and publishing it once; the store's
   `HoldForever` policy (the same path that already keeps a finished file's
   last frame on screen forever) serves it indefinitely with **no decode and
   no pacing**. A held channel (still, paused, or EOF-held) additionally
   **heartbeat-republishes** the held `Arc` at cadence with advancing stamps
   (`publish_arc`, `crates/multiview-framestore/src/tile.rs:253`) so the
   freshness ladder, readiness, tally, and fault reporting stay honest —
   steady-state cost: one `Arc` clone per cadence period (media-playout
   §7.3/§8/§13). Stills with alpha publish the NV12+A payload per
   [ADR-0058](ADR-0058.md).

6. **Embedded audio rides the existing per-source audio path.** A player whose
   asset has audio gets a per-source `AudioStore` + decode thread exactly like a
   file source today (`audio_ingest_loop` spawn, `pipeline.rs:3917`), routed
   onto the program bus under the [ADR-0059](ADR-0059.md) seam; AFV and
   play-on-take audio start/fade with the same transition window as video;
   stinger embedded audio mixes over the background crossfade. Silent assets
   contribute silence (the bus's existing posture).

7. **Protocol posture.** v1 exposes the **native API only** (REST bare verbs per
   [ADR-W021](ADR-W021.md): `/api/v1/media/players/{id}/load|cue|play|pause|stop|seek`;
   events on the realtime stream per [ADR-RT008](ADR-RT008.md)). The transport
   semantics deliberately match the openly published CasparCG AMCP model —
   cue-shows-first-frame, background-load auto-chain — cited as a de-facto
   reference, not a wire-compatibility target. VDCP and AMP are legacy
   RS-422-era external-playout-server protocols: **out of scope**, revisited
   only on a concrete interop demand.

8. **Admission.** A playing/cued clip player costs one decode (a source like
   any other) and is admission-accounted ([ADR-E007](ADR-E007.md)); a held
   still or an EOF-held clip costs only the heartbeat republish (one `Arc`
   clone per cadence period — no decode, no pacing); mezzanine RAM is capped
   per [ADR-0058](ADR-0058.md). Player count is config-bounded.

## Rationale

- **Maximum reuse of the proven path.** Pacing (inv #4), source-relative
  timestamps + latch-on-tick (inv #3), hold-last-good (inv #2), prime gating,
  supervised reconnect, live add/remove, and per-source audio all exist and are
  production-exercised on the exact thread the player extends. The net-new
  surface is a transport state machine on that thread plus the library/import
  resources — no new data-plane channels, no new invariant exposure.
- **Library/player separation is load-bearing**, not cosmetic: assets are
  durable desired state (config-as-code exportable), player state is live
  engine state (Device-registry precedent) — collapsing them would either
  persist transport positions into config or make assets unaddressable from
  multiple channels.
- **Import-time validation is the only honest place to fail.** A stinger that
  rejects at take time defeats the entire point of a pre-validated transition
  ([ADR-0055](ADR-0055.md) arms are admission/validation points, takes are not).
- **Invariant posture:** players are *sampled* per tick like every source —
  they never pace (inv #1); transport commands cross on wait-free latest-wins
  slots (inv #10); a stalled/underrunning player degrades through the tile state
  machine and its EOF policy (inv #2, [ADR-R011](ADR-R011.md)).

## Alternatives considered

- **A — Players as a new ingest engine on the `multiview-input`
  `IngestPump`/`FrameProducer` scaffold.** Rejected: the pump is an unwired
  parallel twin of the production `ingest_loop` (zero CLI consumers — verified);
  building transport there would ship a second, divergent ingest implementation
  with its own timing/reconnect/caption behaviour. The repo already carries this
  divergence as a named trap; media players must not widen it.
- **B — Decode-on-take (no cue step).** Rejected: cold-start dishonesty. Taking
  an uncued clip pays open+probe+decode latency on air; cue/pre-warm
  ([ADR-P004](ADR-P004.md) — "one machinery for look and take") makes readiness
  a queryable, prime-gated state instead of a hope.
- **C — Unbounded in-memory assets (pre-decode the whole library to RAM).**
  Rejected: a handful of 1080p60 clips exhausts memory
  (≈ 5.18 MB/frame NV12+A — [ADR-0058](ADR-0058.md) budgets). Only
  stinger-eligible clips pre-decode into the bounded mezzanine; everything else
  streams from disk through the paced decode thread under hard caps,
  pool-allocated at load (safety rule 5).
- **D — Stills via the overlay image cache instead of sources.** Rejected:
  overlays are input-decoupled by design ([ADR-R008](ADR-R008.md)) and baked
  off-thread on the egress consumer — not bus-selectable, not tally-derivable,
  not frame-coordinated with switcher state ([ADR-0054](ADR-0054.md)). A still
  is a *source*.
- **E — VDCP/AMP transport control in v1.** Rejected: proprietary-held /
  vendor-specific RS-422-era protocols for external playout servers; nothing in
  scope drives Multiview's players from broadcast automation today. The native
  API + AMCP-style semantics cover the operator need.

## Consequences

- **Positive**
  - Stills, clips, loops, VT rolls, and stinger media become ordinary
    bus-selectable sources with honest cue/ready state, frame-accurate starts,
    and configurable EOF behaviour — the full MVP media scope of
    [media-playout §16](../research/media-playout.md) (library + 2 players:
    stills + clips, loop, play-on-take, EOF policy).
  - Take-time failures are designed out: every stinger/asset is validated,
    probed, and (where needed) transcoded at import.
  - One ingest implementation continues to carry all transport semantics; the
    `IngestPump` divergence does not widen.
- **Negative / cost**
  - The transport state machine lives on the production ingest thread —
    `pipeline.rs` is the serial LANE-CORE territory, so this work is sequenced,
    not parallel (work-schedule SW lanes).
  - New control-plane surface: library CRUD + import operation + player verbs +
    events, with OpenAPI/AsyncAPI + SPA forms ([ADR-W021](ADR-W021.md),
    [ADR-RT008](ADR-RT008.md)).
  - Import-time transcode adds an on-disk mezzanine footprint for non-native
    alpha formats (bounded, per-asset, disclosed in the asset metadata).
  - `Demuxer::seek` gains its first production callers and therefore needs the
    test coverage its zero-caller status never forced (frame-accuracy across
    keyframe brackets, loop-point seams, post-seek pacer re-anchor).
- **Neutral**
  - `load`/`cue`/`play`/`pause`/`stop`/`seek` are all Class-1 hot (inv #11):
    heavy work happens off-thread (LiveSourceHub / ingest thread); the frame
    boundary sees only O(1) binding/slot mutations. `auto_off` is Class-1 with
    a consequence the **switcher state machine** applies: bus selections fall
    to the cell's configured `on_loss` slate and a DSK fed by the channel
    drops off-air-safe per [ADR-R011](ADR-R011.md).
  - The still store is not a new mechanism — it is the EOF-hold path used on
    purpose.
