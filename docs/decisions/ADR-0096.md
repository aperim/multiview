# ADR-0096: Licensing live wire (heartbeat client) — gate, resolved by Conspect API v0.6.1

- **Status:** Accepted — opened as a gate (2026-06-16), **resolved the same day by Conspect API v0.6.1**; CONSPECT-3 implementation in progress.
- **Area:** Management
- **Date:** 2026-06-16
- **Source:** operator (finalized the conspect.studio `/v0` licensing reference, "closing the loop" on end-to-end licensing)

## Context

The runtime licensing/entitlement subsystem
([ADR-0050](ADR-0050.md)/[0051](ADR-0051.md)/[0052](ADR-0052.md)/[0053](ADR-0053.md);
account-side design [conspect-account-architecture](../research/conspect-account-architecture.md)) is
**~90% shipped**. The `multiview-licence` crate holds the always-compiled state model
(lease/ladder/fingerprint/verify/challenge/store), the three never-off-air engine seams
**S1/S2/S3** are wired and proven by the CONSPECT-2 chaos gate (PR #94), and the control
routes, mesh plane, and web screens are delivered (CONSPECT-4..11). The engine reads licence
posture as **two wait-free atomics per tick** (`config_locked`, `watermark`) — never licence
logic or a network call on the hot path (invariants #1, #10).

**The only gap is the device→server network client.** `crates/multiview-licence/src/heartbeat.rs`
does not exist and there is no `[features] heartbeat`. This was gated on three details the
device API had to publish — **D1** key-trust, **D2** device auth, **D3** the lease signing
pre-image — plus operator confirmations **O1–O4**. Per [ADR-0050](ADR-0050.md)/[ADR-0051](ADR-0051.md)
the confirmations gate **only the live wire**; everything upstream is testable against an
in-process fake.

### Resolution — Conspect API v0.6.1 (2026-06-16)

v0.6.1 is content-identical to the captured v0.6.0 baseline (a version bump); the answers were
already in the **rich operation/field descriptions** plus the **live, public
`/.well-known/conspect-licensing-keys.json`** (an earlier truncated fetch had hidden the prose).
The gate is **closed** for the device heartbeat client:

- **D1 — key-trust → RESOLVED.** `GET https://api.conspect.studio/.well-known/conspect-licensing-keys.json`
  (public, unauthenticated) publishes: a pinned **ECDSA-P256 root**; `lease_keys` = **dual-pin
  Ed25519 intermediates**, each carrying `root_sig`; an `update_keys` set; a `revocation` list
  (`root_revocation_sig`); and the `attestation_contract`. Trust bootstrap: **pin the root →
  for each intermediate verify its `root_sig` (`ecdsa-p256-sha256`, raw r‖s, base64url) over the
  deterministic-CBOR key pre-image `[key_id, key_type, statement, public_key, valid_from, valid_until]`
  → accept current+next within validity → honour the revocation list → then verify a lease against
  the attested intermediate.** (There IS a device-facing key endpoint; roots are not obtained
  out-of-band.)
- **D2 — device auth → RESOLVED.** The licensing surface authenticates with the **account JWT
  Bearer chain today** (`Authorization: Bearer <jwt>`; **Engineer** role / `account:licence:write`
  for activate, **operator** role for heartbeat; **`Idempotency-Key` header required** on every
  mutation). The device PoP key is **captured and stored** (`devicePublicKey`, Ed25519) but
  authenticating a request *with* it is **deferred server-side too** (slice 5d) — so the client
  sends the public key now and does **not** sign requests with it yet. (Not a PAT/API-key model.)
- **D3 — lease signing → RESOLVED.** `lease.signature` is a **bare Ed25519 (EdDSA) signature,
  lower-case hex (64 bytes), NO COSE**, computed over `lease.leaseBytes` — and `leaseBytes` is the
  **STANDARD-base64** (RFC 4648 §4, **not** base64url) encoding of the **RFC 8949 §4.2.1
  deterministic-CBOR** signed body. Verify the signature over the *decoded* `leaseBytes` against
  the attested intermediate for `signerKeyId`, then parse the CBOR for the authoritative fields
  (`gpu_limit`, `hardware_class`, `not_after`, chain fields) — the JSON lease fields are a
  convenience subset. **Every lease response is its own golden vector** (it carries the exact
  signed bytes + signature), so no separate KAT is needed.

Never-off-air is server-confirmed: a revoked entitlement returns `200` with `lease: null` + the
current `enforcementState` rung (revocation by **non-reissue**; no push kill verb).

**One residual (O4, non-blocking):** the default `{orgId}` a **free** self-host activates against
(omit `claimCode`) lives in the external `licensing-architecture.md §5`, not the OpenAPI — the
paid/claim-code path is fully specified, so the client treats `orgId` as config and the free-tier
default is filled when §5 is supplied. O3 (the 6-char claim-code glyph alphabet) is a
server/portal concern — the device passes the user's code through, so it does not block the client.

## Decision

Implement the remainder (CONSPECT-3 heartbeat client, then CONSPECT-12 live-wire/hardware
validation) as **one end-to-end LANE-CONSPECT wave** against the v0.6.1 wire above — wired,
tested, no stub (rule 6):

- **Feature flags (default-off):** `[features] heartbeat` in `crates/multiview-licence/Cargo.toml`
  and `heartbeat = ["multiview-licence/heartbeat"]` in `crates/multiview-cli/Cargo.toml`, appended
  to the `nvidia`/`apple`/`linux-vaapi`/`full` presets **only** (mirrors `mesh-mdns`). Default
  `cargo check --workspace` + `cargo deny` stay a pure, LGPL-clean, network-free shell. New deps
  (behind `heartbeat`): `p256` (RustCrypto, the root ECDSA-P256 verify), a deterministic-CBOR codec,
  and a rustls-based HTTP client — all MIT/Apache, deny-verified.
- **`crates/multiview-licence/src/heartbeat.rs`** (cfg `heartbeat`): an object-safe
  `trait LicenceServer` implemented by the real HTTP client **and** an in-process `FakeLicenceServer`;
  a `HeartbeatClient` loop that sleeps to `nextDue`, verifies the returned signed lease via the D1/D3
  chain above, and drives the existing `store.install_binding` convergence (the same path the
  offline-lease install and the mesh relay already use). The security-critical verify lives in the
  testable crate; the HTTP transport may sit at the cli boundary.
- **Spawn point:** `spawn_heartbeat` in `multiview-cli/src/licence.rs` (modelled on
  `spawn_mesh_discovery`), called beside it in `main.rs`. The task holds **only** `Arc<LeaseStore>`
  + pinned root + `Arc<dyn LicenceServer>` — never an engine handle/channel/lock (invariant #10);
  its sole output is `install_binding`, propagating to S1/S2/S3 via the existing `refresh_signal`.
- **Tests (TDD):** heartbeat-OFF reports `unlicensed-build` honestly (never fake `active`);
  heartbeat-ON against the fake → Active(35d) wait-free; tamper/forged/unattested-intermediate/revoked
  → rejected; no raw identifier in the payload; the CONSPECT-2 never-off-air chaos gate is extended
  to SIGKILL/stall/partition the heartbeat task and assert bit-unchanged output on every rung.

### Constants that must match the portal verbatim (never round/re-derive)

`LEASE_FULL=35d · LEASE_GRACE=14d · LEASE_HARD=90d · ACTIVATION_WINDOW=31d · CLAIM_CODE_TTL=15m ·
CLAIM_REJECT_WINDOW=24h · TRANSFER_WINDOW=72h · FINGERPRINT_MATCH_STRONG=100 ·
FINGERPRINT_MATCH_THRESHOLD=70 · CLAIM_CODE_LEN=6 · lease signing = Ed25519; root attestation =
ECDSA-P256`. "A portal showing 35 days and a machine enforcing 30 is a support incident."

## Rationale

The wire shapes + enforcement ladder were stable; the load-bearing unknowns were the **trust and
signing** details (D1/D2/D3) — exactly what makes a heartbeat client correct and secure. v0.6.1
publishes them precisely (a public key-trust endpoint with an explicit attestation contract, a
byte-exact lease pre-image, and a named auth model), so the client can be built without guessing.
Recording the gate and its resolution as a named service dependency (this ADR) is the
rule-42/rule-6 alternative to silent debt or a stub.

## Alternatives considered

| Alternative | Rejected because |
| ----------- | ---------------- |
| **Stub/guess the wire before v0.6.1** | Ships a guessed trust/signing model as if done (rule 6); a wrong pinned-root or canonicalisation is a security defect, and `todo!()` on a security boundary is the worst place for it. |
| **Scaffold the trait+fake first, defer the HTTP client** | Reasonable, but the operator finalized the API within hours, so the whole wave lands against the real wire with no throwaway scaffolding. |
| **Leave it undocumented** | Loses the captured baseline, the precise gap, and the now-verified wire — re-derived cost every session. |

## Consequences

- **Unblocked:** CONSPECT-3 is in implementation against a fully-specified, security-reviewed wire.
- **Honest:** the default build stays a pure LGPL-clean, network-free shell with `unlicensed-build`
  honesty ([ADR-0050 §7](ADR-0050.md)); the new client deps are feature-gated + deny-clean.
- **Invariant guard:** the heartbeat task is, by construction, unable to back-pressure the engine
  (#10); revocation-by-non-reissue means no licence path takes running output off air (#1) — both
  re-proven by the extended chaos gate in the implementing wave.
- **Residual:** the free-tier default `{orgId}` (O4) awaits `licensing-architecture.md §5`; it gates
  only the free auto-issue onboarding default, not the heartbeat/renew core.
