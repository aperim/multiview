# ADR-0051 — Conspect mesh: the `multiview-mesh` crate — always-on mDNS, signed digests, end-to-end-signed relay

**Status:** Proposed (2026-06-10)
**Area:** Account / Mesh — the local-discovery + heartbeat-relay plane of the Conspect subsystem
**Drivers:** A machine with **no internet path** to the licence server must still be able to keep its
entitlement lease live ([ADR-0050](ADR-0050.md)). On a LAN, an **online** neighbour can **relay** an
offline machine's heartbeat to the server and carry the signed response back. This needs (a) local
peer **discovery** and (b) a **relay** path that the relayer cannot read, forge, or alter. There is
**no** mDNS or mesh infrastructure in the tree (greenfield — the same absence the
[managed-devices](../research/managed-devices.md) brief notes for device discovery).
**Research:** [conspect-account-architecture](../research/conspect-account-architecture.md) §9.
**Related:** [ADR-0050](ADR-0050.md) (entitlement — the signed assertions mesh carries),
[ADR-0042](ADR-0042.md) (IPv6-first — mDNS binds + multicast groups),
[ADR-0041](ADR-0041.md) (SAP discovery — the **untrusted-inventory + confirm-adopt** doctrine reused
here), [managed-devices](../research/managed-devices.md) (the control-plane-actor-over-channels
isolation shape, inv #10), [ADR-RT004](ADR-RT004.md)/[ADR-P001](ADR-P001.md) (isolation invariant #10).

## Context

The relay's threat model is the crux: a relaying machine is **untrusted** with respect to the payload
it carries. It must be a **dumb carrier** — it forwards bytes it cannot interpret or modify, and the
licence server still authenticates the *originating* machine end-to-end. If relay integrity depended on
trusting the relayer, a malicious neighbour could mint or steal entitlements. It must not.

Discovery has its own hazard: an announced peer is **untrusted**. Auto-trusting LAN announcements
(the classic mDNS footgun) would let any host on the segment inject itself. We reuse the doctrine
already proven for SAP-discovered sessions ([ADR-0041](ADR-0041.md)) and managed devices: discovery
populates an **untrusted inventory**; a relationship is formed only by explicit operator
**confirm-adopt**.

## Decision

### 1. One new leaf crate: `crates/multiview-mesh`

Lib `multiview_mesh`. Depends on `multiview-core`, `multiview-licence` (for the signed
digest-summary + assertion types it announces and relays), `multiview-events`, `tokio` (net, behind a
`mdns` feature), `ed25519-dalek`, `thiserror`, `tracing`. Dependency direction stays acyclic:
`mesh → {licence, core, events}`, `cli → mesh`. Default build is a **pure shell** (the relay logic and
signed-payload handling are pure-Rust and always compiled/tested); the `mdns` feature pulls the socket
+ a maintained, deny-clean mDNS-sd-class crate (the exact dependency is an operator-confirm item, brief
O5).

### 2. Discovery — always-on mDNS, IPv6-first, signed summaries

The mesh announces and browses a Conspect service type over IPv6-first link-local multicast
(`ff02::fb` mDNS; IPv4 `224.0.0.251` legacy interop) per [ADR-0042](ADR-0042.md). Each announcement
carries an **Ed25519-signed** summary: the machine's **salted** fingerprint digest set + a signed
entitlement summary (level + lease bounds). It **never** carries raw identity (serials/MACs/URLs) —
data minimisation (brief §8). "Always-on" means discovery runs whenever the account plane runs; it is
not gated on being offline (a machine advertises its willingness to relay, and learns its neighbours,
continuously).

### 3. Untrusted inventory + confirm-adopt (the ADR-0041 doctrine)

Discovered peers populate a **bounded, untrusted** inventory exposed at `GET /api/v1/mesh/peers`. A
peer is **never** auto-trusted or auto-relayed. The operator **confirm-adopts** a peer
(`POST /api/v1/mesh/peers/{id}/adopt`, Class-1) before it participates in relay; adoption is reversible
(`DELETE`). This is the same untrusted-discovery → explicit-bind contract as SAP and managed devices.

### 4. Relay — end-to-end signed; the relayer is a dumb carrier

An online machine relays an adopted offline neighbour's heartbeat request to the licence server and
carries the signed response back. The relay payload is **end-to-end signed** by the **originating
machine** (request) and the **licence server** (response): the relaying machine lacks both keys, so it
**cannot read, forge, or alter** the assertion. Relay integrity therefore does **not** depend on
trusting the relayer — a tampered or spoofed relay payload fails the originator/server signature check
and is rejected. Relay is **opt-in per machine** (`PUT /api/v1/mesh/relay`): a machine can be a willing
relayer or decline, and is disclosed in the Mesh screen. A relayer **earns no entitlement** from
carrying traffic.

### 5. Bounded + isolated (invariant #10 by construction)

The mesh plane is a **control-plane actor over watch/broadcast channels** — the proven
managed-devices/NMOS isolation shape. The engine never sees it. A relayer enqueues at most a **bounded**
set of neighbour requests (drop-oldest), never blocks on a neighbour (UDP/multicast is non-blocking; a
full queue drops), and never holds a lock the engine holds. Mesh traffic can lose *neighbour relay
attempts* under pressure; it can never stall the engine or even the local heartbeat.

### 6. Feature ladder + licensing

- Crate feature: `mdns` (the socket + mDNS library, off by default); the relay/signing core is always
  compiled.
- cli: enabled by the `account` aggregation; shipped presets turn `mdns` on.
- `ed25519-dalek` is `MIT OR Apache-2.0`; the mDNS library must be deny-clean (operator-confirm O5).

## Consequences

- **Offline machines stay licensed** via a LAN neighbour, with **zero trust** placed in the relayer —
  the end-to-end signatures make the relayer a courier, not an authority.
- **No LAN-injection hazard** — discovery is untrusted + confirm-adopt, reusing the ADR-0041 doctrine.
- **Greenfield, collision-free** — a brand-new leaf crate touching no WebUI-session file; built early.
- **Privacy-preserving** — announcements carry salted digests + signed summaries, never raw identity.
- **Isolation preserved** — the mesh is an off-hot-loop actor over bounded drop-oldest channels;
  invariant #10 holds by construction.
- **Operator confirmations** gate the live wire: the licence-server relay framing + the server key
  (shared with [ADR-0050](ADR-0050.md) O1/O2) and the mDNS dependency choice (O5).

## Alternatives rejected

- **Trust the relayer (relayer terminates + re-issues the assertion).** Lets a malicious neighbour mint
  or steal entitlements; makes relay integrity depend on every LAN host. Rejected for end-to-end
  signing with the relayer as a dumb carrier.
- **Auto-adopt discovered peers.** The classic mDNS footgun — any segment host injects itself. Rejected
  for untrusted-inventory + explicit confirm-adopt (ADR-0041 doctrine).
- **A central broker / cloud relay only (no LAN mesh).** Defeats the whole point (an air-gapped-from-
  -internet but LAN-connected box can't heartbeat) and adds a hosted dependency. Rejected — LAN relay
  is the offline-resilience story; cloud is the normal path.
- **Folding mesh into `multiview-licence` or `multiview-control`.** Couples discovery/socket code to
  the entitlement state model or the API crate (collision with the WebUI session), and loses the clean
  leaf-crate test boundary. Rejected for the dedicated leaf crate.
- **IPv4-first mDNS.** Violates the IPv6-first directive ([ADR-0042](ADR-0042.md)). Rejected —
  `ff02::fb` primary, IPv4 `224.0.0.251` legacy interop only.
