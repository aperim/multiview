# ADR-0001: Hybrid media engine: FFmpeg/libav for demux/decode/encode + custom Rust for compositing and serving

- **Status:** Proposed
- **Area:** Core Engine
- **Date:** 2026-06-02
- **Source brief:** [core-engine.md](../research/core-engine.md)

## Decision

Use FFmpeg/libav (libavformat/libavcodec/libavutil) for demux, hardware decode, and hardware encode, and implement compositing and the serving/output side in custom Rust + GPU-native code.

## Rationale

libav provides mature, battle-tested demux/decode/encode for every required protocol and HW backend (NVENC/NVDEC, VAAPI, QSV, VideoToolbox) with automatic software fallback via the generic hwaccel path. Compositing and serving are exactly where libav is weak/uneven (no CUDA stack filter; xstack has no fit/crop; hls muxer can't do LL-HLS), so owning them in Rust is the differentiator and enables zero-copy control.

## Alternatives considered

Pure-FFmpeg filter-graph mosaic (rejected: no per-cell fit/crop, vendor-uneven, no LL-HLS, no zero-copy control). GStreamer end-to-end (rejected: heavier dependency, GLib/Tokio friction, against the custom-serving decision). Pure-Rust media stack (rejected: re-implementing five vendor codec SDKs is infeasible).

## Consequences

Must own an unsafe FFI boundary over libav and a custom GPU compositor + custom serving stack. Gains full control of zero-copy, layout, and LL-HLS. Effective license governed by the FFmpeg build, not the Rust crate.
