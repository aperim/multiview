# ADR-0025: Streaming encode â€” bound memory + continuous output on the run-until path

- **Status:** Accepted
- **Date:** 2026-06-04
- **Area:** `mosaic-cli` run/encode orchestration (`RealPipeline::drive`)
- **Relates to:** invariants **#1** (output-clock never stalls), **#10** (no engine back-pressure), **#7** (encode-once-mux-many â€” partially, see Follow-ups), the resilience pillar "bulletproof continuous output". Found by the discovery fan-out (cli-run-build #1).

## Context

`RealPipeline::drive` collected **every** composited tick into an unbounded
`Arc<Mutex<Vec<CollectedTick>>>` (a per-tick *full-canvas* `Arc::new(canvas.clone())`),
then ran `bake_overlays` + `encode_outputs` only **after** the engine loop returned.
`run_until(&stop)` (wired to real Ctrl-C â€” the "continuous output" daemon mode)
passes `max_ticks = None`, so the loop is unbounded:

- **Memory:** ~3 MB/frame Ă— 25 fps â‰ˆ 270 GB/hour â€” the daemon OOMs within minutes.
- **Latency:** no output bytes are written until the run *ends* â€” the opposite of
  continuous output; a Ctrl-C'd HLS/file emerges in one terminal burst.

A 3-architect design panel confirmed the bug and converged on a streaming design.

## Decision

**Stream bakeâ†’encodeâ†’mux off the hot path through a bounded queue; bound memory to
`O(queue)`, not `O(ticks)`.** Implemented entirely within `mosaic-cli` by reusing
the existing pull-based `mosaic_output` sinks unchanged:

1. **Hot loop â†’ bounded `sync_channel`.** The per-tick `state_of` closure keeps its
   on-hot-loop caption/fault *sampling* (constraint: cues must be sampled before the
   small live window evicts them), builds a `StreamItem { Arc<canvas>, dynamics }`
   (one cheap `Arc` clone â€” no more than today), and hands it to the queue. The
   *only* hot-loop op touching the consumer is a wait-free send.
2. **Send policy by mode (the load-bearing split).** `run_for(N)` (offline render)
   uses **block-for-exact**: a full queue back-pressures the *renderer* (legitimate â€”
   offline is not a live clock), so every tick is encoded â†’ exactly N frames, never
   dropped. `run_until` (live) uses **drop-on-overload**: `try_send`; on `Full` the
   incoming frame is dropped and a counter incremented â€” the engine never blocks
   (inv #1/#10). Offline therefore **cannot** silently truncate a file; live degrades
   visibly, never stalls.
3. **One off-hot-path bake consumer.** A `std::thread` (mirroring `IngestSupervisor`,
   `forbid(unsafe)`) receives `StreamItem`s, runs the overlay **rasterise** once per
   frame (the bake *render* moves here; sampling stayed on the hot loop), and fans
   the baked `Arc<Nv12Image>` to each output's own bounded channel.
4. **Per-sink streaming via the unchanged `sink.run()`.** Each configured sink runs
   on its own thread driving a `StreamingFrameSource` (a `VideoFrameSource` over its
   bounded receiver): `next_frame()` blocks on `recv` â†’ `Ok(Some)`, channel closed â†’
   `Ok(None)` = end-of-program â†’ the sink writes its trailer. The tested
   `FileSink/SegmentSink::run` are reused verbatim.
5. **Teardown / finalisation.** After the engine loop: `supervisor.shutdown()`, drop
   the hot-loop sender â†’ the bake consumer drains, fans the remainder, drops the
   per-sink senders â†’ each `sink.run()` sees end-of-program and finalises â†’ join all
   threads â†’ fold stats into `PipelineReport`.
6. **`faltered` redefined honestly:** `dropped > 0` (live overload), surfaced in the
   report. Offline `dropped == 0` â‡’ `!faltered` â€” every existing exact-frame test
   stays green.

No new dependency (`std::sync::mpsc::sync_channel` gives blocking `send`, `try_send`,
and `recv`). `mosaic-cli` stays `forbid(unsafe)`; `mosaic-output` is untouched.

## Consequences

- Memory is `O(queue capacity)` (a few NV12 buffers, ~tens of MB) for any run length;
  the daemon runs continuously and streams output as it is produced.
- Offline `--ticks N` keeps exact, all-frames, deterministic semantics (block-for-exact).
- **Gates (TDD, sleep-free via `ManualTimeSource`/`CooperativePacer`):** (1) a blocked
  fake sink + live policy: the engine still emits all N ticks, peak queue â‰¤ cap,
  `dropped > 0` (the chaos/OOM guard); (2) offline produces exactly N frames,
  `dropped == 0`, `!faltered`; (3) streaming-not-batch â€” frame 0 is encoded while the
  engine is still ticking (today it is encoded only after the loop ends); (4) clean
  stop writes the trailer.

## Alternatives considered

- *Single consumer that encodes the canvas **once** and fans encoded packets to N
  muxers (a new `PacketMuxSink` trait).* This is the panel's preferred long-term shape
  â€” it also fixes invariant #7 (today the CLI encodes **per sink**, so file+HLS encode
  twice) and the per-GOP throwaway seed-encoder wart. **Deferred** to a follow-up: it
  refactors the tested `mosaic-output` sink internals, a larger surface than the OOM
  fix needs. Tracked.
- *Hard-cap the collected `Vec` (drop-oldest) + keep encode-at-end.* Rejected â€” it
  drops frames into a still-batch encode (incomplete output), not real streaming.

## Follow-ups (tracked, not in this change)
