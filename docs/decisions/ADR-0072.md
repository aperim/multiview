## Context

The operator wants Multiview to ingest **broadcast cues** from several transports — physical
GPIO/relay closures, studio SCTE-104, in-stream SCTE-35, HLS/DASH manifest cues, BXF schedules —
and turn each into an action: run a macro, cut/auto the switcher, load/play a VT, show/hide a
lower-third (DSK), set an aux route, fade an audio bus, mute a source, start/stop recording, emit
SCTE-35 or a manifest cue downstream, fire a GPO, send a WebSocket event, or call a webhook.

The decoders differ per transport (ADRs 0073–0076), but the **dispatch must be uniform** and must
not invent a privileged path into the engine. Two facts make this tractable and verified in-tree:

1. A normalized cue value already exists for SCTE: `crates/multiview-input/src/scte/mod.rs:88`
   `CueEvent` with `CueKind::{SpliceOut,SpliceIn,TimeSignal}` (`:72`) and exact `90k→ns`
   projections (`:107-115`).
2. Every action the operator listed **already exists** as a bare-verb command or subsystem
   operation: cut/auto/DSK/aux/T-bar ([ADR-W021](ADR-W021.md)), audio fade/mute
   ([ADR-0059](ADR-0059.md)), macro run ([production-switcher §10.2](../research/production-switcher.md)),
   recording arm/disarm ([ADR-0037](ADR-0037.md)), and the webhook notifier builder
   (`crates/multiview-control/src/notify/webhook.rs`). The BUILT frame-boundary `CommandDrain`
   (`crates/multiview-cli/src/control.rs`) and `Command` enum
   (`crates/multiview-control/src/command.rs:174`) are the one apply path.

The risk to avoid: a second, cue-specific command channel that could bypass the live-apply
classifier or back-pressure the engine.

## Decision

1. **Define one normalized `Cue` value** (shared type in `multiview-core`, internally-tagged
   serde, never `untagged`): `{id, source, kind, event_id?, pts?, break_duration?, observed_at,
   attributes}` (brief §1.1). The SCTE `CueEvent` is the SCTE-specific seed; `Cue` is the superset
   across all five ingresses. The SCTE→`Cue` adapter is a field map over the existing exact-rational
   projections — no new time math.

2. **Define one `CueAction` value** (internally-tagged enum in `multiview-core`) covering the full
   operator vocabulary (brief §6). Each variant **desugars onto an existing `Command` or subsystem
   op** — switcher verbs become the identical `Command` a live operator sends; effects
   (emit-SCTE, emit-manifest, fire-GPO, WebSocket, webhook) run on the control-plane dispatch or
   packager task. No `CueAction` reaches the engine except via the existing command bus.

3. **Define the cue rule table** as an operator-authored, additive `cues` config block on
   `MultiviewConfig` (the routing/switcher additive-block precedent, [ADR-M012](ADR-M012.md)):
   `CueRule {name, match: CueMatcher{source?, kind?, attribute_eq, min_interval?}, actions:
   Vec<CueAction>, enabled}`. Validation is `#[serde(default)]`, internally-tagged unions only.

4. **The cue dispatcher is a pure value machine** in `multiview-control` in the salvo/tally/alarm
   house style: consume a `Cue`, match enabled rules, emit the ordered action list. It takes no
   lock the engine holds and never awaits. Switcher actions are submitted through the existing
   bounded command bus with `try_send`; the macro action reuses the
   [production-switcher §10.2](../research/production-switcher.md) sequencer verbatim. **One apply
   path; a cue is just another client.** The dispatcher computes an `effective_at` per cue (derived
   from the SCTE splice/time-signal data or BXF boundary where present, else `observed_at`) and
   submits its actions at that tick through the existing frame-boundary wait-step — so a cue with a
   future splice/preroll time fires neither early nor late, and an already-expired cue is dropped or
   fired-with-warning per an operator-selectable per-rule policy (brief §1.2). Scheduling is a
   control-plane timer, never a data-plane wait (inv #1).

5. **Classification is inherited, not invented** (inv #11, brief §1.3): a cue action carries the
   apply class of the verb it desugars to ([ADR-M012 §9](ADR-M012.md)/[ADR-0034 §8](ADR-0034.md)).
   The MVP cue-action set is entirely Class-1 (Hot); a structural change is out of scope for cues.
   The plan/take dry-run ([ADR-W021](ADR-W021.md)) previews a rule's classification before arming.

6. **A `cue` lifecycle event** publishes every decoded cue and dispatched action on the existing
   drop-oldest realtime broadcast lane ([ADR-RT008](ADR-RT008.md)) for the SPA cue log —
   observational only, never blocking. The operator-authored *send-WebSocket* action is a
   **distinct** `cue.user` event, not this telemetry.

7. **Emit actions are outward-facing and RBAC-gated.** Emitting SCTE-35, a manifest cue, or a GPO
   closure has real downstream consequences; emit actions require an explicit `emit` capability
   scope distinct from operating the local switcher (CLAUDE.md §0 confirmation posture).

8. **No new crate**: types in `multiview-core`, schema in `multiview-config`, dispatcher + rule
   table + REST in `multiview-control`, wire event in `multiview-events`, panel in `web/`.

## Consequences

**Positive.** The full cue vocabulary lands as config + a value machine over already-built verbs;
the classifier, idempotency, and correlation apply for free; "run macro" makes arbitrarily complex
cue responses a one-line rule. A cue can re-originate (GPI→emit-SCTE; SCTE-out→GPO+webhook).

**Negative / cost.** `multiview-core` grows the `Cue`/`CueAction`/`CueRule` types and
`multiview-control` the dispatcher + a new REST/resource domain (the sync-groups six-piece
template). Emit actions add an RBAC scope and an outward-facing confirmation posture. A large
rule table is O(rules) per cue — bounded, but worth a match-index if rule counts grow (named
follow-on, not MVP).

**Neutral.** Cue rate is human/automation rate; the dispatcher is off the hot path by construction.

## Alternatives considered

1. **A cue-specific command channel into the engine.** Rejected — a second apply path could bypass
   the live-apply classifier and risk inv #10; desugaring onto the existing `Command` bus gives one
   audited path.
2. **Hard-code cue→action mappings.** Rejected — the operator needs an editable rule table; a fixed
   mapping is unmaintainable and untestable per deployment.
3. **A scripting/expression engine for matchers.** Rejected for the MVP — a typed `CueMatcher`
   (source/kind/attribute-eq/min-interval) covers the stated asks without an embedded interpreter
   (no `dyn Any`, no eval surface); revisit only on a real need.
4. **Cues emit events that *clients* turn into commands.** Rejected — round-tripping every cue
   through a client adds latency and a failure mode; the dispatcher submits directly, and the
   *send-WebSocket* action remains available for genuinely external consumers.

## How this honors the invariants

- **#1 (output clock):** the dispatcher and all decoders run on control-plane tasks; no cue path
  blocks or paces the output-clock thread. Switcher actions apply at the existing frame-boundary
  drain exactly as live commands do.
- **#10 (isolation):** every ingress→dispatcher→bus hop is bounded drop-oldest with `try_send`; the
  engine never awaits a cue producer or consumer. A flood of cues is coalesced/dropped, never a
  stall (the macro-sequencer/notify isolation proof, [ADR-RT004](ADR-RT004.md)). Overflow is
  *class-aware*: safety-paired cues (out↔in, on↔off) coalesce-or-replace a stale same-key entry
  rather than blind-evicting their pair, and every drop increments a per-class counter that raises an
  operator alarm (brief §7). This changes which entry is evicted on overflow, never adding an
  unbounded queue or an engine await.
- **#11 (Class-1/2):** actions inherit their verb's classification; the plan/take dry-run surfaces
  it before applying.
