# ADR-0021: Input timing and frame-sync — best-effort PTS normalisation + wall-clock pacer + sample-at-tick frame-sync

- **Status:** Accepted — **as-built**. Point 4 (sample-at-tick frame-sync) shipped + verified 2026-06-03; points 2–3 (`PtsNormalizer` + `Pacer` + `ReorderBuffer`) wired on the raw-RTP / ST-2110 ingest 2026-06-14 (ENG-2b, see [§As-built (2026-06-14)](#as-built-2026-06-14--eng-2b-pacer--reorderbuffer-wired-on-the-raw-rtp--st-2110-ingest)). Remaining cosmetic de-dup: retiring `PtsWallClock` on the libav path (point 1).
- **Area:** Streaming/Timing
- **Date:** 2026-06-03
- **Source brief:** [input-timing-and-sync.md](../research/input-timing-and-sync.md)
- **Builds on:** [ADR-T001](ADR-T001.md) (monotonic output clock), [ADR-T002](ADR-T002.md) (per-tile hold-last-good + dup/drop), [ADR-T003](ADR-T003.md) (per-input PTS normalisation), [ADR-T004](ADR-T004.md) (HLS ingest pacing, not `-re`), [ADR-T006](ADR-T006.md) (drift loop); refines the Layer-C input half of [ADR-0020](ADR-0020.md)

## Decision

The **input side** of every source (file/VOD and live) flows through one canonical pipeline, and the **only** ad-hoc per-frame timestamp pacer in the running ingest loop (`multiview-cli/src/pipeline.rs::PtsWallClock`) is **retired** in favour of the existing, tested `multiview-input` components:

1. **Per-frame timestamp acquisition:** schedule by `best_effort_timestamp` (= `guess_correct_pts(pts, pkt_dts)`, set on every decoded frame; display order, B-frame-correct), falling back to raw `pts`, then to a **genpts** synthesis. Never schedule by `pkt_dts`.
2. **Normalisation** via `multiview-input/src/normalize.rs::PtsNormalizer`: delta-based 33-bit/32-bit wrap unwrap into a 64-bit counter (never libavformat `correct_ts_overflow`); genpts fallback from the **measured/declared `r_frame_rate` cadence** (not a fixed constant); discontinuity re-anchor (`EXT-X-DISCONTINUITY` / TS indicator / `|jump| > ~10 s`); first-frame anchoring `offset = master_now − first_pts`; strictly-monotonic guard; exact-rational rescale to i64 ns. The mpegts ~1.44 s start offset is absorbed by anchoring (never stripped specially).
3. **Pacing** via `multiview-input/src/pacer.rs::Pacer`: anchor-first-PTS-to-now, release each frame at `anchor_wall + (pts − pts0)`, bounded `5/4` (1.25×) catch-up for small drift (never an instant seek), re-anchor on discontinuity, behind the bounded drop-oldest `jitter.rs::ReorderBuffer`. The pacer feeds on **normalised** PTS, not raw decoded PTS.
4. **Frame-sync (sample-at-tick):** the output clock (`out_pts = f(tick)`, [ADR-T001](ADR-T001.md)) samples each per-tile store at tick `N` via `nearest_at_or_before(t_N)`, holding `last_good()` on starvation. The pacer gates *when a frame enters the store*; the output tick decides *which stored frame is shown* — these are separate concerns.

The genpts fallback **moves out of `StreamVideoDecoder`** (which hard-codes `FALLBACK_STEP_NS = 33_366_667` ≈ 29.97 fps for any stream) into `PtsNormalizer`, which derives the frame period from the per-input declared cadence. `StreamVideoDecoder` keeps the correct `best_effort_timestamp`-then-`pts` selection.

## Rationale

A file/VOD tile played **ultra-fast then froze** because two divergent PTS paths exist and the bulletproof one is unused by ingest: the running loop feeds raw `decoded.meta.pts` into `PtsWallClock`, which anchors on the first frame and sleeps `now − base_instant ≥ pts − base_pts`. When `best_effort_timestamp` intermittently returns `AV_NOPTS_VALUE` (observed on `net_news.ts`) the decoder's hard-coded 29.97 fallback (and any run of near-constant PTS) makes per-frame `delta ≈ 0`, so every wait returns immediately and the source **races** (~10 s into ~1 s); when real PTS resumes the pacer schedules a far-future release and the tile **freezes**.

`best_effort_timestamp` is the correct field: FFmpeg `master` `libavcodec/decode.c` sets `frame->best_effort_timestamp = guess_correct_pts(dc, frame->pts, frame->pkt_dts)` with the comment *"Attempt to guess proper monotonic timestamps for decoded video frames which might have incorrect times"*, choosing reordered PTS when PTS is at least as reliable as DTS (faulty-pts/dts counters), else DTS (verified 2026-06-03). `avcodec_receive_frame` returns frames in presentation order (empirically verified on the refsrc B-frame captures — strictly increasing PTS), so a normalised stream is directly schedulable; the monotonic guard remains because display-order is *not* an explicit API guarantee. `-re` is the wrong tool: it is *"equivalent to setting -readrate 1"*, *"Mainly used to simulate a capture device or live input stream (e.g. when reading from a file)"* and *"Should not be used … with an actual capture device or live stream as it may cause packet loss"* (FFmpeg `master` `doc/ffmpeg.texi`, 2026-06-03) — and is CLI-only besides; Multiview owns its libav-linked pacer. The sample-at-tick model is GStreamer's live-aggregator behaviour: it *"waits on the clock and if a pad does not have a buffer in time it ignores that pad"* (GStreamer docs, 2026-06-03). The fix honours invariants #1/#2/#3/#4: the pacer gates ingest only, the output samples last-good-or-placeholder, all time is i64 ns / exact rationals.

## Alternatives considered

- **Naive 1:1 frame stepping (the current `PtsWallClock` + 29.97-constant genpts) — the bug.** *Rejected.* Anchors on raw first-frame PTS, re-anchors only on a backward step, and synthesises a fixed 29.97 cadence for any no-PTS stream. Collapsing/near-constant PTS → instant flood (race); a forward discontinuity → far-future deadline (freeze); 25/24 fps no-PTS tails play 11–25 % off-speed.
- **Decode-all-then-replay (buffer the whole source, then play paced).** *Rejected.* Unbounded memory (violates the data-plane bounded-memory rule / invariant #10), impossible for live sources, adds latency, and still needs a pacer at replay. The bounded `ReorderBuffer` + streaming pacer is the bounded equivalent.
- **Output paced by the input ("master source").** *Rejected — breaks invariant #1.* A stalling/bursting/wrapping source would freeze or time-warp the whole multiview; this is exactly the failure invariant #1 and [ADR-0020](ADR-0020.md) forbid. Cross-source alignment for genuinely PTP-locked feeds is handled in Layer C by mapping onto the common timeline, still *sampled, never pacing*.
- **Strip the mpegts start offset / trust libavformat `correct_ts_overflow`.** *Rejected.* The offset is harmless once anchored (deltas preserved); the libav wrap heuristic fired a false rollover at ~13h14m on a bogus SDP `rtptime` (MediaMTX #622). Own the unwrap.

## Consequences

- **One ingest pipeline.** `open_and_stream` routes file/VOD and live through `PtsNormalizer` + `Pacer` + `ReorderBuffer`; `PtsWallClock` and the diagnostic instrumentation are removed. The pacer's clock is injected, so the path is deterministically testable.
- **`StreamVideoDecoder` change is small and safe.** It stops owning the genpts fallback (drops `FALLBACK_STEP_NS`/`last_pts_ns`) and yields the raw `best_effort_timestamp`-or-`pts` (possibly `None`) plus the stream's `r_frame_rate`; the normaliser does genpts from the real cadence. The `best_effort_timestamp`-then-`pts` selection is preserved.
- **Determinism gate is flake-free.** `normalize(raw, master_now_ns)` and `submit(pts, now_ns)` are pure functions; tests assert exact emitted `MediaTime` sequences and `Release` decisions against a golden schedule with an injected clock — no sleeping. The frame-sync table asserts the exact per-tick composited frame id (CPU golden-frame).
- **Regression coverage is mandatory** for the adversarial matrix (brief §6): CFR 24/25/30, VFR, B-frames/non-monotonic-received, no-PTS, mpegts ~1.44 s offset, mid-stream discontinuity, PTS gap, 33-bit/32-bit wrap boundary, off-output-fps 24/25/29.97/30/50/60. The decisive guard: no-PTS-at-25 + 24 fps must emit a smooth measured-cadence schedule, failing the old 29.97 constant.
- **Soak/GPU tier separate.** End-to-end no-race/no-freeze under real network/GPU and across the wrap boundary is a ≥72 h zero-gap soak + SSIM/PSNR assertion, not part of the flake-free unit gate.
- **No change to the output side or the commodity default timing model** ([ADR-0020](ADR-0020.md)); this ADR fixes and unifies the *input* half of Layer C.

## As-built (2026-06-03)

The race-then-freeze bug was fixed in commit `7aaec73` (`fix(timing): sample inputs by output media-time (latch-on-tick), not latest-wins`). The deployed fix is **point 4 (sample-at-tick frame-sync)** as the *primary* mechanism — diagnosis showed the decisive root cause was the **output-side sampling reading latest-wins** while a fast producer (or a slow output loop) raced ahead, not the input PTS path. The narrower fix proved sufficient and was preferred to the full points-1–3 reroute. Specifically:

- **`multiview-framestore`** — the per-tile store gained a bounded, **media-time-ordered ring** (drop-oldest, lock-free `ArcSwap`) plus **`read_at(now)`** that latches the frame *nearest-but-not-after* the output media-time (binary search; earliest-retained fallback; backwards-stamp re-anchor). This *is* point 4's `nearest_at_or_before(t_N)` + hold-`last_good()`-on-starvation, realised at the framestore layer. `multiview-engine` drive samples `store.read_at(out_media_time)`.
- **`multiview-cli/src/pipeline.rs`** — the running ingest loop now stamps each published frame with **source-relative media time** (`pts − first_pts`). `PtsWallClock` was **retained as a read-ahead throttle** (not retired): once sampling is latch-on-tick, a producer that runs ahead is harmless (the bounded ring drops oldest), so the full `Pacer` reroute was unnecessary for *correctness*.
- **`multiview-ffmpeg::StreamVideoDecoder`** — **keeps** the genpts fallback but derives the inter-frame step from the stream's **declared cadence** (`with_declared_fps`, reproducing `30000/1001` exactly) instead of the hard-coded `33_366_667`. The cadence-correctness goal of point 2 was met *in the decoder*, not by moving genpts into `PtsNormalizer`.
- **Deterministic gates (failing-before / passing-after):** `multiview-framestore` `sample_by_media_time` (publish frames `0..=250` stamped `i·40 ms`; sampling at `1.0 s` latches frame 25, not 250) and `multiview-ffmpeg` `decode_stream` cadence tests. Verified end-to-end: a 25 fps burned-timecode diagnostic tracks source-TC ≈ 1:1 against output time and never falters.

### Correction (2026-06-04) — the "verified 1:1 by eye" above was overlay-masked

The As-built "verified end-to-end by eye … tracks source-TC ≈ 1:1 … never falters" claim was measured on frames carrying the **burned-in timecode + animated overlays (clock, audio meters)**, which change every frame and **masked a real defect**: in a **debug** build the single-threaded CPU compositor was ~10–55× slower than real time and the decoded **picture stalled** while those overlays animated. The latch-on-tick sampling here (point 4) is correct and the output-clock cadence held, but "smooth video" was **not** actually verified — md5/eyeball of overlay-laden, lossy-encoded frames is unreliable. The real smoothness defect was root-caused and fixed under **[ADR-0022](ADR-0022.md)** (parallelize the compositor + LUT the per-pixel colour math) plus a startup first-frame prime (task #40), and is now verified the right way: **content-aware, overlays-OFF** (`tblend=difference`→`signalstats` YAVG vs a ground-truth ffmpeg encode) — render real-time, rendered motion ≈ ground-truth, first frame real content not the cold slate. See [[verify-dont-assume]] "never verify VIDEO with animated-overlay frames".

**Not built at 2026-06-03 (see ENG-2b As-built below for the wiring):** points 1–3's routing through `multiview-input::PtsNormalizer` + `Pacer` + `ReorderBuffer`, and the retirement of `PtsWallClock`. Those components existed and were unit-tested but remained unused by ingest. They address an *additional* class — 33-bit/32-bit PTS wrap, mid-stream `EXT-X-DISCONTINUITY`/TS-indicator re-anchor, bounded 1.25× catch-up, jitter reordering — **not** exercised by the file/VOD race the as-built fix resolves. The deployed latch-on-tick sampling is what guarantees no-race/no-freeze on the file/VOD path. (The §Consequences "one ingest pipeline" / "`StreamVideoDecoder` drops the genpts fallback" bullets describe a plan that is **not** how the file/VOD path is built.)

## As-built (2026-06-14) — ENG-2b: Pacer + ReorderBuffer wired on the raw-RTP / ST-2110 ingest

ENG-2b closes points 2–3 **on the path where they actually matter** — the `FrameProducer`-based raw-RTP / ST-2110 ingest, where the network reorders, bursts on connect, and the 32-bit RTP timestamp wraps (~13.25 h at 90 kHz). The pieces are wired into `multiview-input::source::IngestPump` (the producer-agnostic ingest core that ST-2110 flows through):

- **`ReorderBuffer` (jitter) — already wired, now exercised on the live path.** `IngestPump` holds an optional bounded `ReorderBuffer<ProducedFrame>` keyed by **raw** PTS *before* normalization (the normalizer enforces monotonicity by arrival order, so it must see display order). It is **bounded at `jitter_depth + 1` and drops-oldest, never grows** (invariants #5/#9): a frame below the release watermark is rejected (`DroppedLate`), and the oldest is evicted (`DroppedToMakeRoom`) when a newer frame arrives at capacity.

- **`Pacer` — newly wired, behind the reorder buffer, fed the normalized PTS.** A new **`PacePolicy`** on `IngestConfig` selects the ingest-side pacing:
  - **`Passthrough` (default)** — file / VOD / the latch-on-tick path. No ingest pacing; a producer that runs ahead is harmless (the bounded media-time ring drops oldest and the output samples latch-on-tick). This preserves the point-4 as-built fix above unchanged.
  - **`WallClock`** — the raw-RTP / ST-2110 / live policy. `IngestPump` builds a `Pacer` and, per **point 3**, places it **behind** the reorder buffer, feeding it the **normalized** PTS (reorder on raw PTS → normalize → pace → publish). Each frame is released at `anchor_wall + (normalized_pts − pts0)`, smoothing a connect burst or jittery arrival *into the store*.

- **The pacer is ingest-side only (invariant #1).** The normalized timeline is anchored **source-relative** (first frame at 0); the pacer — not the normalizer — owns the wall-clock relationship, so the stored `meta.pts` stays a stable source-relative instant and the **output clock still re-stamps from its own tick counter**. A bursting / wrapping / stalled input never paces or stalls the program output. The pacer is **clock-injected** (`pump_one_paced(now_ns)` and the supervised-loop `run_paced_to_end` over an injected `PaceClock`), so the path never blocks the engine — it returns a `PaceStep::{WakeAt,Pending,Eos}` telling the ingest task when to next poll — and is deterministic + sleep-free under test. On the live path `Ok(None)` from the producer means *nothing ready this poll* (not EOS — a real end is a fault the supervisor handles), so the clock drains the pending queue across polls.

- **Bounded pending queue.** Frames normalized but not yet due are held in a `pending` queue capped by `pace_pending_max` (default 8); at the cap the oldest is published early rather than the queue growing (drop-into-the-single-slot, never unbounded — invariants #5/#9).

- **`PtsWallClock` stays.** It already satisfies inv #4 on the (now-normalized) timeline for the libav demux path, and the raw-RTP path gets its pacing from `IngestPump`'s `Pacer`; retiring `PtsWallClock` remains the cosmetic de-dup point-1 envisioned, not a correctness gap.

**Verification (deterministic, flake-free, clock-injected — no sleeping):** `multiview-input/tests/st2110_ingest.rs` drives a **real `St2110Producer`** through `run_paced_to_end` — a back-to-back frame burst releases spaced by one 90 kHz frame period (smoothed, not flooded), and a **32-bit RTP wrap-crossing soak** (a run of 30 fps frames rolling `2^32 → 0`) releases with uniform spacing and no regression / explosion (asserted against the exact float-free `rescale`). `multiview-input/tests/pacer_ingest.rs` pins the pump contract directly (passthrough immediate; wall-clock paced by normalized PTS; bursty-arrival smoothed; out-of-order reordered then paced monotonic; reorder overflow drops-oldest), and `pacer_reorder_properties.rs` is a `proptest` proving arbitrary out-of-order RTP yields a strictly-monotonic bounded sequence and that wrap-crossing sequences never regress. The content-aware overlays-OFF GPU/soak wrap verification (ENG-2b item (b)) remains a soak-tier check, honestly noted as gated where the GPU tier is unavailable in CI.

**Other paths (unchanged, by design):** the **WebRTC** ingest is the compressed-media-event seam (`WebRtcProducer::next_event`, not a `FrameProducer`); its RTP de-jitter is owned by the `multiview-webrtc` `RtpRing` and the keyframe-gated depacketizer, and as a low-latency contribution path it does not add a wall-clock ingest pacer — wiring `IngestPump` there is not in scope for ENG-2b. The **NDI** and **libav demux** producers present frames in presentation order and use the `Passthrough` policy (the output clock paces emission).
