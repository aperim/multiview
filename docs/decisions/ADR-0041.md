# ADR-0041 — SAP (Session Announcement Protocol) discovery + announcement: one engine, unifying the AES67 SAP/SDP plan

**Status:** Accepted (2026-07-11) — the essence-agnostic SAP engine (RFC 2974 packet codec, multicast group set, bounded drop-oldest session table, announce schedule) is implemented in `multiview-input/src/sap/` with the adversarially-corrected wire details honoured (3-bit version, `auth_len` counted in 32-bit words, ±1/3 announce jitter, `max(10×period, 1 h)` purge, full-group-join, reject `E=1`, cap `C=1` zlib inflate at `MAX_SDP_PAYLOAD` (Decision 8), never hash 0), the socket listener/announcer behind the `st2110` feature and loopback-validated. The listener rate-limits datagrams into the O(n) RCU fold with a fixed-window budget (`SapRateLimiter`, Decision 3) and purges on a cadence rather than per-datagram, so a spoofed-origin flood cannot force the expensive clone at line rate and starve the shared control-plane runtime (inv #10; panel finding F4). The general RFC 8866 `sdp/` model + the `st2110/sdp.rs`→`Aes67Profile` collapse remain a follow-on gated on the ADR-0040 media path (multicast RTP ingest / `Output::Udp`).
**Area:** streaming/timing (T) — discovery/signalling (implementation)
**Drivers:** plug-and-play interop with the multicast world VLC/Dante-AES67/RAVENNA already speak
(SAP/SDP), and the open-standards + LGPL-clean mandate.
**Research:** [sap-discovery](../research/sap-discovery.md).

multicast carries), [ADR-0033](ADR-0033.md) (AES67 — whose audio-only SAP/SDP plan this unifies and
whose §6/§9 placement clause this supersedes), [ADR-0034](ADR-0034.md) (routing / `StreamInventory`),
[ADR-0035](ADR-0035.md) (`HealthWarning`), [ADR-0042](ADR-0042.md) (**IPv6-first** — SAP/SDP here is
IPv6-first per that ADR: join the IPv6 SAP group, `c=IN IP6` first-class/no-TTL; IPv4 is legacy).
**SAP here = Session Announcement Protocol, not subtitles.**

## Context

SAP (RFC 2974, Experimental) periodically multicasts an SDP describing a media session to a well-known
group on UDP 9875; receivers build a browsable list (VLC's "Network Streams (SAP)"). SAP carries only
the SDP descriptor — the media rides RTP/UDP multicast (the [ADR-0040](ADR-0040.md) transport).
Multiview wants to **discover** announced sessions (and let an operator bind one as a Source) **and
announce** its own multicast outputs (so VLC/Dante discover them).

No SAP code exists today. Two narrow SDP parsers exist to generalize: `nmos/is05.rs:340`
`parse_sdp_transport` (video `c=`/`m=video` only, self-described "not a full RFC 8866 implementation")
and the `webrtc/sdp.rs:138` `RtpMap`. Critically, the AES67 brief/backlog
([ADR-0033](ADR-0033.md) §6, AES67-5/-6) already plan a **narrow, audio-only** SAP+SDP under
`multiview-input/src/st2110/{sap.rs,sdp.rs}` (m=audio L16/L24, ptime, ts-refclk, mediaclk). A general
SAP/SDP layer that must serve video, audio, **and** MP2T cannot be a second implementation, and
`st2110/` is the home of essence-*specific* depacketizers (v20/v30/v40) — the wrong home for an
essence-agnostic engine. The NMOS IS-04/IS-05 node is the richer, trustworthy parallel discovery
already built; SAP is the simpler, unauthenticated, VLC/Dante-compatible LAN path.

Adversarial verification corrected several wire details that the implementation must honour: the SAP
version field is **3 bits** (value 1); the announce jitter is **±1/3** of the interval (the "2/3" is
the window width); receivers purge after **`max(10×period, 1 h)`**; the `application/sdp` payload-type
string is normally **omitted** (receiver detects `v=0`); `auth_len` counts **32-bit words**
(`skip auth_len*4` — VLC's byte-count parse is a bug); `m=video <port> udp 33` is a **convention, not
RFC-normative** (no `MP2T/AVP` token; the `/15`-style number on `c=` is a **TTL, not a CIDR prefix**);
VLC also joins the admin-scoped groups, so a receiver must join the **full group set**; SAP auth/`E`
are **optional and unimplemented in practice** (not formally deprecated); and the ~30 s cadence is
**Dante's choice, not an AES67-2018 requirement**.

## Decision

1. **Build one essence-agnostic SAP engine** in `multiview-input/src/sap/` (RFC 2974 packet codec +
   multicast group set + bounded session table + supervised listener/announcer) and **one general SDP
   model** in `multiview-input/src/sdp/` (RFC 8866 video + audio + MP2T parse/generate), both
   off-by-default under the existing **`st2110`** feature (no FFI, `unsafe_code = forbid`,
   LGPL-clean). The SAP packet codec is **SDP-content-agnostic**. Keeps discovery on the control plane
   (inv #10).

2. **The planned AES67 SAP/SDP (AES67-5/-6, [ADR-0033](ADR-0033.md) §6) become consumers/profiles** of
   this layer: `st2110/sdp.rs` collapses to an `Aes67Profile` validator/extractor atop `sdp/`; the
   AES67 receiver **filters** the shared SAP inventory; NMOS IS-05 `transport_file` and the generalized
   `is05.rs:340` video parser also delegate to `sdp/`. **No second SAP, no parallel SDP parser.** **This
   ADR supersedes the audio-only-placement clause of ADR-0033 §6/§9.** (DRY; one wire-contract per
   protocol.) Land the `sap/`+`sdp/` extraction **before or jointly with** AES67-5/-6 to keep that
   convergence a one-line re-point.

3. **SAP receive** is a supervised off-hot-path tokio **listener** that joins the group set on 9875,
   parses into a **fixed-capacity bounded drop-oldest** session table keyed on (msg-id hash,
   originating source) with per-source + global rate limits and implicit `max(10×period, 1 h)` purge,
   and publishes a newest-wins inventory snapshot via the `LatestState`/`ArcSwapOption` wait-free
   pattern (`isolation.rs`). Respects inv #1 (never on the output clock) + inv #10 (wait-free publish,
   engine never awaits, bounded drop-oldest).

4. **Discovered sessions are untrusted hints:** surfaced as candidates requiring explicit operator
   **confirm-to-bind**, **never auto-ingested**; the table is hard-capped + rate-limited +
   zlib-size-capped because SAP is unauthenticated/spoofable and gives no native protection. Respects
   inv #10 + bounded memory. Discovery is **opt-in** (join no groups until enabled).

5. **SAP announce** is an **independent tokio timer** (never the per-tick output loop) generating SDP
   from the authoritative output config and emitting it to the **scope-selected** SAP group on 9875
   TTL 255 at a deliberate **≥30 s** cadence (default; the RFC bandwidth-fair timer is an option),
   with a stable non-zero per-output hash, `"application/sdp\0"` prefix, and a courtesy `T=1` deletion
   on teardown. Respects inv #1 (announcer cannot pace output) + inv #11 (Class-2 migration sends a
   delete). **Announce ships with the multicast egress it announces** ([ADR-0040](ADR-0040.md) MC-4) —
   no announce-without-a-stream.

6. **Bind** a discovered raw-UDP-TS session via `SourceKind::Ts { url: "udp://@group:port" }` today
   (or the typed `SourceKind::Udp` once ADR-0040 MC-3 lands); RTP-framed-non-TS sessions are surfaced
   **"unbindable"** until the multicast RTP ingest kind exists (the ADR-0040 gap). A SAP-discovered
   Source is a **sampled** input feeding the last-good-frame store; binding it to a tile is **Class-1**
   live-apply. Respects inv #1/#2/#11.

7. **Choose the SAP group from the media address scope** (RFC 2365/2974), not TTL (TTL = 255 fixed);
   model the SAP group and the SDP `c=` media group as **two distinct typed fields**; receive joins the
   full four-IPv4 + IPv6 set (to catch AES67/Dante `239.255.255.255`). The **proto token** (`RTP/AVP`
   vs `udp`), not the PT, is the framing discriminator. (VLC/Dante interop correctness.)

8. **Wire-correctness + security hardening** (verified): reject `E=1` (no SAPv2 algorithm; VLC
   rejects); ignore inbound `T=1` deletions against confirmed inputs (hijack vector); **skip `auth_len`
   as words × 4** with bounds-checks (do **not** replicate VLC's byte-count bug); never emit or accept
   hash 0; cap zlib decompressed size; handle PT 33 with **and** without `a=rtpmap`; the version field
   is the top 3 bits. Respects correctness/security/spec-compliance.

9. **Crate/feature/placement:** new `sap/` + `sdp/` under `multiview-input` (`st2110` feature); the
   discovery endpoint extends the `multiview-control` NMOS surface (`GET /api/v1/discovery`, listing
   SAP **+** NMOS, clearly labelled by trust — **never auto-bridge SAP into the NMOS registry**). The
   media gaps (RTP-framed ingest, `Output::Udp` egress) are **owned by ADR-0040** (MC-3/MC-4); SAP
   declares the cross-deps and does not re-solve them.

## Consequences

- Multiview becomes plug-and-play **discoverable by and able to discover** VLC/Dante-AES67/RAVENNA
  gear over open SAP/SDP, with **no FFI and no GPL escalation** (pure-Rust under the existing `st2110`
  feature, `unsafe_code = forbid`).
- **One** RFC 2974 codec and **one** RFC 8866 model serve video + audio + MP2T; the AES67-5/-6 plan
  collapses to thin profiles (ADR-0033 §6/§9 superseded), eliminating a second SAP/SDP stack.
- Discovery is a **bounded, untrusted, operator-confirmed** surface that physically cannot
  back-pressure the engine (inv #1/#10); it is complementary to — never merged with — the trusted NMOS
  registry.
- Announce is gated on the multicast egress (ADR-0040 MC-4), so the two land together.
- The implementation backlog (`SAP-0..SAP-10`) is in
  [docs/development/sap-discovery-backlog.md](../development/sap-discovery-backlog.md); it depends on
  ADR-0040 (MC-3/MC-4) for the media path and shares the `channel_bridge` drop-oldest fix (AES67-4).

## Alternatives rejected

- **Build SAP separately from AES67** (a second RFC 2974 codec + a second SDP parser under `st2110/`
  for audio, and a different one here for video/MP2T). Rejected: duplicates the wire contract, two
  places to fix RFC bugs, violates DRY and the no-second-SAP mandate; AES67-5/-6 are re-pointed at the
  shared layer instead.
- **Put the general SAP engine under `st2110/`** (as the AES67 brief originally planned). Rejected:
  `st2110/` is the home of essence-*specific* depacketizers; an essence-agnostic engine + general SDP
  model belong in sibling `sap/` and `sdp/` modules.
- **Auto-ingest discovered SAP sessions** (join + decode on discovery). Rejected: SAP is
  unauthenticated/spoofable; auto-joining lets any LAN host steer the engine to arbitrary multicast
  groups (join-amplification / resource exhaustion) or feed malicious media. Discovery → operator-
  confirmed candidate → bind only.
- **Skip SAP in favour of NMOS-only discovery.** Rejected: NMOS does not interoperate with VLC or
  Dante-AES67 gear (which speak only SAP/SDP); SAP is the plug-and-play LAN path and the
  VLC-discoverability requirement. SAP is complementary to the already-built NMOS, not replaced by it.
- **Auto-bridge SAP-discovered sessions into the NMOS IS-04 registry.** Rejected: NMOS is a
  higher-trust, registry-based world that does not mention SAP; injecting spoofable SAP hints pollutes
  it. Keep SAP a clearly-labelled lower-trust list alongside NMOS.
- **Honour inbound SAP `T=1` deletions** to drop discovered/confirmed sessions. Rejected: a third
  party can spoof a deletion to hijack/deny an announcement (VLC ignores them for this reason); expire
  by timeout, and never let a network packet delete an operator-confirmed running input.
- **Implement the strict RFC 2974 bandwidth-limited `max(300 s, …)` timer as the only announce
  cadence.** Rejected as the default: a 300 s floor makes sessions appear sluggishly and
  AES67/Dante/VLC use a short fixed interval; default to ≥30 s for interop and offer the bandwidth-fair
  timer as an option for many-session scale.
- **Replicate VLC's `auth_len`-as-bytes parse.** Rejected: RFC 2974 defines it as 32-bit words
  (`skip auth_len*4`); copying VLC's bug mis-parses any authenticated packet. A regression test asserts
  we do not copy it.
- **Support encrypted (`E`) SAP / set the RTP marker bit.** Rejected: `E` has no SAPv2 algorithm and
  VLC rejects it — reject on receive, never set on send; the marker bit is irrelevant to discovery.
