# ADR-0052 — Conspect two-pipe model: mandatory licensing heartbeat vs opt-in telemetry, consent (LWW), consent-independent metrics retention

**Status:** Proposed (2026-06-10)
**Area:** Account / Telemetry — the contact + consent + retention plane of the Conspect subsystem
**Drivers:** Two machine→Aperim contacts exist and must **never** be conflated: a **mandatory monthly
licensing heartbeat** (the minimum contact that keeps an entitlement lease live) and an **opt-in daily
product telemetry** stream (anonymised analytics). Co-mingling them — one toggle, one transport, shared
copy — would make "opt out of telemetry" look like "lose your licence", and would let a privacy switch
silently affect entitlement. The separation is a **hard operator directive** (spec).
**Research:** [conspect-account-architecture](../research/conspect-account-architecture.md) §7, §8.
**Related:** [ADR-0050](ADR-0050.md) (the heartbeat *grants the lease* — this ADR owns its
*separation* from telemetry), [ADR-0051](ADR-0051.md) (relay carries the heartbeat, never telemetry),
[ADR-0053](ADR-0053.md) (the local metrics buffer feeds the support context-pack),
[ADR-M006](ADR-M006.md) (config-as-code reference-only secrets — telemetry/context never carry secret
values), [ADR-RT004](ADR-RT004.md) (isolation invariant #10 — both pipes are off-hot-loop).
**Extends:** `multiview-telemetry` (additive; the existing tracing/Prometheus/health surface is
untouched).

## Context

The licensing heartbeat is **not** "telemetry you can't turn off." It is the *licensing keep-alive*: it
carries only what licensing needs (a salted machine-fingerprint digest + a lease request + signed
assertions, brief §8) and it is mandatory for the official build's licence to stay live. Telemetry is
*product analytics*: anonymised, aggregated usage counters, **opt-in**, default off, revocable. These
are different purposes, different payloads, different cadences, and must be different *everything* the
user touches — transport, consent, copy, API surface, retention.

A third, subtle requirement: the machine keeps a **local** rolling metrics buffer for its **own**
diagnostics (the support context-pack, [ADR-0053](ADR-0053.md)). That buffer must be **independent of
telemetry consent** — turning telemetry *off* stops the *outbound daily pipe*, it does not stop the
machine keeping a bounded local buffer for the operator's own support exchanges. Mis-selling "opt out"
as "the machine keeps nothing" would be dishonest.

## Decision

### 1. Two pipes, separate in every dimension

| | **Licensing heartbeat** | **Product telemetry** |
|---|---|---|
| Purpose | Keep the entitlement lease live (ADR-0050) | Anonymised product analytics |
| Cadence | **Monthly minimum** (within `ACTIVATION_WINDOW = 31d`) | **Daily** when enabled |
| Consent | **Mandatory** for the official build's licence (it *is* the keep-alive, not analytics) | **Opt-in**, default **off**, revocable |
| Payload | Salted fingerprint digest + lease request + signed assertions | Aggregated/anonymised counters; **never** raw identifiers |
| Transport | Entitlement endpoint (`multiview-licence` `heartbeat`) | Telemetry endpoint (distinct host/path) |
| API | `…/account/licence`, `…/account/licence/heartbeat` | `…/account/telemetry`, `…/account/consent` |
| UI | The **Licence** screen | The **Data** screen |
| Retention | Heartbeat/lease history, bounded, licensing-scoped | Local buffer is **consent-independent** (§3) |

The settings UI, the API docs/OpenAPI descriptions, and the privacy copy **must never** present the two
as one switch. This is asserted by a test that no telemetry field appears in a heartbeat payload and
vice-versa.

### 2. Consent is a versioned single-value document; last-writer-wins

Telemetry consent is `{ telemetry: bool, updated_at, updated_by }` at `GET/PUT /api/v1/account/consent`
(`If-Match`). Concurrent edits (machine UI vs portal) resolve **last-writer-wins** by `updated_at` —
the simplest correct rule for a single preference where the only risk is staleness, not merge-loss.
The **heartbeat consent is NOT in this document**: it is implicit in running the official build, and
the Licence-screen copy says so plainly. There is exactly one consent surface, and it governs only the
*outbound telemetry* pipe.

### 3. Consent-independent local metrics retention

`multiview-telemetry` keeps a **bounded, auto-pruned, local** rolling metrics buffer (engine-seam S5,
sampled off the hot loop). It is retained **regardless of telemetry consent** because it serves the
*operator's own* support context-pack ([ADR-0053](ADR-0053.md)), not Aperim analytics. Turning telemetry
**off** stops the *outbound daily pipe*; it does **not** empty or stop the local buffer. The Data
screen states this distinction explicitly. The buffer never leaves the machine without an explicit
operator-approved data request (ADR-0053). A test asserts that disabling telemetry leaves the local
buffer populating.

### 4. Data minimisation across both pipes

Neither pipe transmits raw serials/MACs/media URLs/stream contents/user paths (brief §8). The heartbeat
carries salted digests + signed assertions only; telemetry carries aggregated/anonymised counters and
histograms only, with anonymisation applied **before** anything enters the daily pipe. A test asserts
no raw identifier appears in either payload.

### 5. Isolation (invariant #10)

Both pipes and the local sampler are **off the hot loop**: separate tokio tasks publishing/reading via
watch/`arc_swap`/bounded drop-oldest, never holding a lock the engine holds, never `.await`ed by the
engine. A wedged telemetry pipe loses *outbound analytics*, never a tick and never the heartbeat. The
heartbeat task publishes `LicenceStatus` ([ADR-0050](ADR-0050.md) S4) wait-free.

### 6. Feature ladder + licensing

The outbound telemetry pipe is additive in `multiview-telemetry`, **off by default** (opt-in). The
local buffer is always available. No new licence-escalating dependency; nothing touches GPL/proprietary
boundaries.

## Consequences

- **Honest privacy story** — opting out of telemetry never threatens the licence, and never silently
  empties the operator's own local diagnostics. The two are explained as two things because they are.
- **Licence keep-alive survives privacy choices** — the heartbeat is the licensing contact, separate
  from analytics consent, relayable offline ([ADR-0051](ADR-0051.md)).
- **Operator keeps their diagnostics** even with telemetry off — the consent-independent local buffer
  powers support without exporting anything until explicitly approved.
- **Minimal, anonymised** — salted digests + aggregated counters only; no raw identifiers anywhere.
- **Isolation preserved** — both pipes are off-hot-loop; invariant #10 holds.

## Alternatives rejected

- **One "phone home" switch governing both.** The dishonest, directive-violating design — makes a
  privacy opt-out look like a licence-killer and lets a privacy choice affect entitlement. Rejected;
  the separation is the whole point of this ADR.
- **No local buffer when telemetry is off.** Mis-sells "opt out" as "keeps nothing" and strips the
  operator of their own diagnostics. Rejected for consent-independent local retention with explicit-
  approval egress.
- **CRDT/merge consent.** Over-engineered for a single boolean preference where staleness is the only
  risk. Rejected for last-writer-wins by `updated_at`.
- **Raw identifiers in telemetry "for better analytics".** A privacy and correlation hazard with no
  product need. Rejected — aggregated/anonymised counters only, salted digests for the heartbeat.
