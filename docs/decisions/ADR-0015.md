# ADR-0015: YouTube live ingest via an external runtime-discovered resolver (yt-dlp)

- **Status:** Proposed
- **Area:** Core Engine
- **Date:** 2026-06-03
- **Source brief:** [io/youtube-live.md](../io/youtube-live.md)
- **Relates to:** [ADR-0008](ADR-0008.md) (NDI runtime-load precedent), [ADR-0012](ADR-0012.md) (LGPL-clean default build), [ADR-0009](ADR-0009.md) (data plane vs control/IO plane), [ADR-T004](ADR-T004.md) (HLS ingest pacing), [ADR-T003](ADR-T003.md) (PTS normalisation), [ADR-R001](ADR-R001.md)/[ADR-R003](ADR-R003.md) (continuous output + supervision), [ADR-M007](ADR-M007.md) (CapabilityReport gate), [ADR-M006](ADR-M006.md) (reference-only secrets)

## Decision

Support **live YouTube** as an input by adding a `youtube` source kind to the adjacently-tagged source
union, implemented as a **resolver** that invokes an external **`yt-dlp`** binary — **discovered at
runtime, never vendored or linked** — to turn a YouTube watch/live/channel URL into a concrete,
FFmpeg-readable **HLS master-playlist URL** (`*.googlevideo.com`). That URL is fed into Multiview's
**existing HLS ingest path** (libav demux/decode + the custom HLS input pacer + PTS normalisation +
supervised reconnect); a YouTube source is a thin wrapper over `hls`. The resolver and a **periodic
re-resolution loop** (refreshing the URL before its `expire`-timestamp TTL — ~6 h for live HLS) run
off the data plane on the control/IO plane. `yt-dlp` is an **optional runtime dependency** behind an
off-by-default `youtube` Cargo feature, probed via `CapabilityReport`; the default Multiview build neither
contains nor requires it. Pin the `web_safari` player client (PO-token-free live HLS) and avoid `ios`.
Resolution failures degrade the **tile** (LIVE → STALE → NO_SIGNAL) and raise an operator alarm; they
never stall the output clock (invariants #1/#2/#10).

## Rationale

YouTube publishes no stable manifest URL: the playable HLS/DASH manifest lives behind the private
InnerTube `player` API and requires solving a frequently-rotated JavaScript `n`-signature and, for some
formats, a PO token. This is exactly the brittle, fast-moving scraping surface that must **not** sit on
the critical path of a *bulletproof continuous output* engine. Delegating extraction to `yt-dlp` — the
community-maintained reference implementation that Streamlink and many tools also rely on — and
consuming only its **processed output URL** isolates the fragility in a separate, operator-updatable
process. The live-HLS case is favourable: per yt-dlp's PO Token Guide (2026-03-10), HLS live streams do
not require a PO token (except the `ios` client), and the default `web_safari` client returns pre-merged
video+audio HLS renditions — so a live master URL is typically obtainable without tokens and is readable
by libav. Running `yt-dlp` as an out-of-process subprocess (it is public-domain / Unlicense) imposes **no
linking or copyleft obligation**, keeping the default build LGPL-clean. The runtime-discovery + feature-gate
+ `CapabilityReport` posture mirrors the proven NDI model (ADR-0008): the capability is simply reported
unavailable when the binary is absent, never crashed-on.

## Alternatives considered

- **Pure-Rust YouTube client (e.g. `rustypipe`) in-process** — attractive (no external binary, no
  subprocess) but moves YouTube's n-sig/PO-token/player-client churn onto Multiview's own release cadence:
  if the crate lags a player change, *every* YouTube tile breaks in-process. Deferred as a **future
  alternative backend** behind a `youtube.resolver = yt_dlp | rustypipe` knob, to adopt once it is proven
  to track YouTube as reliably as yt-dlp.
- **Re-implement the InnerTube flow + n-sig solver directly in Multiview** — rejected: the highest-maintenance,
  most-breakage-prone code imaginable on a continuous-output engine; we would own the entire fragile surface.
- **Vendor / bundle yt-dlp's Python** — rejected: pulls a Python runtime and its dependency licenses into
  the product and defeats the LGPL-clean default, while still requiring updates for n-sig churn.
- **Hand-copied manifest URLs** — rejected: resolved googlevideo URLs expire (~6 h for live HLS, HTTP 403
  past `expire`); binding must be by YouTube URL with automatic re-resolution.
- **`--live-from-start`** — rejected for the live-edge use case: it downloads from the broadcast's start
  via DASH `is_from_start` processing, the opposite of joining the live edge now.

## Consequences

- **Operational fragility is expected and handled.** Extraction can break when YouTube rotates its player
  JS / n-sig; operators must keep `yt-dlp` current (and ideally provide a JS runtime such as Deno).
  Multiview records the resolver version, surfaces a staleness/extraction-failure alarm, and degrades the
  tile rather than the output. The resolver runs as a supervised subtask with a hard timeout and bounded
  backoff (ADR-R003); a hung `yt-dlp` is killed, not awaited (invariant #10).
- **Re-resolution is load-bearing.** The implementation must parse the `expire` Unix-timestamp deadline,
  refresh ahead of it (lead time), do a make-before-break source swap (Class-1-style, ADR-R004/ADR-M005),
  and also re-resolve on a sustained 403 burst. Treat the ~6 h figure as an upper-bound guard, not the
  source of truth.
- **Licensing stays clean by construction.** The subprocess boundary + Unlicense means no obligation
  attaches to the Multiview build; the default build excludes the feature entirely. (An operator who
  *redistributes* a PyInstaller-bundled yt-dlp should review that bundle's dependency licenses — Multiview
  ships neither.)
- **Security & ToS.** Cookies are secret-ref only (ADR-M006); `yt-dlp` is spawned with an argument vector
  (no shell); stderr is captured/redacted. Lawful use (YouTube ToS, content rights) is the operator's
  responsibility, surfaced as an explicit UI acknowledgement (mirroring the NDI license notice).
- **Scope/placement.** This is **backlog**, building on **M4 (Full I/O)**; it is not started. The `youtube`
  module lives inside `multiview-input` (no new crate, per conventions §3). Implementation is phased in
  [io/youtube-live.md §10](../io/youtube-live.md). Because YouTube's player surface moves often, the
  cited yt-dlp line references and the ~6 h TTL must be re-verified at implementation time.
