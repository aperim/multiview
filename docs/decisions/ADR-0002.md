# ADR-0002: rsmpeg as the primary libav binding

- **Status:** Proposed
- **Area:** Core Engine
- **Date:** 2026-06-02
- **Source brief:** [core-engine.md](../research/core-engine.md)

## Decision

Use rsmpeg (FFmpeg 8.x) as the primary safe libav binding, with raw ffi:: access for custom zero-copy frame pools, behind an unsafe boundary module in mosaic-sys/mosaic-ffmpeg.

## Rationale

rsmpeg is the only mainstream Rust binding with safe wrappers for the full hwaccel lifecycle (AVHWDeviceContext::create, set_get_format, set_hw_device_ctx, AVHWFramesContext, hwframe_transfer_data) AND shipped hw_decode/vaapi_encode examples, while still exposing complete raw FFI for custom AVHWFramesContext pools and av_hwframe_map. This minimizes error-prone C-style hwaccel plumbing.

## Alternatives considered

ffmpeg-the-third / ffmpeg-next (rejected as primary: no safe hwaccel API, all hwaccel via raw as_mut_ptr; ffmpeg-next is maintenance-only). ez-ffmpeg/rsmedia/ac-ffmpeg (rejected: too abstracted or WIP for a custom on-GPU engine).

## Consequences

The get_format callback remains unsafe extern "C" on the decode thread (no fully-safe hwaccel path exists in any binding). Must pin exact FFmpeg version against rsmpeg's CI matrix; audit the specific safe methods used; libclang required for bindgen or ship pre-generated bindings.
