## Context

All raw libav\* FFI is owned by `crates/multiview-ffmpeg`, behind the off-by-default `ffmpeg`
feature, via the **`ffmpeg-next`** binding (+ `ffmpeg-sys-next` build crate; both **WTFPL**).
The binding is pinned `ffmpeg-next = "7.1"` (`crates/multiview-ffmpeg/Cargo.toml:29`, mirrored
in `crates/multiview-output/Cargo.toml:53`, `crates/multiview-cli/Cargo.toml:53`); the lockfile
resolves `ffmpeg-sys-next 7.1.3` (`Cargo.lock:1298`), capping the *current* build at FFmpeg 7.1
/ libavcodec 61.

Today the deploy images obtain libav\* from **third parties**:
- `deploy/Dockerfile` â€” Debian-trixie apt `libav*` (`:71-77` build, `:115-124` runtime).
- `deploy/Dockerfile.nvidia` â€” the **jellyfin-ffmpeg7 apt repo** + a `LD_LIBRARY_PATH`
  override (`:92-109`).

Both are rejected for the product: opaque provenance (no per-flag manifest for an SBOM),
a foreign patchset on a foreign release cadence, and a **codec grab-bag** â€” jellyfin-ffmpeg is
built `--enable-gpl` with x264/x265, so the NVIDIA image installs a **GPL artifact under a
"default build" banner**, defeating the repo's `gpl-codecs`-is-opt-in invariant. CLAUDE.md Â§7
and `conventions.md` Â§7 require the **default build to stay LGPL-clean** with GPL strictly
opt-in.

Verified facts that shape the decision:
- **Licensing gate (from FFmpeg's `configure`):** only `cuda_nvcc`, `cuda_sdk`, `libnpp`
  (nonfree), `libx264`/`libx265` (GPL), `libfdk_aac`/`openssl` (nonfree) taint a build.
  **NVENC, NVDEC, cuvid, ffnvcodec, cuda-llvm, vaapi, libvpl (QSV), videotoolbox are
  LGPL-clean.** We scale with `scale_cuda` via `--enable-cuda-llvm` + `nv-codec-headers`
  (no CUDA SDK), never `scale_npp`.
- **Binding ceiling:** `ffmpeg-sys-next` latest is **8.1.0**, whose build.rs maps `ffmpeg_8_1`
  â†’ libavcodec **62**. We are **not** capped at 7.x; the 8.1 move is a one-line binding bump +
  `cargo update` + soname-reference churn in both Dockerfiles.
- **Sonames:** 7.1 â†’ libavcodec **61**; 8.x â†’ **62** (8.0â†”8.1 ABI-compatible; 7.1â†’8.x is not).

## Decision

**Build our own FFmpeg** from a pinned upstream source tarball, replacing jellyfin-ffmpeg and
the apt/PPA sources entirely.

1. **Version.** Ship **FFmpeg 7.1.4** (latest 7.1 patch; libavcodec **61**) in the first cut â€”
   it matches the pinned binding, so it lands the provenance win with **zero Rust-code change
   and zero soname churn**. **Then** bump to **FFmpeg 8.1.1** (libavcodec **62**) as a *gated*
   follow-on after moving the binding to `ffmpeg-next = "8.1"` and re-verifying `--locked`.
   The builder takes `FFMPEG_VERSION` as an ARG so this is a one-line change.
2. **Source provenance.** Pinned `ffmpeg-<V>.tar.xz` from `ffmpeg.org/releases`, verified by
   **GPG** (key `FCF986EA15E6E293A5644F10B4322F04D67658D8`) **and** SHA-256
   (checksums TODO at implementation â€” see below).
3. **Licensing profiles.** Two configure profiles selected by a `FF_LICENSE` builder ARG:
   - **`lgpl` (default, LGPL v2.1+):** the Â§"enable-flags" line below; **no** `--enable-gpl`,
     **no** `--enable-nonfree`, **no** `--enable-libnpp/--enable-cuda-nvcc/libfdk_aac/openssl`.
   - **`gpl` (opt-in, `-gpl` tag):** identical **plus** `--enable-gpl --enable-libx264
     --enable-libx265`; whole build becomes GPLv2+. Pairs with the existing `gpl-codecs` cargo
     feature. **Never the default.**
   - **`--enable-nonfree` is banned** â€” it produces an unredistributable binary.
4. **Linkage.** **Shared, our-built** libav\* by default (`--enable-shared --disable-static`,
   copied builderâ†’runtime; `ldconfig`) â€” kills jellyfin/PPA while keeping the LGPL story
   trivial (no relinkability paperwork). A **static** single-binary option (no runtime soname
   coupling) is a scoped follow-on (the `ffmpeg-sys-next` `static` feature; LGPL-2.1 Â§6
   relinkability satisfied because we already ship the pinned source tarball).
5. **Reproducible multi-arch builder.** A `deploy/Dockerfile.ffmpeg` builder stage (pinned
   `debian:trixie@sha256:`, `SOURCE_DATE_EPOCH`, pinned `nv-codec-headers` tag
   `n13.0.19.0`) producing `/opt/multiview-ffmpeg`, cached as its own digest-referenced image.
   **linux/amd64 + linux/arm64** via per-arch native buildx; QSV (`libvpl`) flags guarded
   amd64-only. It **replaces** the apt `libav*` (`deploy/Dockerfile`) and the **jellyfin block**
   (`deploy/Dockerfile.nvidia:92-109`).
6. **Rust-binding constraint.** FF-0 keeps `ffmpeg-next = "7.1"` (resolved `ffmpeg-sys-next
   7.1.3`, libavcodec 61). The 8.1 bump moves all three declarations to `"8.1"`, runs
   `cargo update -p ffmpeg-sys-next`, and is **gated** on a green `--locked` build + the
   soname-reference updates in both Dockerfiles and the `hwdecode.rs:118` comment.
7. **Reduce-reliance stance.** FFmpeg **stays** as the codec + hwaccel engine (no production
   pure-Rust H.264/HEVC encoder exists; pure-Rust decoders are not live-grade; HW codecs are
   vendor SDKs FFmpeg portably wraps). We commit to **shrinking the surface to codec + hwaccel
   only** over time â€” egress containers, ingest protocols, and audio decode move toward
   pure-Rust (much already in-house), behind feature flags, A/B-tested. **Elimination is a
   non-goal.**

## Consequences

**Positive**
- Provenance: pinned source + GPG + SHA-256 + our exact flags â†’ attestable SBOM; no foreign
  patchset/cadence.
- Licensing: a genuinely LGPL-clean default (fixes the NVIDIA image's GPL mislabel) with GPL
  strictly opt-in and nonfree banned.
- **NVENC/NVDEC in the LGPL build** â€” the `select_encoder` fallback-to-mpeg2 concern on the
  NVIDIA image disappears *without* going GPL.
- Smaller attack surface (`--disable-everything` allowlist â‰ˆ the codecs/muxers/protocols
  actually used) vs the grab-bag.
- We own the soname line end-to-end; no third-party bump can break the binding silently.

**Negative / costs**
- +5â€“15 min build per arch (mitigated by a cached, digest-pinned builder image).
- We own FFmpeg CVE-tracking and flag upkeep (tracked alongside `cargo deny`).
- The 8.1 bump is a coordinated change (binding + lockfile + soname refs in both Dockerfiles +
  `hwdecode.rs:118`) â€” deliberately sequenced after the builder exists.
- LGPL-static (if FF-3 is taken) carries a relinkability obligation â€” satisfied by shipping the
  pinned source tarball, but it is a documented obligation.

## Alternatives rejected

1. **Keep jellyfin-ffmpeg / apt / PPA.** Rejected: opaque provenance, foreign patchset/cadence,
   ships GPL x264/x265 under a "default" banner (a licensing defect), uncontrolled soname.
2. **Use the `ffmpeg-sys-next` `build` feature (let the crate fetch + compile FFmpeg).**
   Rejected: it is *a* build from source but not *our pinned, GPG-verified, flag-audited*
   source; we want explicit control of the tarball, checksum, and configure line.
3. **Jump straight to FFmpeg 8.1.1 in FF-0.** Rejected as the *first* step: it couples the
   provenance win to a binding bump + soname churn across two Dockerfiles + a `--locked`
   re-verify. Sequenced as the gated FF-0b instead, keeping FF-0 a zero-code-change drop-in.
4. **Static-link by default in FF-0.** Rejected as default: adds an LGPL-2.1 Â§6 relinkability
   obligation for marginal benefit over our-built shared `.so`. Offered as the scoped FF-3
   option.
5. **`--enable-nonfree --enable-cuda-nvcc --enable-libnpp` (NVIDIA's guide line).** Rejected:
   produces a nonfree/unredistributable build; unnecessary because `--enable-cuda-llvm` +
   `scale_cuda` give LGPL-clean NVENC/NVDEC/scaling without the CUDA SDK.
6. **OpenSSL for TLS.** Rejected: OpenSSL is nonfree-incompatible for FFmpeg; use **gnutls**.
7. **Eliminate FFmpeg (pure-Rust codecs).** Rejected: no production pure-Rust H.264/HEVC
   encoder; pure-Rust decoders not live-grade; HW codecs are vendor SDKs. We shrink the
   surface instead (Â§Decision 7).
