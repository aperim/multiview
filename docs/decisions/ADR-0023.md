# ADR-0023: Region-limited, LUT'd overlay bake — end the full-canvas colour round-trip (inv #5)

- **Status:** Accepted
- **Date:** 2026-06-04
- **Area:** compositor / overlay sub-pass (CPU reference bake)
- **Relates to:** [ADR-0022](ADR-0022.md) (real-time software compositor — the sibling fix), [ADR-0016](ADR-0016.md) (overlay rendering), invariants **#5** (NV12-throughout) and **#8** (fixed colour order).

## Context

`multiview_compositor::overlay::apply_overlays_to_nv12` (the CPU overlay bake the CLI
software path uses to burn labels / clocks / meters / fault badges / DVB-sub cues
into the program canvas) re-converts the **entire** NV12 canvas every frame:

1. decode **every** pixel YUV→linear (the transcendental `tile_yuv_to_canvas_linear`, EOTF per channel),
2. `blend_overlays` over the seeded buffer (this part already walks only each primitive's footprint), then
3. re-encode **every** pixel linear→YUV (the transcendental `canvas_linear_to_output_yuv`, OETF per channel).

Two problems:

- **Performance.** The dominant cost is the two full-canvas transcendental passes,
  **un-LUT'd** (it does not use the SSIM-gated transfer LUT that ADR-0022 built for
  the compositor). Measured: every 1280×720 demo renders in ~24 s for 10 s of
  content (**2.4× real-time**) **regardless of overlay or tile count**
  (`clockdemo` 1-tile 24.7 s ≈ `faults` 4-tile 22.7 s), while the *same* layouts
  rendered overlays-OFF are real-time (9-tile 11.2 s/10 s). The cost tracks canvas
  pixels, not overlay content — the signature of a full-canvas round-trip.
- **Invariant #5 (efficiency).** It materializes a full-frame linear RGBA buffer
  and re-converts pixels **no overlay touches**. Invariant #5 says *never
  materialize full-frame RGBA; YUV→RGB at tile size* — the bake violated the
  spirit of that for the ~90 % of the canvas no overlay covers. (Note: the
  YUV→linear→YUV round-trip is *effectively lossless* for in-range SDR values —
  verified: a vivid saturated-chroma frame round-trips **byte-identically** — so
  this is a **performance**, not a pixel-correctness, defect. The fix preserves
  output; it just stops doing the redundant work.)

## Decision

**Region-limit the round-trip — one even-aligned dirty rect per overlay
primitive, copy everything else through byte-identically.**

1. Compute one dirty rect **per primitive footprint** (not a single union box, so
   the gaps between spread-out overlays — labels/meters/clock scattered across the
   frame — stay untouched and only the real footprints are processed). Each rect is
   **snapped out to even x/y** so every NV12 2×2 chroma block is wholly inside or
   outside (matching `write_nv12_pixel`'s 2×2 model). Per-primitive bounds come
   from the same geometry each `blend_*` walks: Glyph/FilledRect/Line/Image = their
   rect; Stroke = endpoints ± (`half_thickness` + 1px AA); Ring = centre ±
   (`outer_radius` + 1px AA) — using the same `bbox_span` the blend uses, so the
   cover is exact.
2. Copy the input NV12 Y/UV planes through verbatim, then decode→`blend_overlays`→
   re-encode **only those rects** (walked in raster order, so chroma matches).

**Use the transcendental oracle, not a LUT.** Region-limiting already cuts the
processed pixels to the small overlay footprint, so the per-pixel oracle cost is
negligible here — and using the same oracle as the reference keeps the bake output
**byte-identical** to the full-canvas reference *inside* the rects. The transfer
LUT (ADR-0022) belongs on the full-canvas per-tile **composite** hot path, not this
small per-frame bake; pulling it in here would only add a ≤2-code-value
LUT-vs-oracle drift for no real speed benefit.

`blend_overlays` and every `blend_*` primitive routine are **unchanged** — the GPU
sub-pass oracle equivalence (ADR-0016) is preserved. The previous full-canvas body
is kept as `apply_overlays_to_nv12_reference` (the correctness oracle for tests).

## Consequences

- **Output is preserved.** Inside the overlay rects the bake is **byte-identical**
  to the full-canvas reference (same oracle, same raster-order chroma); outside
  every rect it is exact NV12 passthrough (the old full-canvas round-trip merely
  drifted a code value or two on saturated chroma — our passthrough is bit-exact).
  Transcendental work drops from O(canvas) to O(overlay footprint) — the win.
- The CPU bake is the software/CLI path; the product composites overlays on GPU
  (`gpu_subpass`). This makes the software path efficient and inv-#5-clean; the GPU
  path is unaffected.
- **Gates (TDD):** (1) pixels outside the overlay rects are byte-identical to the
  input planes; (2) an empty overlay list returns the input unchanged; (3) the
  optimized output is byte-identical to the full-canvas reference inside the rects
  and exact passthrough outside — exercising every primitive kind, so a too-tight
  rect that clips an overlay edge fails it; (4) `overlay_dirty_rects` unit tests
  pin the per-primitive rect geometry (even-snap, clamp, off-screen, ring AA);
  (5) a perf bench asserts the region-limited bake is much faster than the
  full-canvas reference for a small overlay on a large canvas. Existing
  `overlay_subpass` / `overlay_efficiency` tests assert the blend math and are
  untouched (the blend path is unchanged).
- Verified end-to-end by re-rendering a gallery WITH overlays in release and
  confirming it is now ~real-time **and** the picture is unchanged (overlays-OFF
  content-aware motion still matches the ground-truth encode, per ADR-0022's gauge).

## Alternatives considered

- *Cache the rasterised glyph/clock bitmaps* — already done at the atlas layer
  (ADR-0016 T2); the cost here is the colour round-trip, not rasterisation.
- *Blend in a dirty-region-sized buffer with offset coordinates* — would require
  changing `blend_overlays`'s absolute-coordinate math (tested, GPU-oracle). Kept
  the full-size buffer (only the seed/encode loops are region-limited) to leave the
  blend math untouched; the buffer allocation is cheap relative to the transcendental
  savings and can be pooled later if needed.
