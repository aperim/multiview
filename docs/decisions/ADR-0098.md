# ADR-0098: Production switcher programme decomposition — sub-slice dependency order and parallel/serial boundaries

- **Status:** Proposed
- **Area:** Core Engine / Control Plane / Configuration (production switcher programme sequencing)
- **Date:** 2026-06-19
- **Source:** Agent session; extends [ADR-0054](ADR-0054.md) (switcher architecture),
  [ADR-0055](ADR-0055.md) (transition engine), [ADR-0056](ADR-0056.md) (keyers),
  [ADR-0057](ADR-0057.md) (media players), [ADR-0072](ADR-0072.md) (cue dispatcher),
  [ADR-M012](ADR-M012.md) (config model), [ADR-W021](ADR-W021.md) (REST API),
  [ADR-RT008](ADR-RT008.md) (realtime events), [ADR-0058](ADR-0058.md) (alpha media),
  [ADR-0059](ADR-0059.md) (switcher audio), [ADR-T015](ADR-T015.md) (timing)

## Context

The production switcher programme is specified across eleven Proposed ADRs
([ADR-0054](ADR-0054.md)–[ADR-0059](ADR-0059.md), [ADR-0072](ADR-0072.md),
[ADR-M012](ADR-M012.md), [ADR-W021](ADR-W021.md), [ADR-RT008](ADR-RT008.md)) that
collectively define a **large, interdependent feature set**: an M/E (mix/effects) stage
inside one Program with PGM/PVW buses, a transition engine with multiple families
(cut/mix/dip/wipe/DVE/stinger), upstream and downstream keyers, fade-to-black,
media library and players, audio-follow-video, macros and a cue dispatcher, with
complete REST/realtime/config surfaces.

None of this is built: all eleven ADRs are Proposed and the codebase carries zero
production switcher logic — only foundational seams exist (bounded command bus,
output-clock hook, salvo/tally machines, program-bus audio, frame-boundary apply
points, per-tile opacity + linear premultiplied compositing).

**Rule 6 (AGENTS.md) forbids partial shipping:** a thing is "done" only when
wired end-to-end. This ADR breaks the programme into **dependency-ordered,
individually ship-complete sub-slices** that can land wave-by-wave over parallel
lanes (where the crate territories permit) and serialized lanes (where hot files
force sequencing). The decomposition enables the **rule-6-compliant build**: each
slice ships fully wired and gate-tested, blocking/blocking relations are explicit
upfront, and the Conductor can fan out independent work to bounded parallelism
without creating cherry-pick hazards.

### Parallel vs. Serial Assignment Rules

**SERIAL lanes** (rewrite the same hot files, must be one owner per rule 32):
- **Pipeline lane:** MP-5 → Slice 1 → Slice 2 → Slice 3b → Slice 2.3
  - Reason: all touch `pipeline.rs` (ingest thread, drive_streaming, frame-boundary control)
  - Owner: one agent
  - Sequential within the lane; parallel land to stable commits after each slice passes CI

- **Schema lane:** MP-5 → all schema.rs edits below it
  - Reason: all add fields to `schema.rs` (switcher, transition, USK, DSK, FTB, audio, macros, cues blocks)
  - Owner: one agent (coordinates with pipeline lane ownership)
  - Sequenced with schema.rs edits (one per commit), tested after each addition

**PARALLEL lanes** (disjoint territories, can fan out):
- **Compositor lane:** Slice 3a (USK) + Slice 3b (DSK/FTB key math)
  - Reason: `multiview-compositor` module-scoped, no contention with pipeline/schema
  - Gates: CPU reference + GPU SSIM/PSNR per existing pattern
  - Unblocks: Slices 1–2 composition passes can land in parallel

- **Control API lane:** [ADR-W021](ADR-W021.md) + [ADR-RT008](ADR-RT008.md) routes/events
  - Reason: `multiview-control` routes/events, zero pipeline contention
  - Routes: one per slice (M/E, transition, USK, DSK, FTB, players, audio, macros, cues)
  - Gates: OpenAPI / AsyncAPI, typed_resources tests
  - Can land in parallel per slice (once schema is stable)

- **Media library** (Slice 4 CRUD)
  - Reason: config-only, no engine surface
  - Scope: schema.rs media_library definition + CRUD REST routes + SPA forms
  - Can land independently (zero engine or pipeline dependencies)

- **Macros** (Slice 7)
  - Reason: config-only, no new data-plane surfaces
  - Scope: schema.rs macros[], sequencer task, macro-run REST, SPA macro recorder
  - Can land once Slices 1–6 are committed (all verbs exist)

- **Cue dispatcher** (Slice 8)
  - Reason: config-only, dispatcher is a pure value machine
  - Scope: schema.rs cues[], CueRule/CueAction types in core, dispatcher in control
  - Can land once Slices 1–7 are committed; tests via mock cues + verification verbs

### Recommended Build Order

The precedence is:
1. **Foundation:** MP-5 (programs schema) — ships alone, unblocks everything downstream
2. **Serial pipeline lane:** Slices 1 → 2 → 3b → 2.3 (4 commits, 1 owner)
3. **Parallel lanes** (start immediately after MP-5):
   - **Schema lane:** schema.rs all-blocks (switcher, transition, audio, keyers, macros, cues, media_library, media_players)
   - **Compositor lane:** USK + DSK + FTB key math, CPU + GPU paths
   - **Control API lane:** routes + events per slice (start as soon as schema is stable per slice)
   - **Media library:** independent CRUD + import scaffold
4. **Integration:** Slice 5 (media players) + Slice 2.3 (stinger) depend on Slice 4 shipping; land after media library committed
5. **Final:** Slices 7–8 (macros + cue dispatcher) land once Slices 1–6 + 5 are all committed

### MVP vs. Post-MVP Boundaries (per ADR references)

**MVP:**
- Slice 1: M/E bus selector (PGM/PVW)
- Slice 2 Wave 1: cut, mix, dip (transition engine)
- Slice 3a: USK linear + luma (keyer types)
- Slice 3b: DSK linear + luma, FTB solid cover
- Slice 5: media players (load, cue, play, pause, stop, loop)
- Slice 6: AFV (audio mute/fade during transition)
- Slice 7: macros (record/edit/play n-step)
- Slice 8: cue dispatcher (full taxonomy)

**Post-MVP (Deferred; explicit in ADRs):**
- Slice 2 Wave 2: wipes (per-tile SDF masks), DVE sub-pixel placement, fit/crop/rotation wiring
- Slice 2 Wave 3: stinger transitions (blocked on ADR-0058 alpha media)
- Slice 3a/3b: chroma keyers, pattern-based mattes, soft-edge garbage matte (ADR-C007 kernel gates pinned)
- Slice 4: import-time transcode (MVP rejects non-native; post-MVP normalizes on-disk mezzanine)

## Rationale

- **Rule-6 compliance:** Each slice ships end-to-end (data plane + control plane + config + tests), never partial. The precedence ensures no slice lands without its dependencies.

- **Fan-out safety:** Compositor, control API, media library, macros, and cue dispatcher can run as independent lanes because they touch disjoint file territories (no shared hot files). Schema.rs and pipeline.rs coordination is explicit and sequential.

- **Invariant posture:**
  - **#1 (output clock):** All switcher logic is frame-boundary-driven (transitions, keyers, FTB sampled per tick). Cues are dispatched on control tasks, never blocking the clock.
  - **#10 (isolation):** All control surfaces (cue dispatcher, macro sequencer, control API) use bounded drop-oldest queues. The engine never awaits a client.
  - **#11 (Class-1/2):** All switcher operations are Class-1 (hot/seamless at frame boundary) except RT-12 aux routing (Class-2, deferred). The classification is inherited from the base verb or explicitly stated per ADR.

- **Parallel opportunities:** 4–6 lanes can work in parallel (compositor, control API, media library, macros, cue dispatcher) while the serial pipeline lane executes Slices 1–2–3b–2.3 sequentially. The schema lane is coordinated (one owner, but schema blocks can be merged per slice). This bounds the critical path to ~6–8 weeks for the full programme (estimate per lane + CI + review).

## Alternatives Considered

| Alternative | Rejected Because |
| ----------- | --------------- |
| One mega-PR for all 11 ADRs | Violates rule 6 (no partial shipping); cherry-pick hazards across 2000+ lines; single gate failure blocks all downstream; unfeasible review scope. |
| Schema.rs first, all code second | Leaves schema unvalidated against real ingest/output paths; risks cascading refactors after code lands; control-plane tests cannot run without code. |
| Disjoint slices in true parallel (no serialization) | `pipeline.rs` and `schema.rs` are genuinely hot files; concurrent edits on the same lines create merge conflicts and require force-pushes; rule 32 forbids this. Sequencing under one owner is mandatory. |
| Defer all compositor work (Slice 3) to post-transition | Keyers are frame-coordinated and participate in transitions (ADR-0055 Decision 7); deferring decouples the design and creates a false separation that costs test integration later. |
| Build stinger transitions before media players | Stinger requires media players to exist (ADR-0055 Wave 3 explicitly gates on ADR-0058 + player API). Media players must land first. |

## Consequences

- **Positive**
  - The programme lands wave-by-wave, each slice individually shippable and gate-tested. Rule-6 compliance is structural.
  - Parallel lanes maximize throughput: compositor, control API, media library, macros, and cue dispatcher can be owned by different agents and execute concurrently.
  - Schema and pipeline coordination is explicit and owned by one agent per lane, eliminating merge conflicts and cherry-pick hazards.
  - Every slice's dependencies are declared upfront, so the Conductor can validate the build order before assigning work.
  - MVP boundaries are clear: Slices 1–8 (minus Wave 2/3 transitions) ship complete; Wave 2/3 and post-MVP features follow with zero impact on the shipped core.

- **Negative / cost**
  - The serial pipeline lane (Slices 1–2–3b–2.3) is the critical path (~6 weeks). It cannot be parallelized; any slip delays all downstream work.
  - Schema.rs grows large (~500–700 lines of switcher config blocks). Validation must be comprehensive at each step (the sequenced review gate is load-bearing).
  - Integration risk: compositor + control API + media library work in parallel against an evolving schema. The schema owner must publish a stable commit per slice; each parallel lane rebases once per commit.
  - Each slice's gate must validate not only its code but its integration point (e.g., Slice 2 gate must prove drive_streaming + transitions + render-plan are correct together; cannot be deferred to post-slice-5 integration testing).

## Open Questions / Residual Risk

1. **Wave-2 compositor work (wipes + DVE + fit/crop/rotation).** This is the largest single work item (honest GPU + CPU oracle changes). Estimates suggest 4–6 weeks. If this is not on the critical path, it can land in parallel after Slice 1–2 complete, but the risk is that it may be (if dependent tests require Wave 2 in scope). The ADR-0055 phasing is explicit; this decomposition defers it post-MVP per design. **Mitigated:** The compositor lane (Slice 3a/3b) lands MVP keyers (luma + linear) on existing GPU/CPU paths (opacity already wired); Wave 2 is explicitly documented as a future lane, not blocking MVP.

2. **Media player on the ingest thread.** ADR-0057 Decision 4 mandates that player transport extends `pipeline.rs`'s production ingest thread, not the unwired `multiview-input` scaffold. This is the correct design (reuses verified machinery) but serializes Slice 5 on the pipeline lane. If a parallel media-player lane attempts to exist, it will conflict. **Mitigated:** This decomposition explicitly marks Slice 5 as SERIAL on pipeline.rs; the owner of the pipeline lane also owns Slice 5 (or coordinates tightly with a separate Slice 5 owner on the same lane).

3. **Alpha media (ADR-0058) is a separate programme.** Stinger transitions (Slice 2.3) are gated on ADR-0058 shipping (NV12+A format, framestore + ingestion paths). This decomposition does not include ADR-0058; it is a prerequisite (already referenced, ready to be split separately). **Mitigated:** The task is clear: Slice 2.3 unblocks after ADR-0058 lands.

4. **API / SPA surfaces are voluminous (200+ REST routes + events + forms).** Control API gates (typed_resources, OpenAPI spec, SPA components) are per-slice and significant. They are tractable as parallel work (disjoint routes per slice) but require careful coordination of the SPA state machine (state.rs additions). **Mitigated:** Control API routes can fan out per slice once schema is stable; SPA work is additive (new route handlers, new form pages); the state machine can be extended linearly without conflicts if schema changes are landed per-slice and sequenced.

5. **Macro sequencer and cue dispatcher are "downstream" systems that should not gate the core switcher.** Both (Slices 7–8) are control-plane-only, but they are part of the full specification (ADRs 0072 + production-switcher §10.2). They could be deferred to a post-MVP follow-on programme, but ADR-0072 explicitly treats them as part of the same push (single CueRule/CueAction/Cue type definitions). **Mitigated:** They are deferred to last (after Slices 1–6 + 5 commit) but are included in this decomposition so the Conductor knows their place. If the MVP window requires de-scoping, macros and cues can become a follow-on programme without affecting the core switcher (Slices 1–6 + 5 ship complete without them).

## How This Honors the Invariants

- **Invariant #1 (output clock):** Every switcher operation samples per tick; nothing blocks or paces the clock. Transition progress, keyer math, FTB, and AFV all run on the frame-boundary hook at `runtime.rs:430-439`. Cue dispatch and macro sequencing run on control tasks, never holding the output-thread lock.

- **Invariant #10 (isolation):** Every control surface (cues, macros, command bus) uses bounded drop-oldest queues. The engine never awaits a client. Player transport rides the existing ingest-thread pattern (wait-free command slots, no engine await). Media library import is off-thread (control-plane task, `202` operation).

- **Invariant #11 (Class-1/Class-2):** All switcher operations are Class-1 (hot, seamless at frame boundary) in MVP. RT-12 aux routing (Class-2, explicit five-phase make-before-break) is deferred to a later programme (not part of this switcher core).
