# ADR-0022: Real-time software compositor — parallelize + LUT the per-pixel colour pipeline; content-aware smoothness gate

- **Status:** Accepted
- **Area:** Efficiency / Compositor (mosaic-compositor)
- **Date:** 2026-06-03
- **Builds on:** [ADR-T001](ADR-T001.md) (output clock), [ADR-T002](ADR-T002.md) (hold-last-good), [ADR-E001/E002](ADR-E002.md) (decode-at-display-res, NV12-throughout), [ADR-C003/C006](ADR-C006.md) (fixed colour order), [ADR-0021](ADR-0021.md) (latch-on-tick sampling)
- **Supersedes the verification method** used for the demo galleries (md5/screenshot of overlay-laden, lossy output).

## Context

A user observed the multiview "plays in bursts / a few fps / the video pauses while the audio meters keep moving." A 3-angle diagnosis workflow (architecture + overlays-off symptom map + **instrumented** worktree run) converged on a result that **refutes** the first guesses (producer stall / ring eviction) and corrects my own measurements:

- **Producer is healthy** — `open_and_stream` publishes all 250 frames and EOFs in ~10 s real-time (instrumented count = 250).
- **Framestore `read_at` is healthy** — it advances 1:1 with `out_pts = f(tick)`, returns `Live` frames, **never** sticks or falls back to earliest-retained; the bounded ring fills to 250 < `RING_CAPACITY` (256) so **nothing the consumer still needs is evicted** (`drain(0..overflow)` never fires). The latch-on-tick design (ADR-0021) works.
- **The bottleneck is the CPU reference compositor.** `compose` (`crates/mosaic-compositor/src/pipeline.rs`) is a single-threaded per-canvas-pixel × per-covering-tile loop running the full fixed colour pipeline — per pixel: `tile_yuv_to_canvas_linear` (YUV→RGB matrix → **EOTF linearize** → primaries convert) + premultiplied `over` in linear + `canvas_linear_to_output_yuv` (**OETF**). The `eotf`/`oetf` are transcendental (`pow`) per channel per pixel. Measured: **median ~931 ms/tick** for a 1080p 6-tile canvas in a **debug** build (~100 s to render 10 s of content, ~55× slower than real time), **flat vs tile count** (1 tile is equally slow → fixed per-tick cost, not multi-tile lag), with **24 CPU cores idle** (single-threaded). A **release** build cuts this to ~20 s (still ~2× slower than real time at 1080p×6).
- **A real startup held-frame transient** exists (first ~1–3 s, then it recovers to full rate) — confirmed by a content-aware metric (`tblend=difference` → `signalstats` YAVG, floor 2.0) against a ground-truth ffmpeg encode of the same source.
- **Verification-method failure (the root of the "fooled into smooth" report):** **md5 of the mpeg2-encoded `program.ts` is unreliable both ways** — mpeg2 quantization noise makes byte-held frames hash *distinct* (false "smooth"), and animated overlays (audio meters, clock, burned-in timecode) change every frame so any frame/crop md5 or eyeball that includes them looks "advancing" even when the picture is frozen; conversely a crop inside a legitimate 2–3-tick upsample-hold window looks "frozen". The earlier "timing verified 1:1" and "gallery smooth" claims were measured this way and are **not trustworthy**.

## Decision

1. **Parallelize the per-canvas compose loop across cores** using `std::thread::scope` over row-chunks (data-parallel, each thread owns disjoint output rows — naturally race-free; **no new dependency**, keeping the crate `unsafe`-free and the dep posture unchanged). Target: real-time (≤ one tick period per composite) for 1080p with a realistic tile count on commodity CPUs.
2. **Replace per-pixel transcendental colour math with lookup tables.** The 8-bit-input **EOTF** becomes a 256-entry LUT per transfer characteristic (built once); the **OETF** (linear f32 → 8-bit code) becomes a quantized LUT / fast monotone mapping. The **fixed colour order (invariant #8) is preserved exactly** — only the *evaluation* of `eotf`/`oetf` changes, not the pipeline order or the linear-light blend. Correctness is gated by the existing CPU golden-frame / SSIM tests (the un-LUT'd reference math remains the oracle; LUT output must stay within the SSIM/PSNR threshold).
3. **Build and benchmark in release** (the demos used debug). Add a per-tick composite-cost benchmark with a budget assertion.
4. **Characterize and fix the startup held-frame transient** if it persists after (1)–(3): once the loop runs at real-time, the consumer no longer lags the producer, which should shrink it; any residual (ring still filling at t=0) is handled by priming/at-least-one-frame-per-tile before the first tick, or accepting first-frame hold within the documented last-good contract (ADR-T002).
5. **New verification gate (replaces md5):** a **content-aware, overlays-OFF** smoothness check — render with the `overlay` feature off (no meters/clock), and assert inter-frame motion via `tblend=difference`→`signalstats` against a ground-truth ffmpeg encode of the same source (or a RAW/lossless tap), plus the per-tick perf budget. A moving overlay or mpeg2 noise can never again mask a frozen picture. Re-verify every demo gallery against this gate before re-claiming "smooth".

## Consequences

- Real-time **software** multiview becomes feasible on commodity multi-core CPUs; the GPU compositor (M2, wgpu/CUDA/Metal/VAAPI) remains the high-density / high-resolution route, not a correctness prerequisite.
- The colour math stays bit-for-bit defined by the reference (LUTs validated against it); invariant #8 order is untouched.
- The prior md5/screenshot demo verification is **superseded** by the content-aware overlays-off gate; the committed `docs/proof/` gallery and FEATURES/timing "verified" claims must be re-measured against it.
- This is the actual "absolutely as efficient as possible" work for the compositor; it had been masked because every smoothness check ran through animated overlays + a lossy encoder.
