# ADR-0024: Native caption master-fetch over libav avio (no `curl`, no HTTP dependency) + bounded retry

- **Status:** Accepted
- **Date:** 2026-06-04
- **Area:** caption ingest (`multiview-cli::captions`, `multiview-ffmpeg`)
- **Relates to:** #36 (caption ingest), [ADR-0019](ADR-0019.md) (native multi-form caption ingest), invariants **#1** (output-clock) and **#10** (isolation).

## Context

Native HLS WebVTT caption ingest must read the **master** playlist *text* to find
its `TYPE=SUBTITLES` rendition — libav opens an HLS URL as a single program and
does not surface the master's separate subtitle rendition as a decodable stream.
The MVP (#24) fetched that text by **shelling out to the `curl` binary**. Two
problems surfaced:

1. **Deploy wart / divergent stack.** `curl` is a *foreign runtime dependency*
   (must exist in the deployment image), and its TLS/redirect/failure semantics
   differ from the libav stack that fetches the actual media segments.
2. **The real bug — silent disable.** `caption_plan_for` returned `None` on *any*
   single fetch failure, logging only a `warn!`. A re-render this session showed an
   **empty caption band** because that one-shot `curl` fetch missed (a transient
   blip / `curl` absent) — silently disabling captions for the whole run.

## Decision

**Fetch the playlist text in-process over libav's own `AVIOContext`, and make the
fetch resilient.**

1. **libav avio, not a new dependency.** `ffmpeg-next` exposes no safe AVIO read
   wrapper, but the raw FFI (`avio_open2`/`avio_read`/`avio_closep`/`av_dict_set`)
   is available via `ffmpeg::ffi`. A new `multiview_ffmpeg::fetch_url_text(url,
   max_bytes, allowed_protocols)` drives it — in `multiview-ffmpeg`, the **only** crate
   allowed `unsafe` (every block carries a `// SAFETY:` note; the context + options
   dict are freed on **every** return path). This reuses the already-linked libav
   stack (same TLS as the segment fetch) and adds **no** crate to the `cargo deny`
   graph. (Alternative — a small HTTP crate like `ureq`+`rustls` — was rejected: it
   needs human sign-off, a second TLS stack, and a `deny.toml` re-audit, for no
   benefit over the linked libav.)
2. **Security.** The URL scheme is validated in Rust against an explicit allow-list
   (`"http,https,tls,tcp"`) **before** opening — a hard guarantee — and the same
   list is handed to libav as `protocol_whitelist` (defence-in-depth), so a stray
   `file:`/`concat:`/`subfile:` master URL can never be read. A `rw_timeout` (30 s,
   matching the old `curl --max-time 30`) bounds a hung connection; `max_bytes`
   bounds the body.
3. **Resilience — bounded retry, budgeted.** The master fetch is retried up to 3×
   with a 250 ms backoff (`LibavFetcher` → `fetch_with_retry`), under a **total
   30 s wall-clock budget**: a *fast* transient failure retries (recovering the
   blip), but a *hung* connection that eats ~one `rw_timeout` stops after the first
   attempt rather than stacking ~3× timeouts onto the serial pipeline build (a
   review finding). A *persistent* failure still yields `None` (best-effort —
   captions must never fail a live source's pipeline build). This runs at
   pipeline-build time / on the caption reader's own ingest thread — it never
   touches the output clock (inv #1) or back-pressures the engine (inv #10).
4. **Testability — a fetcher seam.** `caption_plan_for` delegates to
   `caption_plan_for_with(source, &dyn PlaylistFetcher)` over a
   `resolve_caption_plan` seam, so the fetch→parse→pick→resolve path **and** the
   `None`-on-failure contract are exercised **offline** with a canned fetcher (no
   network). `multiview-cli` stays `forbid(unsafe)` — all FFI is behind the
   `multiview-ffmpeg` boundary.

## Consequences

- No `curl` binary dependency; no new Rust dependency; default (ffmpeg-free) build
  unaffected (`avio_fetch` + the `Fetch` error arm are behind the `ffmpeg` feature).
- **Tests:** 3 avio unit tests (`file:` read, byte-budget cap, blocked-scheme
  refusal), 4 CLI seam/retry tests (incl. `a_persistent_fetch_failure_yields_no_plan`
  — the direct regression guard for the empty-band bug, and retry recover/give-up).
- **Verified live:** the `caption_ingest` `#[ignore]` test, now routed through avio
  (not `curl`), fetched the real bipbop master over **HTTPS** and decoded a real cue
  (`"English subtitle 1 -Unforced-"`) — proving HTTPS-over-avio works end-to-end.
- The new `Error::Fetch` arm + the `multiview-cli` seam are API-surface changes carried
  through the standard adversarial cross-vendor review.
