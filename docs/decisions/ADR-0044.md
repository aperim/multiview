# ADR-0044: Local display output via DRM/KMS (display-kms sink, scanout affinity, mailbox flip policy)

- **Status:** Accepted
- **Accepted:** 2026-06-21 (operator greenlit the Devices epic)
- **Area:** Core Engine
- **Date:** 2026-06-10
- **Source brief:** [display-out.md](../research/display-out.md)

## Decision

Add local HDMI/DisplayPort output as a new module `display/` in `multiview-output`, behind a new
off-by-default Cargo feature **`display-kms`**, built on the **`drm` crate (Smithay, MIT, pure-ioctl
— no C linkage) plus the `gbm` crate** for scanout-buffer allocation; the default build stays
pure-Rust and LGPL-clean. The display sink is a **raw-frame sink, never a `PacketMuxSink`**
(`crates/multiview-output/src/sink.rs`): it consumes the **pre-encode NV12 canvas** through the same
isolation pattern the preview taps prove out (`crates/multiview-preview/src/tap.rs`, [ADR-P001](ADR-P001.md))
— concretely a wait-free single-slot latest-frame mailbox (the `LatestSlot` primitive,
`crates/multiview-framestore/src/latest.rs`). The engine publishes each tick's canvas and moves on;
it never awaits the sink, and encode-once-mux-many ([ADR-E003](ADR-E003.md)) is untouched because no
encoder or muxer is involved.

A **dedicated sink thread owns the DRM fd** and runs the flip loop: on page-flip-complete event →
take the latest mailbox frame → convert/import as needed → `atomic_commit(NONBLOCK | PAGE_FLIP_EVENT)`.
The kernel enforces **at most one in-flight atomic commit per CRTC** (a second nonblocking commit
fails `EBUSY`), which yields mailbox conflation for free; if the mailbox holds no new frame the loop
does nothing, because **KMS repeats the current framebuffer at no cost**. `TEST_ONLY` commits
validate the full configuration (plane formats, modifiers, modes) at startup and on hotplug;
`ALLOW_MODESET` commits (tens of ms, link training) run only at startup/reconfigure, **never on the
frame path**. The buffer strategy is per display block, verified against kernel source: **Intel
Gen9+ and Raspberry Pi vc4 get NV12 direct scanout** (vc4 additionally NV12/P030 with the Broadcom
SAND128 modifier, the decoder's native tiling) — dmabuf → `ADDFB2` → plane, zero copies; **AMD
DCE11** (the HP t630 test unit's class) cannot scan out NV12 via the DRM uAPI (the primary plane's
`plane_cap.nv12` is false in `dce110_resource.c`, and amdgpu_dm grants YUV formats only to
primary/universal planes — the silicon's YUV underlay pipe is never exposed), so it takes one
mandatory wgpu NV12→XRGB pass rendered into a GBM-allocated scanout dmabuf; **NVIDIA** is supported
only via wgpu's DRM surface (`VK_EXT_acquire_drm_display`), tier-2, because the proprietary driver
documents that GBM buffer submission to its DRM KMS is unsupported. GPU→KMS completion ordering uses
`IN_FENCE_FD`/`OUT_FENCE_PTR` via the generic atomic-property API, with DRM syncobjs as the fallback.

Two supporting decisions complete the design. **Scanout affinity becomes a HAL/placement input**:
KMS scanout requires the framebuffer to live on the connector-owning GPU, so KMS card-node/connector
discovery is added to capability probing (`crates/multiview-hal/src/probe.rs`, which today sees only
render nodes), a sink-locality constraint is added to selection (`crates/multiview-hal/src/select.rs`
— `PipelineDemand` gains sink locality), and the placement controller
(`crates/multiview-engine/src/placement.rs`) gates `MigrationPlan`s so composite is never migrated
off a display-sink GPU — which would force the per-frame GPU→host→GPU copy [ADR-0018](ADR-0018.md)
forbids. Trivial on single-GPU hosts, but modelled in the types for multi-GPU. **Mode selection** is:
EDID preferred mode by default; prefer the mode whose **exact rational** refresh matches the engine
cadence (59.94 Hz is 60000/1001 — never float fps); **CVT-RB computed forced modes** for EDID-less
chains (a real fleet condition), with per-connector overrides in `multiview-config`. **One canvas
per head, no spanning**: each connector is an independent sink fed by its own (possibly identical)
layout via the existing `WallConfig`/`HeadConfig` model (`crates/multiview-config/src/wall.rs`);
same-node heads are flipped in a single atomic commit covering both CRTCs where possible. Config
surface: an `Output::Display { … }` variant in the internally-tagged `Output` enum
(`crates/multiview-config/src/schema.rs`), shipped wired through `build_outputs`
(`crates/multiview-cli/src/pipeline.rs`) and the SPA in the same slice.

## Rationale

This shape makes invariants #1 and #10 hold **by construction** rather than by discipline: the only
engine↔sink contact is a wait-free overwrite of a single-slot mailbox, the only blocking waits
(vblank, fences, WSI present on NVIDIA) happen on a thread the engine never joins, and the failure
mode of a wedged display is a frozen monitor showing the last framebuffer while program output
continues untouched. The pure-ioctl drm-rs stack keeps the off-by-default feature free of GPL
exposure and the default build free of native deps, and its API coverage (atomic commits with all
four flags, `ADDFB2` with modifiers, property access, syncobjs, flip events with timestamps) is
sufficient without a compositor framework — for a single fullscreen canvas the right altitude is
~1–2k lines over drm-rs + gbm. The per-hardware buffer strategy follows verified driver reality
instead of an idealized NV12-everywhere story, and the affinity work is required, not optional: the
display sink is the first GPU-resident raw-frame consumer, and without a placement gate the
controller would happily migrate composite away from the scanout GPU. Refresh ≠ tick is reconciled
by repeat/drop (at 60.000 Hz tick vs 59.94 Hz display, one duplicated frame every ~16.7 s) plus
rational mode matching; the optional display-locked clock mode for dedicated display nodes is
decided separately in [ADR-0045](ADR-0045.md).

## Alternatives considered

- **GBM + EGL surface (the proven LibreELEC/Kodi path).** Rejected as primary: it introduces a GLES
  renderer beside the wgpu compositor, doubling the shader/colour surface for no capability we need.
  Retained as the documented fallback only if wgpu-on-Vulkan misbehaves on a target (e.g. v3dv edge
  cases on Raspberry Pi).
- **Depend on Smithay's `DrmCompositor`.** Rejected: it drags in Smithay's renderer/element
  abstractions to solve multi-element plane negotiation we don't have — our scanout content is one
  fullscreen canvas. It remains the design prior art for TEST_ONLY validation and plane fallback
  logic.
- **Run as an X11/Wayland client.** Rejected: requires a whole compositor stack on a headless
  appliance, adds compositor scheduling jitter and latency, denies exact mode/refresh control, and
  forecloses direct-KMS-only capabilities (HDR metadata, plane control). The node owns DRM master,
  full stop — incumbent desktop/signage stacks are disabled, not coexisted with.
- **Encode + local-decode loopback** (point a local decoder process at our own served stream).
  Rejected: pays an encode rendition, a decode, and seconds-class latency to reach a monitor on the
  same box, and adds a second process to supervise. The display-node tier already covers the remote
  case; local heads deserve the direct path.

## Consequences

The chaos/soak gate must prove the isolation claim: a deliberately wedged display sink (unplugged
connector, blocked fence) may freeze its monitor but never moves the output clock — this becomes a
standing CI scenario alongside the existing engine chaos gate. AMD DCE11 pays the NV12→XRGB pass on
every displayed frame (~0.7 GB/s at 1080p60, single-digit % of that iGPU; ~2.7 GB/s at 4K60 —
estimates, to be measured); NVIDIA display-out is explicitly **tier-2** (no NV12 planes, WSI present
semantics, vendor-documented experimental KMS) and is never the recommended display-node hardware.
The fence path is a known unknown: drm-rs has no first-class `IN_FENCE_FD`/`OUT_FENCE_PTR` helpers,
so a short spike validates the generic-property route (syncobjs as fallback) — a wrong fence
strategy shows up as flip-latency spikes, so flip timestamps and fence-wait durations are exported
as telemetry from day one. Hardware-specific regressions are real (vc4 has shipped kernel windows
with broken NV12 scanout, rpi/linux #5727; vc4 also serializes atomic commits across CRTCs,
rpi/linux #5094, so dual-head Pi commits can slip a frame), so a **golden-scanout test on the deploy
kernel** — TEST_ONLY capability probe plus a captured-output check — is a release gate for each
display target, and dual-head Pi budgets for occasional slips. Scanout phase across distinct
displays is never aligned (0–1 refresh residual; no genlock-grade sync without framelock hardware),
and the duplicated-frame cadence artifact (~every 16.7 s at 60 vs 59.94 Hz) is documented rather than
hidden. Finally, the `display-kms` feature adds another off-by-default native dependency to
`multiview-output` (Mesa `libgbm`), consistent with the crate's existing feature-gated
GStreamer/libav native deps; the default workspace build remains pure-Rust.
