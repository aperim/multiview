# ADR-0014: Encode the multiview once per output; size density to physical NVENC chips

- **Status:** Proposed
- **Area:** Core Engine
- **Date:** 2026-06-02
- **Source brief:** [core-engine.md](../research/core-engine.md)

## Decision

Compose the multiview once and encode it once per output codec/profile (plus ABR variants), feeding many cheap decodes; schedule encode jobs against the active physical NVENC chip count (not the 12-session cap), use a single canonical low-latency profile (CBR, no B-frames, no lookahead, intra-refresh/forced-IDR), and fall back to CPU encode when NVENC capacity is exhausted.

## Rationale

NVDEC decode is effectively uncapped; NVENC throughput is bounded per physical NVENC engine, and most consumer GeForce have only one. A single stream cannot exceed one NVENC's throughput without SFE. Per-tile re-encode would exhaust both throughput and the per-system session cap. B-frames/lookahead are the dominant latency adders.

## Alternatives considered

Per-tile transcode (rejected: session/throughput/bandwidth hostile). Size capacity off the 12-session headline (rejected: conflates admission with throughput). Quality-tuned NVENC with B-frames/lookahead in the live path (rejected: adds frames of latency).

## Consequences

Density planning must detect SKU/active-NVENC-count at deploy time and may need datacenter/Quadro GPUs for high fan-out. SFE only helps single very-high-res outputs (HEVC/AV1, Ada+). An optional VOD/quality profile re-enables B-frames/lookahead for non-live outputs.
