# ADR-0027: Synthetic sources are first-class — bars / solid / clock as in-process `SourceKind`s through the one uniform ingest path

- **Status:** Proposed
- **Area:** Core engine / Input / Config (source model)
- **Date:** 2026-06-05
- **Source brief:** [core-engine §9.1, §13](../research/core-engine.md), [resilience-and-av](../research/resilience-and-av.md), [streaming-gotchas §1](../research/streaming-gotchas.md)
- **Relates to:** [ADR-0010](ADR-0010.md) (declarative config schema), [ADR-T002](ADR-T002.md) (last-good-frame + tile state machine), [ADR-R001](ADR-R001.md) (output clock), [ADR-0025](ADR-0025.md) (streaming ingest/encode)

## Context

A multiview's inputs are not only remote feeds. Colour **bars** (with **tone**), a
full-screen **clock**, and a **solid**-colour slate are standard, load-bearing
sources every operator expects — for line-up, as a fallback, as a reference
filler in an empty cell. The product principle is blunt: **a source is a source.**
Bars, a clock, and an RTSP feed must be modelled and handled identically; nothing
downstream of ingest may know or care what *kind* a source is.

Today that principle holds *downstream* of ingest — every kind publishes
`Nv12Image` frames into a per-tile `TileStore`, and the compositor reads
"latest-or-placeholder" identically (invariant #2) — but it **breaks at ingest**
in three ways:

1. **There is no `clock`, `bars`, `solid`, or `tone` source kind.** `SourceKind`
   has a single catch-all `Test`. The full-screen clock exists *only* as an
   overlay (`multiview-overlay::clock` → overlay primitives baked onto an
   already-composited canvas), so it cannot be *placed as a source* in a tile or
   as the whole canvas.
2. **`test` means two different things depending on the build.** The FFmpeg-free
   software driver (`run.rs`) publishes a *static solid colour* per source
   (`test_pattern` → `Nv12Image::solid`, primed once). The full libav pipeline
   (`pipeline.rs`) shells out to `ffmpeg -f lavfi -i testsrc` and decodes the
   result as a finite file. Same config, different picture, and the synthetic
   path needs a subprocess + libav even though it is conceptually pure pixels.
3. **Synthetic-frame generation is duplicated across the two run drivers** rather
   than being one set of generators behind one ingest path.

The only legitimate axis between the two drivers is *whether the libav decoders
are compiled in for network/file decode*. Synthetic sources need no libav — they
are pure pixels — so they should be produced in-process, identically, in both
builds.

## Decision

Make synthetic sources **first-class, in-process `SourceKind`s** produced by a
pure-Rust generator and driven through the **same** ingest→`TileStore`→compositor
path as any decoded feed.

1. **New `SourceKind` variants** (LGPL-clean, no libav), internally tagged by
   `kind` like the rest:
   - `bars` — SMPTE/EBU 75 % colour bars (the classic line-up signal; the audio
     companion *tone* is emitted on this source's audio bus — see Consequences).
   - `solid { color }` — a single-colour slate (e.g. `#101014`).
   - `clock { face, tz, … }` — a full-frame analog/digital clock, disciplined by
     the same `ClockModel`/`TimeRef` used by the clock *overlay*.
   - `test` is **retained as a back-compat alias for `bars`** (no breakage; the
     schema stays `schema_version = 1`, additively).

2. **A pure-Rust synthetic generator** (`SyntheticSource`) that, given the canvas
   geometry, the output-clock instant (`MediaTime`/wall time), and the colour tag,
   yields one `Nv12Image`:
   - `bars` fills NV12 colour bands directly (no libav, no subprocess).
   - `solid` is `Nv12Image::solid`.
   - `clock` composes existing pieces — a solid background `Nv12Image` with the
     existing clock rasterizer (`clock_face` → `apply_overlays_to_nv12`) baked
     **full-frame** — so a clock can fill a tile *or* the whole canvas. No new
     clock-drawing code; the overlay rasterizer is reused to draw a standalone
     frame.

3. **One uniform, converged ingest path.** Synthetic sources publish into their
   `TileStore` at the canvas cadence from a generator task that is a peer of a
   decode thread (`publish_arc(frame, at)` each tick). Both the software driver
   and the full libav pipeline drive synthetic sources through this *same*
   generator; the full pipeline **stops shelling out to `ffmpeg testsrc`**. The
   FFmpeg-free build and the full build now render byte-for-byte the same
   synthetic picture.

4. **Synthetic sources are live.** They publish per tick, so the clock advances
   and bars may carry a moving element — they ride the same `LIVE` tile state as
   any healthy feed (and, being in-process, never go `STALE`).

## Rationale

- **The principle, enforced in code.** After this, the *only* `match kind` left at
  ingest is "synthesize in-process vs open+decode via libav" — intrinsic, not a
  privilege. Everything else (framestore, state machine, compositor, encode-once,
  output fan-out) stays kind-agnostic.
- **Invariants preserved.** Generation is cheap, bounded, and writes the same
  lock-free single-slot store; a slow generator cannot stall the output clock
  (inv #1) — the compositor reads last-good (inv #2) — and cannot back-pressure
  the engine (inv #10). It is pure-Rust and LGPL-clean, so it lives in the default
  build (no `gpl-codecs`, no `ffmpeg`).
- **Clock-as-source falls out of existing parts.** Rendering the clock onto a
  solid full-frame NV12 reuses the *exact* rasterizer the overlay path already
  uses; we add a *placement* (a source tile/canvas), not a new renderer.
- **Removes a real divergence + a subprocess.** "Same config, same picture" across
  builds; no `ffmpeg` child process for a test pattern.

## Alternatives considered

- **A — Keep `test` as `ffmpeg testsrc` only (status quo).** Rejected: unavailable
  in the FFmpeg-free build, diverges software vs full, cannot express
  clock/bars/solid as sources, and shells out to a subprocess for pure pixels.
- **B — Keep the clock overlay-only.** Rejected: the operator model is
  clock-*as-source* (fills a tile or the canvas). Overlay-only conflates content
  (a source) with annotation (an overlay on top of a source) and cannot be placed
  in the layout grid as an input.
- **C — A dedicated `multiview-generators` crate.** Deferred, not rejected: the
  generator initially lives where both drivers are (`multiview-cli`), reusing
  `multiview-compositor` (`Nv12Image`, the overlay rasterizer) and
  `multiview-overlay` (`ClockModel`). It can be promoted to its own crate (or into
  `multiview-input`) once the API settles, without changing the contract.
- **D — Animate via the compositor overlay each tick, leave the source static.**
  Rejected for the clock: that keeps the clock an overlay, not a source, and
  re-introduces the "static synthetic frame" smell in the software driver.

## Consequences

- **Positive**
  - Pure-Rust **bars + full-screen clock + solid** sources in *every* build; the
    UI, previews, and outputs show a real line-up signal and a real clock with no
    libav and no GPL.
  - The two run drivers converge on one synthetic-frame path; the software-vs-full
    divergence and the `ffmpeg testsrc` subprocess are removed.
  - Clock/bars/solid are placeable as a tile source **or** full-canvas — exactly
    the operator model.
- **Negative / cost**
  - New `SourceKind` variants ripple to the config schema + validation (colour and
    clock fields), the WebUI source-kind picker, the OpenAPI/JSON-schema, and the
    docs. `test` is kept as an alias so existing configs keep working
    (`schema_version` unchanged, additive).
  - Synthetic generators add bounded per-tick CPU. This is cheap (a band fill / a
    small overlay bake) and the resource-adaptive degradation loop (inv #9) already
    sheds tiles cheapest-impact-first if a host is saturated.
  - **Tone (audio)** is the companion to bars and is *not* in this slice's video
    scope: it is emitted on the `bars` source's audio bus and is tracked with the
    audio subsystem (`multiview-audio`), not the video generator.

## Implementation plan (TDD, sliced)

1. Pure generator (`SyntheticSource`) + unit tests: bars band boundaries/colours,
   `solid` uniformity, `clock` hands at known wall-times (e.g. 03:00:00 → hour
   hand at 90°). Pure, deterministic (injected time), no I/O.
2. Config: add `bars`/`solid`/`clock` variants (+ `test` alias) to
   `multiview-config` `SourceKind`, with validation; round-trip TOML/JSON tests.
3. Wire into the software driver: per-tick publish so a `clock` source advances;
   test that a clock tile's content changes across ticks and differs from `solid`.
4. Converge the full pipeline: replace `generate_test_clip` with the generator
   task; assert the FFmpeg-free and full builds produce the same synthetic frame.
5. Surface: WebUI source-kind picker + forms, OpenAPI/JSON-schema regen, docs +
   examples; (later) `tone` on the `bars` audio bus.
