# ADR-0058: Alpha media path — an optional NV12+A frame payload (2.5 B/px) for media/graphics fills, premultiplied once in linear light at the compositor boundary

- **Status:** Proposed
- **Area:** Core Engine / Compositor / Efficiency (frame payload, alpha media)
- **Date:** 2026-06-11
- **Source brief:** [media-playout.md](../research/media-playout.md)
- **Relates to:** invariant #5 / [ADR-E002](ADR-E002.md) (NV12-throughout —
  never RGBA per tile), invariant #8 / [ADR-C003](ADR-C003.md) (premultiplied
  linear-light blending) + [ADR-C006](ADR-C006.md) (fixed color-pipeline order),
  [ADR-0056](ADR-0056.md) (keyer math — the alpha consumption point),
  [ADR-0057](ADR-0057.md) (media library import policy — the alpha production
  point), [ADR-0055](ADR-0055.md) (stinger transitions),
  [ADR-R008](ADR-R008.md)/[ADR-0016](ADR-0016.md) (overlay stack is
  input-decoupled — what this ADR deliberately does **not** touch),
  [ADR-T002](ADR-T002.md) (last-good-frame), [ADR-E007](ADR-E007.md)
  (admission/degradation)

## Context

Keyed media — stingers ([ADR-0055](ADR-0055.md) Wave 3), stills/graphics fills
for DSKs ([ADR-0056](ADR-0056.md)), lower-third clip loops — needs **per-pixel
alpha in the frame currency**. Today no alpha survives ingest anywhere
(verified in this worktree):

- The working frame payload has no alpha plane: `Nv12Image` is Y + interleaved
  UV only (`crates/multiview-compositor/src/pipeline.rs:169`); the compositor
  `Tile` carries one **uniform** opacity float (`pipeline.rs:413`, `:426`).
- `PixelFormat::Rgba` exists but is documented *"do not materialize per tile"*
  (`crates/multiview-core/src/pixel.rs:16`) and is unused on the run path
  (inv #5).
- NDI BGRA alpha is explicitly dropped at ingest
  (`crates/multiview-input/src/ndi/convert.rs:369`).
- The only alpha-carrying surface is the overlay path: premultiplied-RGBA cue
  bitmaps through a **content-keyed, upload-once LRU** texture cache
  (`crates/multiview-compositor/src/overlay/gpu_image.rs:101`) — designed for
  *static* cues, baked **off-thread** on the egress consumer
  ([ADR-R008](ADR-R008.md)), and therefore neither bus-selectable nor
  frame-coordinated with on-clock switcher state ([ADR-0054](ADR-0054.md)).

The blend substrate, by contrast, is already correct: compositing is
premultiplied Porter-Duff source-over in linear light
(`crates/multiview-compositor/src/blend.rs` — `LinearRgba::premultiplied` +
`over`; inv #8 step 5), the CPU kernel computes per-pixel source alpha at
exactly the right point (`fold_tile_into_band`,
`crates/multiview-compositor/src/pipeline.rs:1075`, opacity applied at
`:1109`), and the GPU kernel applies the uniform tile opacity at the same
step (`crates/multiview-compositor/src/gpu/shaders/composite.wgsl:132`) with
per-tile `texture_2d_array` plane bindings (`composite.wgsl:45-46`) and spare
`TileParams` lanes (`dst_size.zw`, `opacity_transfer.zw` —
`composite.wgsl:19`/`:21`). The missing piece is a frame payload that carries
the per-pixel alpha to that point.

## Decision

1. **NV12+A.** Add an **optional** frame-payload extension for media/graphics
   fills: the NV12 planes plus **one full-resolution R8 alpha plane** —
   **2.5 B/px** (vs 1.5 opaque NV12, vs 4.0 RGBA). The framestore is already
   generic over its payload (`TileStore<T>`,
   `crates/multiview-framestore/src/tile.rs:167`), so the NV12+A image rides
   the identical lock-free latest-slot / `read_at` / `is_primed` /
   `HoldForever` semantics (inv #2) with **zero framestore changes**. Scope is
   hard: NV12+A is for **media-player and graphics-fill sources only**
   ([ADR-0057](ADR-0057.md)) — never per-tile RGBA video (inv #5), and never
   bolted onto the overlay stack (overlays stay input-decoupled per
   [ADR-R008](ADR-R008.md)).

2. **Alpha is straight in the payload; premultiplication happens exactly once,
   in linear light, in-kernel.** The mezzanine/payload stores **straight**
   alpha; import normalizes premultiplied-authored assets and records the fact
   ([ADR-0057](ADR-0057.md) probe metadata; externally supplied fill+key pairs
   carry `KeySource.premultiplied` per [ADR-0056](ADR-0056.md)). In both
   kernels the per-pixel alpha multiplies into the **existing** premultiply
   step — after range-expand → YUV→RGB → linearize, before the linear-light
   `over` — so the fixed color-pipeline order (inv #8,
   [ADR-C003](ADR-C003.md)/[ADR-C006](ADR-C006.md)) is preserved unchanged:
   per-pixel A composes multiplicatively with the uniform `Tile.opacity` at
   `fold_tile_into_band` (CPU) and the `opacity_transfer.x` application (GPU).
   This is "premultiplied at the compositor boundary": the compositor is the
   only place premultiplication ever occurs.

3. **Format policy (verified FFmpeg alpha behaviour; enforced at import per
   [ADR-0057](ADR-0057.md)).**
   - **Decodes with alpha — accept:** ProRes 4444 / 4444 XQ (intra,
     heavy-bitrate but cheap multithreaded CPU decode), QuickTime Animation
     (`qtrle`, legacy RLE, universally produced), PNG/TGA image sequences
     (universal interchange; per-frame decompress cost), VP9-with-alpha via
     `libvpx-vp9` (alpha side-stream; common stinger format in open-source
     production tooling).
   - **Decodes opaque — reject or import-transcode:** HEVC-with-alpha. FFmpeg
     returns its alpha layer fully opaque (8.0 added only *encode* support via
     libx265), so accepting it would silently produce an opaque stinger. The
     import pipeline rejects it or transcodes it to a mezzanine format above.
   - Stinger-eligible clips additionally require exact canvas resolution,
     exact output frame rate, and an explicit `trigger_point`
     ([ADR-0057](ADR-0057.md) §3) — a wrong-cadence stinger cannot be
     frame-indexed by tick offset and is rejected at import, not discovered at
     take.

4. **Stinger mezzanine: pre-decoded, bounded, pool-allocated.** A stinger must
   never decode on the take path. At **load/cue** the player pre-decodes the
   clip into an in-memory NV12+A frame ring indexed by **tick offset**
   (integer frames at the output cadence — [ADR-T015](ADR-T015.md)); at take,
   the player thread publishes the mezzanine frames into its `TileStore`
   stamped `start_tick_pts + k × frame_period`, so the clock thread's
   latch-on-tick `read_at` resolves to exactly ring index
   \[tick − start_tick\] — equivalent selection by pure array indexing, but
   the engine touches only the `TileStore`, never the mezzanine (Decision 5's
   dataflow: the engine never knows a tile is media). Zero decode, zero
   per-take allocation (buffers come from a pool sized at load; safety
   rule 5). Budgets (1080p, 1920×1080 = 2 073 600 px):

   | Form | Bytes/frame | 3 s @ 60/1 (180 frames) |
   |---|---|---|
   | NV12 (opaque) | 3 110 400 (1.5 B/px) | ≈ 0.56 GB |
   | **NV12+A** | **5 184 000 (2.5 B/px)** | **≈ 0.93 GB** |
   | RGBA | 8 294 400 (4.0 B/px) | ≈ 1.49 GB |

   NV12+A is a **37.5 % saving** over RGBA at the worst point (the resident
   mezzanine); UHD scales ×4 (≈ 20.7 MB/frame), which is why the cap is hard:
   a configurable per-player + global mezzanine budget, validated at import
   and admission-accounted ([ADR-E007](ADR-E007.md)). Clips beyond the cap use
   a bounded **decode-ahead ring** (pre-fill at cue, refill from the paced
   decode thread during playout, with an import-validated lead that covers the
   trigger window); the fully resident form is the default for stingers.

5. **Where alpha flows — and where it does not.**
   - **Flows:** import/transcode ([ADR-0057](ADR-0057.md)) → NV12+A payload in
     the player's `TileStore` → the drive's tile resolution (an optional alpha
     plane reference alongside the fill image; for linear keys from alpha
     media, the A plane **is** the key) → keyer math at the verified insertion
     points ([ADR-0056](ADR-0056.md)) → premultiplied linear-light `over`.
   - **Does not flow:** the program canvas stays `Nv12Image` (alpha is fully
     consumed at the blend; the composite output and every encode/output path
     are byte-for-byte unchanged); the overlay stack is untouched; ordinary
     video sources still decode to plain NV12 (NDI ingest continues to drop
     BGRA alpha for normal sources — an NDI fill+key *pair* is expressible
     later via the linear-key fallback, Alternative C).

6. **Kernel impact (both backends, CPU oracle first).**
   - **CPU:** `fold_tile_into_band` gains one byte load + normalize + multiply
     per covered pixel when the tile has an alpha plane (no cost for opaque
     tiles — branch on the tile form). Gated by the existing CI budget test
     (`tile_driven_meets_budget_9up_1080p`,
     `crates/multiview-compositor/tests/composite_tile_driven.rs:337` — 40 ms
     release budget at 1080p/9 tiles), extended with an alpha-tile variant.
   - **GPU:** one additional R8 `texture_2d_array` (alpha layers, pooled
     grow-only like the existing plane arrays) + one `textureLoad` + multiply
     in the tile loop; the `TileParams` spare lanes carry the alpha layer
     index + a has-alpha flag — **no struct growth**. The `MAX_TILES = 64`
     layer budget (`crates/multiview-compositor/src/gpu/compositor.rs:36`) is
     unchanged; only media tiles consume an alpha layer. GPU output verified
     against the CPU oracle via the established SSIM/PSNR pattern.

## Rationale

- **2.5 B/px is the efficiency-pass answer** (standing review): per-pixel alpha
  costs exactly one extra plane, preserves NV12 as the color carrier (the
  whole 4-axis color pipeline — range/matrix/transfer/primaries — keeps
  operating on YUV exactly as today), and keeps the type-level guarantee that
  ordinary video never pays for alpha.
- **The blend math already exists.** Straight alpha → `premultiplied()` →
  linear `over` is the shipped, tested compositing contract
  ([ADR-C003](ADR-C003.md)); this ADR adds a *source* of per-pixel alpha, not a
  new blend domain — which is why inv #8's order survives untouched.
- **Frame coherence by construction.** A single NV12+A payload means fill and
  key can never skew by a frame — they are one image. (The two-source fallback
  cannot guarantee this; see Alternative C.)
- **Invariant posture:** the mezzanine is sampled per tick (`f(tick)` indexing
  — inv #1; never pacing), holds last-good on underrun (inv #2 →
  [ADR-R011](ADR-R011.md) EOF/underrun policy), is bounded and pool-allocated
  (safety rule 5), and is loaded/decoded entirely off the clock thread
  (inv #10).

## Alternatives considered

- **A — RGBA frames for alpha media (`PixelFormat::Rgba` payload).** Rejected:
  4 B/px is +60 % RAM over NV12+A at the worst point (the resident stinger),
  violates the inv #5 posture [ADR-E002](ADR-E002.md) pins (`pixel.rs:16` says
  *"do not materialize per tile"*), and forces an RGB side-path through a
  color pipeline whose detection/tagging axes are YUV-centric.
- **B — Route alpha video through the overlay image cache.** Rejected: the
  cache is **content-keyed and upload-once by design**
  (`overlay/gpu_image.rs:101`) — video defeats the key every frame, devolving
  to full-canvas RGBA8 uploads (1080p: 8 294 400 B ≈ 8.3 MB/frame,
  ≈ 498 MB/s at 60 fps); and overlays bake off-thread after the preview taps
  ([ADR-R008](ADR-R008.md)), so an overlay-borne stinger could be neither
  frame-coordinated with the transition state machine nor bus-selectable nor
  tallied. The cache stays what it is: a static-cue path.
- **C — Alpha via a second NV12 source (key's luma = alpha).** **Workable —
  kept as the linear-key fallback**, not the primary path: two ordinary NV12
  sources (fill + key) with the keyer computing alpha from the key source's
  post-range-expand Y ([ADR-0056](ADR-0056.md) `KeySource{fill, key}`). This
  remains the right shape for *external* fill+key feeds (e.g. a graphics
  engine delivering two SDI-style streams) and for non-alpha codecs delivered
  as pairs. It is not the path for library media because it doubles decode and
  the two `TileStore`s latch independently per tick — under stress fill and
  key can skew a frame, which keying makes visible as a fringe. NV12+A is
  frame-coherent by construction.
- **D — Subsampled (4:2:0) alpha plane (2.0 B/px).** Rejected: keying quality
  demands full-resolution alpha edges — chroma-resolution alpha fringes on
  text and thin graphics, exactly where stingers and lower thirds live. The
  0.5 B/px premium is the price of correct edges.
- **E — Accept HEVC-with-alpha natively.** Rejected: FFmpeg decodes it opaque
  (decode-side alpha unsupported; 8.0 added encode-only via libx265) — the
  failure is silent (a fully opaque stinger covering the program), the worst
  kind. Reject or import-transcode ([ADR-0057](ADR-0057.md)).

## Consequences

- **Positive**
  - Stingers, alpha stills, and graphics-fill clips become first-class at
    2.5 B/px with zero impact on opaque sources, the program canvas, encodes,
    outputs, or the overlay stack — the alpha is produced at import, carried
    by the generic framestore, and consumed entirely inside the blend.
  - The DSK MVP ([media-playout §16](../research/media-playout.md) — stills
    with alpha via NV12+A for the 2 DSKs) and the Wave-3 stinger transition
    land on one payload design.
  - Take-tick cost for a resident stinger is array indexing + one extra
    texture fetch per covered pixel — no decode, no allocation, no readback
    change.
- **Negative / cost**
  - Both composite kernels (CPU reference + wgpu) grow an alpha-aware tile
    form, each with oracle/SSIM coverage and a budget-gate extension — the
    main implementation cost, owned by the compositor lane (Lane B) alongside
    the [ADR-0056](ADR-0056.md) keyer math.
  - Resident mezzanine RAM is real (≈ 0.93 GB for a 3 s 1080p60 stinger) and
    must be admission-accounted and capped; UHD stingers will routinely need
    the decode-ahead ring rather than full residency.
  - A second payload form in the engine's generics (`TileStore<Nv12aImage>` or
    an image-enum payload) ripples type signatures through drive/tile
    resolution — mechanical but wide.
- **Neutral**
  - The encode path never sees alpha; "encode discards alpha" is true in the
    strongest sense — there is nothing to discard, because the canvas never
    carries it.
  - P010/HDR media with alpha is out of scope for v1 (the mezzanine is 8-bit
    NV12+A); [ADR-C007](ADR-C007.md) owns HDR canvas behaviour under
    transitions.
