# ADR-0089: Output orientation — per-output rotation/flip via canvas-rotate or a display-rotation tag, sourced from the layout manager, flowing to program + preview

- **Status:** Proposed
- **Area:** Output / Compositor / Config / Control / Web
- **Date:** 2026-06-13
- **Source brief:** [output-metadata-and-orientation.md](../research/output-metadata-and-orientation.md) §2–§3
- **Extends:** [ADR-C006](../decisions/ADR-C006.md) (tag the output explicitly, then verify — invariant #8 "tag, never convert"), [ADR-M002](../decisions/ADR-M002.md) (per-output `EncodeProfile` / rendition geometry + pinned-resolution Class-2)
- **Relates to:** [ADR-M005](../decisions/ADR-M005.md) (live-apply class + dry-run plan), [preview-subsystem](../research/preview-subsystem.md) / [ADR-P001](../decisions/ADR-P001.md) (preview isolation + fidelity labels), [display-out](../research/display-out.md) (`Head.orientation` — a *different* axis), [ADR-T001](../decisions/ADR-T001.md) / [ADR-RT004](../decisions/ADR-RT004.md) (invariants #1 / #10)

## Context

The layout manager needs an **orientation** setting on an output that **flows to program and preview** so a downstream player (VLC was the example) orients the picture correctly — without forcing a re-render where the player can do it.

There are two genuinely different mechanisms, routinely conflated:

- **(a) a display-rotation TAG** — the pixels stay landscape; the container/bitstream declares a rotation (MP4/MOV `tkhd` display matrix; ffmpeg `displaymatrix` side data; `-display_rotation`) and a tag-aware player rotates on render. **Zero pixel cost; VLC/ffmpeg/browsers honor it; tag-blind players do not.** (verified — ffmpeg `libavutil/display`, `mov_read_tkhd`, ffprobe `displaymatrix`)
- **(b) ROTATE THE CANVAS** — the composited pixels are produced rotated, so **every** sink shows correct pixels with no cooperation. **One lossless quarter-turn sampling pass; 90°/270° swap W↔H (a new rendition geometry).**

The codebase already has the right vocabulary, verified in-tree, but no per-output orientation:

- `QuarterTurn` (`None`/`Cw90`/`Cw180`/`Cw270`, with `degrees()` + `swaps_axes()`) at `crates/multiview-core/src/layout.rs:32-62`, used today only as **per-cell tile** rotation (`Cell.rotation`, `layout.rs:144`; lossless sampling transform, `layout.rs:28`). Unit-tested (`crates/multiview-core/tests/layout_broadcast.rs:159-170`).
- `Orientation` (`Landscape`/`Portrait`, `swaps_axes()`) at `layout.rs:71-85`, used on **display-head scanout** (`Head.orientation`, `layout.rs:460`; `HeadConfig.orientation`, `crates/multiview-config/src/wall.rs:35`) — a *different* concept ([display-out](../research/display-out.md)).
- No `display_rotation`/`tkhd`/`transpose`/orientation field anywhere in `multiview-output` — the per-output orientation surface is net-new.

Three rotation axes must stay distinct: per-cell tile rotation (inside the canvas), display-head portrait scanout, and **this** per-output presentation orientation.

## Decision

Add a per-output **`OutputOrientation`** that **reuses the existing core `QuarterTurn`** (no new rotation enum), selectable between the tag and pixel mechanisms.

1. **Surface.** Additive `orientation: Option<OutputOrientation>` on every `Output` variant (`crates/multiview-config/src/schema.rs:683`; additive like `audio`/`gpu_pin`; `#[non_exhaustive]`). Fields:
   - `turn: QuarterTurn` (`multiview-core`, `layout.rs:32`) — `none`/`cw90`/`cw180`/`cw270`.
   - `mode: { auto | tag | pixels }`.
   - `flip: { none | horizontal | vertical }` reusing the cell-transform flip vocabulary ([layout-and-config §4](../templates/layout-and-config.md)).
2. **Mode semantics.**
   - **`tag`** — emit the display-rotation tag only (mechanism a). **Validation rejects `tag` on transports with no rotation tag** (MPEG-TS, RTSP, NDI) — explicit, never a silent no-op.
   - **`pixels`** — produce a rotated-canvas rendition (mechanism b); always available. A non-`none` `flip` forces `pixels` (no container "flip" tag exists).
   - **`auto`** (default) — `tag` where the transport carries one (MP4/MOV/HLS-fMP4), else `pixels`. This is **tag-capable-player-correct and zero-cost where supported, pixel-correct only for transports without a robust rotation tag** (the `pixels` fallback). On a tag-transport a tag-blind player still shows landscape — surfaced in preview + the open question on `auto`.
3. **Apply.**
   - **Tag path:** at **mux init**, write the `tkhd` 3×3 display matrix for θ (and `AV_*_DATA_DISPLAYMATRIX` side data on copy/remux paths); optional H.264/HEVC display-orientation SEI as advisory only. No compositor change.
   - **Pixels path:** apply the **whole-canvas** quarter-turn (the same lossless sampling `QuarterTurn` already does per tile, `layout.rs:28`) into that rendition's encode target; odd turns carry a **swapped (H×W) `EncodeProfile` geometry** ([ADR-M002](../decisions/ADR-M002.md)). 180°/flip keep geometry.
4. **Source + flow.** Orientation is set in the layout/output manager and is part of the versioned config. "Flowing to program" *is* applying the tag or producing the rotated rendition. **Preview must reflect it:** the per-output real-encoded preview already shows reality; the pre-encode program-preview tap shows rotated pixels on the pixels path and landscape pixels **with an explicit "tagged θ — player-applied" overlay** on the tag path (the [preview-subsystem](../research/preview-subsystem.md) fidelity-label rule). The orientation rides each preview descriptor so the client renders the correct bounding box.
5. **Live-apply class.** Tag changes are a mux-init change (Class-1 where the container can re-emit an init segment; Class-2 where the muxer must restart). `turn` on the **pixels** path that changes geometry (odd turns) is **Class-2** (pinned-resolution change, [ADR-M002](../decisions/ADR-M002.md), make-before-break); 180°/flip on the pixels path keep geometry and can be Class-1. The classifier surfaces which via the dry-run plan ([ADR-M005](../decisions/ADR-M005.md)) before applying.
6. **Capability-gated + verified.** An `OrientationPlan` (chosen mechanism + live-apply class + any rejection) is returned with the dry-run plan; the [ADR-C006](../decisions/ADR-C006.md) ffprobe gate is extended to assert the `displaymatrix` (tag path) or the rendition dimensions (pixels path) match the request.

## Consequences

- **Positive.** The VLC case is the zero-cost tag (`auto` on MP4/HLS-fMP4); tag-blind transports (TS/RTSP/NDI) get correct pixels automatically; the operator can force either mechanism. One vocabulary (`QuarterTurn`) across cell/head/output rotation. Program and preview agree, with honest labelling when a tag is in play.
- **Negative / cost.** The pixels path at 90°/270° is a distinct-geometry rendition ⇒ a separate composite+encode counted against the [ADR-M002](../decisions/ADR-M002.md) session budget (the one real cost, paid only when chosen). Net-new mux display-matrix code + the whole-canvas turn in the compositor + the swapped-geometry `EncodeProfile` wiring + the extended ffprobe gate.
- **Neutral.** 180° and the tag path never change geometry; flip is pixel-only. `OutputOrientation` stays distinct from `Head.orientation`; a future ADR may unify if they converge.
- **Risk surfaced as an open question (brief §Open questions).** `auto` preferring the tag means STBs that ignore the matrix would show landscape; a per-deployment "assume tag-blind" toggle that flips `auto`→`pixels` is a candidate. SEI display-orientation for TS/RTSP is advisory-only pending a player-support survey.

## Alternatives considered

1. **Tag-only (never rotate pixels).** Rejected: MPEG-TS/RTSP/NDI carry no dependable rotation tag, so tag-only would leave those outputs wrong on every player. Both mechanisms are required.
2. **Pixels-only (always rotate, never tag).** Rejected: it throws away the zero-cost VLC-correct path and, at 90°/270°, forces a second encode even when a tag would do — wasteful where the transport can tag.
3. **A new orientation enum distinct from `QuarterTurn`/`Orientation`.** Rejected: `QuarterTurn` already models quarter turns losslessly and is tested; a new vocabulary would fragment the model. We reuse it.
4. **Reuse `Head.orientation` (`Landscape`/`Portrait`) for network outputs.** Rejected: that type is a scanout portrait/landscape flag for KMS heads ([display-out](../research/display-out.md)), not a quarter-turn-plus-mechanism; conflating them would mis-model both. Kept distinct (consolidation deferred).
5. **Put orientation on the canvas (one global rotation).** Rejected: orientation is per-output — one program legitimately feeds a landscape HLS and a portrait file. It is an output property.
6. **Rotate via an FFmpeg `transpose` filter on every output.** Rejected: that is a per-output filter pass (extra cost, color-pipeline risk) where the compositor already owns a lossless GPU quarter-turn; and it cannot produce the tag path at all.

## How this honors the invariants

- **#1 (output-clock).** Both mechanisms are applied at **encode/mux** (tag = mux init + side data; pixels = the compositor's existing per-rendition render, off the clock-emit path). Nothing here runs on, blocks, or paces the tick loop; `out_pts = f(tick)` is unchanged.
- **#10 (isolation).** Preview reflecting orientation stays **read-only, drop-oldest** ([preview-subsystem §2](../research/preview-subsystem.md) / [ADR-P001](../decisions/ADR-P001.md)); a slow preview freezes its own image and can never back-pressure the engine. The layout manager setting the orientation is a control-plane config edit, applied via the existing seam — not a channel the engine awaits.
- **#8 (tag, never convert).** The tag path is invariant #8 applied to orientation — label the presentation without touching pixels. The pixels path is a geometry transform inside the existing linear pipeline; it reorders no color stage. Both end at the [ADR-C006](../decisions/ADR-C006.md) ffprobe verification gate.
- **#7 (encode-once).** The **tag** path adds no encode (mux-init metadata, fans to all sinks of the rendition). A **pixels-path** rotation/flip changes pixel content, so it is a separate encode whenever outputs differ — a 90°/270° turn additionally swaps geometry (W↔H); a 180°/flip keeps geometry (no swap) but still produces distinct pixels and so is a separate pixel rendition when another output of the same program is unrotated. The rule is the same one any divergent rendition follows ([ADR-M002](../decisions/ADR-M002.md)): no geometry swap for 180°/flip, but it may still require a distinct pixel rendition.
