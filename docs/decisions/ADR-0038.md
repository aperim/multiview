# ADR-0038 — Per-source wall-clock trust + multi-source time-of-day sync

- **Status:** Proposed
- **Date:** 2026-06-08
- **Deciders:** Multiview engine + input + output
- **Area:** streaming / timing (T-series)
- **Related:** [ADR-T003](ADR-T003.md) (unified timing — built ON), [ADR-T001] (output-clock),
  [ADR-T009] (framestore COW), [ADR-0032](ADR-0032.md) (HLS serving + `utc0@tick0` PDT anchor),
  [ADR-0020] (the WHEN/WHAT timing-layer split + slew-not-step; the PTP-is-reference-not-pacer
  principle — note repo **ADR-0033** is AES67 audio I/O, *not* the PTP-reference ADR), inv #11
  live-apply class. **Brief:** [wall-clock-sync](../research/wall-clock-sync.md).
- **Supersedes / refines:** none. Additive; default-off ⇒ zero behaviour change.

## Context

Multiview ingests many live sources whose absolute time-of-day relationship is operationally
important (multi-camera coverage of one event; "what real time is this content?"). Today the engine
has **no notion of a per-source wall-clock**: `multiview-input/src/normalize.rs` anchors each input's
first frame to `master_now` and advances by the input's own PTS deltas — i.e. **reclock-to-house by
arrival**, with no cross-source common timeline. A repo-wide `rg` confirms the surface is greenfield:
no `WallClockRef`/`WallClockTier`/`SyncMode`/`sync_mode`/output-PDT exists in `crates/`.

Real transports *do* carry wall-clock standards (RTCP SR, HLS EXT-X-PROGRAM-DATE-TIME, MPEG-TS TEMI,
SMPTE ST 12-1 SEI, RP188/ATC over ST 2110-40, PTP), but **many real sources carry none** (plain RTSP
without SR, HLS without PDT, SRT/RTMP/file, synthetic sources). The product needs to:

1. **Honestly classify** each source's wall-clock trust (and surface it), without authoring it.
2. Let the operator **Use** or **Discard** a source's wall-clock per source.
3. Offer a per-program **time-of-day sync mode** that aligns multiple cameras to a common reference
   instant — **without ever stalling the output clock** (inv #1) and **without a second decoded frame
   copy** (decode-once, inv #5/#7).

The hard constraints: a naïve decoded MAX-window alignment buffer is multi-GB per source *and*
violates decode-once; arrival-anchored sources fundamentally cannot be content-synced (the capture
information was discarded); a synced output must advertise the *content's* time, not emission "now";
and none of this may give control/preview a way to back-pressure the engine (inv #10).

## Decision

We add per-source wall-clock trust + a per-program time-of-day sync mode, composed entirely from
existing engine machinery, default-off.

**1. R1 — Detect → Trust → Use|Discard → reclock-to-house.** Per source, a detector extracts a media-
PTS→wall-clock affine map from whatever standard the transport carries. The **existing** lock-state
classifier (`ptp.rs` `LockState` + `sysref.rs` `classify_system`) is **reused per source** (via a thin
per-origin synthetic-sample adapter — non-PTP origins disable the PTP path-delay guards and drive
Holdover from staleness) to yield `WallClockTier { Trusted, Suspected, None }` (`Locked`→Trusted,
`Holdover`→Suspected, `Freerun`/`Acquiring`→None). The operator verb `WallClockChoice { Use, Discard }`
is config-authored; the tier is **measured at runtime, never authored**. **Discard/None reuses the
as-built reclock-to-house path verbatim** (`normalize.rs`); the net-new branch is the **Use** path —
rebase onto the common wall-clock timeline. A **bare timecode (SEI/RP188) caps at Suspected** and can
never enter `ContentSynced`.

**2. Honest content-sync rule.** A discarded/house-clocked source **cannot** be truly content-synced —
it is "house-clocked / live" (lowest latency, roughly arrival-aligned, never frame-accurate). True
multi-cam content-sync requires a **Trusted** common-timeline wall-clock, and is **tier-qualified**:
*frame-accurate* on Tight (PTP), *time-of-day-accurate (ms-class)* on Loose (SR/PDT). The badge must
say so and never over-promise sub-frame phase on Loose.

**3. R2/R3 — Encoded, pre-decode, per-program alignment buffer.** To align a synced source to `T`,
hold its **compressed packet stream** in a **net-new `ReorderBuffer<EncodedPacket>`** inserted between
the demuxer and the decoder, byte-bounded to `MAX × peak_bitrate`, **drop-oldest**, released at
`deadline = wall_clock(pts) + D` via the (now-wired) pacer; the decoder runs `D`-behind real-time so
the **normal-sized** framestore (`RING_CAPACITY = 256`) holds the frame for `T`. **Exactly one decoded
copy** (decode-once). The buffer is **per-program**, `D` shared across the program (`D` = slowest synced
source `≤ MAX`); a single-source/passthrough program syncs to nothing ⇒ **no buffer**.

**4. R2/R3 — MAX→OFFLINE, never stall.** In sync mode the engine keys the **non-blocking** framestore
read on `T = our_now − D` (the **single** net-new data-plane change at the `drive.rs` sample seam;
today `let now = tick.pts;`). A source `> MAX` behind `T` latches an aged frame and rides the
**existing** STALE→NO_SIGNAL ladder to last-good/slate — `SyncStatus::Offline` is a **thin telemetry
overlay** mapping onto NO_SIGNAL, **not** a new render state and **not** a degraded out-of-sync picture.
**OFFLINE threshold = MAX**, so MAX bounds both buffer memory and the offline decision. We pin
`nosignal ≤ MAX` so OFFLINE-at-MAX truly fails to slate. `D` never enters `pts_at`/`deadline_nanos`;
the cadence stays house-locked (inv #1/#3).

**5. R4 — Cadence house-locked, label = synced instant.** Output cadence stays `out_pts = f(tick)`
(pure, unchanged). The advertised wall-clock **label** is a separate projection:
`label_pts(N) = utc0 + rescale(N, cadence) − D`. FreeRun (`D=0`) = our wall-clock; Syncing (`D>0`) =
`T`, the reference instant the tiles are aligned to — **never** emission "now". `D` is subtracted
**once at the carrier-write seam**, never on the data plane, and is a **latched, hysteresis-damped
per-program constant** changed only at Class-2 seams. Three greenfield carriers: HLS
EXT-X-PROGRAM-DATE-TIME, H.264/H.265 SMPTE ST 12-1 SEI, and NTP↔RTP (RTCP SR for true RTP; NDI's own
100 ns timecode for NDI — a **distinct** path). Apply the TAI-UTC offset (37 s) + PTP
`currentUtcOffset` before any UTC label, in integer ns. On any `D` change, step the label behind a
Class-2 plan + `EXT-X-DISCONTINUITY` / SEI jam — never silently.

**6. Tight (PTP) tier — discipline, never jam.** The `ptp.rs` servo's `frequency_ppb` (today computed
but unwired) is applied as a **bounded rate-slew** to a new cadence-rate-scale input on `OutputClock`,
keeping `pts_at` a pure integer function and **never** stepping `seed_nanos` mid-run; the SMPTE-Epoch
origin is seeded at start/relock only (mid-run re-jam = Class-2). PTP-lost rides
Locked→Holdover→Freerun→NTP fallback; the label follows the badge.

**7. Surface.** Config: per-source `wall_clock: Option<SourceWallClock{choice}>`; per-program
`sync: {mode, max_window (validated tier-bounded Duration newtype), tier}`; default FreeRun. API:
per-source rides source CRUD; per-program sync = **Class-2/Reset-lite** through the existing
`RouteClass` classifier with a mandatory plan/dry-run, plus a new `Program.sync.{mode,max_window,tier}`
capability-matrix row. Realtime: `WallClockTrust` on Inputs, `SyncStatus` on Tiles, conflated
drop-oldest (inv #10). UI: trust badge + Use/Discard toggle, program sync controls, per-tile sync
indicator, program rollup + `D` readout.

## Consequences

**Positive.**

- Honest, runtime-measured trust per source; the operator decides Use/Discard; the UI never
  over-promises phase it does not have.
- Real multi-camera time-of-day sync where a Trusted wall-clock exists, with `D ≤ MAX` added latency,
  bounded memory, and **exactly one** decoded copy.
- Default-off ⇒ **zero behaviour change**; Discard/None reuses the as-built data plane verbatim.
- Inv #1/#3 preserved structurally: the cadence path (`pts_at`/`deadline_nanos`) is byte-for-byte
  unchanged; the only data-plane edit is *which instant* the non-blocking read samples.
- OFFLINE reuses the bulletproof fail-to-slate ladder — a `> MAX`-behind source degrades exactly like a
  dead source, never a torn/out-of-sync picture, never a stall.

**Negative / costs.**

- New `ReorderBuffer<EncodedPacket>` + decoder-`D`-behind + a wired pacer (the existing post-decode
  `ReorderBuffer` and the unwired pacer are **not** reusable as the delay buffer — using them as-is
  would buffer decoded frames and blow memory; this is the most error-prone slice).
- New extractors per standard (RTCP SR receiver, HLS media-playlist/PDT scanner, TEMI parser, SEI-TC
  side-data hook, RP188-decode) — each additive but real net work.
- A sync-mode change is **Class-2** (alters `D`): outputs reset, consumers reconnect — surfaced via the
  plan/dry-run, never silent.
- The Tight tier requires a new bounded cadence-rate-scale input on `OutputClock` (the servo is
  currently computed-but-unconsumed); must be implemented as rate-only slew, never a seed step.

**Risks / mitigations (all folded into the design).**

- *Decoded-buffer regression* → property test: `buffer_bytes ≤ MAX × configured_max_bitrate`; forbid a
  `ReorderBuffer<DecodedFrame>` at MAX depth.
- *Stall instead of OFFLINE* → the read is a pure non-blocking atomic snapshot; chaos/soak test: a
  source pushed `> MAX`-behind transitions OFFLINE→NO_SIGNAL within one MAX window, output clock never
  blocks; cadence-invariance test asserts `D` never reaches `pts_at`/`deadline_nanos`.
- *Wrong output label* → test that the label projection and data-plane PTS diverge by exactly `D` and
  that `pts_at` is byte-identical for `D=0` vs `D>0`; `D` latched/hysteresis-damped; TAI-UTC offset
  applied.
- *Over-trusting a label* → test that two sources with identical embedded TC but differing arrival/PTS
  are **not** content-synced; SEI/RP188 caps at Suspected.
- *Servo jam* → bounded per-update slew + accumulated-deviation clamp; adversarial PtpSample soak
  asserting strict-monotonic `out_pts` and in-tolerance cadence through step/outlier/loss.

## Alternatives considered

- **Decoded MAX-window buffer (rejected).** ~3.11 GB/source at 10 s (12.44 GB for a 2×2) **and** a
  second decoded copy — violates decode-once #5/#7 and bounded-memory. Encoded pre-decode is ~50–500×
  cheaper depending on bitrate and adds zero decoded copies. *(See Efficiency.)*
- **Stall the output clock until the slowest source catches up (rejected).** Violates inv #1
  outright. We instead key the **non-blocking** read on `T` and let a missing source fall to last-
  good/slate.
- **Declare a `> MAX`-behind source merely "out of sync" and composite it anyway (rejected).** Emits a
  visibly wrong/torn picture. We declare **OFFLINE** → the existing fail-to-slate ladder.
- **A new parallel trust state machine (rejected).** The engine's `LockState`/`classify_system` already
  partition the lifecycle along the exact trust boundary; we reuse it per source.
- **Label the synced output with emission "now" (rejected).** Falsely advertises "now" while showing
  content from `now − D`. We label `T = now − D` (the content's reference instant).
- **Build a second servo to slave emission to a PTP source (rejected).** Violates the PTP-is-reference-
  not-pacer principle. We slew the *rate* only, never slave the pacer to a source.
- **Author the trust tier in config (rejected).** Trust is a runtime measurement; config carries only
  the Use/Discard verb.

## Efficiency

The feasibility rests entirely on placing the delay in **compressed bytes pre-decode**.

A decoded 4K NV12 frame (inv #5 = 1.5 B/px) = `3840 × 2160 × 1.5 = 12.44 MB`.

| Buffer (per source) | MAX = 0.5 s | MAX = 1 s | MAX = 10 s | 2×2 @ 10 s |
|---|---|---|---|---|
| **Decoded** NV12 @ 25 fps (prohibitive; + 2nd copy) | 0.156 GB | 0.311 GB | **3.11 GB** | **12.44 GB** |
| **Encoded** H.264 @ ~5 Mbps (streaming) | 0.31 MB | 0.625 MB | **6.25 MB** | **25 MB** |
| **Encoded** H.264 @ ~25–50 Mbps (broadcast) | 1.6–3.1 MB | 3.1–6.25 MB | 31–62 MB | 125–250 MB |

- **~500× cheaper at 5 Mbps; ~50–100× at 25–50 Mbps.** The load-bearing point is not the headline ratio
  but that the encoded buffer is bounded at **`MAX × peak_bitrate`** and adds **zero** decoded copies.
- **Framestore stays bounded:** decoder runs `D`-behind, producing frames only around `T`; the ring
  keeps `RING_CAPACITY = 256`. COW publish is `O(capacity)` on the input thread, never the output clock.
- **Latency:** added `= D ≤ MAX`; MAX hard-capped (≤ 30 s) bounds **worst-case `D`** and buffer memory.
  Single-source/passthrough = `D = 0`.
- **Invariant-safe degradation:** if a decoder cannot sustain `D`-behind, the encoded buffer fills then
  drop-oldest, and the source goes OFFLINE — never a memory blow-up.
