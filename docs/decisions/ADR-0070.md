# ADR-0070: Object-store / S3 offload of recordings — asynchronous, bounded, multipart, lifecycle-governed, never back-pressuring

- **Status:** Proposed
- **Area:** Output / Storage / Config / Control / Events / Telemetry / Web
- **Date:** 2026-06-13
- **Source brief:** [recording-storage-offload.md](../research/recording-storage-offload.md) §3
- **Extends:** [ADR-0037](ADR-0037.md) (the recorder's bounded drop-oldest write path, dual-axis retention, and disk-pressure disarm — **the isolation discipline this ADR reuses verbatim for the uploader**), [ADR-0032](ADR-0032.md) (rolling segmenter + atomic publish — the closed-segment source the uploader taps)
- **Builds on:** [ADR-R003](ADR-R003.md) (`backon` capped-exponential + full-jitter back-off), [ADR-0035](ADR-0035.md) (`HealthWarning` model), [ADR-R009](ADR-R009.md) (`/livez`/`/readyz` — a storage fault must not restart-loop the container), [ADR-0042](ADR-0042.md) (IPv6-first)
- **Relates to:** [ADR-0071](ADR-0071.md) (the Iceberg catalog this offload feeds), [ADR-0052](ADR-0052.md) (consent/retention boundary), invariants #1/#10

## Context

The operator asked for **S3 storage** and **offload of ISOs/recordings to S3**. The recorder design ([ADR-0037](ADR-0037.md)) writes closed, durable segments to local disk via the rolling segmenter (`close_current` → `<seg>.tmp` → `rename(2)`; manifest `atomic_write` → `sync_all` → `rename` — the `multiview-output` segmenter/HLS-live seams, to confirm against the implementation). Offload must move those closed files to an S3-compatible object store **without** ever stalling the engine, the recorder, or the live transports.

The cardinal constraint is the recorder's, restated: a slow/full/unreachable object store can **never** back-pressure the data plane. The offloader operates on **closed files on disk**, asynchronously; the worst a wedged bucket can do is leave local copies pending — which local retention + disk-pressure disarm ([ADR-0037 §D](../research/iso-program-recording.md)) already govern. There is **no S3/object-store/multipart/statvfs code in the tree today** (this is net-new) — but it inherits the [ADR-0037](ADR-0037.md) isolation pattern.

The S3 HTTP API is **de-facto open** — MinIO, SeaweedFS, Ceph RGW, Garage, and AWS S3 all implement it. S3 **multipart upload** requires a 5 MiB minimum part size (except the last part) and is the standard path for large objects; **S3 lifecycle** rules (`ExpirationInDays`, `AbortIncompleteMultipartUpload`) govern long-term retention and orphaned-part cleanup (verified against AWS S3 docs).

## Decision

1. **The uploader taps closed segments, never live packets.** When the rolling segmenter publishes a closed segment + atomic manifest, the writer task enqueues that **path + small metadata** (not the bytes) into a **bounded work queue**. The uploader reads a file that already exists and is durable. The offloader and the engine share **no channel, no lock, no buffer** — only a directory of closed files and a bounded queue of filenames. This is the structural reason invariants #1/#10 hold.

2. **A bounded, never-back-pressuring uploader (reuse [ADR-0037 §E](../research/iso-program-recording.md)).** A dedicated **off-hot-path uploader task** drains the bounded queue. Full queue → the local file is simply retained under existing retention and a `storage-offload-backlog` warning is raised; nothing blocks, nothing grows in RAM (the queue holds paths, not media). The in-memory queue is **only a wake-up hint, never the source of truth**: the **durable segment inventory / local index** ([ADR-0037](ADR-0037.md), [iso-program-recording.md §A](../research/iso-program-recording.md)) carries a per-segment `offloaded` flag, and a periodic **reconciler** scans for closed-but-not-offloaded segments and re-enqueues them — so a path dropped from a full queue or lost across a restart is *eventually offloaded by the scan*, never permanently forgotten. Objects above a threshold use **S3 multipart upload** (≥ 5 MiB parts, streamed so a multi-GiB ISO never loads wholly into RAM; bounded in-flight parts). On any S3 error → **`backon` capped-exponential + full-jitter** back-off ([ADR-R003](ADR-R003.md), the same crate the recorder uses), capped at minutes. Persistent failure leaves the local copy in place and latches the warning — it **never** deletes a local file it has not confirmed durable remotely.

3. **Idempotent keys + incomplete-upload reaping.** Object key = a deterministic function of `(recorder_id, media_kind, segment_start_utc, sequence)`; a crash-then-retry is a safe same-bytes overwrite. An S3 lifecycle rule (`AbortIncompleteMultipartUpload` after N days) reaps crashed multipart uploads so a partial upload never orphans paid parts.

4. **Delete-after-confirm (normal retention) vs emergency disk-pressure (always wins).** Under **normal retention**, a local segment becomes prune-eligible **only after** the uploader confirms the object is durable **and** the catalog row is written ([ADR-0071](ADR-0071.md)). "Durable" means an **integrity-checked** upload (multipart checksum / stored content hash validated against the local file) — **not merely HTTP 200 + ETag**, which is a commit confirmation, not an integrity proof across all S3-compatible stores. **Emergency disk-pressure prune is a distinct path that always wins** (the recorder never blocks to keep bytes): at the disk-pressure threshold the recorder first disarms *new* recording, then — if still over threshold with a wedged backlog — prunes the **oldest not-yet-offloaded** local segments, emitting a loud `storage-offload-data-loss` warning with the dropped object keys (never silent). Default policy prefers dropping oldest un-offloaded copies over stalling; an opt-in "disarm-only, never drop un-offloaded" knob (risking a full disk) is an Open question. A wedged *catalog* (upload succeeded, row pending) is **not** data-loss — the bytes are durable and only normal prune is gated. The object store is an *addition*, never a precondition for local correctness.

5. **Two retention domains, not confused.** **Local** retention is the recorder's existing dual-axis (time AND size) prune + disk-pressure disarm, **unchanged** — local disk becomes a cache/staging area. **Object-store** retention is enforced by **S3 lifecycle rules** on the bucket; Multiview *declares* a recommended lifecycle (optionally applies it via `PutBucketLifecycleConfiguration` where supported), but the **bucket owner's policy is authoritative** — we document the recommendation and never silently override.

6. **Vendor-neutral client; IPv6-first.** Target the S3 HTTP API nominatively (no vendor SDK bundled, no vendor spec redistributed — the [managed-devices.md](../research/managed-devices.md) posture). Default Rust client: the generic **`object_store`** crate (one async trait over S3/GCS/Azure/local-fs, `put_multipart`, production-proven), so the same code targets any S3-compatible endpoint or a local path for air-gapped sites. Endpoint URLs accept bracketed IPv6 literals and must dual-stack-resolve ([ADR-0042](ADR-0042.md)). Credentials are **reference-only** (env/1Password/file), never literals in committed config (gitleaks/`cargo deny` gates apply).

7. **Config/API/realtime/UI are additive.** A `[storage]` `offload: Option<OffloadPolicy{endpoint, bucket, prefix, region, credentials_ref, multipart_part_mib, lifecycle}>`; operational `POST /recordings/{id}/offload` (force, `202` + op-id, shed-to-`503`) and `GET /storage/status`; a re-snapshotable `Event::StorageStatus{offload_backlog, last_uploaded_utc, bucket_reachable, bytes_offloaded}` (conflated drop-oldest); additive `#[non_exhaustive]` `WarningCode` variants `storage-offload-backlog` / `storage-offload-failing` / `storage-offload-data-loss` (emergency disk-pressure dropped an un-offloaded segment, Decision 4) at `GET /api/v1/health` (**not** `/livez`/`/readyz`, [ADR-R009](ADR-R009.md)).

## Consequences

**Positive.**
- Recordings/ISOs offload to any S3-compatible store with bounded RAM, multipart streaming, and lifecycle-governed long-term retention — the operator's full ask, vendor-neutral.
- The uploader is **structurally isolated** (operates on closed files; shares no channel with the engine), so a wedged bucket degrades to "local copies pending + a warning," never to a stalled frame.
- `object_store` gives a local-filesystem fallback for air-gapped sites at zero extra code.

**Negative / costs.**
- Net-new S3/multipart/lifecycle code with real footguns (multipart abort handling, ETag confirmation, credential refresh, IPv6 resolution). The chosen crate's behaviour must be verified against a live MinIO/SeaweedFS (Open question in the brief).
- Offload roughly doubles read I/O on closed segments (read-to-upload) — sequential and off-peak-schedulable, recommended on a separate staging volume so it never contends with ingest writes.
- Two retention domains add operator-facing complexity (local cache vs bucket lifecycle); documented explicitly to avoid "why is my disk still full / why did my cloud copy vanish."

**Neutral.**
- Bucket lifecycle authority stays with the operator; Multiview only recommends/optionally-applies.

## Alternatives considered

1. **Stream live packets directly to S3 (skip local disk).** *Rejected* — couples the engine/recorder to network latency; a slow PUT would back-pressure the packet stream (violates #1/#10). Offload only closed, durable files.
2. **Unbounded upload queue / hold bytes in RAM.** *Rejected* — violates bounded-memory; the queue holds paths, full → retain-local + warn.
3. **Block local prune until offload completes (offload as a hard precondition).** *Rejected* — a wedged bucket would then fill local disk faster than disk-pressure can disarm, or stall prune. Delete-after-confirm gates only **normal** retention prune; the **emergency disk-pressure path (Decision 4) is separate and always wins**, dropping oldest un-offloaded local copies with a loud data-loss warning rather than stalling.
4. **Bundle a vendor SDK (aws-sdk-s3 only).** *Rejected as the default* — couples to one vendor and historically IPv4-centric. `object_store` is vendor-neutral with a local-fs fallback; `aws-sdk-s3`/`rusty-s3` remain fallbacks.
5. **Reimplement lifecycle/expiry in Multiview for the bucket.** *Rejected* — S3 lifecycle is the standard, owned by the bucket; we recommend rules and optionally apply, never reimplement a server-side expiry loop.
6. **No back-off (retry tight on failure).** *Rejected* — would hammer a recovering endpoint; `backon` capped-jitter ([ADR-R003](ADR-R003.md)), same as the recorder.

## How this honors the invariants

- **#1 (output-clock untouchable).** The uploader runs on a dedicated task over closed files; it never runs on, awaits, or shares a lock with the output-clock thread. `out_pts = f(tick)` is independent of bucket reachability, upload latency, or failure.
- **#10 (isolation).** The engine/recorder hands a *closed file path* to a bounded drop-oldest queue and returns; the uploader is never awaited. A full/slow/unreachable bucket → drop-to-retain-local + warn + `backon` back-off + auto-resume, never a stall, never RAM growth. This is the [ADR-0037 §E](../research/iso-program-recording.md) discipline applied one layer out.
