# ADR-0085: SIP media ↔ engine bridge — reuse the WebRTC RTP/SRTP media engine; audio into switcher-audio mix-minus

- **Status:** Proposed
- **Area:** Calling / media bridge (`multiview-sip` + `multiview-webrtc` + `multiview-input` + `multiview-audio` + `multiview-output`)
- **Date:** 2026-06-13
- **Source brief:** [sip-calling.md](../research/sip-calling.md)
- **Builds on:** [ADR-0084](ADR-0084.md) (the SIP stack, crate, transports, source/output seam), [ADR-0048](ADR-0048.md) (the shared str0m media endpoint), [ADR-0078](ADR-0078.md) (mix-minus / N-1 / talkback safety), [ADR-0059](ADR-0059.md) (`AudioReader` multi-cursor view + `repoint_crossfade`), [ADR-0049](ADR-0049.md) (encode-once fan-out + Opus return rendition)
- **Relates to:** [ADR-T013](ADR-T013.md) (RTP-audio → `AudioStore` rebase seam), [ADR-T014](ADR-T014.md) (WHIP media path), [ADR-0034](ADR-0034.md) (per-stream routing — the call video is a routable input stream), [ADR-0042](ADR-0042.md) (IPv6-first SDP), [ADR-T001](ADR-T001.md)/[ADR-RT004](ADR-RT004.md) (invariants #1/#10)

## Context

[ADR-0084](ADR-0084.md) pins SIP as signaling-only and routes the negotiated media to the shared
[ADR-0048](ADR-0048.md) WebRTC endpoint. This ADR pins **how the SIP media actually bridges into the
engine**: the SDP↔str0m adapter, the SRTP keying-model straddle, and — load-bearing — that the SIP
party receives a **mix-minus return** so it never hears itself.

The reuse is deep and rests on verified seams: the pure `MediaEngine` trait + `WebRtcProducer`
("sampled, never pacing") at `crates/multiview-input/src/webrtc/transport.rs:201`/`:213`/`:547` and the
keyframe-gated `H264Depacketizer` at `:227`; RTP wrap via `WrapBits::Rtp32`
(`crates/multiview-input/src/normalize.rs:42`/`:54`); audio in via `AudioStore::publish`
(`crates/multiview-audio/src/store.rs:171`); the mix-minus + crossfade via `ProgramBus::repoint_crossfade`
(`crates/multiview-audio/src/program.rs:217`) and the [ADR-0078](ADR-0078.md) `AudioBusKind::MixMinus`;
return egress via the encode-once `PacketRouter`/`PacketSink`
(`crates/multiview-output/src/fanout.rs:140`/`:152`).

The one genuine SIP-specific complication is **media keying**: WebRTC mandates DTLS-SRTP, but classic
SIP commonly uses SDES-SRTP ([RFC 4568](https://www.rfc-editor.org/rfc/rfc4568.html), keys in the
TLS-protected SDP) or plain RTP. The bridge must straddle both — verified industry practice for any
SIP↔WebRTC interworking (gateways convert DTLS/SRTP↔plain-RTP): the SDP transport profile signals
which — `UDP/TLS/RTP/SAVP(F)` for DTLS-SRTP ([RFC 5764](https://www.rfc-editor.org/rfc/rfc5764.html)),
`RTP/SAVP` + `a=crypto` for SDES-SRTP, `RTP/AVP` for plain RTP.

## Decision

### 1. The SDP↔media adapter is the only net-new media code

`multiview-sip` owns a small **SDP↔str0m adapter**:

1. Parse the far-end SDP (m-lines, rtpmap/fmtp, direction, `a=rtcp-mux` presence, SRTP keying).
2. Configure a str0m `Rtc` session **in RTP mode** for the call (the mode WHIP ingest already uses,
   [ADR-T014](ADR-T014.md)) on the shared endpoint, **or** the SIP SRTP path of §3 for SDES/plain-RTP
   far ends.
3. Emit the SDP answer/offer Multiview sends, **IPv6-first** (`c=IN IP6`, no TTL —
   [ADR-0042](ADR-0042.md)), reusing the IPv6-aware SDP path the WebRTC answer already needs.

**SDP port model (non-ICE reality).** A non-ICE SIP peer does not perform WebRTC ICE/DTLS demux; it
sends/expects RTP/RTCP at concrete advertised `m=` ports. The shared `webrtc.udp_port` is reused
**only** when the call RTP/RTCP-muxes on it (Multiview offers/requires `a=rtcp-mux`) and inbound demux
is unambiguous by 5-tuple + SSRC/PT (the [ADR-0048](ADR-0048.md) §4 tuple demux extended per-call).
A far end that cannot `rtcp-mux`, or any tuple/SSRC/PT ambiguity (symmetric-NAT source rewrite, SSRC/PT
collision), gets a **per-call RTP/RTCP port pair from a bounded pool** — never silently-forced
multiplexing; pool exhaustion is an admission denial (logged 488/486), not an isolation hazard.

Everything downstream is **reused verbatim**: depacketize → decode → `TileStore` (inbound video),
Opus/G.711 decode → `AudioStore::publish` (inbound audio), `PacketRouter` route()+1 (outbound video),
the mix-minus return (outbound audio). No second decode path, no second timing model.

### 2. Inbound media path (call → tile + strip)

- **Video:** RTP → the existing `MediaEngine` seam → `H264Depacketizer` (keyframe-gated, bounded) → a
  packet-fed H.264 decoder (the [ADR-T014](ADR-T014.md) decoder, `ffmpeg`-gated; decode-at-display-res,
  inv #6) → NV12 → `TileStore` publish, with `PtsNormalizer` running `WrapBits::Rtp32` (inv #3).
  `ColorInfo` tagged from the H.264 VUI, default BT.709 (inv #8 — the compressed-ingest policy).
- **Audio:** RTP → Opus (RFC 7587) or G.711 (8 kHz µ-law/A-law) depacketize/decode → 48 kHz PCM →
  `AudioStore::publish` → a switcher input strip ([switcher-audio.md §4](../research/switcher-audio.md)),
  rebased per [ADR-T013](ADR-T013.md). DTMF arrives as RFC 4733 telephone-event → a control event,
  **never** mixed into program audio.
- **Lifecycle/keyframes:** PLI is sent at call start + on loss while the keyframe gate is closed,
  rate-limited (≥ 2 s) — the [ADR-T014](ADR-T014.md) policy; far ends that ignore PLI recover via their
  encoder GOP (documented per-peer, like the OBS profile).

### 3. The SRTP keying straddle (the SIP-specific carve-out)

| Far end | Keying | Path |
|---|---|---|
| WebRTC-style SIP / WebRTC-SIP gateway / modern PBX | **DTLS-SRTP** (`UDP/TLS/RTP/SAVP(F)` + `a=fingerprint`/`a=setup`, [RFC 5764](https://www.rfc-editor.org/rfc/rfc5764.html)) | str0m's native DTLS-SRTP — fully reused |
| Classic secure SIP | **SDES-SRTP** (`RTP/SAVP` + `a=crypto`, [RFC 4568](https://www.rfc-editor.org/rfc/rfc4568.html)) | keys in the TLS-protected SDP; SRTP transform on the shared socket. **MUST ride `sips:`/TLS** — which protects keys only hop-by-hop through proxies (not end-to-end), so SDES is permitted only for trusted-direct/trusted-proxy plants |
| Legacy SIP | **plain RTP** (`RTP/AVP`) | only when `allow_insecure_media` is set; off by default |

(unverified) Whether str0m's SRTP transform can be driven off-WebRTC for SDES must be validated; if
not, the SDES/SRTP-context handling is implemented in `multiview-sip` against
[RFC 3711](https://www.rfc-editor.org/rfc/rfc3711.html)/[RFC 4568](https://www.rfc-editor.org/rfc/rfc4568.html)
using a vetted SRTP crate, with keys still flowing through the shared socket/demux. **SDES is a v1
requirement, not optional** — classic SIP/TLS plants overwhelmingly key media with SDES (or plain RTP),
so "full SIP/TLS calling" is not met without it; the open question (brief #1) is the *mechanism*
(reuse str0m vs a vetted SRTP crate), the largest implementation risk. A **DTLS-SRTP-only first cut**
is a stopgap for an early hardware-validation milestone (interops with modern PBXes and WebRTC-SIP
gateways only) and is explicitly **not** the "full" v1 — not a silent deferral.

### 4. The SIP party gets a mix-minus return (mandatory, no self-audio)

A SIP participant **must** receive `program − that participant + talkback`, or it hears itself echoed.
This is [ADR-0078](ADR-0078.md)'s `AudioBusKind::MixMinus` reused without modification:

- The call's inbound audio strip is **auto-excluded** from its own return (the
  [ADR-0078](ADR-0078.md) auto-exclude-self safety invariant — structural, not a checkbox; an explicit
  override is required to defeat it).
- The return is realized over the [ADR-0077](ADR-0077.md) send matrix as `base − excluded` — **not** a
  second decode — using the [ADR-0059](ADR-0059.md) `AudioReader` multi-cursor view (each return bus
  gets its own wait-free cursor over the shared stores; a stalled/shed return never perturbs the
  program cursor — inv #10 across buses).
- **Talkback can never reach Program** (the [ADR-0078](ADR-0078.md) validator refuses
  talkback→Program); folding producer talkback into a SIP return is safe by construction.
- Per-return delay/gain/limiter/mute are available; a dying excluded source silence-fills so the return
  **completes over silence and never stalls** ([ADR-0078](ADR-0078.md)/[ADR-R011](ADR-R011.md)).

### 5. Return encoding (encode-once for audio; route()+1 for video)

- **Return audio** is encoded **once per distinct mix-minus rendition** (inv #7 for audio,
  [ADR-0049](ADR-0049.md) §6.1): a mix-minus is a *different mix* than program, so it is one extra
  encode per **distinct** return — parties excluded identically share a return; Opus where the far end
  wants it, a cheap G.711 narrowband encode otherwise. (Caveat: true per-party mix-minus excludes a
  different strip per party, so N talking parties ≈ N distinct returns — sharing only helps listen-only
  parties excluding the same set; budget N-party cost as ≈ N returns.)
- **Return/outbound video** is the program (or configured bus) rendition fanned to the call's
  `PacketSink` — **route()+1, zero extra video encode** (inv #7) — under the H.264 B-frames-off +
  repeat-headers constraint ([ADR-0049](ADR-0049.md); RTP carries no DTS). A *clean*/different video
  bus pays exactly one extra encode for that rendition.

### 6. Video as a routable input stream

The inbound call's video `TileStore` is a first-class routable input stream
([ADR-0034](ADR-0034.md)/[decoupled-routing.md](../research/decoupled-routing.md)): it can feed any
layout cell, a PiP, or a full-frame guest. A call with no video contributes no tile; its audio strip is
still on the mixer (audio-only PSTN legs).

## Consequences

- **SIP costs a dial-plan + an SDP, not a media stack.** One decode per inbound call, one encode per
  distinct return rendition, zero extra video encode for the program/bus return — the same efficiency
  shape as WHIP/WHEP.
- **Echo is designed out.** Auto-exclude-self + talkback-never-to-Program make the catastrophic
  remote-contribution failures (self-echo, producer mic on air) *unrepresentable*, reusing
  [ADR-0078](ADR-0078.md) — no new DSP.
- **The keying straddle is the real cost.** SDES-SRTP may need a SIP-side SRTP crate; the DTLS-SRTP-first
  fallback keeps v1 shippable and honest.
- **Conferencing scales by distinct returns, not raw party count** — N parties excluded identically
  share one return; the [ADR-0059](ADR-0059.md) `AudioReader` multi-cursor contention under N returns
  is soak-tested before scaling (the open question).
- Downstream: a call-control UI (ring/answer/hangup, return health, mix-minus indicator), capability-matrix
  rows, OpenAPI, and metrics (call counts, PLIs, return ring drops) in `multiview-telemetry`.

## Alternatives considered

- **Re-decode per return (a full mix per SIP party).** Rejected per [ADR-0059](ADR-0059.md) §5 /
  [ADR-0078](ADR-0078.md): the `AudioReader` cursor split gives each return a wait-free cursor over the
  shared stores at one `AtomicI64` cost — no second decode, no N× ring memory.
- **Send program (not mix-minus) to the SIP party.** Rejected: that is the echo bug the operator's
  requirement exists to prevent; mix-minus with auto-exclude-self is mandatory.
- **A second video encode per call.** Rejected (inv #7): outbound video is a `PacketSink` route()+1 on
  the existing program/bus rendition; only a distinct *clean*/breakaway video bus pays one extra encode.
- **Re-stamp inbound SIP RTP from the output tick (`out_pts = f(tick)`).** Rejected: inbound media is a
  sampled source; it uses the existing `PtsNormalizer`/`WrapBits::Rtp32` rebase (inv #3), and the output
  re-stamps from the tick counter — the same split every other ingest uses.
- **Mix DTMF into program audio.** Rejected: DTMF is a control signal (RFC 4733 telephone-event),
  surfaced as an event, never on air.

## How this honors the invariants

- **#1 (output-clock):** the SDP adapter and SRTP context live in `multiview-sip`/the shared driver,
  never on the tick path. Inbound media is *sampled* through the bounded keyframe-gated
  `MediaEngine`/`AudioStore` path; the mix-minus return is a consumer-side `ProgramBus`/`AudioReader`
  the engine never awaits; outbound video rides the existing fan-out through a bounded drop-oldest
  per-call ring. A SIP party that stalls loses *its* media, never a tick.
- **#10 (isolation):** every engine↔SIP edge is a bounded drop-oldest ring; the engine never awaits the
  UA, the far end, the SRTP context, or a return consumer. A stalled return's `AudioReader` cursor falls
  behind and reads silence-fill — it cannot perturb the program bus's cursor or any other return
  (inv #10 across buses), proven by the same wedged-consumer chaos gate as [ADR-0078](ADR-0078.md).
- **#3/#6/#7/#8** are upheld by reusing the WHIP paths verbatim: `WrapBits::Rtp32` + tick re-stamp
  (#3), decode-at-display-res (#6), route()+1 video + one-encode-per-distinct-return audio (#7),
  VUI-tagged BT.709-default color (#8).
