# ADR-0060: Structured, resource-scoped logging taxonomy + libav log routing/correlation

- **Status:** Proposed
- **Area:** Telemetry / cross-cutting (`multiview-telemetry`, `multiview-ffmpeg`,
  `multiview-input`, `multiview-output`, `multiview-control`, `multiview-events`)
- **Date:** 2026-06-13
- **Source brief:** [observability-logging.md](../research/observability-logging.md)
- **Extends:** [ADR-R009](ADR-R009.md) (telemetry/health/SLI surface), the BUILT libav→`tracing`
  bridge (`crates/multiview-ffmpeg/src/log_bridge.rs`), the BUILT subscriber builder
  (`crates/multiview-telemetry/src/tracing_init.rs`), and the BUILT `Topic::Logs` lane
  (`crates/multiview-events/src/topic.rs:57`).
- **Relates to:** [ADR-MV001](ADR-MV001.md) (X.733 alarm engine — the lifecycle layer above log
  records), [ADR-0052](ADR-0052.md) (two-pipe consent/retention boundary the local log buffer
  respects), [ADR-RT004](ADR-RT004.md) / [ADR-T001](ADR-T001.md) (invariants #10 / #1).

## Context

A live multiview's normal operating condition is glitchy inputs, and libav reports them by
logging — HLS opens, reconnects, decode errors, AVIO statistics. The BUILT bridge
(`log_bridge.rs:14-24`) already routes every libav line into `tracing` with the libav
**component** class (`hls`, `hevc`, `https`) as a field, and a bounded anti-flood suppressor
(`log_bridge.rs:191-308`) collapses repeats. But the operator's log sample shows the defect: a
line like

```text
INFO  libav  component="hls"   Opening 'https://cdn.example/cnn/master.m3u8' for reading
ERROR libav  component="hevc"  Error constructing the frame RPS.
```

carries **no resource identity** — not *which source*, *which output*, or *which layout*. CNN,
BBC, and ABC all decode HEVC and pull HLS, so `component` alone cannot tell the operator which
tile is flapping. The recovery is already handled (the tile state machine rides LIVE→STALE→
NO_SIGNAL, [resilience-and-av.md](../research/resilience-and-av.md) §1) and the flood is already
de-duplicated; the missing dimension is purely **attribution**.

The codebase already keys work by stable ids — `source_id`
(`crates/multiview-engine/src/drive.rs:67`, `:327-400`; `crates/multiview-cli/src/pipeline.rs`),
`output_id` (`crates/multiview-output/src/display/sink.rs:278`,
`crates/multiview-output/src/display/audio/sink.rs:404-465`) — but those ids reach logs only as
ad-hoc fields on individual call sites, and **never** reach libav's own lines, which fire on
libav-owned decoder/IO threads outside any of our spans (`log_bridge.rs:50`).

## Decision

Adopt a **resource-scoped span hierarchy** plus a **libav→resource correlation map**, so every
log record — ours and libav's — names the resource it came from, with logging strictly on the
best-effort side of invariant #10.

1. **Span hierarchy as the attribution spine.** A shallow `tracing` span tree:
   `run{run_id}` → (`source{resource_id,label}` | `output{resource_id,label,transport}` |
   `layout{resource_id}` | `program{}`). A log event inherits its parent span's fields, so any
   code running inside the right span is attributed automatically. `source`/`output` are
   long-lived (entered by the owning task/thread); `layout` is short-lived (wraps an
   `apply_layout`/`rebind_cell`, e.g. `drive.rs`); `program` carries the engine's rare program-
   level events. Because the `layout` span is short-lived, source/output records sample a coarse
   **`active_layout_id`** (outputs: `program_layout_id`) from engine state off-loop and attach it
   as a plain field, so a reconnect/decode line still answers "which layout is on air" without a
   per-frame layout span. **The output-clock loop logs nothing per tick** (per-tick observability
   is numeric metrics, not logs), and the layout sample is read off-loop, never on the tick.

2. **Stable resource-id vocabulary.** `resource_id` is the **config id** (`Source.id`,
   `Output.id`, `Layout.id`) — stable across restarts, what the API addresses, what alarms
   (MV001) and stats ([system-stats-attribution.md](../research/system-stats-attribution.md))
   already use — never a pointer or array index. Shared fields: `run_id`, `resource_kind`,
   `resource_id`, `label`, `transport` (outputs), optional `corr` (the realtime job key,
   `topic.rs`), optional OpenTelemetry `trace_id`/`span_id` (carried on the wire as the W3C Trace
   Context `traceparent` `trace-id`/`parent-id`; OTel export, decision 5).

3. **libav→resource correlation, priority-ordered, honest.** The bridge's `route()`
   (`log_bridge.rs:534`) resolves attribution before emit:
   - **(A) thread/task-local `ResourceContext`** set when a thread/task owns a source/output
     (the `Demuxer::open` seam, e.g. `crates/multiview-input/src/libav.rs`) — covers libav
     lines emitted synchronously on our own threads (the operator's `Opening '…cnn…'` case). It is
     **scoped, never stale**: a RAII/`task_local!` guard sets it on entry to an owned
     demuxer/decode region and **clears it on exit**, so a line emitted outside an owned region (an
     unrelated or nested unregistered context) sees no context and falls through to (B)/(C) rather
     than inheriting whichever resource last ran on the thread.
   - **(B) `AVClass` registration map** `context_ptr → resource_id`, recorded when we open a
     libav context and removed on `Drop`; resolved from `avcl` directly, or by walking libav's
     `parent_log_context_offset` to the owning format context, or weakly via `item_name`. The map
     must specify which concrete objects are registered (the per-resource `AVFormatContext*`, and
     reachable child `AVIOContext`/protocol/HLS sub-demuxer contexts), remove an entry on `Drop`
     **before** the pointer can be reused, and carry a per-entry generation/epoch so a reused
     address never resolves to a dead resource — a mismatch falls through to (C). Tests cover
     nested HLS/protocol/decoder-thread contexts and an open→close→reopen pointer-reuse case.
   - **(C) fall through to `component`-only** when neither resolves — the record **omits**
     `resource_id` rather than guess. Never a wrong id.
   The suppressor key gains `resource_id` (`(level, message)` → `(level, resource_id, message)`,
   `log_bridge.rs:259`) so per-source floods suppress independently.

4. **Five surfaces, one record.** stderr/file (BUILT `SubscriberBuilder`,
   `tracing_init.rs:42-115`) + an optional **systemd journal** layer (native structured fields,
   `journalctl RESOURCE_ID=cnn`) + a bounded **control-API log tail** (`GET /api/v1/logs` filter
   by `resource_id`/`kind`/`level` + live tail on the BUILT `Topic::Logs`, `topic.rs:57`) +
   the **MV001 alarm engine** (a recurring attributed error is an alarm input — logs are evidence,
   MV001 owns the X.733 lifecycle) + the **ADR-0052 consent-independent local buffer** (S5, for
   the support context-pack).

5. **OpenTelemetry export is opt-in, off by default.** An optional `tracing-opentelemetry` layer
   attaches OpenTelemetry `trace_id`/`span_id` (correlated on the wire via the W3C Trace Context
   `traceparent` `trace-id`/`parent-id`) and exports to an operator-run collector for trace↔log
   correlation. It is the operator's own diagnostic plane — **not** the Aperim telemetry pipe
   (ADR-0052) — governed by neither the licensing heartbeat nor the product-telemetry consent
   toggle, and gated behind a deny-clean feature so the CI-green baseline is untouched.

6. **Bounded, async, never back-pressuring (inv #10).** The log-tail ring is fixed-capacity
   **drop-oldest**; the live stream is a `broadcast` a slow reader cannot stall; the suppressor
   lock is held only for an O(cap≤256) scan off the engine thread (`log_bridge.rs`); the
   `AVClassMap` is bounded by open-context count. **Every** sink — including the file/journal/stdout
   writers, which are *not* inherently non-blocking — sits behind a **bounded, lossy non-blocking
   appender** (drop-oldest, count the drops); a wedged sink drops records, never stalls a writer,
   and engine/output-owned threads never write a sink synchronously. No log surface shares a lock
   with the engine, and the engine awaits no log consumer.

7. **Privacy boundary.** Full media URLs stay in **local** logs (the operator needs them); a log
   selected into a **context-pack for egress** is **redacted to host + `resource_id`** (ADR-0052
   §8). Logs never enter the daily telemetry pipe and leave the box only inside an explicitly
   operator-approved, redacted context-pack.

## Consequences

**Positive.**
- The operator's defect is fixed: every libav line (and every log of ours) names its source /
  output / layout via a stable config id, de-flooded per-resource, filterable in the UI.
- Maximum reuse — the bridge, the suppressor, the subscriber builder, and `Topic::Logs` all
  exist; this ADR adds attribution, a correlation map, a journal layer, and the tail producer.
- One vocabulary spans logs, alarms (MV001), per-resource stats, and the SPA.

**Negative / costs.**
- Net-new FFI surface: the `parent_log_context_offset`/`item_name` walk is more bounded pointer
  reads of libav structs (the `class_name` pattern the bridge already does, `log_bridge.rs:500-519`),
  each null-checked with a `// SAFETY:` note, falling through to `component`-only on any miss.
- The `parent_log_context_offset` coverage must be hardware-validated on the **deploy** ffmpeg
  version (the project's "validate ingest fixes on the deploy ffmpeg" rule) — a miss is harmless
  (fall-through) but coverage is worth measuring.
- The suppressor key gaining `resource_id` raises key cardinality; cap may need to scale with
  declared source count (still fixed at build, still bounded) to avoid eviction-churn re-floods.

**Neutral.**
- `tracing-opentelemetry` is off by default; the CI-green, deny-clean baseline is unchanged until
  a site opts in.
- The log-tail ring and the ADR-0052 S5 buffer may be the same or separate stores (open question
  in the brief §8) — both bounded, sharing the record shape.

## Alternatives considered

1. **Keep `component`-only (status quo).** Rejected — it is exactly the operator's complaint;
   the codec family is not the resource.
2. **Per-call resource fields instead of spans.** Rejected as the *only* mechanism — it cannot
   reach libav's own lines (they fire outside our call sites, on libav threads) and it litters
   every call site; a span covers all in-scope lines including libav's via §3.
3. **Attribute libav lines by the `avcl` pointer string.** Rejected — a pointer is unstable
   across runs and meaningless to a human; the resolved **config id** is the stable handle.
4. **Guess the resource from the most-recent thread-local even when the libav line came from an
   unregistered context.** Rejected — a *wrong* id is worse than none; mechanism C omits
   `resource_id` honestly (tested).
5. **Log per-tick engine state for full visibility.** Rejected — a log call on the output-clock
   loop is an invariant-#1 hazard (format cost, contended subscriber, blocking writer); per-tick
   observability is numeric metrics sampled off-loop ([system-stats-attribution.md](../research/system-stats-attribution.md)).
6. **An unbounded in-memory log buffer / blocking log sink for the API tail.** Rejected —
   violates inv #10; the tail is a bounded drop-oldest ring + broadcast a slow reader cannot
   stall.
7. **Route logs into the Aperim product-telemetry pipe.** Rejected — conflates operator
   diagnostics with anonymised analytics and would mis-sell "opt out of telemetry" as "logs
   stop"; logs are local-first and egress only via an approved, redacted context-pack (ADR-0052).
8. **Make `tracing-opentelemetry` a mandatory dependency.** Rejected — it would dirty the
   CI-green, deny-clean baseline and force an export plane on operators who do not run a
   collector; kept opt-in behind a feature.

## How this honors the invariants

- **#1 (output clock).** The output-clock loop and the decode→composite→encode→mux tick path
  emit **no** log per tick (a hard rule, checked by a reviewer grep); attribution adds nothing to
  the loop because the loop does not log. libav-line attribution happens on libav's own
  decoder/IO threads (`log_bridge.rs:50`), off the engine loop. No log call, lock, or sink ever
  sits on the tick path. Inv #1 holds by construction.
- **#10 (isolation).** Every log surface is bounded and async: the suppressor LRU is fixed-cap
  with lock held only for an O(cap) scan off the engine thread (`log_bridge.rs`); the file/journal/
  stdout writers sit behind a **bounded, lossy non-blocking appender** (a plain `tracing` writer is
  not inherently non-blocking, so this is mandatory); the control-API log tail is a fixed-capacity
  **drop-oldest** ring + a `broadcast` a slow consumer cannot fill; the `AVClassMap` is bounded by
  open-context count; the OTel exporter (opt-in) batches off-thread. No log surface shares a lock
  with the engine, and the engine awaits no log consumer — a wedged log reader loses log lines,
  never a tick. Inv #10 holds by construction.
