## Context

[ADR-0057](ADR-0057.md) Decision 4 gives a media player an EOF policy of
`hold_last_frame | loop | black | auto_off`. The `loop` policy loops **forever**:
on clean EOF it seeks to the in-point in place and keeps decoding, with no exit.
That is the right behaviour for a station ident or a background bed, but it is not
what a vamp is.

A **vamp** is a fill of *unknown length*: an opener bed under a presenter, a
"please stand by" roll, a sting held under a hand-off — material the operator
plays on a loop *to fill dead air until a cue lands*, then exits cleanly on that
cue. The two halves are in tension:

- The fill must loop seamlessly for an a-priori-unknown number of laps (the cue
  may come in 3 seconds or 3 minutes).
- The exit must be **frame-clean**: it lands on a loop boundary (or a declared
  exit frame), never mid-lap with a visible jump, and the operator arms it
  *ahead* of the boundary so the cut is deterministic, not a reflex race.

A forever-`loop` cannot express "stop at the next seam"; nothing in
[ADR-0057](ADR-0057.md), [ADR-0072](ADR-0072.md), or
[media-playout](../research/media-playout.md) models an *armed, boundary-aligned
exit*. This ADR adds exactly that, reusing the machinery that already exists.

What **is** true today, and what this ADR builds on (every path verified in the
worktree at this ADR's base):

- **The player channel and its transport are designed but unbuilt**
  ([ADR-0057](ADR-0057.md)): a media player is a **pre-declared, bus-selectable
  channel** owning one supervised ingest thread from boot, so `load`/`cue` change
  *what* the thread plays and never spawn/tear a thread on the engine path —
  which sidesteps the [ADR-W018](ADR-W018.md) live-spawn restriction. This ADR
  does **not** introduce a new `SourceKind`; the vamp is a transport mode of that
  same channel.
- **The playout substrate is BUILT.** `PtsWallClock`
  (`crates/multiview-cli/src/pipeline.rs:9192`) paces decoded frames to wall-clock
  and stamps `publish_time(pts)` as **source-relative media time** `pts − first_pts`
  (`pipeline.rs:9231-9244`); its backwards-PTS branch **re-anchors rather than
  stalls** on a discontinuity/wrap (`pipeline.rs:9208`); the decode→normalize→pace→
  publish loop (`pipeline.rs:8925-8995`, `store.publish(image, pacer.publish_time(pts))`
  at `:8995`) already produces one monotonic nanosecond timeline (the 33-bit-wrap
  monotonicity test, `pipeline.rs:10842`).
- **The seek primitive is BUILT with zero production callers.** `Demuxer::seek`
  (`crates/multiview-ffmpeg/src/demux.rs:576`) is a keyframe-bracketed `Input::seek`.
  Strict random-access detection exists: `ReadPacket::is_idr`
  (`crates/multiview-ffmpeg/src/demux.rs:118`) inspects the coded NAL/OBU bytes via
  `idr::is_idr` (`crates/multiview-ffmpeg/src/idr.rs:77`, with `CodecKind` `:38` and
  `NalFraming` `:57`) — distinct from `ReadPacket::is_key` (`demux.rs:102`), which
  trusts the container flag and admits open-GOP CRA/recovery-point frames.
- **The exit-verb shape is BUILT for salvos.** `Command::ArmSalvo`/`TakeSalvo`/
  `CancelSalvo` (`command.rs:259-289`) with REST handlers
  (`routes/salvos.rs:211/244/277`) and a correlation key
  `CorrKey::Salvo { salvo, phase: SalvoPhase{Armed|Taken|Cancelled} }`
  (`realtime.rs:78-141`) is the proven arm/take/cancel triad: arm stages an intent,
  take commits it, cancel withdraws it — each a single unambiguous correlated outcome.
- **The realtime contract already named this feature.**
  [ADR-RT008](ADR-RT008.md) reserves `CorrKey::MediaPlayer { player, state }`, the
  `media.player_state` lossless lifecycle event (`cued/playing/paused/stopped/eof`),
  and the per-player snapshot shape `{asset, state, position_frames}`.

What does **not** exist yet, and is therefore net-new (called out so no comment or
doc claims behaviour the code lacks — rule 27): there is **no `PtsNormalizer` type
and no `ProducedFrame.discontinuity` field** in the tree; discontinuity handling
today is the `PtsWallClock` backwards-PTS re-anchor (`pipeline.rs:9208`) plus the
loop's monotonic guard. There is **no `avcodec_flush_buffers` binding** anywhere in
`multiview-ffmpeg` (verified: zero hits across `crates/multiview-ffmpeg/src/`). Both
are surfaces the player MVP must add; this ADR pins what they must do.

## Decision

### 1. A new transport state `Vamping { exit_armed: bool }`, distinct from `Playing(loop)`

The media player's transport state machine ([ADR-0057](ADR-0057.md) Decision 4)
gains a state **`Vamping { exit_armed: bool }`**, separate from playing under an
`eof_policy: loop`. The distinction is load-bearing:

- **`Playing` under `loop` policy** loops forever with no terminal condition — the
  as-designed [ADR-0057](ADR-0057.md) behaviour, unchanged.
- **`Vamping { exit_armed: false }`** loops the **vamp segment** (in→out, see §2)
  seamlessly, *waiting to be told to leave*. It is reported distinctly on the wire so
  an operator panel shows "vamping" not "playing".
- **`Vamping { exit_armed: true }`** is the same loop with an exit committed: it will
  transition out at the **next vamp boundary** (the next time playout reaches the
  vamp out-point), then settle into the post-exit state (§4). Arming is reversible
  until the boundary fires (cancel, §3).

`Vamping` is entered by a `vamp` verb (or `play` with the asset's vamp segment
selected); it is left only by the armed exit firing at a boundary, by `stop`
(immediate re-cue to in-point, [ADR-0057](ADR-0057.md)), or by an engine fault that
drives the tile state machine. The state lives on the player thread exactly like the
rest of the transport machine — engine-owned, mirrored read-only, never persisted in
config ([ADR-0057](ADR-0057.md) Decision 1).

### 2. Integer-frame entry / vamp / exit fields — declared, never raw URL/float

The asset/player carries the cue geometry as **integer frame counts** at the asset's
exact rational cadence ([ADR-T015](ADR-T015.md), inv #3) — never raw `?in=…` URL
query fields, never float seconds:

- `in_point_frames` / `out_point_frames` — the clip's overall play window (already
  foreseen by [ADR-0057](ADR-0057.md) Decision 2's "optional in/out loop points
  (frames)").
- `vamp_in_frames` / `vamp_out_frames` — the **vamp segment**: the sub-range that
  loops while vamping. Defaulting `vamp_in = in_point` and `vamp_out = out_point`
  makes the whole clip the vamp loop; narrowing them lets an opener play a one-shot
  head (`in_point … vamp_in`), vamp a middle bar (`vamp_in … vamp_out`), and play a
  one-shot tail (`vamp_out … out_point`) on exit — the openers/loops/VT-rolls use
  case [ADR-0057](ADR-0057.md) Decision 4 cites.

The four fields obey `in_point ≤ vamp_in < vamp_out ≤ out_point` (strict on the vamp
pair so a vamp segment is at least one frame; a degenerate `vamp_in == vamp_out` is
rejected at validation, not silently degraded — the [ADR-T015](ADR-T015.md) §3 dip
precedent). The REST surface accepts **either** `*_frames` **or** `*_ms` per field
(exactly one), and a millisecond input converts **once at the API boundary** with the
pinned exact-rational rule (`frames(ms) = max(1, floor((2·ms·num + 1000·den) /
(2000·den)))`, round-half-up, min 1 — [ADR-T015](ADR-T015.md) §2). Responses echo
`frames` (canonical) plus a display-only `ms`, so the operator sees the quantisation.
No float fps/seconds is stored or re-converted anywhere.

### 3. Exit verbs `arm` / `take` / `cancel`, mirroring the BUILT salvo triad

The vamp exit is **not** a `202` long-running operation and **not** a privileged
engine channel — it is a command-bus action mirroring the named-salvo arm/take/cancel
triad verbatim:

- **`POST …/media/players/{id}/exit/arm`** → `Command::ArmMediaExit { player, op }`:
  set `Vamping.exit_armed = true`. The exit will fire at the next vamp boundary.
  Idempotent: arming an already-armed vamp is a no-op echo.
- **`POST …/media/players/{id}/exit/take`** → `Command::TakeMediaExit { player, op }`:
  arm **and** request the soonest boundary in one verb (the "go now, at the next
  clean seam" button). Functionally `arm` on a not-yet-armed vamp; on an already-armed
  vamp it is a no-op echo. It never forces a mid-lap cut — "take" still means *at the
  next vamp boundary*, never *this instant* (a mid-lap exit would defeat the clean-seam
  guarantee; an operator who wants an immediate hard change uses the switcher's
  cut/`stop`, not the vamp exit).
- **`POST …/media/players/{id}/exit/cancel`** → `Command::CancelMediaExit { player, op }`:
  set `Vamping.exit_armed = false`. The vamp keeps looping. Valid only while
  `exit_armed` and the boundary has not yet fired; a cancel that races a
  just-fired boundary is reported as a no-op (the boundary won), never a wrong
  state.

These three `Command` variants are added alongside `ArmSalvo`/`TakeSalvo`/
`CancelSalvo` (`command.rs:259-289`), with the same `op` operation-id field and the
same `kind()` strings (`arm_media_exit`/`take_media_exit`/`cancel_media_exit`). They
flow REST → bounded command bus → the CLI frame-boundary `CommandDrain`, which writes
a wait-free latest-wins slot the player thread reads between frames
([ADR-0057](ADR-0057.md) Decision 4) — the engine never awaits the player, the player
never paces the engine (inv #1/#10).

### 4. Firing at the next vamp boundary via `effective_at`; output-anchored stamping

The armed exit fires at the **next vamp boundary** — the next output tick at which the
player's playout reaches `vamp_out_frames`. Concretely, on the player thread, when
the frame about to be published is the vamp out-point **and** `exit_armed` is true,
the player transitions out of `Vamping` *instead of* seeking back to `vamp_in`:

- it plays the one-shot tail (`vamp_out … out_point`) if one is declared, then
- settles into the asset's terminal `eof_policy` (`hold_last_frame` default, or
  `black`/`auto_off` — [ADR-0057](ADR-0057.md) Decision 4), emitting
  `media.player_state = eof` ([ADR-RT008](ADR-RT008.md)).

The boundary tick is the player's own `effective_at` ([ADR-0072](ADR-0072.md)
discipline): the exit is *requested* by the control plane but *applied* at the tick
the player's playout machine selects, through the existing frame-boundary path — so it
fires neither early (no mid-lap cut) nor late (the very next seam, not the one after).
It is a function of the **tick counter and the playout position**, never a wall-clock
timer (inv #1/#3).

Stamping stays **output-anchored** across every lap and across the exit, exactly as
[ADR-0057](ADR-0057.md) Decision 4 pins for player frames: a published frame carries
`publish_at(k) = anchor_pts + k × frame_period`, where `anchor_pts` is the output
media time of the start tick and `k` increments monotonically across loop laps and
through the exit tail. This is the player's own discipline, **not** the boot path's
source-relative `publish_time = pts − first_pts` (`pipeline.rs:9244`), which is correct
only because boot ingest starts at run start — it would smear a mid-asset cue and would
reset to near-zero on every loop wrap. `anchor_pts` accumulates one vamp-segment length
per lap as exact rationals (inv #3); the monotonic published timeline never goes
backwards across a wrap.

### 5. The two non-negotiable engine rules at every loop/vamp boundary and every seek

These hold for **`Playing(loop)`** and **`Vamping`** identically — the vamp loop is a
loop, with the same seam hazards [ADR-0057](ADR-0057.md) Decision 4's `loop` policy
faces, and they are the reason a naive `Demuxer::seek` is insufficient:

1. **The loop/vamp boundary MUST mark a discontinuity explicitly.** When playout wraps
   `vamp_out → vamp_in` (or seeks for any reason), the player must **force the
   `PtsWallClock` re-anchor** (`pipeline.rs:9208`) — it must not rely on the automatic
   backwards-PTS detection, because on a short vamp segment the post-wrap PTS does not
   move ~10 s backwards (it moves a few hundred ms, or *forward* if `vamp_in > 0`), so
   the monotonic guard would otherwise clamp the wrap into nanoseconds and freeze/
   time-warp the tile. The player therefore signals the discontinuity at the wrap
   instant, and stamping continues output-anchored (§4) so the published timeline stays
   strictly monotonic. *(Net-new surface: this is an explicit "mark discontinuity"
   signal on the player→pacer path; no `PtsNormalizer.mark_discontinuity` /
   `ProducedFrame.discontinuity` exists in the tree today — the player MVP adds it.)*
2. **`avcodec_flush_buffers()` MUST run after every seek.** After each
   `Demuxer::seek` (the loop-back seek, the cue seek, and any user `seek`), the decoder
   must be flushed or stale reordered B-frames from before the wrap leak past the seam
   as a one-off glitch. *(Net-new surface: there is no flush binding in
   `multiview-ffmpeg` today — verified zero hits; the player MVP adds exactly one safe
   FFI wrapper for `avcodec_flush_buffers`, behind the `ffmpeg` feature, in the FFI-owning
   crate per CLAUDE.md safety rule 4.)*

In addition, **entry, exit, and every loop seam snap to a STRICT IDR**: the seek targets
an `is_idr` frame (`demux.rs:118` / `idr::is_idr`), never merely `is_key` (`demux.rs:102`)
— open-GOP CRA/recovery-point frames decoded cold produce green/garbage, which on a
vamp seam would be a visible flash every lap. A failed seek falls back to a clean re-open
via the existing supervised reconnect bracket in the ingest thread
([ADR-0057](ADR-0057.md) Decision 4) — the proven recovery shape.

> **Implementation refinement (2026-06-19, as built — supersedes rule 1's
> mechanism, not its intent).** This ADR was authored against `main` @88e636d2,
> where the discontinuity-marking surface was net-new. The implementation
> (lane `feat/media-player-vamp-exit`) takes a **stronger** approach than rule 1
> as originally worded: the media player **synthesizes its own monotone,
> output-anchored timeline** in the pure transport core
> (`MediaPlayer::on_decoded` → `PlayerAction::Publish { at }`, where
> `at = anchor + emitted × frame_period`, exact integer rationals, `emitted`
> advancing per published frame and **never resetting on a wrap**). The player
> ingest path therefore **stamps from this synthesized timeline directly and
> does not route the decoded frame's raw PTS through `PtsNormalizer` at all** —
> not on a loop/vamp wrap, not on the first non-looping play, not on a cue/seek.
> Because no source PTS ever reaches the normalizer's monotonic guard on the
> player path, the frozen/time-warped-tile failure rule 1 guards against
> **cannot occur**, so an explicit `PtsNormalizer::mark_discontinuity()` at the
> wrap is **unnecessary for the player path** (and `mark_discontinuity` does in
> fact now exist in the tree at `crates/multiview-input/src/normalize.rs:167`,
> used by the boot/live ingest path — not the player). The intent of rule 1 —
> *the published timeline stays strictly monotonic across every wrap, never
> clamped into nanoseconds* — is upheld **by construction** and proven by the
> `core_never_wedges_and_stamps_never_regress` property test plus the
> `ffmpeg`-gated looping-clip integration test (the published stamp at the loop
> seam equals the previous stamp + one `frame_period`, across multiple laps).
> **Rule 2 (`avcodec_flush_buffers()` after every seek) is RETAINED unchanged
> and remains non-negotiable** — it is implemented as `StreamVideoDecoder::flush`
> (`crates/multiview-ffmpeg/src/decode_stream.rs`), called after every seek;
> stale reordered B-frames are a decoder-state hazard independent of the stamping
> timeline. The STRICT-IDR seam requirement is likewise retained.

### 6. Correlation and lifecycle events: `CorrKey::MediaPlayer { player, state }`

The arm/take/cancel verbs correlate via the reserved
**`CorrKey::MediaPlayer { player, state }`** ([ADR-RT008](ADR-RT008.md)), mirroring
`CorrKey::Salvo { salvo, phase }` (`realtime.rs:119-135`): the key is derived
identically from the command (at `202` time, `CorrKey::for_command`) and from the
matching outcome event (at projection time, `CorrKey::for_event`), so the realtime
layer recovers the op id and stamps it onto the outcome envelope's `corr` without an op
id ever entering the `Event` enum or the engine hot loop. The exit transitions surface
as `media.player_state` lossless lifecycle events on the `switcher` topic (the existing
drop-oldest broadcast lane, [ADR-RT008](ADR-RT008.md)): the discrete state changes
`vamping` (armed/unarmed reflected in the snapshot, not as a per-frame event) and the
terminal `eof`/`stopped`. The connect-time `switcher` snapshot's per-player shape
`{asset, state, position_frames}` ([ADR-RT008](ADR-RT008.md)) carries `state =
Vamping{exit_armed}` so a late-joining panel sees the armed exit. Progress/position is
**not** a per-tick event — it rides the conflated snapshot, never a lossless per-frame
lane (inv #10).

### 7. Apply classification: every vamp/exit verb is Class-1 Hot

`vamp`, `exit/arm`, `exit/take`, and `exit/cancel` are **all Class-1 Hot**
([ADR-R004](ADR-R004.md), inv #11): they mutate the player's transport-state slot and,
at the boundary, drive store writes + the playout state machine — **no encoder reset, no
new output session, no SPS/PPS/IDR discontinuity on the program output**. The output
clock keeps emitting one stamped frame per tick throughout; the vamp's loop seam and its
exit are absorbed inside the player's own tile timeline (the re-anchor of §5), never the
program encoder. The plan/take dry-run ([ADR-W021](ADR-W021.md)) previews the class as
Class-1 before arming. This is consistent with [ADR-0057](ADR-0057.md) Decision 4
(`load`/`cue`/`play`/`pause`/`stop`/`seek` are all Class-1) and
[ADR-0072](ADR-0072.md) Decision 5 (the MVP cue-action set is entirely Class-1).

### 8. Scope boundary: video vamp/loop ships first; audio-on-loop is a distinct subsystem slice

The vamp/loop/exit machinery this ADR specifies is the **video tile** behaviour: the
transport state machine, the in-place loop seek + flush, the output-anchored stamping, and
the boundary exit all operate on the player's `TileStore` (video). [ADR-0057](ADR-0057.md)
Decision 6 separately establishes that a player whose asset has audio gets a **per-source
`AudioStore` + decode thread**, routed onto the program bus under the
[ADR-0059](ADR-0059.md) switcher-audio seam — i.e. audio is a **distinct subsystem**
(`multiview-audio`: per-input decode/mix/route + R128 + the program bus), not a sub-field
of the video tile.

This delta therefore ships in two slices, and the boundary is **deliberate, not an
incidental cut**:

1. **The video vamp/loop player (this ADR, now).** A pre-declared player channel that
   loops a clip over its `[in_point, out_point)` window, vamps its `[vamp_in, vamp_out)`
   sub-window with a clean armed exit, is frame-accurate and API-drivable end-to-end
   (config boot-spawn → transport core → in-place loop → command routing). This is a
   **complete, coherent, usable feature** — the operator's "VT vamp/loop with entry + exit"
   ask, fully realized. A boot-spawned player is **silent by design** (no `AudioStore`
   wired yet), exactly as a video-only tile is — *not* silent because the feature is
   half-built.
2. **Audio-on-loop (a follow-on `multiview-audio` integration slice).** The player's
   embedded audio joining the program bus and **looping on the same wrap instant** as the
   video, with the equal-power `GainRamp` crossfade at the loop seam that kills the click
   (`ProgramBus::repoint_crossfade`, the [ADR-0059](ADR-0059.md)/[ADR-T015](ADR-T015.md) §5
   sample-exact ramp). This rides the existing per-source audio path
   ([ADR-0057](ADR-0057.md) Decision 6, `audio_ingest_loop`) extended with the same
   seek-back-on-wrap the video executor performs, and is tracked as a committed follow-on,
   **not** deferred indefinitely.

Splitting here is rule-6-clean because these are **two subsystems** (video tile vs. the
audio bus), not two halves of one feature: the video player is whole and shippable without
the audio integration, exactly as the multiviewer composites a silent tile today and the
program bus is a separate concern.

> **Implementation refinement (2026-06-20, as built — task #135, [ADR-T019](ADR-T019.md)).**
> The audio-on-loop slice ships as [ADR-T019](ADR-T019.md). It upholds this §8
> *contract* — the player's embedded audio loops on the **same wrap instant** as the
> video, with a **`GainRamp` crossfade** at the loop seam that kills the click — while
> refining the *mechanism* exactly as §5 did for the video player. Rather than a decoder
> "seek-back-on-wrap" (`multiview_ffmpeg::AudioFileDecoder` has no seek, and an audio
> container has no IDR, so a container audio seek lands on a packet boundary =
> sample-imprecise = a click every lap), the audio loops by **sample-exact
> buffer-and-replay**: the `[vamp_in, vamp_out)` segment is decoded **once** into a
> bounded buffer and replayed as an **overlap-add loop** whose period is exactly the
> decoded 48 kHz sample count of that *time range* (= the audio duration of `vamp_len`
> asset frames, drift-free at any cadence, inv #3 — *not* a `SampleClock::total_at` tick
> delta, which would conflate source-frame and output-tick indices). The seam crossfade
> is applied **inside the player's own buffer** with a **correlation-adaptive law**
> (equal-power for a decorrelated seam, linear for a correlated/tonal one — a literal
> equal-power-everywhere fade swells +3 dB on a sustained tone), positioned by the bus's
> absolute read cursor so a realign under load still lands inside a faded seam — *not* a
> `ProgramBus::repoint_crossfade` (which crossfades a store **swap** for the switcher's
> A→B breakaway, not a loop seam). See [ADR-T019](ADR-T019.md) for the full decision, the
> alternatives, the cross-vendor-review fixes, and the invariant posture.

## Rationale

- **A separate `Vamping` state is the minimal honest model.** Overloading
  `Playing(loop)` with a hidden "armed" bit would make the wire ambiguous (an operator
  cannot tell a forever-loop from a vamp awaiting its cue) and would put exit-boundary
  logic on the forever-loop path it must not affect. A distinct state with an explicit
  `exit_armed` bit is queryable, snapshot-able, and keeps the forever-loop behaviour of
  [ADR-0057](ADR-0057.md) untouched.
- **Reusing the salvo arm/take/cancel triad is load-bearing, not cosmetic.** Arming an
  exit *ahead* of the boundary is exactly the salvo semantics — stage an intent, commit
  at a boundary, allow withdrawal — and it inherits the salvo's correlation discipline
  (`CorrKey`, single unambiguous outcome) and its bounded-bus apply path for free, with
  no new engine channel and no new invariant exposure.
- **Integer frames everywhere** is forced by inv #3 and [ADR-T015](ADR-T015.md): a vamp
  seam that must land on an exact output frame has to be denominated in the tick
  counter's units. Float seconds at 1001-family cadences accumulate the classic
  ~3.6 s/hour error; a vamp that runs for minutes would drift its seam off the frame
  grid. ms→frames converts once at the boundary with the pinned half-up/min-1 rule.
- **The two engine rules are the whole correctness story of a loop.** Without the
  explicit discontinuity mark, a short-segment wrap is invisible to the auto backwards-
  PTS heuristic and the monotonic guard freezes the tile; without the post-seek flush,
  every lap leaks a B-frame glitch. These are not optional hardening — they are the
  difference between a clean vamp and a tile that visibly stutters once per lap.
- **Invariant posture:** the player is *sampled* per tick like every source (inv #1);
  the exit is *requested* by the control plane and *applied* by the player's own
  `f(tick, position)` at the boundary (inv #1/#3); exit commands cross on the wait-free
  latest-wins slot (inv #10); a stalled/underrunning vamp degrades through the tile
  state machine (inv #2). The player is a **pre-declared channel**, so arming/firing a
  vamp exit never spawns or tears an engine-path thread — it sidesteps the
  [ADR-W018](ADR-W018.md) live-spawn restriction entirely.

## Alternatives considered

| Alternative | Rejected because |
| ----------- | ---------------- |
| **Overload `eof_policy: loop` with a separate "exit at next loop" command, no new state.** | The wire could not distinguish a forever-loop from a vamp-awaiting-cue, and the boundary-exit logic would sit on the forever-loop path it must not perturb. A distinct `Vamping { exit_armed }` state is queryable and snapshot-able; the cost is one enum variant. |
| **Exit as a `202` long-running operation** (like media import, [ADR-0057](ADR-0057.md) Decision 3). | A `202` models off-engine work that completes asynchronously; an exit is an *instantaneous-at-a-boundary* state mutation that must correlate to a single tick, exactly the salvo arm/take/cancel shape — not the import shape. The salvo triad already exists and carries `CorrKey`. |
| **Take = exit immediately at the current frame** (hard cut out of the vamp mid-lap). | Defeats the clean-seam guarantee that is the entire point of a vamp; a mid-lap exit shows a visible jump. An operator who wants an instant hard change uses the switcher cut or `stop`, not the vamp exit. "Take" means *at the next boundary*. |
| **Float seconds / ms as the stored vamp/exit geometry** (raw `?in=`/`?out=` URL fields). | Inv #3 / [ADR-T015](ADR-T015.md): `29.97` may never reach the cadence math; a minutes-long vamp would drift its seam off the frame grid. Frames are the invariant under cadence change; ms converts once at the API boundary. |
| **Rely on the automatic backwards-PTS re-anchor at the loop wrap** (no explicit discontinuity mark). | A short vamp segment's post-wrap PTS does not jump ~10 s backwards (it moves a little, or *forward* when `vamp_in > 0`), so the auto heuristic never trips and the monotonic guard clamps the wrap into nanoseconds — a frozen/time-warped tile. The boundary must mark the discontinuity explicitly. |
| **Skip the post-seek decoder flush** (trust `Demuxer::seek` alone). | Stale reordered B-frames from before the wrap leak past the seam as a per-lap glitch. `avcodec_flush_buffers()` after every seek is mandatory; it is the one new FFI surface the player needs. |
| **Snap seams to `is_key` (container keyframe flag) rather than `is_idr`.** | Open-GOP CRA/recovery-point frames decoded cold render green/garbage; on a vamp seam that is a flash every lap. The strict `is_idr` test (`demux.rs:118`) inspects the coded NAL/OBU bytes and is the only safe random-access point. |
| **A new `Vamp`/`Clip` `SourceKind` for the vamp player.** | [ADR-0057](ADR-0057.md) settled this: a media player is a pre-declared, bus-selectable *channel*, not a new source kind, so load/cue swap content without spawning/tearing a thread on the engine path — sidestepping the [ADR-W018](ADR-W018.md) live-spawn restriction. The vamp is a transport *mode* of that channel. |

## Consequences

- **Positive**
  - Operators get the real vamp workflow — loop-to-fill an unknown duration, then a
    deterministic frame-clean exit on cue — on top of the existing player channel, with
    no new engine channel, no new `SourceKind`, and no new data-plane invariant exposure.
  - The exit inherits the salvo triad's correlation (`CorrKey::MediaPlayer`), bounded-bus
    apply, and arm/cancel reversibility for free; the realtime contract
    ([ADR-RT008](ADR-RT008.md)) already reserved every wire name this needs.
  - Entry/vamp/exit geometry is cadence-portable (frames) and drift-free across a
    minutes-long vamp (inv #3).
- **Negative / cost**
  - The transport state machine grows a `Vamping { exit_armed }` state and the boundary-
    exit logic on the production ingest thread (`pipeline.rs`, the serial LANE-CORE
    territory — sequenced, not parallel, per [ADR-0057](ADR-0057.md) Consequences).
  - Two **net-new surfaces** must land with the player MVP and are non-optional: (a) an
    explicit "mark discontinuity / force re-anchor" signal on the player→`PtsWallClock`
    path (no `PtsNormalizer`/`ProducedFrame.discontinuity` exists today), and (b) a safe
    `avcodec_flush_buffers` FFI wrapper in `multiview-ffmpeg` behind `ffmpeg` (none exists
    today). Both need their own RED tests (a synthetic frame producer + an all-intra
    fixture, no GPU): a short-segment loop that proves the published timeline stays
    monotonic across the wrap, and a post-seek decode that proves no pre-wrap B-frame leaks.
  - New control-plane surface: three exit verbs + their `Command` variants + `CorrKey`
    projection + OpenAPI/AsyncAPI + an SPA "arm exit / cancel exit" control
    ([ADR-W021](ADR-W021.md), [ADR-RT008](ADR-RT008.md)).
  - `Demuxer::seek` gains its first production callers and therefore the frame-accuracy /
    loop-seam / post-seek re-anchor coverage its zero-caller status never forced
    (the [ADR-0057](ADR-0057.md) Consequences obligation, now also covering the vamp seam).
- **Neutral**
  - `vamp`/`exit-arm`/`exit-take`/`exit-cancel` are all Class-1 Hot (inv #11): the program
    encoder never resets; the loop seam and exit are absorbed in the player's own tile
    timeline.
  - This ADR is a pure delta on [ADR-0057](ADR-0057.md) — it adds one transport state,
    four declared frame fields, three exit verbs, and pins two engine rules already
    implied by any loop; it changes none of [ADR-0057](ADR-0057.md)'s other decisions.

## Adversarial self-review — the single biggest residual risk

The dominant residual risk is the **exit-arm / boundary-fire race on a very short vamp
segment** (§3–§5). `exit/arm` writes `exit_armed = true` into a wait-free latest-wins
slot that the player thread reads *between* frames; the boundary check fires when the
frame about to be published is `vamp_out`. If the vamp segment is only a handful of
frames, the operator's arm can land in the same drain window as the boundary the player
is already crossing — so "the next boundary" is ambiguous (this lap's seam, which the
player may have already committed to seeking, or the next one). Resolved wrongly, an
arm could either be **missed for a whole extra lap** (the operator perceives the exit as
laggy/ignored), or **double-applied** (arm fires this seam *and* the latch is still set
next seam). The salvo precedent does not fully cover this: a salvo take fires once at a
single chosen tick, whereas the vamp boundary recurs every lap, so the latch must be
**consumed exactly once** at the first boundary at-or-after the arm tick and cleared
atomically in the same player-thread step that commits the exit — with the boundary
comparison done against the *publish* tick the player will actually emit, not the decode
position (which runs ahead). This is the one place a property/soak test is mandatory
before claiming the feature works: a synthetic 2–3-frame vamp segment, an arm injected
at every possible offset within the lap (including the exact boundary tick), asserting
exactly-once exit at the first seam at-or-after the arm and no extra lap — over
`ManualTimeSource` with zero wall-clock dependence. A secondary, lower residual: the
net-new "mark discontinuity" signal is asserted here as the fix for the short-segment
monotonic-guard freeze, but its exact placement on the player→pacer path is unverified
against a built player (the player does not exist yet), so the RED test for wrap
monotonicity is what must prove it — not this ADR's reasoning.
