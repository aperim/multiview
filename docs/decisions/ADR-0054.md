## Context

Multiview today composites **one scene per program per tick** and switches it atomically:
`CompositorDrive::compose(tick)` samples each layout cell's lock-free `TileStore`, derives
a transient z-ordered placement list from pooled scratch, and dispatches one composite
(BUILT — `crates/multiview-engine/src/drive.rs:449-551`, `ComposeScratch` at
`drive.rs:670`). The live "cut" primitives are `rebind_cell` (O(1) source re-point,
`drive.rs:376-402`) and `set_layout` (validated `Arc<Layout>` swap, `drive.rs:318-324`),
applied at the **frame-boundary control seam**: the per-tick hook the runtime invokes
between `clock.tick()` and `compose(tick)` (BUILT —
`crates/multiview-engine/src/runtime.rs:430-439`). A production switcher needs what none
of this provides: a **preview (PVW) bus** rendered alongside program, **transitions** that
evolve over ticks (mix/dip/wipe/DVE/stinger, [ADR-0055](ADR-0055.md)), **keyers and a
fade-to-black master stage** ([ADR-0056](ADR-0056.md)), **aux/clean feeds**, and tally
derived from what is actually on air ([ADR-MV006](ADR-MV006.md)).

Four verified constraints shape the architecture:

- **The per-tick hook receives no `Tick`.** Its signature is
  `FnMut(&mut CompositorDrive<Nv12Image>)` (`runtime.rs:401`, invoked at
  `runtime.rs:439`), so nothing at the seam can compute deterministic per-tick progress
  today — the hook cannot know which tick it is mutating for.
- **`ProgramSet` programs have independent clocks.** Each program owns its own
  `OutputClock` over one shared `TimeSource` (BUILT —
  `crates/multiview-engine/src/programset.rs:5,207`); there is no master clock, so two
  programs cannot guarantee the same scene lands on the same tick.
- **The output←program crosspoint is an empty seam.** Engine `RouteIntent` has exactly
  `Video`/`Audio`/`Subtitle` variants (`crates/multiview-engine/src/route.rs:54-81` — no
  `Output`), there is no `Command::RouteOutput`, and `PacketRouter::move_sink` /
  `EncodeOnceDriver::move_sink` (BUILT in `crates/multiview-output/src/fanout.rs:212,331`)
  have **zero callers outside `multiview-output`** — exactly as [ADR-R010](ADR-R010.md)
  admits (RT-12 in the [decoupled-routing backlog](../development/decoupled-routing-backlog.md)).
- **Overlays bake off-thread, after the taps.** The hot loop publishes the pre-overlay
  canvas `Arc` per tick (`preview.store(...)` at
  `crates/multiview-cli/src/pipeline.rs:2122`, `ProgramSlot` =
  `Arc<ArcSwapOption<Nv12Image>>` at `crates/multiview-cli/src/preview.rs:28`) and the
  bake consumer burns overlays in afterwards — so anything keyed there is **not
  frame-coordinated** with on-clock switcher state.

## Decision

1. **An M/E (mix/effects) stage lives INSIDE one [ADR-0030](ADR-0030.md) Program — never
   as a sibling program.** PGM and PVW are two *scene states* of one M/E, both evaluated
   by the same `CompositorDrive` against the **same `Tick`** under the program's one
   `OutputClock`. The PVW bus is a **second compose against the same `Tick`** (the
   `TileStore`s are `Arc`-shared, lock-free multi-reader by design — no extra decode). A
   transition mathematically requires both scenes in one composite — mix/wipe arithmetic
   is impossible across two independently encoded streams — and two `ProgramSet` programs
   tick on independent clocks (`programset.rs:5,207`), which breaks the frame-aligned
   take. One clock, one composite, two scene states: inv #1 holds by construction because
   the transition is *sampled* per tick, never pacing.

2. **The switcher state machine is a pure value machine in `multiview-engine`, drained at
   the per-tick control seam.** Same house style as the salvo/tally/alarm machines
   (`crates/multiview-engine/src/salvo.rs:158/169/189` — arm/cancel/take mutate own state
   and RETURN a batch for the engine to apply at the frame boundary): the machine consumes
   drained switcher commands + the current `Tick` and returns the resolved bus/transition/
   keyer state; it never does I/O, never awaits, never allocates per tick. It lives where
   `CommandDrain` already lives — the hook at `runtime.rs:430-439`. **The hook signature
   is widened to receive the `Tick`** (`FnMut(&mut CompositorDrive<Nv12Image>, Tick)`),
   with a back-compat adapter wrapping the existing tick-less closures (the
   `no_control_hook()` shape at `runtime.rs:167`) so every current caller compiles
   unchanged. This is the one signature change on the inv-#1 seam and it carries a
   chaos/soak proof per [ADR-R011](ADR-R011.md).

3. **The per-tick placement derivation is promoted to a render-plan resolver.**
   `compose()` already rebuilds a transient placement list from `layout.cells` each tick
   out of pooled scratch (`drive.rs:476-494`, `Placement` at `drive.rs:643`,
   `ComposeScratch` at `drive.rs:670` — zero steady-state allocation). That derivation
   becomes an injectable, synchronous resolver:
   `fn(scene_states, switcher_state, tick) -> render plan` — the placements (plus, in
   later waves, per-tile mask/transform/key parameters per [ADR-0055](ADR-0055.md)/
   [ADR-0056](ADR-0056.md)) for every composite this tick: PGM (with in-flight transition
   applied), PVW, and any admitted clean/aux variants. It is allocation-pooled exactly
   like `ComposeScratch`, runs on the clock thread, and is a pure function of its inputs —
   so the whole switcher render is testable with a `ManualTimeSource` and can never stall
   the clock (inv #1). When no switcher block is configured the resolver degrades to
   byte-identical current behaviour (one scene, one composite).

4. **DSK/FTB composite ON-CLOCK in the render plan; the off-thread overlay bake remains
   for monitoring overlays.** Downstream keyers and fade-to-black must be
   frame-coordinated with on-clock switcher state (a DSK that joins a transition, an FTB
   ramp — [ADR-0056](ADR-0056.md)): they become render-plan stages evaluated in the same
   tick as the M/E composite. The existing off-thread egress bake
   (`apply_overlays_to_nv12` on the bake consumer, ADR-0023) is **not**
   frame-coordinated — it runs after the hot loop publishes the canvas
   (`pipeline.rs:2122`) — and stays exactly where it is for monitoring-style overlays:
   tile labels, meters, state badges, captions, the enforcement watermark. The split is
   explicit and budgeted: on-clock stages are admission-accounted (Decision 8); the
   monitoring bake keeps its drop-oldest isolation (inv #10).

5. **Build vs route: compose inside one program; route only what needs its own encode.**
   Everything that must be frame-coordinated (PGM, PVW, transitions, keyers, FTB,
   multi-box compositions) is *built* inside the one program per Decisions 1–4. Aux buses
   and feeds that genuinely need an **independent encode** (a distinct rendition,
   inv #7 — [ADR-E003](ADR-E003.md)) are *routed*: they are [ADR-0030](ADR-0030.md)
   `ProgramSet` programs bound through the RT-12 output←program crosspoint — which must
   therefore finally be built: `RouteIntent::Output` (the enum at `route.rs:54` is
   `#[non_exhaustive]` for exactly this), `Command::RouteOutput`, and the engine bridge
   into `PacketRouter::move_sink`/`EncodeOnceDriver` (`fanout.rs:212,331` — BUILT,
   proof-tested, zero engine callers today). Class-2 output moves ride
   [ADR-R010](ADR-R010.md)'s five-phase make-before-break verbatim. No new mechanism is
   invented on either side of the split.

6. **Clean feeds are tap points along ONE program's render plan; the bus tap taxonomy is
   `{program (post-FTB), clean (pre-DSK), preview, me[n], aux[j]}`.** Adopt
   [ADR-0037](ADR-0037.md)'s CLEAN/DIRTY vocabulary: *clean* = pre-DSK, *dirty/program* =
   post-DSK/post-FTB. The anchor exists today: the hot loop already publishes the
   pre-overlay canvas `Arc` per tick into the wait-free `ProgramSlot`
   (`pipeline.rs:2122`) **before** the bake consumer burns overlays — the as-built
   program preview is effectively a clean feed. With DSK/FTB moving on-clock
   (Decision 4), the clean tap moves with them to the render-plan seam (the canvas after
   the M/E stage, before the DSK stage); each tap replicates the `ProgramSlot` wait-free
   pattern, and a tap that needs its own encode crosses to the *route* side of Decision 5.
   Mid-transition, a clean feed shows the transition (it is pre-DSK, not pre-M/E) — the
   de-facto industry meaning of "clean".

7. **No new crate.** Per the [ADR-I004](ADR-I004.md) precedent (broadcast-multiviewer
   features as modules inside the existing crates) the switcher adds **modules, not
   crates** — there is no FFI, no license boundary, and no dependency-direction change to
   justify one, so [conventions §3/§4](../architecture/conventions.md) needs **no edit**.
   Module map: switcher state machine + render-plan resolver + bus/tap types in
   `multiview-engine`; compositor primitives (scene pre-pass/fusion, masks, transforms,
   keyer math) in `multiview-compositor`; shared vocabulary types (bus ids, transition
   descriptors) in `multiview-core`; the `switcher` config block + validation in
   `multiview-config` ([ADR-M012](ADR-M012.md)); the REST/realtime domain in
   `multiview-control` ([ADR-W021](ADR-W021.md)/[ADR-RT008](ADR-RT008.md)); the operator
   panel in `web/`.

8. **The PVW composite and every additional render-plan pass are admission-accounted load
   items, shed before program.** Today nothing accounts a second composite (the engine
   `ControlLoop` is SCAFFOLD; the live shed is egress drop-on-overload only). The PVW bus
   gets a **new degradation-ladder rung between the preview rungs and the
   tile/shared-resource rungs** (the BUILT ladder orders preview rungs 1–5 — including
   `DropOffAirCueDecoders`, rung 4 — ahead of tile rungs 6–8 and program rungs 9–13,
   `crates/multiview-hal/src/degradation.rs:28-68`): reduced-rate/reduced-resolution PVW
   first, then PVW suspension — always before any program tile is touched (inv #9,
   [ADR-E007](ADR-E007.md)). Clean/aux render-plan variants are accounted the same way.
   An **in-flight transition is program-affecting** and is never shed mid-flight
   (complete-then-degrade; AUTO demotes to CUT at arm/admission — pinned in
   [ADR-0055](ADR-0055.md)). [ADR-P007](ADR-P007.md) pins the PVW-bus monitor tiers and
   the cue/pre-warm integration ([ADR-P004](ADR-P004.md)'s worker is the PVW arm
   mechanism).

9. **MP-5 (the `programs` schema root) is a prerequisite slice.** The
   `programs: Vec<ProgramSpec>` root, its backward-compat desugar, and per-program
   cross-validation are explicitly deferred to MP-5 today
   (`crates/multiview-config/src/program.rs:17-20,168-169`), and
   `validate_output` accepts any non-empty `OutputCrosspoint.program` string with no
   check that the program exists (`crates/multiview-config/src/routing.rs:434-441`) — a
   latent inconsistency the aux-bus side of Decision 5 would inherit. The switcher
   program therefore lands MP-5 first: schema root + desugar + validation that every
   output crosspoint's program exists. This is pure additive config work
   ([ADR-M012](ADR-M012.md)) and unblocks aux buses, RT-12, and Class-2 migration in one
   slice.

## Rationale

- **One clock or no take.** Frame-aligned cut/take is the product: the only structure
  that guarantees PGM and PVW agree on "this tick" is one `OutputClock` evaluating both
  scenes in one `compose` pass. The verified alternative (two `ProgramSet` programs)
  cannot — independent clocks (`programset.rs:207`) make "the same frame" undefined, and
  a mix across two encoded outputs is not computable at all.
- **The seam already exists; it is one widening away.** Every live mutation today flows
  through the `runtime.rs:430-439` hook (route drain, capped at `MAX_REPOINTS_PER_TICK`,
  `crates/multiview-cli/src/control.rs:598`); the salvo machine already proves the
  pure-value-machine + frame-boundary-apply shape. Adding `Tick` to the hook and
  promoting the existing placement derivation to a resolver reuses the two
  most-hardened pieces of the engine rather than building a parallel path.
- **Build-vs-route keeps inv #7 honest.** Compositing N buses in-program costs in-shader
  work on one canvas; routing a bus to its own output costs a full encode + mux. The
  split puts each bus on the cheap side that is *correct* for it, and the expensive side
  reuses the already-built-and-tested `PacketRouter` instead of a new mover.
- **The clean-feed anchor is as-built behaviour, not new machinery.** The pre-overlay
  publish at `pipeline.rs:2122` already separates clean from dirty; Decision 6 names and
  formalizes it rather than inventing a tap system.
- **Invariant posture:** #1 (transition sampled per tick; resolver synchronous on the
  clock thread; nothing new awaits), #10 (taps are wait-free `ArcSwap` slots / bounded
  drop-oldest; control rides the existing bounded bus; the engine never awaits a
  consumer), #7 (encode-once per rendition; aux = explicit extra rendition), #9
  (Decision 8), #11 (every switcher take classified, [ADR-M012](ADR-M012.md)).

## Alternatives considered

- **A — Each M/E (and the PVW bus) as a separate [ADR-0030](ADR-0030.md) Program.**
  Rejected. Verified: programs have independent `OutputClock`s over a shared time source
  (`programset.rs:5,207`) — no tick identity across programs, so a take cannot be
  frame-aligned and a mix/wipe across two encoded program outputs is mathematically
  impossible without decoding both back to pixels (one tick of added latency, an NV12
  generation loss, and a full extra encode per bus against inv #7). `ProgramSet` remains
  exactly right for what it models: genuinely independent outputs — hence the
  build-vs-route split, not a rejection of multi-program.
- **B — A new `multiview-switcher` crate.** Rejected per the [ADR-I004](ADR-I004.md)
  precedent: a crate boundary is warranted by FFI, a license/feature boundary, or a
  reusable dependency island — the switcher is none of these; it is engine state +
  compositor math + config + control surface, each of which already has a home crate. A
  new crate would force `conventions.md` §3/§4 edits, a dependency-direction carve-out,
  and a circular pull (the engine would depend on the switcher crate which depends on
  engine types).
- **C — Off-clock transition animation** (a tokio task interpolating opacity/geometry and
  pushing updates at the seam). Rejected outright: progress would be wall-clock-derived
  (float seconds against inv #3) and delivery-jittered (a delayed update smears the
  transition across the wrong ticks), and a stalled animator wedges a half-finished
  transition on air. Progress must be `f(tick.index)` — exact, replayable, and incapable
  of pacing anything (inv #1). Pinned in [ADR-0055](ADR-0055.md).
- **D — Drive transitions by mutating the layout per tick via `set_layout`.** Rejected,
  verified: `set_layout` re-validates the whole layout on every call
  (`drive.rs:318-324` — `layout.validate()`) and swapping a rebuilt `Arc<Layout>` per
  tick allocates on the hot loop, violating the scratch-pool discipline `compose`
  itself enforces (`drive.rs:453-466`). Even `rebind_cell`'s cheaper path copy-on-writes
  the cell vector when the `Arc` is shared (`drive.rs:392`). Per-tick variation belongs
  in the resolver (Decision 3), which derives placements *from* immutable scene states —
  the layout is data, the render plan is the per-tick function of it.
- **E — Keep DSK/FTB on the off-thread overlay bake.** Rejected: the bake runs after the
  hot-loop tap publish (`pipeline.rs:2122`) on the egress consumer — a keyer there
  cannot be frame-coordinated with a transition (the DSK would join a take one-or-more
  ticks late, visibly), and FTB there would not gate the clean/aux taps correctly.
  Monitoring overlays keep that path; on-air stages move on-clock (Decision 4).

## Consequences

- **Positive**
  - A real switcher (PGM/PVW, frame-aligned take, transitions, keyers, FTB, clean/aux
    feeds) lands as an extension of the five most-hardened existing seams — the output
    clock, the control-hook drain, the placement derivation, the `ProgramSlot` tap, and
    the `PacketRouter` — instead of a parallel engine.
  - `compose()`'s current behaviour is the resolver's degenerate case: no switcher block
    ⇒ byte-identical output, so the whole layer is regression-gated by construction.
  - RT-12 finally gets built (Decision 5), unblocking aux buses, [ADR-0037](ADR-0037.md)
    recording feeds, and Class-2 migration ([ADR-R010](ADR-R010.md)) with one shared
    mechanism; MP-5 (Decision 9) retires the unvalidated-program-string latency in
    `validate_output`.
- **Negative / cost**
  - The PVW bus is a second full composite per tick (and scene pre-render during
    overlapping-scene transitions adds more — [ADR-0055](ADR-0055.md)): real GPU/CPU
    load, mitigated by Decision 8's admission accounting + new ladder rung and by GPU
    pass fusion on the linear `Rgba16Float` canvas (the
    `composite_with_overlays` chaining precedent,
    `crates/multiview-compositor/src/gpu/compositor.rs:326`).
  - Widening the hook signature touches every `EngineRuntime` caller (mechanical, with
    the back-compat adapter) and the inv-#1 seam itself — it carries a mandatory
    chaos/soak gate ([ADR-R011](ADR-R011.md)) and a `ManualTimeSource` determinism suite.
  - The render-plan resolver becomes the single hottest new abstraction in the engine;
    its allocation-pooling discipline must be property-tested (zero steady-state
    allocation across arbitrary switcher-state sequences) before anything ships on it.
  - Two compose passes sharing one `TileStore` map per tick is new read-concurrency
    surface — the stores are lock-free multi-reader by design, but the chaos gate must
    cover a wedged PVW consumer with the program compose unaffected (inv #10).
