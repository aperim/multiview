# ADR-M009: Device stream binding — source-candidate, output-target, and display-head projections

- **Status:** Proposed
- **Area:** Management
- **Date:** 2026-06-10
- **Source brief:** [managed-devices.md](../research/managed-devices.md)

## Decision

A managed device ([ADR-M008](ADR-M008.md)) is **never itself a Source or an Output**: its driver projects it into the existing media graph through exactly three facets, and the engine only ever sees ordinary Sources and Outputs. **(a) Source candidates** — streams the device serves or can push, listed at `GET /api/v1/devices/{id}/source-candidates`; adopting one creates an ordinary managed Source using an existing `kind: rtsp|srt|ndi` variant of the internally-tagged `SourceKind` union (`crates/multiview-config/src/schema.rs`) annotated with a `device_ref`. The ingest path is untouched — pacer, jitter buffer, timestamp normalization, supervised reconnect (`crates/multiview-input/src/reconnect.rs`), and the framestore ladder apply as-is; the driver's only extra duties are keeping URL/credentials current and cross-linking device status in the UI. Where a vendor does not document its served URL paths (true of the ZowieBox-series RTSP/RTMP servers, whose ports are published but mount paths are not — ZowieTek API documentation v1.5, 2025, vendor-published), the candidate's URL field is **operator-suppliable** and flagged unverified, never guessed. **(b) Output targets** — the device's decode-side entry points, listed at `GET /api/v1/devices/{id}/output-targets`; binding one creates an ordinary managed Output with `device_ref`, and the driver writes the device's decode-table entry to point at the Multiview-served RTSP/SRT/RTMP/NDI output. Encode-once-mux-many governs cost: pointing the device at an **existing rendition is free fan-out (Class-1**, "shares encoder" badge); a **new rendition is C1-additive and admission-gated** (409 when the budget rejects; placement re-assessed per the instant-apply doctrine). **(c) Display heads** — display-capable devices (display nodes per [ADR-0045](ADR-0045.md); a vendor decoder's HDMI out) expose head assignment — Program, a specific output/rendition, or a `WallConfig`/`HeadConfig` head (`crates/multiview-config/src/wall.rs`) — as device-record state, never as a new graph node type.

Device mode is **converged, never toggled blind**: the published ZowieBox API exposes no encoder/decoder mode endpoint, while live firmware enforces a real workmode state (decode-table calls reject with a workmode error in encoder workmode — observed on current firmware) and close-before-open mutual exclusion on the decode table; the vendor-provided SDK is to be checked for a workmode-switch call during implementation. The device record carries `desired_mode`; the driver converges actual state toward it in close-before-open order, and `POST /devices/{id}/set-mode` returns `202` with the device-side impact declared **before** apply ("device restarts its pipeline; N bound sources from this device ride the tile ladder to NO_SIGNAL; no Multiview outputs are affected"). Program output is never interrupted by any device operation. Bindings survive device loss by construction: a bound Source rides the normal `SourceState` ladder (LIVE→STALE→RECONNECTING→NO_SIGNAL, `crates/multiview-framestore/src/state.rs`) with supervised reconnect; an Output keeps being served whether or not the device is currently pulling; on reconnect the driver re-converges desired mode, decode bindings, and sync offset. Capability-matrix classification (CP = control-plane only; C1/C2 per invariant #11; DEV = device-side reset, program output unaffected):

| Operation | Class |
|---|---|
| Adopt / remove / edit device record; probe / status / identify | CP |
| Bind device stream as Source (additive input; normal tile ladder) | C1 |
| Send program to device — existing rendition (free fan-out) | C1 |
| Send program to device — new rendition (admission-gated, 409 on budget) | C1-additive |
| Display-node content/layout reassignment (node-local make-before-break) | C1 |
| Local display head modeset (blanks that head briefly) | C2 |
| Device mode convergence (encoder⇄decoder; declared impact pre-apply) | DEV |
| Device LAN/mDNS/port change, reboot, firmware (some vendor sets return no HTTP response) | DEV |

## Rationale

Projection keeps invariant #10 by construction: drivers are control-plane pollers publishing into watch/broadcast channels, so nothing device-related can touch the engine's data plane (the same proof shape as the NMOS module). It buys the product's resilience doctrine for free — the tile ladder and supervised reconnect already are the device-loss story, with no device-specific failure paths to write. It preserves encode-once-mux-many: a device-bound output is just another consumer of an existing rendition unless the operator explicitly buys a new one through the admission gate. And it is the vendor-neutrality/YAGNI guard in structural form: three fixed projections plus a fixed status shape model "a network encoder/decoder", not any vendor's feature tree.

## Alternatives considered

**Device as a Source/Output subtype** (`SourceKind::Device` / `Output::Device` variants): rejected — duplicates transport handling the `rtsp`/`srt`/`ndi` variants already own, drags driver code and its feature gates toward engine-adjacent crates, and breaks the 1:N reality (one device simultaneously projects several source candidates, an output target, and a display head; a device's lifecycle is not a stream's lifecycle). **Auto-adopt + auto-bind of discovery results**: rejected — discovery output is untrusted inventory requiring explicit confirm-adopt (ADR-0041 doctrine); auto-binding would let any LAN actor inject sources into the program. **Modelling the vendor's full feature tree** (device-side routing matrices, layouts, per-device ABR, recording-first workflows): rejected — YAGNI plus the vendor-neutrality guard; the projections are everything the media graph needs, a wider surface invites doc-drift bugs against undocumented firmware behaviour, and trade-dress/ToU exposure grows with every vendor concept copied.

## Consequences

The Sources/Outputs pickers gain "From device" sections fed by the projection endpoints (which land in the OpenAPI spec first). `device_ref` has a real lifecycle: `DELETE /devices/{id}` returns 409 while Sources/Outputs still carry its `device_ref` (the problem body lists them), so unbinding is an explicit operator step; deleting a bound Source/Output makes the driver best-effort-clear the matching device table entry (the device may be offline — convergence retries). The driver owns reconciliation between our desired bindings and the device's actual tables: edits made through the vendor's own UI are detected as drift and overwritten on the next convergence pass — documented to operators, not silent. Per-facet honesty is standing: undocumented vendor URL paths mean some source candidates ship operator-suppliable rather than turnkey until hardware verification pins the path conventions; binding never upgrades a device — vendor decode legs stay Tier-C bounded-drift ([ADR-M010](ADR-M010.md)) and their close-before-open switching shows a visible gap on that device's display; and the "free fan-out" badge is only free while the rendition already exists — capacity truth lives in the admission gate, never the picker. Devices that are IPv4-only end-to-end are handled as legacy-interop islands under conventions §10; our own candidate/target URLs and examples stay IPv6-first.
