# ADR-0009: Hybrid concurrency: dedicated-thread data plane + Tokio control/IO plane

- **Status:** Proposed
- **Area:** Core Engine
- **Date:** 2026-06-02
- **Source brief:** [core-engine.md](../research/core-engine.md)

## Decision

Run decode/composite/encode on dedicated OS threads (one decode actor per source, deadline-driven compositor) connected by bounded drop-oldest channels carrying ref-counted pooled frame handles; confine Tokio to network ingest/egress and the control API. Bridge planes with flume (sync<->async), rtrb/ringbuf for hottest hops.

## Rationale

Codec/CUDA/VideoToolbox calls are long synchronous calls that would starve Tokio's cooperative scheduler. Per-source isolation with drop-oldest queues prevents head-of-line blocking; channels carrying handles (not pixels) preserve zero-copy. This mirrors GStreamer's queue-as-thread-boundary + leaky-queue design.

## Alternatives considered

Pure Tokio everywhere (rejected: blocks the runtime, jitter). Pure threads everywhere (rejected: reinvents async networking). rayon for the pipeline (rejected: data-parallel batch, not long-lived actors). Single shared input queue (rejected: one bad source backs up all).

## Consequences

libav context wrappers are Send+!Sync; frame handle Drop returns buffers to per-device pools (avoid async-destructor context). One CUDA context/GPU shared across sessions with per-session streams + events. Thread pinning/RT priority is optional tuning behind a platform trait; correctness must not depend on SCHED_FIFO (absent on macOS).
