# ADR-0074: SCTE-104 + SCTE-35 cue ingest and emit

- **Status:** Proposed
- **Area:** Input / Output / Control plane
- **Date:** 2026-06-13
- **Source brief:** [broadcast-cues.md](../research/broadcast-cues.md) (§3)
- **Extends:** [ADR-0072](ADR-0072.md) (the cue bus + action vocabulary),
  [ADR-0034](ADR-0034.md) (SCTE-35 already a crosspoint; `pts_adjustment` continuity)
- **Relates to:** the BUILT SCTE parsers (`crates/multiview-input/src/scte/`),
  [ADR-0075](ADR-0075.md) (manifest cues carry the same SCTE-35 bytes),
  [ADR-T003](ADR-T003.md) (re-stamp from the tick counter, inv #3)

## Context

The operator wants two SCTE surfaces: a **studio/baseband cue message** (SCTE-104, over TCP or SDI
VANC) and a **compressed-stream cue** (SCTE-35, in an MPEG-TS PID) — both **ingested** (react to the
cue) and **emitted** (originate the cue downstream).

The decoders already exist and are pure, bounded, panic-free, verified in-tree:

- SCTE-35 `splice_info_section` (`crates/multiview-input/src/scte/splice35.rs`): `table_id 0xFC`
  (`:23`), `splice_insert 0x05` / `time_signal 0x06` (`:29-32`), CRC-32/MPEG-2 validated, MSB-first
  bit reader, projecting `CueEvent`. SCTE-35 PIDs are discovered in the PMT
  (`crates/multiview-input/src/mpegts/selection.rs` `scte35_pids`). A re-emit primitive exists:
  `SpliceInfoSection::reserialize_with_pts_adjustment` (`splice35.rs:309`) rewrites the 33-bit
  `pts_adjustment` and CRC (via `mpegts/crc.rs`), refusing encrypted sections it cannot vouch for.
- SCTE-104 `multiple_operation_message` (`crates/multiview-input/src/scte/scte104.rs:161`
  `Scte104Message::parse`): `splice_request_data` op `0x0101` (`:25`), `time_signal_request_data`
  op `0x0104` (`:28`), `SpliceInsertType` (`:33`), projecting `CueEvent` via `cue_events()` (`:242`).

External grounding: SCTE-104 is the operational trigger automation sends to an encoder, carried
over **TCP/IP** and into **VANC via SMPTE ST 2010** (DID/SDID `0x41/0x07`) in **ST 291M** ancillary
data. A SCTE-104 `multiple_operation_message` can carry any of several operations; the MVP parser
**recognizes** `splice_request_data` (OpID `0x0101`) and `time_signal_request_data` (OpID `0x0104`)
and warns-and-ignores unsupported operations (so other op types never stall ingest)
([broadpeak](https://www.broadpeak.io/scte-104-the-key-to-streamlined-broadcasting/),
[SMPTE ST 2010](https://global.ihs.com/doc_detail.cfm?document_name=SMPTE%20ST%202010&item_s_key=00513497),
[EEG inserter data sheet](https://www.ai-media.tv/wp-content/uploads/Data-Sheets_SCTE_104_Inserter.pdf)).
SCTE-35 carries `splice_insert`/`time_signal` + `segmentation_descriptor` in a TS PID
([broadpeak SCTE-35](https://www.broadpeak.io/scte-35-a-standard-for-live-video-in-band-event-signaling/),
[AWS MediaTailor](https://docs.aws.amazon.com/mediatailor/latest/ug/ca-scte-35-messages.html)).
What is missing: the **transports** (a TCP listener for SCTE-104; observation taps for SCTE-35) and
a **full SCTE-35 builder** for emit (the in-tree code is a decoder + a `pts_adjustment`-only
re-serialiser — [decoupled-routing.md](../research/decoupled-routing.md) §DATA confirms "no SCTE
restamp target / re-serialiser" for a full build).

## Decision

1. **SCTE-104 ingest over TCP (MVP).** A control-plane TCP listener in `multiview-control`
   (dual-stack `[::]`, IPv6-first) accepts an automation system's stream of
   `multiple_operation_message`s; each → `Scte104Message::parse` → `cue_events()` → `Cue`
   ([ADR-0072](ADR-0072.md)). The listener is bounded best-effort drop-oldest; a slow/dead peer
   cannot stall anything (inv #10).

2. **SCTE-104 over SDI VANC (deferred, same decoder).** The VANC path maps SCTE-104 into ST 291M
   ancillary data per ST 2010; Multiview has no SDI capture today, so this is gated behind whatever
   SDI/ST-2110-40 ANC ingest lands (`crates/multiview-input/src/st2110/v40.rs` is the natural home —
   it already parses ANC). The decoder is the **identical** `Scte104Message::parse`; only the
   carriage differs. Named follow-on, not dropped.

3. **SCTE-35 ingest by observation.** The cue subsystem observes the decoded `splice_info_section`
   as it passes through ingest on the already-discovered PID and emits a `Cue` (`SpliceOut` on
   out-of-network, `SpliceIn` on return, `TimeSignal` for bare `time_signal`). Pure observation —
   the engine samples inputs, never paces to them — so reading the SCTE-35 content adds no data-plane
   coupling. `segmentation_descriptor` type is surfaced into `Cue.attributes` so a rule can match
   `BreakStart` vs `ProgramStart`; full descriptor-body parsing is a sized extension of `splice35.rs`.

4. **SCTE-35 emit into the program transport.** An `emit SCTE-35` action ([ADR-0072](ADR-0072.md))
   builds a fresh `splice_info_section` (`splice_insert` with `out_of_network`/`break_duration`, or
   `time_signal` + `segmentation_descriptor`) and inserts it on a dedicated output PID in the
   MPEG-TS muxer (`multiview-output`). **PTS is stamped from the output tick counter** (inv #3 —
   never a raw input PTS); the [ADR-0034](ADR-0034.md) `pts_adjustment` continuity is reused so a
   re-stamped passthrough keeps cue alignment. This is a *new builder* over the existing decoder +
   re-serialiser. It is a data packet fanned with the program — inv #7 (encode-once-mux-many) is
   untouched (SCTE-35 is signalling, not a re-encode).

5. **SCTE-104 emit upstream (deferred).** A TCP client that sends a `multiple_operation_message`
   (Multiview as an automation source to a downstream encoder). Lower priority — Multiview is
   usually the encoder, not the automation system. Named follow-on.

6. **Emit is RBAC-gated** ([ADR-0072](ADR-0072.md) §7) — originating an ad-break cue has real
   downstream consequences and requires an explicit `emit` capability scope.

## Consequences

**Positive.** The full SCTE round-trip lands on tested parsers: ingest reacts, emit originates, and
the same emitted bytes feed the HLS/DASH manifests ([ADR-0075](ADR-0075.md)) so one cue is signalled
consistently across TS/HLS/DASH. A cue can re-originate (GPI→emit-SCTE; SCTE-out→GPO+webhook,
[ADR-0072](ADR-0072.md)).

**Negative / cost.** A new TCP listener (bounded, isolated), a SCTE-35 *builder* (net-new — the
in-tree code only decodes + re-serialises `pts_adjustment`), and `segmentation_descriptor` matching
depth (brief §8 Q2). The VANC and SCTE-104-emit paths are deferred (named, gated), not shipped.

**Neutral.** Ingest observation is on the existing ingest task at cue rate; emit adds a few bytes
per cue to the program mux.

## Alternatives considered

1. **Route SCTE-35 only (no react/emit), per the existing crosspoint.** Rejected — the operator
   wants to *act on* and *originate* cues, not only pass them through; [ADR-0034](ADR-0034.md)
   continuity is reused, not replaced.
2. **Emit SCTE-35 by re-stamping an ingested section.** Rejected for originated cues — a fresh
   `splice_insert`/`time_signal` is needed when Multiview is the cue source; re-stamping is for the
   passthrough-continuity case [ADR-0034](ADR-0034.md) already owns.
3. **SCTE-104 over UDP/multicast.** Rejected for the MVP — TCP is the standard automation-to-encoder
   carriage; multicast adds no value for a point-to-point automation link.
4. **Block on the SCTE-104 peer for ordered delivery.** Rejected — would violate inv #10; the
   listener is bounded drop-oldest and a flood is coalesced.

## How this honors the invariants

- **#1 (output clock):** ingest observation and the TCP listener run on input/control tasks; emit
  runs on the mux task and re-stamps from the tick counter (inv #3). Nothing blocks or paces the
  output-clock thread.
- **#10 (isolation):** the SCTE-104 listener and the cue queue are bounded drop-oldest with
  `try_send`; the engine never awaits the automation peer. SCTE-35 ingest is pure observation of an
  already-sampled input.
