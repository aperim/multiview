# ADR-0083: URL-as-input `SourceKind` — headless web page source: JS-on-load / on-page-event, interval refresh, conditional refresh on page-text

- **Status:** Proposed
- **Area:** Input / Config / Compositor-adjacent (synthetic ingest path) / Control / Web
- **Date:** 2026-06-13
- **Source brief:** [url-input.md](../research/url-input.md)
- **Extends:** [ADR-0080](ADR-0080.md) (motion-graphics headless render engine — the **overlay** surface; this ADR is its **source-side twin**, sharing the engine), [ADR-0027](ADR-0027.md) (synthetic sources are first-class `SourceKind`s through the one uniform ingest path)
- **Relates to:** [ADR-P001](ADR-P001.md) (read-only / drop-oldest / shed-first sampled-consumer pattern reused for DOM-text polling), [ADR-T002](ADR-T002.md) (last-good-frame + tile state machine), [ADR-R004](ADR-R004.md) (Class-1/Class-2 live-apply, inv #11), [ADR-T001](ADR-T001.md) (output clock, inv #1), [ADR-RT004](ADR-RT004.md) (isolation, inv #10), [ADR-E001](ADR-E001.md) (decode/render-at-display-res, inv #6)

> **Scope.** This ADR adds a **web page as a video `SourceKind`** and its operator controls (a JS payload run on load / on a page event; refresh on an interval; conditional refresh when page text is / is-not present). It does **not** specify the headless render engine itself — launch, sandbox, crash-supervision, GPU path, and licence posture are owned by [ADR-0080](ADR-0080.md) and **shared**. A web-page source is a *placement* of that engine (fills a tile), the twin of the overlay placement (draws over program).

## Context

The operator asked for a **URL as an input**: point Multiview at a web page, place it like any other source, **run a JavaScript payload on load or on a page event**, **refresh on an interval**, and **simply refresh if / if-not some text is present on the page**.

A web page is a live picture. By the [ADR-0027](ADR-0027.md) principle — *a source is a source* — it must flow through the exact same path as an RTSP camera or a colour-bars generator: ingest → per-tile last-good store → compositor reads latest-or-placeholder, with nothing downstream of ingest knowing or caring what *kind* it is. The synthetic-source work already proved the shape of an **in-process source that publishes per tick**: `SourceKind` is an internally-tagged, `#[non_exhaustive]` serde union (`crates/multiview-config/src/schema.rs:217`) to which `Bars`/`Solid`/`Clock`/`Timer` were added additively; the generator runs as a peer of a decode thread (`generator_loop`, `crates/multiview-cli/src/synth.rs:913`; `render`, `synth.rs:208`) and publishes into the lock-free single-slot store (`TileStore::publish_arc`, `crates/multiview-framestore/src/latest.rs:75`), which the compositor reads each tick (`crates/multiview-framestore/src/tile.rs:371`).

The motion-graphics brief ([ADR-0080](ADR-0080.md)) introduces a **headless render engine** for HTML/CSS/JS *overlays*. A web-page source needs the **same** engine — only the placement differs. Building a second renderer would be duplication and a second licence/sandbox surface to harden; the right move is to **share one engine, two callers**.

The hard constraints are the invariants: a frozen page, a runaway operator script, a hanging navigation, or a crashed browser child must **never** stall the output clock (#1) or back-pressure the engine (#10), and the page must render at the **tile's displayed size** (#6), not a fixed resolution that is then downscaled.

The capture and control mechanisms are open and verified ([url-input.md §1.3, §2, §3](../research/url-input.md)): the **Chrome DevTools Protocol** drives the browser — `Page.startScreencast` for continuous frame capture, `Page.addScriptToEvaluateOnNewDocument` for pre-document JS injection, `Runtime.evaluate` for on-event scripts and `document.body.innerText` reads, `Page.reload`/`Page.navigate` for refresh. The browser is a heavy, proprietary-ish, operator-supplied dependency; the default Multiview build must stay LGPL-clean, GPU-free, and browser-free.

## Decision

1. **Add `SourceKind::WebPage`** — a new internally-tagged variant on the existing union (`schema.rs:217`), additive and back-compat (the [ADR-0027](ADR-0027.md)/`Timer` precedent). It carries `url`, a `viewport` (defaulting to the tile's display size), an optional `script` block (§2), and an optional `refresh` block (§3). URLs are **IPv6-first**, bracketed literals; auth via the existing `SourceAuth { secret_ref }` pointer (`schema.rs:182`), never plaintext.

2. **Render via the shared [ADR-0080](ADR-0080.md) engine, sampled into the source publish loop.** The browser child renders the page; frames arrive into a **bounded drop-oldest** ring; the source's generator-loop peer takes *latest-or-last-good* each output tick and publishes NV12 into the tile store — the clock-source republish-when-unchanged behaviour, applied to web frames. The **capture surface is a spike-gated candidate, not a verified fact:** CDP `Page.startScreencast` is the candidate, but it yields **compressed (base64 jpeg/png) images** under an ACK flow-control loop — the per-frame base64-decode → image-decode → NV12 color-conversion cost must be **budgeted in the efficiency ladder** (it can dominate CPU; it is not a free sample-and-publish). An ADR-0080 offscreen-render surface that yields raw frames avoids that decode and is preferred if available; the spike picks the surface (frame cadence, decode/base64 cost, ACK behaviour, idle-frame semantics) before implementation. `Page.captureScreenshot` is **rejected** for the live path (7–25 s latency; one-shot fallback only).

3. **JS payload — two composable triggers, both off-engine.**
   - **On load:** `Page.addScriptToEvaluateOnNewDocument` runs the operator's `on_load` script once per navigation, before page scripts, in an isolated JS world by default.
   - **On page event:** the on-load hook installs a `MutationObserver`/listener that signals over CDP (`Runtime.consoleAPICalled` / `Runtime.addBinding`); the off-engine page-control task observes it and runs the operator's `on_event` payload via `Runtime.evaluate`.
   - Operator JS is **sandboxed in the browser child**, resource-bounded (cgroup/rlimit at launch, [ADR-0080](ADR-0080.md)), and has **no path to the engine** — it can only mutate its own page. Throws/timeouts are logged and ride the tile state machine.

4. **Refresh — two laws, both off-engine, both sampled.**
   - **Interval:** re-load every *N* integer seconds (`Page.reload` warm, or `Page.navigate` cold). The source holds last-good across the navigation gap (#2).
   - **Conditional (`refresh_if` / `refresh_if_not`):** every `poll_seconds`, read `document.body.innerText` (optionally CSS-selector-scoped) via `Runtime.evaluate`; match a substring or regex; reload **when text appears** (`if`) or **when text is absent** (`if_not`). Edge-triggered with a `min_interval_seconds` debounce/hysteresis to prevent reload storms. Interval and conditional compose: a conditional reload resets the interval; reloads coalesce latest-wins.
   - The text poll is a **read-only sampled consumer** ([ADR-P001](ADR-P001.md)): it reads a page that already exists; a slow/stuck poll misses a sample, never blocks. Reload requests ride a bounded drop-oldest command channel; the engine is never in the loop.

5. **Supervision + live-apply.** A crashed/hung browser child is reaped and backoff-restarted by a per-source supervisor (the input supervised-reconnect precedent); the tile rides LIVE→STALE→RECONNECTING→NO_SIGNAL (#2), with a **circuit-breaker** that parks the source + warns after K crashes in a window (the [ADR-0037](ADR-0037.md) back-off precedent). Adding/removing/swapping a web-page source is **Class-1 hot** unless the canvas topology changes (then Class-2, [ADR-R004](ADR-R004.md)). Viewport resize on a tile-size change is Class-1 (`Emulation.setDeviceMetricsOverride`).

6. **Feature-gated, vendor-neutral.** The whole capability sits behind an **off-by-default Cargo feature shared with [ADR-0080](ADR-0080.md)**. The browser is **operator-supplied at runtime, never bundled/vendored**, named nominatively with a non-affiliation stance ([managed-devices.md](../research/managed-devices.md) posture). CDP wire shapes are implemented from the open published protocol — no vendor SDK, no vendor spec text redistributed. Default build stays LGPL-clean and GPU-free.

7. **v1 scope.** Video-only (page **audio not captured** — named, deferred to `multiview-audio` routing); sRGB/BT.709, tagged like any input (#8); `file://`/local access validation-gated opt-in; token-injection auth only (no OAuth/cookie-jar flow in v1).

8. **Acceptance (normative).** Two gates are **required for the PR**, not optional commentary: (a) a held-out **chaos gate** — wedge / kill / crash-storm the browser child and a runaway operator payload, and assert **zero output-clock gap** (#1) and bounded RSS / no engine back-pressure (#10); (b) a **capture-surface validation spike** that picks CDP screencast vs an ADR-0080 offscreen-render surface against measured frame cadence, per-frame decode + base64 cost, ACK flow-control behaviour, and idle-frame semantics (Decision #2). Implementation does not start until the spike has set the surface and its budget.

## Consequences

**Positive.**
- A URL is a first-class input — placeable in any tile or full-canvas — flowing the **one uniform ingest path** ([ADR-0027](ADR-0027.md)) with zero downstream awareness. The operator's full ask (URL + JS-on-load/event + interval + conditional-text refresh) is met in one additive `SourceKind`.
- **One engine, two placements.** Sharing the [ADR-0080](ADR-0080.md) renderer avoids a second browser/sandbox/licence surface to harden; this ADR adds only the source-ingest wiring and the refresh/JS/text-condition control surface.
- **Structurally isolated.** Browser-in-its-own-process + off-engine page-control task + lock-free single-slot publish + bounded drop-oldest rings ⇒ a dead/hung/runaway page is contained to its tile; the output clock and the engine never notice. The held-out chaos gate that proves this is a **normative acceptance requirement** (Decision #8), not commentary.
- **Cost-when-unused is zero** (Cargo-gated, browser-free default build).

**Negative / costs.**
- A browser child per web-page source is **heavy** (tens–hundreds of MiB RAM, real CPU, optional GPU co-tenancy) — the dominant cost and the reason this is a deliberate, bounded, opt-in feature. Per-source ceilings + the degradation ladder (#9) bound and shed it.
- New surface area: a CDP client, a screencast→NV12 sampler, the page-control state machine (interval/conditional/coalesce/circuit-breaker), config schema + validation + OpenAPI/JSON-schema + WebUI source-kind form, and the supervisor. Each is testable as a bounded unit, but it is non-trivial.
- Operator JS is powerful and is **operator-trusted content** — sandboxed and bounded, but a hostile/heavy payload degrades *its own* tile (acceptable; isolation holds).

**Neutral / deferred.**
- Page audio, full session/cookie auth, HDR/wide-gamut web content, and structured (DOM-snapshot) text matching are **named follow-ups**, not silent gaps (brief §5).
- If [ADR-0080](ADR-0080.md) ships a non-CDP engine (e.g. a Rust-native renderer), the §3/§4 CDP-specific mechanisms must be re-mapped to that engine's scripting/inspection API — the open question pinned in the brief.

## Alternatives considered

1. **Build a second, source-only web renderer (don't share the overlay engine).** *Rejected.* Duplicates the heaviest, riskiest dependency and doubles the sandbox/licence hardening; the only real difference between the overlay and the source is *placement* (alpha-over-program vs fill-a-tile), not rendering.

2. **Ingest the page via an external screen-grab / virtual-camera tool feeding RTSP.** *Rejected* for v1 as the primary path: it adds an out-of-process moving part with no JS-injection or text-condition control (the operator's explicit asks), and no clean way to render at tile resolution. The web-page source is in-tree and controllable.

3. **`Page.captureScreenshot` per tick for the live frame.** *Rejected.* 7–25 s worst-case latency ([devtools-protocol #28](https://github.com/ChromeDevTools/devtools-protocol/issues/28)) — unusable for continuous capture. `Page.startScreencast` is the event-driven candidate (compressed-image frames + an ACK flow-control loop — a DevTools debugging-capture path, not a verified video primitive); an ADR-0080 offscreen-render surface is the alternative. Screenshot is a one-shot static fallback only. The capture surface is spike-gated before implementation (Decision #2).

4. **Run the JS payload / text poll / refresh on the engine or a shared lock.** *Rejected* — a categorical invariant-#1/#10 violation: any lock or await between the output-clock loop and a web page lets a slow/hung page stall the clock. All three run on the browser child + an off-engine task; the publish is wait-free into a single-slot store.

5. **Re-stamp / pace the output from page frame arrival.** *Rejected.* Inputs are sampled, never pacing (#1); `out_pts = f(tick)`. The page is sampled and held-last-good between frames like every input — we never drive the tick from page output, and we deliberately do **not** virtualise the page clock for determinism (that is the overlay surface's concern).

6. **Conditional refresh via `textContent` / full DOM snapshot by default.** *Rejected as default.* `textContent` includes hidden nodes; a full `DOM.getDocument` snapshot is heavier and its node-ids churn on mutation ([chromedp #762](https://github.com/chromedp/chromedp/issues/762)). Default to selector-scoped `innerText` (visible text — the operator's mental model); offer structured matching as a later option. Caveat (carried from the brief, unverified): `innerText` is layout-dependent (it reflects rendered/visible text and forces a reflow), so it can be more expensive than `textContent` and may vary with viewport/CSS — to verify before implementation.

7. **Bundle/vendor a specific browser for turnkey operation.** *Rejected.* Heavy, licence-entangling, and against the vendor-neutral posture. The operator supplies the browser; Multiview drives it over the open CDP, named nominatively, behind an off-by-default feature.

## How this honors the invariants

- **#1 (output clock untouchable):** the web render, JS payloads, and both refresh laws run on the browser child + an off-engine page-control task — never the output-clock thread. The compositor reads last-good-or-placeholder each tick (`tile.rs:371`); a frozen/reloading/dead page yields a stale frame or slate, never a stall. `out_pts = f(tick)` is unchanged. Inputs are sampled, never pacing.
- **#10 (no back-pressure):** no engine-facing channel a slow page can fill — the render→NV12 hand-off is a wait-free single-slot publish (`latest.rs:75`); the screencast ring and the page-control command channel are bounded drop-oldest; the engine never awaits the producer. A hung/crashed browser child is supervised + circuit-broken, contained to its tile (the [ADR-P001](ADR-P001.md)/object-detection isolation guarantee).
- **#2 (last-good + state machine):** the tile rides LIVE→STALE→RECONNECTING→NO_SIGNAL across reloads and crashes — a web page is treated exactly like a flaky camera.
- **#6 (render-at-display-res):** the browser viewport is the tile's displayed size; cost budgeted in rendered megapixels/sec ([ADR-E001](ADR-E001.md)); re-set on tile resize.
- **#9 (degradation):** a GPU-compositing browser is a co-tenant the degradation ladder sheds cheapest-impact-first before program output is touched.
