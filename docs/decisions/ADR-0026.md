# ADR-0026: Encode-once-mux-many — hoist the single encoder into the bake consumer, fan encoded packets to mux-only sinks

- **Status:** Proposed
- **Area:** Core engine / Output (efficiency)
- **Date:** 2026-06-04
- **Source brief:** [efficiency.md](../research/efficiency.md), [core-engine §9.2](../research/core-engine.md)
- **Relates to:** [ADR-E003](ADR-E003.md), [ADR-E004](ADR-E004.md) (encode-once-mux-many), [ADR-0025](ADR-0025.md) (streaming bake→encode egress)

## Context

Invariant #7 (encode-once-mux-many): composite once, **encode the canvas once per
rendition**, and fan the *same* packets to every transport; separate encode only
when codec/res/bitrate differ.

After ADR-0025 the egress is: the engine hot loop hands each composited tick to a
bounded queue; one **bake consumer** thread bakes overlays into an
`Arc<Nv12Image>` and fans it to one bounded channel per output; each output runs
on its own thread as a `VideoFrameSource` driving a sink (`FileSink`,
`SegmentSink`, `PushSink`). **Each sink builds its own `VideoEncoder`**
(`sink.rs`: `FileSink::run`/`SegmentSink::run`/`PushSink::run` each call
`VideoEncoder::new`). So a config with, say, a file output **and** an HLS output
at the same rendition encodes the canvas **twice** — a direct invariant-#7
violation. ADR-0025 explicitly deferred this redesign.

Two facts from the `mosaic-ffmpeg` API shape the fix:
- `Muxer::add_stream(encoder: &codec::Context, time_base)` builds an output stream
  from **codec parameters only** — no encoder *instance* is required to mux.
- `Muxer::write_packet(stream_index, packet)` takes the packet **by value and
  mutates it** (`set_stream` + `rescale_ts` into the stream time-base). So one
  encoded packet cannot be written to N muxers without **N copies**.

## Decision

**Hoist the single `VideoEncoder` out of the sinks and into the bake consumer.**
The consumer becomes bake → **encode once** → fan encoded *packets* to N
**mux-only** sinks:

1. **One encoder per rendition, owned by the consumer.** `consumer_main` builds
   one `VideoEncoder` (from the existing `EncodeConfig`), and per tick: bake
   overlays → NV12→encoder-format convert → re-stamp PTS from the tick counter
   (inv #3) → `encoder.encode` → drain `receive_packet`.

2. **Fan encoded packets, not frames.** For each encoded packet, clone it once per
   live mux-sink and send the clone to that sink's bounded channel. The packet is
   compressed (≪ the raw NV12 frame), so N clones/tick is cheap. Define a
   `mosaic-ffmpeg` shareable packet (a thin wrapper that exposes `pts/dts/flags`
   and `is_keyframe`, and yields an owned `codec::packet::Packet` per consumer via
   ref-counted `av_packet_ref` or deep clone) so the HLS sink can split without
   re-encoding.

3. **`PacketMuxSink` (new, mux-only) replaces the encode-in-sink path.** It is
   built from the shared encoder's `as_codec_context()` + time-base
   (`add_stream`), writes the header, then writes each received packet
   (`write_packet`), and finalizes the trailer on stop/error (the ADR-0025
   finalize-on-error contract carries over). Two flavours:
   - **single-container** (file / RTMP / SRT push): one muxer, every packet
     written to its one stream.
   - **segmented** (HLS/LL-HLS): rotates the segment muxer on a **keyframe-flagged
     packet** at the GOP boundary (the split logic moves out of the old
     `SegmentSink`, keyed off `packet.is_keyframe()` instead of the encoder's GOP
     counter). One seed of codec params per segment, reused.

4. **Separate encode only when renditions differ.** Outputs are grouped by
   (codec, width, height, bitrate); each distinct rendition gets one encoder; all
   outputs in a group share that encoder's packets. The current build resolves a
   single rendition for all outputs, so v1 ships one encoder; the grouping seam is
   left in place for multi-rendition later (still never per-tile — explicit
   non-goal).

5. **Bounded teardown for a wedged sink (folds in #50).** The consumer fans to
   each sink with a **bounded** send. A *slow* sink legitimately paces the
   consumer (off the hot path — the engine already dropped via the ADR-0025 hot
   queue). A *wedged* sink (never drains, never hangs up) must not hang teardown:
   the egress join is bounded by a grace deadline (mirroring `INGEST_JOIN_GRACE`),
   after which a still-unfinished mux-sink is **detached + logged**, so
   `drive_streaming` always returns. The engine itself is already isolated (inv
   #1/#10) and unaffected.

## Invariants upheld

- **#1 / #10:** the encoder + muxers live on the off-hot consumer/sink threads;
  the engine still only `try_send`s to the bounded hot queue and never awaits the
  consumer. A slow/wedged sink can pace or be detached, never back-pressure the
  engine.
- **#3:** PTS/DTS are still re-stamped from the tick counter before encode; raw
  input PTS never reaches the encoder/muxer.
- **#7:** one encode per rendition, the *same* packets fanned to all transports —
  now actually true across outputs, not just within one sink.
- **Licensing:** the single encoder still honours the LGPL-default / `gpl-codecs`
  opt-in selection; nothing new is linked.

## Alternatives considered

- **Keep per-sink encoders (status quo).** Simple but violates #7 — N encodes for
  N outputs of one rendition, the dominant CPU cost. Rejected.
- **A dedicated encode *thread* between consumer and sinks.** Cleaner separation
  but adds a third hop + another bounded queue for no benefit at one rendition;
  the consumer already runs off the hot path and has spare headroom. Fold encode
  into the consumer; revisit if multi-rendition encode needs its own thread each.
- **Ref-count the packet (av_packet_ref) instead of deep-cloning.** Cheaper for
  many sinks; chosen as the wrapper's internal strategy where safe, but the public
  contract is "each sink gets an owned packet" so `write_packet`'s mutate-in-place
  is sound regardless.

## Consequences

A multi-output config (file + HLS) drops from N encodes to one — the headline
efficiency win of inv #7. The egress restructures (consumer owns the encoder;
sinks become packet-fed); ADR-0025's hot-path isolation, finalize-on-error, and
bounded-memory properties are preserved and extended with a bounded teardown.
Implementation lands TDD-first across `mosaic-ffmpeg` (shareable packet),
`mosaic-output` (`PacketMuxSink`), and `mosaic-cli` (consumer encode + packet
fan-out + bounded join), each gate-green and adversarially reviewed before the
cli integration is wired.
