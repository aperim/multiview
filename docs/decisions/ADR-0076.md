# ADR-0076: BXF / automation metadata exchange

- **Status:** Proposed
- **Area:** Control plane / Config
- **Date:** 2026-06-13
- **Source brief:** [broadcast-cues.md](../research/broadcast-cues.md) (§5)
- **Extends:** [ADR-0072](ADR-0072.md) (the cue bus + the scheduled-cue mechanism),
  [managed-devices.md](../research/managed-devices.md) (clean-room nominative vendor posture)
- **Relates to:** [production-switcher.md](../research/production-switcher.md) §10.2 (the
  macro-sequencer wait-step that schedules a future cue), [ADR-T001](ADR-T001.md)/[ADR-RT004](ADR-RT004.md)
  (invariants #1/#10)

## Context

The operator wants schedule/traffic exchange via **BXF** (Broadcast Exchange Format,
[SMPTE ST 2021](https://www.smpte.org/standards)): the traffic/automation-system schedule of
programs, breaks, events, and as-run logs, exchanged as files or over a SOAP/HTTP web service.

**(unverified — ST 2021 is a paid SMPTE standard; this ADR does not reproduce its schema.)** From
public descriptions, BXF defines an XML document with a schedule (programs/events/breaks at
wall-clock times) and an as-run report, and a web-service (SOAP) binding. BXF is **slow, scheduled,
out-of-band** metadata — minutes-ahead boundaries — **not** a real-time trigger. Multiview already
parses XML (`crates/multiview-input/src/dash/xml.rs` MPD posture) and has the macro-sequencer
wait-step mechanism that can schedule a future action ([production-switcher §10.2](../research/production-switcher.md)).

The vendor/legal posture: ST 2021 is paid; any web-service binding must be a clean-room, nominative
implementation built from the public binding description + operator-supplied schemas — **no vendor
SDK, no redistributed spec** (the [managed-devices.md](../research/managed-devices.md) ZowieTek-driver
posture). The operator decides the proprietary-protocol stance.

## Decision

1. **A BXF poller populates the cue bus ahead of time.** A control-plane poller task in
   `multiview-control` (file-drop watch or SOAP poll, operator-configured, IPv6-first dual-stack
   `[::]`) parses the BXF schedule into a list of **future** cues
   (`ProgramStart`/`ProgramEnd`/`BreakStart`/`BreakEnd` at a wall-clock time) and an **as-run** sink.

2. **Future cues schedule via the existing wait-step mechanism**
   ([production-switcher §10.2](../research/production-switcher.md)): a BXF break at time `T` becomes
   a scheduled cue that, at `T` (sampled at the frame boundary, never paced), enters the dispatcher
   exactly like a live GPI ([ADR-0072](ADR-0072.md)). **BXF *populates* the cue bus; it does not add
   a new dispatch path.** The rule table ([ADR-0072](ADR-0072.md)) decides what each BXF boundary
   does (e.g. a `BreakStart` runs an ad-roll macro).

3. **Parsing is pure XML→value** (the `dash/xml.rs` posture); the SOAP/web-service transport is
   behind an off-by-default feature, a clean-room nominative implementation of the public binding —
   no vendor SDK, no redistributed schema. As-run write-back (Multiview reports what actually fired)
   is the symmetric sink, also feature-gated.

4. **BXF is the lowest-priority, last-wave ingress.** The real-time triggers (GPIO, SCTE-104/35,
   manifest — ADRs 0073–0075) deliver the operator's "broadcast cues" ask; BXF closes the
   traffic-system loop. The MVP proposes **file-drop ingest, read-only schedule first**; SOAP and
   as-run write-back are deferred (brief §8 Q4).

5. **Legal/ToS caveat is explicit** (CODE_OF_CONDUCT.md vendor neutrality): the implementation is
   built from the public binding description + operator-supplied schemas; the operator decides
   whether to engage the proprietary binding at all.

## Consequences

**Positive.** The traffic-system loop closes: a station's schedule drives Multiview's switcher/
recording/cue actions ahead of time, with an as-run report back. Reuses the XML-parse posture and
the scheduled-cue mechanism — no new dispatch path.

**Negative / cost.** A new XML schema mapping (operator-supplied, since the spec is not
redistributed), a feature-gated SOAP client, and an as-run sink. The deferred pieces (SOAP, as-run)
are named, not silently dropped.

**Neutral.** BXF holds one parsed schedule window in memory, pruned as cues fire; the poll is minutes-
cadence, negligible cost.

## Alternatives considered

1. **Treat BXF as a real-time trigger.** Rejected — BXF is scheduled, minutes-ahead metadata; future
   cues are scheduled and fire at their time through the standard dispatcher, not paced.
2. **A dedicated BXF scheduler engine.** Rejected — the macro-sequencer wait-step already schedules a
   future action; BXF reuses it (one apply path).
3. **Bundle/redistribute the ST 2021 schema.** Rejected — paid SMPTE standard; clean-room nominative
   only, operator-supplied schemas ([managed-devices.md](../research/managed-devices.md) posture).
4. **SOAP-first.** Rejected for the MVP — file-drop is far simpler and covers many plants; SOAP is a
   deferred follow-on once a concrete traffic-system integration is on the table.

## How this honors the invariants

- **#1 (output clock):** the BXF poller and the XML parse run on a control-plane task; scheduled cues
  enter the dispatcher and apply at the frame boundary exactly as live cues do — nothing paces the
  output clock.
- **#10 (isolation):** the poller and the cue queue are bounded best-effort; a slow/dead traffic
  system or web service stalls only the poller task; the engine never awaits BXF.
