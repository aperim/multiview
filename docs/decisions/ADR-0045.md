# ADR-0045: Display-node mode (`multiview node` subcommand, enrollment, presentation discipline)

- **Status:** Proposed
- **Area:** Core Engine
- **Date:** 2026-06-10
- **Source brief:** [display-out.md](../research/display-out.md)

## Decision

Ship display nodes as a **`multiview node` subcommand of the existing binary** — a new variant in the intentionally-exhaustive `Command` enum (`crates/multiview-cli/src/cli.rs`), so the subcommand is a compile error until fully wired. A node runs one supervised ingest (RTSP/RTP or MPEG-TS/SRT preferred; HLS accepted) from a central Multiview → hardware decode → single-source full-canvas present → the `display-kms` scanout sink ([ADR-0044](ADR-0044.md)) + ALSA HDMI audio. The ingest path is reused unchanged: `multiview-input`'s pacer, jitter buffer, timestamp normalization and supervised reconnect (`pacer.rs`/`jitter.rs`/`normalize.rs`/`reconnect.rs`) feed a `multiview-framestore` last-good store riding the Live → Stale → Reconnecting → NoSignal ladder ([ADR-T002](ADR-T002.md)), so a node holds last-good and then shows a local slate exactly as a tile would — node resilience equals product resilience by construction, not by re-implementation. This node **is the documented external IP gateway** that roadmap M12 historically required (the M12 focus line previously read "SDI/HDMI reached only via a documented external IP gateway"; [roadmap](../roadmap.md) now serves HDMI first-party per this ADR): HDMI is reached first-party by pointing a display node at a program output, resolving that drift; SDI stays the realm of third-party gateway devices.

Nodes **enroll through the Devices domain** ([ADR-M008](ADR-M008.md), driver `displaynode`): the node generates a keypair on first start and calls `POST /api/v1/devices/enroll` presenting a TTL'd one-time enrollment token (issued in the SPA, displayed once, stored hashed); the operator completes pairing via `POST /api/v1/devices/pair` against a six-character code (plus QR) shown on the attached display. Thereafter the node authenticates by keypair only — no passwords — and receives its assignment (program/output/wall-head binding) and the per-program presentation epoch over the control WebSocket. **Presentation discipline is pure frame choice**: decode into a 2–3-frame queue; on each page-flip-complete event present the frame whose `wall_at(pts) + link_offset` (the outbound `WallClockRef` epoch of [ADR-M010](ADR-M010.md); the exact-affine type at `crates/multiview-core/src/wallclock.rs`) is nearest the predicted next vblank from the KMS flip timestamps; repeat the last frame if the next is early, drop if late. The node is a pull-side consumer with **no feedback channel into the engine**: losing the WebSocket means it keeps the last epoch and free-runs drift-bounded; invariants #1 and #10 hold by construction.

An optional **display-locked clock mode** exists for nodes only, default **off**, applied as a Class-2 change ([ADR-R004](ADR-R004.md)). Invariant #1 forbids inputs, clients, or sinks from pacing the program clock; on a dedicated node the attached display is that box's terminal output, so deriving the node's *local* presentation cadence from vblank timestamps (the mpv display-resample analog) keeps "one fixed-cadence monotonic clock" true while substituting the display's pixel clock for the host timer — the central engine's tick is never touched, and the mode is rejected outside `multiview node`, where a display is one sink among many and sinks never pace the program. It stays default-off because the default repeat/drop reconciliation already costs only ~one duplicated/dropped frame per ~16.7 s at a 60.000 Hz tick against a 59.94 Hz refresh. **Deployment:** bare-metal systemd is the blessed path (minimal Debian/Raspberry Pi OS Lite; no X11/Wayland/display manager; implicit DRM master on first open — no VT, no logind; `User=multiview` with `video render audio` groups; masked getty; watchdog + sd_notify; fbcon restore on crash). A rootful container is workable (`--device /dev/dri --device /dev/snd`); the KMS backend listens on the **kernel** netlink uevent group directly — kernel uevents are delivered to netns owned by the initial user namespace — rather than the udevd-processed stream, so hotplug works without host udev plumbing. Rootless containers lack kernel uevent delivery and fall back to `force_probe` connector polling (2–5 s); the host must not hold DRM master in any container deployment.

## Rationale

One binary means one ingest stack, one resilience doctrine, one timing model, one test surface: the node inherits the hardened pacer/jitter/reconnect/failure-ladder behaviour and every future fix for free, and the heavy dependencies already sit behind feature flags so a node build carries only what it uses. The HP t630 test unit class is verified comfortably within the node-tier budget (1× 1080p60 H.264/HEVC 8-bit ≈ 124 Mpx/s against an unbenchmarked 250–500 Mpx/s decode-ceiling band — 2–4× headroom), and the frame-chooser discipline reaches the proven commodity sync tier (frame-accurate, Tiers A/B of [ADR-M010](ADR-M010.md)) — the same shared-clock + shared-epoch + repeat/drop pattern productized by signage videowall players and measured at sub-ms content skew in network-clock media pipelines. Enrolling through the Devices domain gives the node fleet the same registry, state machine, and alarm UX as every other managed device instead of a bespoke channel.

## Alternatives considered

A **separate node binary** (rejected: forks ingest/resilience/timing into a second codebase that must be kept behaviourally identical by hand, doubling the release/packaging/test surface; the existing subcommand + feature-flag structure already yields a thin artifact). **Full engine on every node** (rejected: a node needs one stream decoded and presented, not the compositor/encoders/control plane; on thin-client budgets the engine steals decode headroom and multiplies the ops/attack surface; where a local multiview is genuinely wanted, tier-2 of [ADR-0044](ADR-0044.md) covers it). **Vendor decoder appliances only** (rejected as the foundation: ZowieBox-class devices are sync Tier C — bounded drift ±100–500 ms, no public buffer/PTP/genlock controls, poll-only management, no last-good/slate discipline; they remain supported peers via the Devices domain, [ADR-M009](ADR-M009.md), never the sync-capable tier). **Browser/kiosk renderer** playing HLS (rejected: drags a full browser/window-system stack onto the node, seconds-class HLS latency, no vblank-granularity frame choice and no path to PTP-disciplined frame accuracy; the browser remains a preview/diagnostic surface).

## Consequences

A new security-relevant fleet surface exists: enrollment-token issuance (one-time display, hashed at rest, TTL), keypair lifecycle and revocation, pairing UX, and rate limiting on `enroll` — with discovered-but-unenrolled nodes treated as untrusted inventory requiring explicit adoption (the [ADR-0041](ADR-0041.md) doctrine). Hardware honesty: a Raspberry Pi 4 is being provisioned for validation while Pi 5 remains **aspirational until hardware is acquired**; Pi 5 has no H.264 hardware decoder (HEVC-only), so program-stream codec choice becomes a per-node concern; the HP t630 test unit is approved as a dedicated display-node target, so the DRM-master takeover and the decode-throughput benchmark (not read-only operations against its incumbent X11 signage stack) are unblocked. EDID-less forced-mode support is **mandatory from day one** (the t630 test unit has an EDID-less head today): CVT-RB fallback, per-connector config overrides, and `video=` kernel-cmdline support; EDID-less heads have no audio path (ELD-gated). The display-locked clock mode adds a second clock-domain code path that needs its own chaos/soak gates before it ships even as an off-by-default option. And no node tier offers genlock-grade alignment: frame-accurate with a 0–1-refresh vsync-phase residual is the ceiling — fine for separated displays, never for zero-bezel walls.
