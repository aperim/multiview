## Decision

License the project MIT OR Apache-2.0; ship the default with a dynamically-linked LGPL-2.1 FFmpeg (no --enable-gpl, no --enable-nonfree; NVENC/NVDEC via MIT nv-codec-headers; scale_cuda not scale_npp; native AAC + GnuTLS) and gate gpl-codecs, nonfree, ndi, and ndi-advanced behind additive Cargo features, enforced in CI with cargo-deny.

## Rationale

NVENC/NVDEC/VAAPI/VideoToolbox need neither GPL nor nonfree, so a fully hardware-accelerated build stays redistributable under LGPL. --enable-nonfree (libnpp/FDK-AAC/OpenSSL) makes the binary non-redistributable; --enable-gpl (x264/x265) makes the whole product GPL. Dual MIT/Apache gives a patent grant while staying GPLv2-compatible (x264/x265 are GPLv2-or-later; Apache-2.0 alone is GPLv2-incompatible).

## Alternatives considered

Always build --enable-gpl --enable-nonfree (rejected: poisons distribution). Software libx264-only default (rejected: drags GPL, loses HW). Apache-2.0-only project (rejected: GPLv2-incompatible vs a GPL FFmpeg build). MIT-only (acceptable but no patent grant).

## Consequences

Must build FFmpeg in-house (distro FFmpeg may be GPL/nonfree), do GPU scaling via scale_cuda/in-house, keep dynamic linking (LGPL relink right) including in containers and macOS bundles, and report the effective license per built artifact. Codec patent licensing (H.264/HEVC/AAC pools) remains a separate obligation.
