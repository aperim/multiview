# ADR-0092: Operator-triggered hardware re-assessment + re-placement with explicit interrupt confirmation

- **Status:** Proposed
- **Area:** HAL / Engine / Control / Web (placement, operability)
- **Date:** 2026-06-13
- **Source brief:** [webui-operability-gaps.md §4](../research/webui-operability-gaps.md)
- **Extends:** [ADR-0035](ADR-0035.md) (self-aware placement — SENSE→DETECT→WARN→PLAN→APPLY, the
  `CapabilityReport`, the off-thread `PlacementController`), [ADR-0018](ADR-0018.md) (adaptive
  affinity-first placement + anti-storm damps), [self-aware-placement.md](../research/self-aware-placement.md)
- **Relates to:** [ADR-R010](ADR-R010.md) (the one make-before-break migration primitive),
  [ADR-0091](ADR-0091.md) (the high-risk confirmation gate this reuses)

## Context

The placement subsystem ([ADR-0035](ADR-0035.md)) builds a closed loop that re-plans **autonomously** on
load signals: a `CapabilityReport` at build/re-plan time, the ADR-0018 `select_device`/
`PlacementController` planner (off-thread, ~1 Hz), and a slow control tick applying O(1) frame-boundary
changes. The controller *proposes only* (`PlacementProposal::{Hold,Shed,Migrate,Split}`,
`crates/multiview-engine/src/placement.rs:117/:363`); a `Migrate` executes as a whole-island Class-2
make-before-break on the supervisor path ([ADR-R010](ADR-R010.md)), never on the clock thread, never
fragmenting a running island (the GPU-placement principle).

What is missing is an **operator-triggered** re-assessment. The operator wants to *refire the hardware
assessment + allocation* on demand — after plugging in a GPU, changing drivers, or freeing a co-tenant
NVR — and to be **made to confirm they know it can interrupt everything** before it moves a running
pipeline.

## Decision

Add an operator-triggered re-assessment that is **dry-run-first** and reuses the built planner and the
one make-before-break primitive — no new placement logic, no new engine channel.

1. **`POST /api/v1/system/placement/reassess?dry_run=true`** re-runs DETECT (refresh the
   `CapabilityReport` — re-probe adapters/encoders, the ADR-0035 throwaway `open_as`/`*_cuvid` probes)
   and PLAN (`select_device` over the current demand + a fresh `DeviceLoad` snapshot) **without
   applying**, returning a typed **`ReassessmentPlan`**: per running pipeline/island,
   `{ stays | would_migrate { from, to, reason, class: Class2 } | would_shed }`, plus any new
   `HealthWarning`s the refreshed report raises (e.g. a now-usable GPU previously silently on CPU — the
   exact ADR-0035 bug). Runs off the clock thread, nothing applied — but the DETECT refresh is **not
   free**: re-probing adapters/encoders (`open_as` / `*_cuvid` throwaway opens) consumes scarce
   decoder/encoder/session slots and could perturb a running pipeline. The re-assess therefore prefers
   **cached/read-only** capability detection, and any live device probe is **admission-budgeted**
   (skipped, not forced, when no spare session exists), returning a `cannot-safely-probe-while-live`
   result for that backend rather than touching a live device. A dry-run never deallocates or restarts a
   running pipeline.
2. **The UI shows the plan and the blast radius.** Every island `stays` ⇒ the apply is a labelled no-op
   (no interrupt). Any island `would_migrate` ⇒ the confirm dialog enumerates exactly which
   outputs/consumers would see a discontinuity (each migration is a Class-2 cutover), reusing the
   [ADR-0091](ADR-0091.md) §4 high-risk gate.
3. **`POST /api/v1/system/placement/reassess`** (apply) requires explicit confirmation and executes each
   accepted migration through the **one** make-before-break primitive ([ADR-R010](ADR-R010.md)):
   spin-up NEW alongside OLD, warm, cut at an IDR, drain+stop OLD — exactly one correctly-signalled
   discontinuity per moved island, never a gap. The anti-storm damps (cooldown / per-GPU budget /
   min-gain, [ADR-0018](ADR-0018.md) §4.6) still gate whether a migration is *worth* proposing; a
   forced re-assess may override the **cooldown** but never the **affinity / no-fragment hard gate**.
4. **Terminal outcomes** (`Migrated` / `RolledBack { reason }`) ride the realtime stream like any other
   migration; a re-assess that admission-fails in VALIDATE ([ADR-R010](ADR-R010.md)) reports a clean
   rejection and leaves every island exactly as it was.

## Consequences

- **Positive.** The operator can force a fresh capability detection + re-placement on demand, sees the
  precise blast radius before committing, and is required to confirm any interrupt — no "apply and hope".
  A re-assess that changes nothing is safe and ceremony-free; one that moves N islands shows N
  discontinuities first.
- **Negative / cost.** A dry-run refreshes the `CapabilityReport` (bounded probe per HW backend) and
  runs one `select_device` pass; the apply pays the make-before-break transient (OLD+NEW concurrently),
  admission-gated so it cannot starve a running output. Whether a *forced* re-assess (that the planner
  would not have proposed) needs a stronger confirmation, and whether manual re-assess should be
  rate-limited, are open (brief Open question 3).
- **Neutral.** Removal of demand returns budget implicitly — placement reads measured NVML load per
  decision, no allocation ledger ([ADR-W018](ADR-W018.md) §7).

## Alternatives considered

- **Apply re-placement immediately on the operator click.** Rejected: the operator explicitly asked to
  confirm the interrupt; dry-run-first is mandatory so the blast radius is shown before any consumer
  moves.
- **Let the operator override the affinity / no-fragment gate.** Rejected: fragmenting or migrating a
  live island against affinity is wrong by construction (the GPU-placement principle); a forced
  re-assess may override the *cooldown* timing, never the hard gate.
- **A second, parallel re-placement mechanism.** Rejected: there is exactly one make-before-break
  primitive ([ADR-R010](ADR-R010.md)); the operator action is a new *trigger* into it, like CTL-6 and
  GPU-5c — never a second cutover path.
- **Run the re-assess on the output-clock loop.** Rejected outright: `observe()`/`select_device` must
  never run on the clock thread ([ADR-0035](ADR-0035.md) §5); the re-assess runs on the off-thread
  placement tick / supervisor path.

## How it honors the invariants

- **#1 (output clock).** DETECT/PLAN run off the clock thread; an applied migration uses two independent
  clocks and cuts at an IDR — there is no instant where a consumer's rendition has no producing clock
  ([ADR-R010](ADR-R010.md) §4). The clock never blocks on a re-assess.
- **#10 (no back-pressure).** The dry-run is a read over a `DeviceLoad` snapshot + a `CapabilityReport`
  refresh; outcomes ride the drop-oldest broadcast; the trigger is *pulled* by the engine off the
  existing desired-state hand-off, never pushed into the clock. No new engine-inward channel.
