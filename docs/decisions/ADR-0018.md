# ADR-0018: Adaptive affinity-first GPU work-placement with deliberate split + closed-loop re-placement

- **Status:** Proposed
- **Area:** Efficiency
- **Date:** 2026-06-03
- **Source brief:** [gpu-placement-engine.md](../research/gpu-placement-engine.md)
- **Builds on:** [ADR-0017](ADR-0017.md) (monitoring + the least-loaded ranking function), [ADR-0004](ADR-0004.md) (zero-copy islands), [ADR-R004](ADR-R004.md) (make-before-break Class-2 migration)

## Decision

Place each tile's `decode -> composite -> encode` pipeline by a **decision engine**, not by "least-busy then load it up." Placement is a **pure function** `select(affinity_set, capability, CostBudget, DeviceLoad[], reserve_headroom, pins) -> Placement` in a new `multiview-hal::select` module (beside the planned `multiview-hal::load`), run **only at admission/reconfig and at migration â€” never per-frame**, in four ordered stages:

1. **Pins win (hard, first).** An operator pin (by stable `DeviceId` â€” NVML UUID / PCI bus id / Metal registryID, never the enumeration index) of a source's decode / the composite / an output's encode is honoured unconditionally; an unsatisfiable pin surfaces a warning + dry-run plan rather than silently relocating.
2. **Hard gates build the candidate set â€” affinity is a GATE, not a score term.** A GPU is a candidate only if it can host the **whole** island and passes: capability (`Capability::supports`), per-engine Mpix/s cost budget (`Planner::admit` per GPU), free-VRAM headroom (`nvmlDeviceGetMemoryInfo().free`), and the per-system NVENC concurrent-session ceiling (discovered at runtime, Multiview tracks its own count, never hard-coded). Cross-GPU is excluded here because consumer GeForce has no P2P, so a split pays a `GPU -> host -> GPU` copy every frame.
3. **Score the survivors, spread-biased (lower = least-loaded).** A composite score = the **DRF dominant-resource share** (the `max` over `{free-VRAM (primary), encoder-util, decoder-util, NVENC-session, compute-busy}` â€” a per-GPU generalisation of the existing `pressure_from_plan`) **plus a small Tetris-style fit/alignment term** (a weighted dot product of the GPU's *free*-resource shape against the pipeline's demand shape, so the last NVENC session isn't burned on the only box with free decode headroom) **minus a tiny empty-GPU spread bonus**. We bias toward spread + reserved headroom (**resilience > density**), the opposite of a batch cost-optimiser. Unknown signals drop out of the score with weight redistributed; ties break to existing-affinity then lowest `DeviceId`.
4. **Cost model is the real safety net.** The score steers; the Stage-2 hard gates decide feasibility.

**Whole-system fluctuating external load is first-class:** util/VRAM/session signals are read for the *whole machine* (they already reflect every consumer), low-passed to a common ~1 s window, and a config **reserve-headroom** is subtracted from each GPU's usable budget before gating/scoring so room is kept for the other tenants.

**Deliberate multi-GPU split, only when it doesn't fit:** if the candidate set is empty even after the degradation ladder shrinks the footprint, the engine splits at the **cheapest cut point** â€” `decode | composite+encode` when the decode engine is the bottleneck (copy the decoded source surface), or `decode+composite | encode` when the NVENC session ceiling is the bottleneck (copy the composited canvas) â€” accounts the host round-trip in the cost model as an explicit `CrossGpuCopy` cost, surfaces it to the operator, and logs it. A split is recorded so the loop can re-converge it onto one GPU when headroom returns. `composite` itself is never split.

**Closed-loop re-placement (invariant #9 extended from degradation to placement + recovery):** an off-hot-path controller in `multiview-engine` senses the wait-free `DeviceLoad` snapshot (incl. external load), detects **sustained** overload via an EWMA over the existing `Hysteresis` controller (transients are ignored), then chooses the cheapest sufficient action: **SHED** via the existing degradation ladder when a cheap local step relieves it, or **MIGRATE** via make-before-break (Class-2, [ADR-R004](ADR-R004.md)) â€” spin up the new island in parallel, cut over at an IDR/GOP boundary, tear down the old â€” when the *imbalance* (not the total load) is the problem. Anti-storm damping (cooldown, per-GPU migration budget, minimum-improvement gate) prevents migration storms. The output clock never drops a frame (inv #1); the controller only *proposes* and never `.await`s the engine or a client (inv #10).

## Rationale

"Least-busy then load it up" is rejected because each naive policy fails a *continuous, never-stalling* live workload: round-robin/first-GPU ignores load and the session ceiling; single-resource least-loaded is the DRF failure (a GPU can be least-loaded on VRAM yet at its NVENC ceiling, because encode/decode are physically distinct ASICs); and bin-packing creates noisy-neighbour hotspots that thermally throttle and jitter the output clock. A live multiviewer wants a **spread/headroom bias gated by hard affinity**, scored over multiple resources, with the hard caps as the safety net.

Affinity must be a **hard gate, not a score term**, because the cross-GPU copy is real and unavoidable on Multiview's commodity targets: NVIDIA disabled peer-to-peer on consumer GeForce (verified â€” an NVIDIA representative confirmed "Peer to Peer is not supported on the RTX 4090", covering the RTX 30 and 40 series; RTX 20-series was the last to support P2P, via NVLink), so without P2P CUDA inter-GPU copies are staged through the host (CUDA Programming Guide). A 1080p NV12 frame is ~3 MB -> ~90 MB/s/stream across PCIe at 30 fps, on both cards. So the copy is modelled **only** on the last-resort split path, never as a routine tradeoff â€” keeping the decode->composite->encode island together ([ADR-0004](ADR-0004.md)).

The scoring model is literature-grounded: the DRF dominant-resource normalisation (Ghodsi et al., NSDI 2011) is exactly what `pressure_from_plan` already computes (worst per-stage `StageUsage::utilization`), generalised per GPU; the Tetris alignment score (Grandl et al., SIGCOMM 2014) is a weighted dot product of available-resource and demand vectors, reused as a fragmentation-aware **fit term** â€” with the verified caveat that Tetris targets batch makespan and tolerates tight packing, so we adopt the dot product but bias toward spread, not density. Make-before-break migration of a live encoder without dropping output is established, glitch-free prior art (SMPTE ST 2022-7 seamless/hitless switching of identical streams; live-encoder failover handoff), and GOP/IDR alignment â€” its precondition â€” is already guaranteed by Multiview's pinned-GOP + `forceIDR` output ([ADR-R004](ADR-R004.md)). Production schedulers (Triton spread-or-pin; Kubernetes only gaining attribute/topology-aware GPU placement with DRA GA in 1.34, Aug/Sep 2025) confirm the spread-or-pin norm and that none reactively fragment a job across devices â€” validating affinity-gated-least-loaded + operator pins.

Placement at admission/reconfig only (never per-frame), a wait-free snapshot read with no lock/await ([ADR-I001](ADR-I001.md)), an off-data-plane controller that only proposes, and migration via the existing make-before-break mechanism keep invariants #1 (output never falters) and #10 (isolation) intact. The split and migration costs are conservatively modelled so the engine prefers degrade-to-fit and migration over a split, and never churns.

## Alternatives considered

- **Static pinning only (no auto policy).** Predictable and operable, and retained as the always-wins override. But it forces operators to hand-place every pipeline and cannot react to fluctuating external load, defeating the multi-GPU commodity-hardware goal. Rejected as the default; kept as the explicit override.
- **Pure least-loaded ignoring affinity (the unrefined hypothesis).** Optimises instantaneous balance but routinely fragments a pipeline's decode/composite/encode across GPUs, paying a per-frame host round-trip (no GeForce P2P) that violates the zero-copy-island design and harms density/latency far more than the imbalance it cures. Rejected â€” affinity gates least-loaded; a split is deliberate and last-resort only.
- **Reactive per-stage load balancing** (move whichever stage is hottest to whichever GPU is coolest, continuously). This is exactly the fragmentation the owner forbids: it would split running pipelines to chase load, add per-frame copies, and (without hysteresis) storm. Rejected â€” running pipelines are kept whole; rebalancing is whole-pipeline, hysteresis-damped, make-before-break migration, not per-stage shuffling.
- **Catch-and-restart-only on overload** (let the supervisor restart in place). Insufficient for requirement #5: restart-in-place does not change *where* the work runs, so an overloaded GPU stays overloaded and the pipeline thrashes. The supervisor's restart path remains for *faults*; the placement loop adds re-analysis + re-selection + migration for *sustained load*. Rejected as the sole mechanism.
- **Tight bin-packing / density maximiser (cloud cost-optimiser).** Frees whole GPUs but creates noisy-neighbour throttling/OOM that jitters a never-stalling output. Rejected â€” we bias spread + reserved headroom (resilience > density); the Tetris dot product is used as a fit term, not a packing maximiser.
- **Exact multi-resource optimal assignment per control tick.** NP-hard; not worth solving on a ~1-4 Hz loop. Rejected in favour of a greedy worst-first heuristic that re-converges over ticks (open question: a small bounded look-ahead).

## Consequences

`multiview-hal` gains a pure, unit-testable `select` module (`select(...) -> Placement` + `SplitPlan`/`CutPoint` types) consuming the existing `Capability`/`CostBudget`/`Planner`/`Hysteresis` types and the planned `load::DeviceLoad`; no new native deps, the default pure-Rust build still places nothing (single-island trivial). `multiview-engine` gains an off-hot-path placement controller that senses, runs the EWMA + `Hysteresis` overload detector, coordinates SHED-vs-MIGRATE with the existing `ControlLoop`, and drives make-before-break migration through the existing supervisor + scene-swap; it proposes only and cannot stall the data plane. `multiview-telemetry` reuses the per-GPU gauges from ADR-0017 and adds placement/migration/split counters (every adaptation logged). `multiview-config` gains the policy as data: per-source/per-output GPU pin, reserve-headroom, scoring weights + `k_fit`/`k_spread`/epsilon, and migration policy (`migration_cooldown`, per-GPU migration budget, `min_gain`, optional freeze-placement) â€” conservative defaults. `multiview-control` + the WebUI expose the placement map, the pin override with a dry-run Class-1/Class-2 plan, and a migration/split feed over the conflated realtime channel.

Costs/risks: the score blends signals with heterogeneous sampling windows (low-passed to ~1 s; hard gates are the real safety); the cross-GPU split cost is a modelled estimate until measured per host (biased conservative so degrade/migrate are preferred); the `min_gain` migration gate needs empirical tuning (too low -> churn, too high -> persistent imbalance); multi-pipeline rebalance is a greedy worst-first heuristic, not a global optimum; a hostile oscillating co-tenant is bounded by the anti-storm damps to "no storm, possibly some shed" with the program never stalling; and on single-GPU hosts (the common commodity case) the engine adds zero behaviour. Tests must assert: a running pipeline is never fragmented unless no single GPU fits; a split is taken only after degrade-to-fit fails and its copy is cost-accounted; a transient spike never triggers a migration while a sustained overload does; a migration is IDR-aligned make-before-break and drops no output frame (chaos/soak); anti-storm damps bound migration frequency; a pin always wins; a blind-vendor probe falls back to VRAM + overall-busy + cost-model without blocking placement; and the controller can never stall the engine (chaos gate).
