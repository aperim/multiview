# ADR-M008: Managed-device registry and compiled-in driver model

- **Status:** Proposed
- **Area:** Management
- **Date:** 2026-06-10
- **Source brief:** [managed-devices.md](../research/managed-devices.md)

## Decision

Add a vendor-neutral **Devices domain** to `multiview-control` (`devices/` module): a registry of operator-adopted hardware — encoder/decoder appliances, our own display nodes, cast targets — with a per-device state machine `DISCOVERED → ADOPTING → ONLINE / DEGRADED / AUTH_FAILED / UNREACHABLE`. The registry reuses the implemented generic versioned resource store (`crates/multiview-control/src/resource_store.rs`) by adding `DeviceKind`/`SyncGroupKind` markers, so devices and sync groups inherit the monotonic `Version` → `ETag`/`If-Match` → 412 concurrency model and the existing CRUD route shape unchanged. Drivers are a **closed `#[non_exhaustive]` enum compiled into the binary** — initially `zowietek`, `displaynode`, `cast` — each declaring fixed probed capability flags `{encode, decode, display, sync: frame-accurate|offset-only|none, audio, reboot, firmware_update}`; a new device family means a new variant + driver module + ADR, never a plugin. A device is never itself a Source or Output: it *projects* source candidates, output targets, and display heads (ADR-M009), and binding a projection creates an **ordinary** managed Source/Output carrying a `device_ref`.

Durability is **config-as-code**: `Device` and `SyncGroup` types live in `multiview-config` (`[[devices]]`/`[[sync_groups]]` TOML), the in-memory stores are seeded from loaded config exactly as sources/outputs are today (`seed_resources` in `crates/multiview-control/src/state.rs`), credentials are write-only `secret_ref`s (export offers ref|redact only), and runtime state — firmware, temperature, achieved skew, discovered-but-unadopted inventory, ad-hoc cast sessions — is never exported. Driver network I/O sits behind **off-by-default per-driver Cargo features** (umbrella `devices-net`), with the full model and axum handlers always compiled and tested socket-free via `tower::oneshot` — the proven posture of the shipped `nmos`/`router` modules, whose features gate only the socket seam. mDNS/DNS-SD discovery is **new shared infrastructure** (no working mDNS exists in the repo today; the NMOS transport is a compile-only seam): built once behind the network feature and shared by `zowietek`/`cast`/`displaynode` browsing, with discovery results treated as **untrusted inventory requiring explicit confirm-adopt** ([ADR-0041](ADR-0041.md) doctrine — candidates are hints, never auto-ingested, bounded and rate-limited). Every driver is a **control-plane-only poller/actor** publishing into watch/broadcast channels; the engine only ever sees ordinary Sources and Outputs, so invariant #10 holds by construction — the same proof shape as the NMOS module.

## Rationale

Three device families arrive at once and peer vendors publish equivalent public HTTP control APIs, so the generic registry is justified day one; reusing the tested store/route/feature-gating machinery means the new domain inherits concurrency, audit, and CI-green-default (socket-free) behaviour instead of re-deciding them; and a closed typed enum keeps capabilities statically checkable and exhaustively matchable (repo typing policy: no `dyn Any`, no open registries) while keeping each vendor driver small, isolated, and removable.

## Alternatives considered

Plugin / `dyn`-trait driver registry (rejected: hot-pluggable third-party drivers add an ABI, test, and security surface no current requirement needs, and open registries defeat the exhaustive-match typing the repo mandates — a new compiled variant per family is cheap and reviewable); generic property-bag capabilities (rejected: untyped key/value capability sets cannot gate the UI or validator exhaustively and drift from real device behaviour — fixed probed flags are the single-machine-readable-gate lesson of ADR-M007); devices as a new engine stage (rejected: devices are management-plane peripherals; placing vendor-HTTP pollers in the engine couples third-party latency and failure to the data plane and puts invariants #1/#10 at risk for zero media-path benefit); SQLite as the durable source of truth (rejected: sources/outputs are config-seeded today and the export story must stay single-sourced for a GitOps-able node fleet — SQLite remains the off-by-default persistence layer per [ADR-W006](ADR-W006.md), never the source).

## Consequences

A new device family always costs a code change plus an ADR — there is deliberately no runtime extensibility, and third parties cannot ship drivers without forking. mDNS discovery is genuinely new network infrastructure with its own security surface; it stays feature-gated, its results stay untrusted-until-adopted, and the default build remains pure-Rust and socket-free. Device credentials impose end-to-end secrets discipline: write-only `secret_ref`s never echoed by the API, export ref|redact only, enrollment tokens shown once and stored hashed. Because the stores are config-seeded and in-memory, runtime adoptions are durable only via config export — consistent with sources/outputs today, riding the same persistence path as the rest of the control plane. Finally, socket-free tests cannot catch vendor doc-vs-firmware drift (observed for the ZowieTek family), so each driver carries a mandatory on-hardware validation item before its typed client is considered verified.
