# ADR-0065: UniFi Protect compatibility — present outputs as adoptable cameras over ONVIF/RTSP + a clean-room nominative adoption shim

- **Status:** Proposed
- **Area:** Output / Control (ONVIF-server facet) / Web
- **Date:** 2026-06-13
- **Source brief:** [unifi-protect-compat.md](../research/unifi-protect-compat.md)
- **Builds on:** [ADR-0006](ADR-0006.md) (in-process RTSP serving — the
  `crates/multiview-output/src/rtsp_server/` seam), the ONVIF Profile S server
  facet (ADR-0063, owned by [onvif-and-ptz.md](../research/onvif-and-ptz.md)),
  [ADR-0037](ADR-0037.md) (best-effort sinks structurally incapable of
  back-pressuring the engine).
- **Relates to:** [managed-devices.md §12](../research/managed-devices.md) /
  ADR-M008..M011 (the clean-room nominative vendor posture this mirrors),
  [ADR-0041](ADR-0041.md) (untrusted-inventory confirm-adopt, applied outbound to

  invariants #1 (output clock), #7 (encode-once-mux-many), #10 (isolation).

## Context

The operator requests "full emulation / compatibility for Multiview outputs to be used by a UniFi
Protect NVR." Verified facts (web-verified 2026-06-13, see the brief §2/§6):

- **UniFi Protect added third-party ONVIF/RTSP camera adoption in the Protect 5.0 application
  generation (mid-2024); 6.x keeps the same path and constraints.** Adoption uses standard ONVIF:
  same-VLAN auto-discovery (or "advanced adoption" with the ONVIF service port appended, the `:8000`
  convention), **Digest + WS-UsernameToken** auth, a correct camera clock/timezone, auto-detect of the
  highest/lowest media profiles (operator cannot choose the mapping), and ideally a working
  `GetSnapshotUri`.
- **Third-party ONVIF cameras get continuous recording + live view only** — **no** Protect smart/AI
  detections, motion-event parity, or two-way audio (those stay UniFi-native). So "used by a Protect
  NVR" realistically means *recorded and viewed*.
- **The load-bearing pieces are existing design dependencies (expected seams, to verify before
  implementation):** an in-process RTSP server (`crates/multiview-output/src/rtsp_server/`:
  `RtspServerSink`, `RtspMount`, `RtspServerHandle::start`) feeding encode-once-mux-many fan-out
  (`crates/multiview-output/src/fanout.rs`), a JPEG still seam
  (`crates/multiview-control/src/routes/preview.rs`), and a control-plane server isolation precedent
  (`crates/multiview-control/src/nmos/mod.rs`). Exact module paths/symbols are confirmed at
  implementation time, not asserted here as present-tree status.
- **UniFi-native cameras (G-series / AI-port)** adopt over a **proprietary, undocumented,
  certificate/ToS-bound UniFi inform protocol**, not ONVIF. Emulating it would require reverse-engineering
  a closed protocol.

The question this ADR answers: how should Multiview make its outputs adoptable by a UniFi Protect NVR
without inventing a vendor emulator and without violating the clean-room / vendor-neutral posture?

## Decision

**Scope, stated plainly: "full UniFi Protect compatibility" in the native-camera sense is NOT
achievable under the clean-room / vendor-neutral constraint and is explicitly not accepted as the
deliverable.** Native Protect parity (smart/AI detections, motion-event parity, two-way audio, native
adoption) rides a proprietary, undocumented, certificate/ToS-bound UniFi inform protocol; reproducing
it would require reverse-engineering a closed protocol. The accepted, served request is therefore
**third-party ONVIF camera adoption for recording + live view** — exactly what Protect grants any
third-party ONVIF camera.

**Within that scope, satisfy the request entirely through the open ONVIF/RTSP standard plus a thin,
named compatibility profile — no emulation of any Ubiquiti protocol — and defer the proprietary native
path to the operator.**

1. **Reuse, don't rebuild.** The adoptable-camera surface is the **generic ONVIF Profile S server facet**
   (ADR-0063, owned by [onvif-and-ptz.md](../research/onvif-and-ptz.md)) wrapping the existing
   in-process RTSP server ([ADR-0006](ADR-0006.md)). This ADR adds **only** the Protect-specific
   compatibility profile + interop matrix on top; the ONVIF SOAP/WS-Discovery server, `GetProfiles`/
   `GetStreamUri`/`GetSnapshotUri`/`GetSystemDateAndTime`, and Digest + WS-UsernameToken auth are
   ADR-0063's deliverables.

2. **Outputs project outward as camera channels (Devices model, inverted).** A new additive config axis
   `outputs.onvif_camera { enabled, service_port=8000, profile, main_rendition, sub_rendition, auth }`
   offers a chosen Multiview output (program, clean feed, a cropped tile, or a wall head) as an ONVIF
   camera with **two media profiles** (main/sub → two existing renditions) so Protect's high/low
   auto-detect maps cleanly. Mapping renditions that already exist is **free fan-out (inv #7)**; config
   validation must require that the named renditions exist. If a named rendition does not exist it is a
   new sub/main encode — **not** free — and must enter the normal admission/resource budget (inv #9),
   never be silently spun up by the ONVIF facet. An output is never itself reconfigured into a Source.

3. **The `unifi-protect` compatibility profile is the only Protect-specific code.** It constrains the
   generic ONVIF responses to Protect's known-accepted shapes (all behavioral, derived from observable
   adoption behavior): exactly two distinct main/sub media profiles, a guaranteed `GetSnapshotUri`
   served from a **sampled, cache-based, rate-limited** per-camera still (never a synchronous on-demand
   encode that could touch the output-clock loop),
   **H.264** main-profile default, RTSP-interleaved-over-TCP as the offered transport, correct WS-Security
   timestamps, and conservative capability advertisement (never advertise analytics/PTZ the output can't
   honor). It is a profile object + a held interop test matrix run against a real Protect controller —
   not a protocol emulator.

4. **The proprietary native (G-series / AI-port) path is a documented non-goal, deferred to the
   operator.** It is described in the brief strictly at the behavioral level (§4.3), with legal/ToS
   caveats; Multiview does **not** implement, emulate, capture, or reverse-engineer it by default.
   Whether to ever pursue it is an explicit operator decision.

5. **Clean-room nominative posture (the managed-devices model).** Build on ONVIF/RTSP/RTP/SDP only; no
   Ubiquiti SDK, spec text, firmware, or protocol capture is committed or redistributed. Nominative
   trademark use only — *"Compatible with UniFi Protect (third-party ONVIF camera adoption); UniFi and
   UniFi Protect are trademarks of Ubiquiti Inc.; this project is independent and not affiliated with or
   endorsed by Ubiquiti."* No badge, no certification claim. The facet is removable; interop is pinned to
   tested Protect builds. Folds into the pending counsel review.

## Consequences

**Positive.**
- The operator's goal (a Multiview output recorded/viewed by a Protect NVR) is met with **near-zero new
  protocol code** — it is the ADR-0063 ONVIF facet + a small profile + tests. Honest scoping, no
  over-promised "emulation."
- Encode-once-mux-many preserved (inv #7): a Protect controller is just another RTSP puller of an
  existing rendition; offering main/sub costs nothing when the renditions already exist.
- Structurally isolated (inv #1/#10): no Protect/RTSP/SOAP path touches the output-clock loop.

**Negative / costs.**
- Net-new (in ADR-0063, consumed here): the ONVIF SOAP server, WS-Discovery responder, WS-Security auth,
  and per-output media-profile description. This ADR adds the `outputs.onvif_camera` config axis, the
  `unifi-protect` profile, possibly a per-output JPEG snapshot tap (open question), and the interop matrix.
- Protect third-party limitations (no AI/smart detections, no two-way audio, no operator profile choice)
  are **vendor-imposed and unfixable from our side** — the docs must set expectations plainly.
- Interop is brittle across Protect releases (third-party support is revocable/changeable); the profile
  must stay a removable, version-pinned facet.

**Neutral.**
- Offering/withdrawing a camera is **Class-1** (toggles a best-effort egress facet; the program output is
  untouched), surfaced in the capability matrix.
- WS-Discovery defaults **off** (manual advanced-adoption), IPv6-first with legacy-IPv4 multicast as the
  interop fallback ([ADR-0042](ADR-0042.md)) — outbound application of [ADR-0041](ADR-0041.md)'s
  untrusted-inventory caution.

## Alternatives considered

1. **Emulate the proprietary UniFi native (G-series/AI-port) adoption protocol** so outputs appear as
   native cameras and unlock smart detections. *Rejected (default).* Undocumented, certificate/ToS-bound,
   requires reverse-engineering a closed protocol — violates the clean-room posture and yields fragile,
   legally exposed interop for a feature the open path already covers for recording/live-view. Behavioral
   description only; deferred to the operator.
2. **Build a bespoke "Protect emulator" service.** *Rejected.* There is no Protect-proprietary handshake
   in the ONVIF adoption path; a separate emulator would duplicate the generic ONVIF facet. The honest
   design is "generic ONVIF + a thin Protect profile."
3. **Ship only the generic ONVIF facet, no Protect profile.** *Rejected as insufficient.* Protect's
   adoption flow has specific, observable expectations (two-profile main/sub, snapshot, auth mode, clock,
   conservative capabilities); without a pinned compatibility profile + interop tests, adoption is
   hit-or-miss. The profile is small but load-bearing.
4. **Make the output itself an ONVIF Source/device record.** *Rejected.* An output is media egress, not a
   device; projecting it outward as a camera channel (the inverted Devices model) keeps the engine and the
   Source/Output graph untouched.
5. **Advertise HEVC main profile by default for efficiency.** *Rejected for v1.* H.264 main profile is the
   broadest-compatibility floor for NVR adoption; HEVC is an interop-test axis, measured not promised.

## How this honors the invariants

- **Invariant #1 (output clock never stalls).** A UniFi Protect NVR is an RTSP *client* of the existing
  in-process server; it pulls frames the engine already produced (`out_pts = f(tick)` is upstream of and
  oblivious to any RTSP session). The ONVIF SOAP/WS-Discovery server is a control-plane HTTP server (the
  NMOS-module shape, `crates/multiview-control/src/nmos/mod.rs`) with no code path to the tick loop. A
  Protect controller that hangs, disconnects, or hammers re-adoption can at worst stall its own session;
  the program output never falters.
- **Invariant #10 (no back-pressure).** RTSP egress uses the existing `BoundedPacketQueue` **drop-oldest**
  fan-out (`crates/multiview-output/src/rtsp_server/`): a slow/greedy or duplicated Protect session drops
  frames on its own queue and never blocks the bake-consumer or the engine. The ONVIF server reads/writes
  only control-plane state via watch/broadcast channels; the engine never `.await`s it. The CI chaos gate
  that already guards #10 over the RTSP/NMOS channels covers these new outbound facets the same way.
  Beyond per-session isolation, the facet carries explicit **global** caps (max concurrent RTSP sessions
  overall and per camera, bounded accept queue, SOAP/WS-Discovery rate limits, auth throttling, fan-out +
  snapshot memory/fd budget) so an adoption loop or many NVRs exhaust only the control-plane caps and are
  rejected there — never stalling an existing session or the engine.
