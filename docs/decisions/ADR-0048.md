# ADR-0048 — WebRTC transport endpoint: the `multiview-webrtc` crate

**Status:** Proposed (2026-06-10); amended 2026-06-13 (TURN added — §5.1, operator hard requirement)
**Area:** cross-cutting — WebRTC transport (ingest + preview + output egress share one stack)
**Drivers:** "full WebRTC support" lands three consumers of one native ICE/DTLS/SRTP stack at once —
WHIP ingest ([ADR-T014](ADR-T014.md)), WHEP preview completion ([ADR-P006](ADR-P006.md)), and the
`webrtc`/`whip_push` program outputs ([ADR-0049](ADR-0049.md)). Without a single pinned owner, each
crate grows its own socket, certificate, and driver loop — three ports to firewall, three
invariant-#10 surfaces to audit, three copies of the same session lifecycle bugs.
**Research:** [webrtc](../research/webrtc.md) (the subsystem brief),
[preview-subsystem](../research/preview-subsystem.md).
**Related:** [ADR-T014](ADR-T014.md) (WHIP ingest semantics + media path),
[ADR-0049](ADR-0049.md) (WebRTC program outputs), [ADR-P006](ADR-P006.md) (WHEP preview
completion), [ADR-W023](ADR-W023.md) (SPA player + fallback), [ADR-0042](ADR-0042.md) (IPv6-first —
binds, candidates, SDP), [ADR-T013](ADR-T013.md) (RTP-audio rebase seam the ingest side feeds),
[ADR-P001](ADR-P001.md) / [ADR-RT004](ADR-RT004.md) (isolation, inv #10),
[conventions §3/§4](../architecture/conventions.md) (crate + feature canon — updated by this ADR).

## Context

Native WebRTC transport exists in the tree today only as the **preview** scaffold. `multiview-preview`
pins the sans-IO stack `str0m 0.16.2` (`crates/multiview-preview/Cargo.toml:42`, optional,
`default-features = false, features = ["rust-crypto"]`) behind its `webrtc-native` feature (PRV-1b),
and `whep/native.rs` wraps it as a `WhepTransport`: real SDP offer→answer with real ICE/DTLS
attributes, CI-runnable socket-free, plus a loopback-bound variant
(`Str0mWhepTransport::bind_loopback`, `whep/native.rs:236`). Meanwhile `multiview-input` carries the
**pure** ingest half — the [`MediaEngine`] seam (`crates/multiview-input/src/webrtc/transport.rs:201`)
that a real ICE/DTLS/SRTP engine must drive, with no concrete engine in-tree.

Three defects/gaps make the status quo untenable as WHIP ingest and WebRTC outputs land:

- **No shared endpoint.** Preview's transport binds its own loopback socket per transport; ingest
  ([ADR-T014](ADR-T014.md)) and the WHIP push client ([ADR-0049](ADR-0049.md)) would each need their
  own socket + certificate + driver, multiplying ports, fingerprints, and isolation audits.
- **An unbounded tombstone map.** `Str0mWhepTransport::close` releases the `Rtc` and media feed but
  *retains the session entry forever* as a closed tombstone so the WHEP `DELETE` stays idempotent
  (`whep/native.rs:419`). Every session ever opened leaks a map entry — fine for a scaffold,
  unacceptable for a 24/7 endpoint (bounded memory — CLAUDE.md safety rule 5 / invariant #9's
  bounded-queues clause).
- **IPv4 placeholders in the pure answer.** The scaffold's hand-rolled answer SDP emits
  `c=IN IP4 0.0.0.0` / `o=… IN IP4 0.0.0.0` (`crates/multiview-preview/src/whep.rs:275/289/306`) —
  the exact defect class [ADR-0042](ADR-0042.md) catalogued — and lacks `a=mid`/`a=rtcp-mux` lines, so
  it is not honest SDP even as a test fixture.

The wider scope (WHIP/WHEP HTTP semantics, media paths, encoders, SPA) is owned by the sibling ADRs;
**this ADR pins the shared transport core**: which stack, which crate, one endpoint model, the
driver/isolation contract, session GC, the config schema, and the feature/licensing ladder.

## Decision

### 1. One new leaf crate: `crates/multiview-webrtc`

A new crate `multiview-webrtc` (lib target `multiview_webrtc`) owns **all** native WebRTC transport
in the workspace — ICE, DTLS, SRTP, the UDP socket, the session map, and the driver task. House
pattern for native code applies:

- `[features] default = []` — the default build is an **empty pure shell** so
  `cargo check --workspace` stays GPU/native-free and deny-clean; `native = ["dep:str0m",
  "tokio/net", …]` enables the real stack.
- Dependencies: `multiview-core`, `multiview-preview` (it implements the pure
  `WhepTransport`/`PreviewMediaSource` seam), `multiview-input` (it drives the [`MediaEngine`] seam),
  `tokio`, `thiserror`, `tracing`. **No `multiview-ffmpeg`, no `multiview-config`** — encode/decode
  stay in the cli adapters ([ADR-P006](ADR-P006.md)/[ADR-T014](ADR-T014.md)), and the cli maps the
  config document to a plain `EndpointConfig` so the crate never grows a config dependency.
- Dependency direction stays acyclic: `webrtc → {preview, input, core}`; `cli → webrtc`. The crate is
  a leaf; nothing media-plane depends on it.

### 2. Relocate preview's native transport into the new crate (PRV-1b preserved)

`multiview-preview`'s `webrtc-native` feature, its optional `str0m` dependency, and `whep/native.rs`
**move** into `multiview-webrtc` (tests preserved/moved; `deny.toml` comments and the work-schedule
PRV-1b note updated — *completed by relocation*, not re-done). Preview keeps the **pure** `webrtc`
seam untouched (`whep.rs`, `whep/transport.rs`, `whep/program.rs`): the trait, the session/token
model, and the fake transport stay where its consumers (control routes) already look. Rationale:
exactly **one str0m owner** in the workspace, because ingest sessions, preview/output egress
sessions, and the push client must share the socket, the certificate, and the driver (§3–§6).
[conventions §3](../architecture/conventions.md) gains the crate row; §4 records the canonical
features and notes preview's `webrtc-native` flag as **removed** (migration note).

### 3. Stack: str0m, pinned `=0.16.2`; any bump beyond 0.16.x is a tracked follow-on

`str0m` (sans-IO, `MIT OR Apache-2.0`) at **`=0.16.2`**, `default-features = false,
features = ["rust-crypto"]` — a **tightening** of `multiview-preview`'s caret `"0.16.2"`
requirement (lockfile-resolved to 0.16.2) to an exact pin, with the same feature shape,
so the licence/advisory closure is **already allowlisted** in `deny.toml` (lines 41–48; note the
0.16.x line pulls `aws-lc-rs` via `dimpl`, not `ring`). Its MSRV (1.81) sits under the workspace MSRV
(1.82, root `Cargo.toml:8`). Known gaps at 0.16.2, stated rather than hidden:

- The **Firefox RTX-SSRC frame-drop fix landed in str0m 0.18.0** — at 0.16.2 Firefox viewers may
  exercise that bug. Mitigation: Firefox is a mandatory hardware-validation target (real browser via
  Playwright on the test box) before this ships as done.
- The **Chrome v144 ICE fix is in 0.16.1** — covered by this pin.
- Picking up **any fix beyond 0.16.x** requires **MSRV 1.85 + Edition 2024** (the requirement
  begins at str0m 0.17.0); that is a workspace-wide toolchain decision, recorded here as a tracked
  follow-on, not smuggled in with this change.

### 4. Endpoint model: one process-wide endpoint, one dual-stack socket, ufrag demux

One `WebRtcEndpoint` per process (feature `native`), owning a **single dual-stack UDP socket** bound
`[::]:port` with `IPV6_V6ONLY=false` (config `webrtc.udp_port`, default **8189**) per
[ADR-0042](ADR-0042.md) — never `0.0.0.0`. **All** sessions multiplex on it: WHIP ingest publishers,
WHEP preview sessions, WHEP output viewers, and the outbound WHIP push client. Demultiplexing is the
established sans-IO single-socket pattern (str0m's `chat` example): **STUN datagrams route by ICE
ufrag** → session map; non-STUN datagrams route by the learned remote `SocketAddr`
(`Rtc::accepts()`). One UDP port to open/forward is the whole firewall story for media; signalling
rides the existing control-plane HTTP listener, with `application/sdp` request bodies capped at
**64 KiB** (an explicit axum body limit on the signalling routes).

> **Amendment 2026-06-15 — feed str0m the CONCRETE local destination, never `[::]`
> (box-validation defect #3).** The socket binds the *unspecified* address `[::]`, but str0m matches
> every inbound STUN binding-request's `destination` against its gathered **local candidates** — the
> concrete `advertised_addresses` (host candidates) + relayed candidates, never `[::]` (str0m 0.16.2
> `ice/agent.rs`: `local_candidates … v.addr() == req.destination`). Feeding str0m the unspecified
> bind addr as each datagram's local destination made it discard **every** inbound STUN check
> ("Discarding STUN request on unknown interface: [::]:PORT") → "no possible pairs" → ICE never
> reached Connected → zero frames, on **both** the host and relay candidate paths. The driver
> therefore recovers the **concrete** local destination of each datagram via `recvmsg(2)` +
> `IPV6_PKTINFO`/`IP_PKTINFO` and resolves it onto a gathered candidate (exact match → same-IP-family
> match for NAT 1:1 / Docker, where PKTINFO reports the private interface IP while str0m only knows
> the public advertised candidate → first concrete candidate). A **relay-unwrapped** datagram is fed
> as arriving on the **relay candidate** address (so the relay pair validates), never `[::]` and never
> the host addr. The `recvmsg` cmsg parsing is the crate's only `unsafe` (`native`+`unix`, `// SAFETY:`
> per block); the receive loop stays bounded and non-blocking (inv #1/#10).

### 5. Full ICE, not ice-lite; IPv6-first candidates + advertised addresses

The endpoint runs str0m's **in-library full-ICE agent** — not ice-lite. Lite implementations of
pion lineage have documented interop gaps, full ICE is the production consensus among self-hosted
WHIP/WHEP servers, and str0m gives us the full agent for free (no extra dependency). Because the
WHIP/WHEP specs forbid server→client trickle, the server **gathers all candidates before
answering**: host candidates for every reachable address — **IPv6 first** in candidate order
([ADR-0042](ADR-0042.md)) — plus the config list `webrtc.advertised_addresses` for NAT 1:1 / Docker
deployments (the additional-hosts pattern self-hosted WHIP servers use), **plus relayed candidates
from any configured TURN server** (see §5.1).

> **Amendment 2026-06-13 — TURN added (operator hard requirement).** This ADR was authored proposing
> **no TURN/STUN client** for the self-hosted posture. The operator subsequently made TURN a **hard
> requirement** ("the WHEP is WebRTC / WHEP over TURN enabled as well") so a Multiview behind a
> symmetric NAT / restrictive firewall can still establish media when host + advertised candidates
> fail. The "no TURN/STUN client" boundary below is therefore **superseded by §5.1**; the crate ships
> an in-crate TURN client. The original boundary text is retained struck through for history.

### 5.1 In-crate sans-IO TURN client (Amendment 2026-06-13)

str0m 0.16.2 is a **full-ICE** agent but ships **no TURN client**: it can *consume* a relay candidate
(`str0m::Candidate::relayed`) but cannot *allocate* one, and its STUN attribute model is private with
no `REQUESTED-TRANSPORT` setter (mandatory for `Allocate`). Because `multiview-webrtc` already owns
the UDP socket beneath str0m's sans-IO core (§4, the `native` transport), the TURN client lives in
that socket-driving layer, **in-crate**:

- **Standards.** A pure STUN/TURN message codec and allocation state machine implementing
  **RFC 5389** (STUN: header, `MESSAGE-INTEGRITY` HMAC-SHA1, `FINGERPRINT` CRC-32, `XOR-MAPPED-ADDRESS`),
  **RFC 5766** and its modern refresh **RFC 8656** (TURN: the 401/`REALM`+`NONCE` long-term-credential
  challenge, `Allocate` / `Refresh` / `CreatePermission` / `ChannelBind`, `Send`/`Data` indications and
  `ChannelData`, `XOR-RELAYED-ADDRESS` / `XOR-PEER-ADDRESS`, `REQUESTED-TRANSPORT`, `LIFETIME`). The
  client learns the relayed transport address from the `Allocate` success and presents it to str0m as
  a relay candidate (§5); relayed media is wrapped/unwrapped transparently beneath str0m.
- **Credential model.** Two schemes, surfaced in config as `webrtc.ice_servers` entries
  (`kind = "turn"` with `credentials`): **long-term** (RFC 5766 §4 static `username`/`password`,
  optional `realm` learned from the 401) and **ephemeral REST / HMAC** — the coturn `use-auth-secret`
  scheme (`draft-uberti-behave-turn-rest-00`): a time-limited `username` of `"<unix-expiry>:<name>"`
  with the password derived `base64(HMAC-SHA1(static_auth_secret, username))`, resolved per allocation
  with a fresh expiry. Secrets are **redactor-named** — the credential fields are `password` and
  `static_auth_secret`, which the control-plane config redactor (`multiview-control::redact_config`)
  drops by key; both `TurnCredential` and `TurnCredentials` carry a `Debug` impl printing
  `<redacted>`. The `username` is not secret (matching coturn's posture).
- **IPv6-first.** TURN runs over IPv6 where the server is reachable over v6 (the ICE-server
  `SocketAddr` may be v4 or v6), and the codec handles `XOR-…-ADDRESS` for **both** address families
  (family `0x01` v4, `0x02` v6); relayed candidates follow the IPv6-first candidate order of §5
  ([ADR-0042](ADR-0042.md)).
- **Pure & offline-testable.** The client is **sans-IO**: it owns no socket and uses no async — it is
  driven by `poll_output()` (emitting `Transmit { destination, payload }` / `Timeout` / `Idle`) and
  `handle_input(datagram, now)` (emitting `Allocated` / `PermissionCreated` / `Data`), exactly the
  drive shape of the rest of the endpoint (§7). The codec and state machine live in the **default
  build** (pure-Rust `hmac`/`sha1`/`md5`/`base64`/`rand`, already in `deny.toml`); only wiring the
  relayed candidate into str0m's socket loop is `native`. Tests pump the client against an
  **in-process fake TURN server** (`tests/fake_turn.rs`) entirely in memory — no socket, no network,
  no str0m, no live coturn — so the full `Allocate`→`CreatePermission`→`Send`/`Data` path is CI-gated
  offline.

Invariant #10 is preserved: the TURN client runs inside the one endpoint driver task (§7); a wedged
TURN server costs *relay reachability for that session only* (the allocation fails or expires), never
an output tick. TURN is purely a reachability mechanism for the existing media sessions.

### 6. One per-run `DtlsCert`

One self-signed DTLS certificate (rcgen via str0m) is minted at endpoint start and shared by every
session — a stable fingerprint per run, **no persistence**. Sessions are per-run by construction
(no session survives a restart), so persisting a certificate would add a private key on disk for
zero benefit.

### 7. Driver task model + bounded channels (invariant #10 analysis)

**One** tokio task owns the socket and all `Rtc` instances: `recv` → `Rtc::handle_input`; then drain
`poll_output()` **fully** — `Transmit` → socket send, `Event` → dispatch, `Timeout` → sleep — the
canonical sans-IO drive loop (the scaffold's shape at `whep/native.rs:303` generalised). Session
registration/teardown crosses a **bounded** command channel; per-session media ingress/egress crosses
**only drop-oldest rings** (the existing `SampleFeed` on the egress side; an `RtpFrame` ring on the
ingest side feeding the [`MediaEngine`] pull). Invariant #10 holds by construction:

- The engine **never awaits** the endpoint task, never sends on a channel it can fill, and holds no
  lock the endpoint holds. A wedged or saturated endpoint loses *preview/ingest media* (rings drop
  oldest), never output ticks.
- A slow/hostile peer back-pressures only its own session's rings; the socket task never blocks on a
  peer (UDP send is non-blocking; a full ring drops).
- Invariant #1 is untouched: nothing here is on the tick path; ingest media is *sampled* through the
  existing keyframe-gated, bounded depacketizer chain ([ADR-T014](ADR-T014.md)).

### 8. Session GC — the tombstone leak is fixed, not inherited

Sessions die two ways: **ICE-disconnect** (str0m surfaces it) or **idle timeout** (config
`webrtc.session_idle_timeout`, default 30 s, no media/STUN activity). Either closes the session
(releases `Rtc` + feeds, exactly as today's `close` does) and leaves a closed tombstone so the WHEP
`DELETE`/status path stays idempotent — but tombstones are now **evicted after 60 s**, replacing the
scaffold's retain-forever map (`crates/multiview-preview/src/whep/native.rs:419`).
`webrtc.max_sessions` (default **64**) caps **preview + output-viewer sessions only**; WHIP ingest
sessions (bounded by the count of configured `webrtc` sources) and the `whip_push` client session
(bounded by configured outputs) are admitted **outside** that pool, so a viewer flood can never
starve a publisher or the push client's reconnect (per-role caps such as `max_viewers` belong to
[ADR-0049](ADR-0049.md)/[ADR-P006](ADR-P006.md)). The session map is therefore bounded by that pool
plus the config-bounded ingest/push sessions plus a 60-second tombstone tail — bounded memory
restored (CLAUDE.md safety rule 5 / invariant #9's bounded-queues clause). Session ids are
**≥128-bit random** (`rand` from the OS RNG, hex/base64url-encoded), never sequential — the
relocated scaffold's sequential minting is replaced — and `DELETE` on a session resource requires
the **same credential class** as the `POST` that created it.

### 9. Config schema: the `webrtc` section

`MultiviewConfig` (`crates/multiview-config/src/lib.rs`) gains an optional top-level `webrtc`
section (absent ⇒ defaults; the endpoint only starts when a `webrtc`-native build has a consumer):

```toml
[webrtc]
udp_port = 8189                      # the single media socket, bound [::] dual-stack
advertised_addresses = ["2001:db8::15"]  # NAT 1:1 / Docker; appended to gathered host candidates
max_sessions = 64                    # preview + output-viewer sessions only (§8); ingest + push sit outside this pool
session_idle_timeout = "30s"         # no media/STUN → close; tombstone evicted 60 s later
cors_allow_origins = ["*"]           # CORS for the media-signalling routes only; default "*"
```

The cli maps this to the crate's plain `EndpointConfig` — `multiview-webrtc` never depends on
`multiview-config` (§1). `cors_allow_origins` applies **only** to the media-signalling routes
(WHIP, WHEP, preview-WHEP, capabilities), with `Access-Control-Allow-Headers: authorization,
content-type` and `Access-Control-Expose-Headers: location, link`; preflight `OPTIONS` is
unauthenticated by browser construction.

### 10. WHEP egress sessions: str0m's own answer; honest SDP in the fake path

Egress signalling stops hand-rolling SDP: the answer returned to a WHEP client is **str0m's own
complete answer** (correct BUNDLE, `a=mid`, `a=rtcp-mux`, fmtp), folded through the existing pure
session model. The pure `WhepSession::build_answer` scaffold (`whep.rs:166`) **stays** for the
fake-transport tests, but its `c=IN IP4 0.0.0.0` placeholders become `c=IN IP6 ::`
([ADR-0042](ADR-0042.md)) and it gains `a=mid`/`a=rtcp-mux` lines so the fake path is honest SDP too.
Media egress uses str0m's **sample API** (`Writer::write`; str0m packetizes H.264/VP8/Opus), and
`Event::KeyframeRequest` (PLI/FIR) is **rate-limited and coalesced (≥2 s floor)** into a force-IDR
callback toward the feeding encoder — a viewer storm cannot inflate keyframe rate unboundedly
(inv #10). To carry program audio, the seam grows: `EncodedSample` gains
`kind: SampleKind { Video, Audio }` with per-kind RTP clocks (video 90 kHz, audio 48 kHz), and
`PreviewMediaSource` gains an optional Opus audio feed — a pre-1.0 seam change with the in-memory
fake and **all in-repo consumers updated in the same push** (details in [ADR-P006](ADR-P006.md)).

### 11. Feature ladder + licensing

- Crate: `native` (off by default) is the only feature; everything native sits behind it.
- cli: `webrtc = ["multiview-preview/webrtc", "multiview-input/webrtc"]` (pure seams only — both
  already exist and add no native deps);
  `webrtc-native = ["webrtc", "dep:multiview-webrtc", "multiview-webrtc/native", "ffmpeg"]`.
- Presets `nvidia`, `apple`, `linux-vaapi`, `full` (`crates/multiview-cli/Cargo.toml`) all gain
  `webrtc-native` — the deployed image serves WebRTC preview out of the box. The **default** build
  stays pure: str0m only ever rides behind `native`, and cargo-deny (which scans default features
  only) sees the same str0m-free default closure as today.
- CI: the `feature-clippy` matrix gains `multiview-webrtc/native` and cli `webrtc-native` legs;
  `multiview-webrtc` tests run under `--features native` (str0m-as-its-own-test-peer over UDP
  loopback needs nothing a CI runner lacks). CI also gains a targeted **`cargo deny check` leg
  running with the cli `webrtc-native` feature graph** (not `--all-features`), so the shipped
  str0m/dimpl/aws-lc closure is tool-gated, not comment-gated. The builder image needs `cmake`
  (aws-lc-rs).
- Licensing is unchanged: str0m and its rust-crypto closure are `MIT OR Apache-2.0`-compatible and
  already enumerated in `deny.toml`; nothing here touches the GPL or proprietary boundaries.

### 12. Efficiency budget (standing review)

- **Memory:** per **egress** session ≤2 MiB bounded (str0m's internal buffers + two drop-oldest
  feeds); per **ingest** session ≤2 MiB transport + up to 8 MiB in-progress access unit
  (`MAX_ACCESS_UNIT_BYTES`) + decoder surfaces. Worst-case pool ≈ `max_sessions` × 2 MiB for
  viewers, plus the config-bounded ingest sessions (≤10 MiB + decoder surfaces each) and one
  ≤2 MiB push session per configured output (§8); tombstones are id+state only and evicted (§8).
- **CPU:** the endpoint task is I/O-bound on one socket; the per-datagram path allocates nothing
  beyond str0m's own. WHEP fan-out is **header-only packetization per viewer** of the same encoded
  bytes — no per-viewer encode (inv #7; the encoder budget lives in
  [ADR-P006](ADR-P006.md)/[ADR-0049](ADR-0049.md)). Aggregate driver ceiling, worst case:
  ≈ `max_sessions` × program bitrate — e.g. 64 × 8 Mbps ≈ 0.5 Gbps, ~50 kpps — serialized on the
  one driver task; a modern core has headroom for that, and if the task ever saturates, viewer
  sessions shed first (the preview rungs) while real outputs and ingest stay prioritized by the §8
  pool split.
- **Degradation:** preview sessions ride the existing topmost preview rungs and are FocusGate-capped;
  real outputs are never shed by the preview ladder ([ADR-0049](ADR-0049.md)).

## Consequences

- **One audited transport.** Socket, certificate, ICE agent, driver loop, session GC, and demux are
  written, tested, and isolation-audited once; WHIP ingest, WHEP preview, output viewers, and the
  push client are *sessions* on it, not stacks of their own. Exactly one crate in the workspace may
  depend on str0m.
- **One UDP port** (`webrtc.udp_port`, default 8189) is the entire media firewall/forwarding story,
  IPv6-first dual-stack; `advertised_addresses` covers NAT 1:1/Docker for the common case. Where host
  + advertised candidates cannot establish (symmetric NAT / restrictive firewall), a configured TURN
  server provides relayed candidates via the in-crate TURN client (§5.1, Amendment 2026-06-13).
- **The tombstone leak dies here** (§8) — the scaffold's retain-forever map does not survive the
  relocation; bounded session memory is restored before the first long-running deployment.
- **`multiview-preview` gets simpler**, not bigger: it keeps only the pure seam; PRV-1b's tests move
  with the code. Anything still naming `multiview-preview/webrtc-native` (docs, CI, deny comments)
  is updated in the same push — the flag is removed, not aliased.
- **The 0.16.2 pin is a stated debt** with a concrete trigger: any fix pickup beyond 0.16.x (the
  Firefox RTX fix landed in str0m 0.18.0; str0m 0.17.0 onward requires MSRV 1.85 + Edition 2024)
  needs the MSRV decision. Until then, Firefox is a mandatory hardware-validation target.
- **Seam ripple is contained:** the `SampleKind` audio extension changes `EncodedSample`,
  `SampleFeed`, `PreviewMediaSource`, and the in-memory fake — all in-repo, all in the same push,
  pre-1.0 (no compatibility shim).
- Stated boundaries (not deferred work): no trickle-ICE/ICE-restart (`PATCH` →
  405 + `Allow: DELETE, OPTIONS` per [ADR-T014](ADR-T014.md) — RFC 9110 generic method semantics;
  trickle is spec-OPTIONAL and server→client trickle is forbidden), no simulcast, no data channels.
  (TURN is **in scope** as of Amendment 2026-06-13 — §5.1; the original "no TURN/STUN client" boundary
  is superseded.)

## Alternatives rejected

- **`webrtc-rs` (the Pion-lineage async stack) instead of str0m.** A much larger dependency closure
  to licence-audit and a fully async, socket-owning design that fights the single-task sans-IO
  isolation model (§7) — with str0m, *we* own the socket, the task, and every channel, which is
  precisely how invariant #10 is proven. str0m 0.16.2 is also already in-tree, pinned, and
  deny-allowlisted from PRV-1b. Rejected.
- **ice-lite.** Tempting for a server (no gathering, host-only), but lite modes in the pion lineage
  have documented interop gaps, full ICE is what production self-hosted WHIP/WHEP servers converge
  on, and str0m's full agent costs no extra dependency. Lite also degrades the NAT-1:1
  `advertised_addresses` story (a lite agent never connectivity-checks alternatives). Rejected.
- **Per-crate sockets** (preview keeps its transport; input and output grow their own). N ports to
  document/firewall, N certificates and fingerprints, N driver loops and demux maps, and N
  invariant-#10 audits — for zero benefit over ufrag demux on one socket, which is the established
  sans-IO pattern. Rejected; this is the core reason the crate exists.
- **A sidecar media server** (run an external open-source WHIP/WHEP gateway next to the binary).
  Breaks the single-binary product, adds an unmanaged auth/config/lifecycle surface, and cannot tap
  the in-process encode-once fan-out (`PacketRouter` route()+1) or the FocusGate preview encoders —
  it would force a second encode per viewer-facing stream, violating inv #7. Rejected.
- **Leave the native transport in `multiview-preview` and have ingest depend on it.** Inverts the
  dependency direction (ingest depending on the preview crate), bloats preview with non-preview
  roles, and still needs the shared-endpoint surgery — in the wrong home. Rejected in favour of the
  leaf crate both sides depend on.
- **Bump straight to str0m 0.20 now.** Picks up the Firefox/Chrome fixes but forces MSRV 1.85 on the
  whole workspace as a side effect of a transport change. Rejected: pin `=0.16.2` (already vetted),
  hardware-validate Firefox, and take the MSRV decision deliberately as a follow-on.
