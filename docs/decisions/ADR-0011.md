# ADR-0011: Cross-platform targets: Linux containers (NVIDIA + VAAPI) and macOS native universal2

- **Status:** Proposed
- **Area:** Core Engine
- **Date:** 2026-06-02
- **Source brief:** [core-engine.md](../research/core-engine.md)

## Decision

Ship Linux as multi-stage containers (an NVIDIA/CUDA variant relying on the NVIDIA Container Toolkit driver injection with NVIDIA_DRIVER_CAPABILITIES including video, and a generic/VAAPI variant via /dev/dri passthrough) and macOS as a native, signed+notarized universal2 (aarch64+x86_64) binary with inside-out-signed bundled dylibs. No Windows.

## Rationale

macOS GPU (VideoToolbox/Metal) cannot run in containers, so macOS must be native. The toolkit injects host driver libs so images stay lean and driver-agnostic, but video capability is mandatory or NVENC/NVDEC silently vanish. universal2 covers Apple Silicon + Intel Macs in one artifact.

## Alternatives considered

Single mega-image with everything (rejected: bloat, conflated driver requirements). Per-arch macOS binaries (rejected: worse UX vs universal2). Docker on macOS for GPU (rejected: impossible — Linux VM, no Metal).

## Consequences

Startup must probe HW (dlopen + throwaway sessions) and fail loudly / fall back to VAAPI/CPU; pin SDK->driver minimum (e.g. SDK 13.0 -> driver >=570). VAAPI needs dynamic render/video GID resolution. macOS needs hardened-runtime + consistent Team ID + @rpath install names + notarytool/stapler; libavformat DNS must be watchdogged.
