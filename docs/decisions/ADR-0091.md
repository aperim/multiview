# ADR-0091: Live-apply-everywhere guarantee + staging (write-but-don't-apply) + pending banner + high-risk gate

- **Status:** Proposed
- **Area:** Web/API stack ↔ control ↔ engine (live-apply, invariant #11)
- **Date:** 2026-06-13
- **Source brief:** [webui-operability-gaps.md §3](../research/webui-operability-gaps.md)
- **Extends:** [ADR-M005](ADR-M005.md) (Class-1/reset-lite/Class-2 + dry-run `/plan`),
  [ADR-M012](ADR-M012.md) §9 (per-operation classification + desired/live split),
  [ADR-W018](ADR-W018.md) / [ADR-W022](ADR-W022.md) (live source/overlay apply + `X-Multiview-Apply`
  header honesty + `LiveApplyCaps`), [ADR-R004](ADR-R004.md) / [ADR-R010](ADR-R010.md)
  (make-before-break Class-2 cutover)
- **Relates to:** [ADR-0059](ADR-0059.md) (audio control seam — un-holds `RouteAudio`),
  [ADR-M004](ADR-M004.md) (per-output field classification), [ADR-M008](ADR-M008.md) (device
  desired-state), the versioned resource store (`crates/multiview-control/src/resource_store.rs`)

## Context

The instant-apply doctrine and invariant #11 require a management change to apply instantly (Class-1
hot at a frame boundary, or Class-2 via make-before-break) and to surface *which* before applying. The
machinery is built: the honest `X-Multiview-Apply: live|restart` header computed per request from a
per-collection `LiveApplyCaps` the binary injects (`crates/multiview-control/src/live_apply.rs`,
`state.rs:622`); live source apply ([ADR-W018](ADR-W018.md)); live overlay apply
([ADR-W022](ADR-W022.md)); live routing + the `/plan` classifier
(`crates/multiview-control/src/routing.rs`, [ADR-M005](ADR-M005.md)/[ADR-M012](ADR-M012.md) §9).

Two gaps remain against the north star:

1. **Some surfaces still answer `restart` honestly** because the live seam is not built — audio routing
   (`routes/audio.rs:156`), output edits (`routes/outputs.rs:126`), devices (`routes/devices.rs:191`).
   The fix for each is its owning ADR's seam, not a header lie ([ADR-W018](ADR-W018.md) honesty bounds).
2. **There is no staging path.** The operator explicitly wants, for high-risk changes, to *write the
   change without implementing it now*, with a visible **pending** state and a **warning bar**, and an
   explicit confirmation before a consumer-disrupting (Class-2) apply.

## Decision

Pin three things over the **already-built** versioned resource store
(`resource_store.rs`, monotonic `Version`→`ETag`/`If-Match`→412) and the built `/plan` classifier —
**no new engine seam is invented here** (those belong to 0059/R010/M004/0094):

### 1. The live-apply coverage audit (drive `restart` to zero)

Every mutation route declares a **live-apply target class** and a tracked backlog item, so a remaining
`restart` is always a named, scheduled gap with a designed seam — never an accepted permanent state.
The remaining surfaces and their targets: audio gain/mute/route → Class-1 via the
[ADR-0059](ADR-0059.md) audio control seam; output non-pinned fields → Class-1, pinned params →
Class-2 migration ([ADR-R010](ADR-R010.md)) per-field ([ADR-M004](ADR-M004.md)); subtitle selector →
Class-1 via the [ADR-0094](ADR-0094.md) re-point seam; devices → [ADR-M008](ADR-M008.md). The audit is
a discipline, not a new mechanism.

### 2. Staging — write-but-don't-apply (opt-in per mutation)

A mutation may carry `apply=stage` (default `apply=now`). A staged write **persists to the resource
store** (survives, exportable, config-as-code round-trips) but **does not enqueue the engine command**;
the response carries `X-Multiview-Apply: staged` plus the would-be class from `/plan`. Staging is
**desired-state only** — it can never freeze live engine state (the [ADR-M012](ADR-M012.md) §7
desired/live split is binding). **Applied and staged reads are explicitly separated:** a normal `GET`
returns the *applied* desired state (what is — or would be — on air) and config-as-code export defaults
to it; staged deltas are visible **only** through `/pending` or an explicit staged-draft view/export
(`?include=staged`), so an unapplied change can never read back as if it were active. Staging is a
third, clearly-labelled *staged-draft* facet on top of the M012 desired/live model. New endpoints:
`GET /api/v1/pending` (staged deltas: target, would-be class, author, time),
`POST /api/v1/pending/apply` (apply in resource-dependency order, each through its own normal live seam
+ the §4 gate), `DELETE /api/v1/pending/{id}` (discard). Existing ETag/If-Match concurrency, BOLA
scoping, and 422 field-path validation apply unchanged — a staged change is just a resource version not
yet applied.

### 3. The persistent pending-changes banner

When the pending set is non-empty the SPA shows a persistent, non-blocking warning bar
("N changes are staged and not yet applied") with **Review / Apply all / Discard all** and a per-change
class badge (Class-1 hot vs Class-2 disruptive), driven by `GET /api/v1/pending` + a `pending.changed`
realtime event on the existing drop-oldest broadcast. The bar also covers the page-local dirty-form case
(snapshot-FormData comparison) so an in-progress edit is never silently lost on navigation — but the two
are **distinct states** with different scope, persistence, and actions and are rendered as separate
sections/badges: *server-side staged* deltas (persisted, shared, BOLA-scoped, applied via
`/pending/apply`) versus *page-local dirty forms* (client-only, never persisted, resolved by Save or
Discard). "Apply all" acts **only** on server-side staged deltas — it can never apply an unsaved
client-only edit. The system keeps running on the applied state until the operator chooses to apply.

### 4. The high-risk gate

Any apply the `/plan` classifier returns as **Class-2** (or the capability matrix flags high-risk)
requires an **explicit confirmation** that names *what will be interrupted* (which outputs/consumers,
the expected discontinuity) before the existing `POST .../migrate` is invoked. Class-1 hot changes need
no gate. The server already returns `202 {operation_id}` for Class-2 and reports the terminal
`Migrated`/`RolledBack` on the realtime stream — the gate adds only the confirmation in front of the
existing call.

## Consequences

- **Positive.** The operator gets the explicit write-but-don't-apply control they asked for, a clear
  pending banner, and a deliberate confirmation before any consumer-disrupting apply — the
  [Terraform plan→apply](https://spacelift.io/blog/terraform-dry-run) and
  [unsaved-changes](https://cloudscape.design/patterns/general/unsaved-changes/) discipline. The audit
  turns "still restart" into tracked, scheduled work driving toward zero download-edit-restart surfaces.
- **Negative / cost.** Staging adds an `apply` mode + three endpoints + a banner + generated types; the
  pending-apply transaction semantics (order, partial-failure) need a decision (brief Open question 1).
  Per-operator vs shared staging scope is open (brief Open question 4).
- **Neutral.** Mem: one resource-store version per staged change + a bounded pending list. The engine
  sees a staged change only when applied, through the normal bounded bus.

## Alternatives considered

- **Instant-apply only, no staging.** Rejected: the operator explicitly wants the option to stage a
  high-risk change. Instant remains the default (`apply=now`); staging is opt-in.
- **A separate `/apply` endpoint per resource.** Rejected per [ADR-W018](ADR-W018.md): invites
  stored-vs-live drift; the apply mode rides the same mutation, classified by the same `/plan`.
- **Auto-apply Class-2 without confirmation.** Rejected: a consumer-disrupting change must be a
  deliberate operator act with a named blast radius (the dirty-state destructive-confirm pattern).
- **Stage in client memory only.** Rejected: a staged change must persist, be exportable, and survive a
  reload — the resource store already provides exactly that.

## How it honors the invariants

- **#1 (output clock).** A `now` apply rides the bounded command bus drained at the frame boundary
  ([ADR-W018](ADR-W018.md)). Bus capacity is checked **before** the mutation is committed: a `now` apply
  that cannot be enqueued is rejected pre-commit (no resource-store write) or auto-staged into `/pending`
  with an operator-visible status — never accepted/written and then silently dropped, which would create
  stored-vs-live drift. Either way the clock never waits. A staged write touches only the resource store;
  the running picture is untouched. A Class-2 apply goes through make-before-break (two independent
  clocks, never a gap, [ADR-R010](ADR-R010.md)).
- **#10 (no back-pressure).** `GET /pending` is a store read; `pending.changed` rides the existing
  drop-oldest broadcast; no new engine-inward channel. Staging adds no path the engine awaits; the gate
  is UI + the existing `/plan`/`migrate` calls.
