> **Vocabulary (generic industry terms, per the production-switcher brief):**
> **USK** = upstream keyer (keys inside an M/E, participates in transitions);
> **DSK** = downstream keyer (keys over the finished M/E output, survives
> transitions); **fill** = the picture a keyer inserts; **key** = the matte
> signal that shapes it; **garbage matte** = a coarse region that forces the key
> off outside it; **FTB** = fade to black, the final master stage. These are
> de-facto industry practice terms; no vendor's model is being copied.

## Context

The production-switcher layer (ADR-0054/ADR-0055) gives one Program two scene
states (PGM/PVW) and a transition engine. A switcher is not complete without
**keying** — graphics/picture insertion that is *frame-coordinated* with
switcher state — and a **fade-to-black** master stage. What exists today,
verified in the worktree:

- **BUILT — one flat tile list, uniform opacity, linear premultiplied `over`.**
  `CompositorDrive::compose` (`crates/multiview-engine/src/drive.rs:449`)
  derives per-tick `Placement`s from `layout.cells` (z-sorted, pooled scratch,
  `drive.rs:476-494`) into `Tile{image, dst rect, opacity}`
  (`crates/multiview-compositor/src/pipeline.rs:412-428`). Both kernels fold
  tiles back-to-front: the CPU reference converts each sample through the front
  half — range-expand → matrix → EOTF → primaries
  (`tile_yuv_to_canvas_linear`, `pipeline.rs:82-119`) — then premultiplies the
  uniform opacity and applies Porter-Duff `over` in linear light
  (`fold_tile_into_band`, `pipeline.rs:1139-1147`;
  `crates/multiview-compositor/src/blend.rs:103-118`). The wgpu kernel runs the
  identical steps 1–5 (`crates/multiview-compositor/src/gpu/shaders/composite.wgsl:109-135`).
  There is **no key primitive anywhere**: one uniform opacity float per tile is
  the only alpha control, and `Nv12Image` carries no alpha plane.
- **BUILT — overlays bake OFF-thread, after the taps.** The hot-loop projection
  publishes the **pre-overlay** canvas `Arc` to the live-preview `ProgramSlot`
  and the DRM/KMS display heads, *then* hands the item to the bake consumer over
  a bounded drop-oldest queue (`crates/multiview-cli/src/pipeline.rs:2118-2129`).
  The consumer thread (`consumer_main`, `pipeline.rs:3015`) bakes monitoring
  overlays — labels, meters, state/fault badges, captions, watermark — via
  `OverlayBaker::draw_list` + `apply_overlays_to_nv12`
  (`StreamBaker::bake`, `pipeline.rs:2523-2565`;
  `crates/multiview-cli/src/overlays.rs:8-10`), then encodes once. This path is
  **input-decoupled by design** (inv #10): it samples state asynchronously,
  drops under overload, and runs *after* the preview/display taps — a keyer
  there can neither be frame-coordinated with on-clock transition state nor
  feed the taps.
- **SCAFFOLD — no video fill for overlays.** `LayerKind`
  (`crates/multiview-overlay/src/layer.rs:183-199`) is
  text/clock/meter/alert/logo/lower-third/subtitle; no variant samples a
  `TileStore`, and `Logo`/`LowerThird` have no renderer. The GPU overlay
  subpass + `composite_with_overlays` pass-chaining
  (`crates/multiview-compositor/src/gpu/compositor.rs:326`) are built in-crate
  but have no production caller.
- **BUILT — per-tile texture-array layers.** The GPU binds one array layer per
  tile (`composite.wgsl:43-47`, `texture_2d_array` luma/chroma planes;
  `gpu/compositor.rs:36` `MAX_TILES = 64`, layers allocated at `MAX_TILES`,
  `gpu/compositor.rs:725-737`) — a second per-tile source is one more layer
  index, structurally cheap.
- **DOC-ONLY — alpha media.** No alpha survives ingest (NV12-throughout);
  ADR-0058 adds the optional NV12+A framestore payload this ADR consumes for
  stills/graphics fills.

The design question: where do keyers live relative to the M/E and the bake,
where exactly does key alpha enter the fixed color pipeline (inv #8), and what
is the master output stage order.

## Decision

### 1. Two keyer families + one master stage (the canvas order is pinned)

```
per tick, inside ONE program (ADR-0054 render-plan resolver):

  M/E stage      = background scene(s) + USKs, transition math (ADR-0055)
       │
       ├── tap: CLEAN feed (pre-DSK)            — ADR-0037 vocabulary
       ▼
  DSK chain      = DSK 1..n in explicit z-order (canvas-level)
       │
       ├── tap: post-DSK (DIRTY, pre-FTB)
       ▼
  FTB            = master fade stage (full-canvas cover, own rate)
       │
       └── tap: PROGRAM (post-FTB)
```

- **Upstream keyers (USK)** are **M/E-scoped scene layers**: each scene state
  (PGM/PVW) carries `background + usk[1..k]`, and USKs **participate in
  transitions** — ADR-0055's next-transition arming selects
  `{background, key[1..k]}` subsets, so a key-only transition mixes a USK on/off
  without touching the background.
- **Downstream keyers (DSK)** are **canvas-level, post-M/E, pre-FTB**, applied
  by the same render-plan resolver in the same `compose(tick)` — i.e.
  **on-clock and frame-coordinated**: the tick that advances a transition is
  the tick that advances a DSK auto ramp, and the clean/post-DSK taps are exact
  per-frame cut points. DSKs **survive transitions** (a mix behind a lower
  third never disturbs the lower third).
- **FTB** is the **final master stage after the DSKs**: a full-canvas cover at
  the top of the final compose list, with its **own rate** (integer frames at
  the output cadence, ADR-T015), states
  `inactive → fading_to_black → black → fading_up`, engageable regardless of
  transition state. REST addressing, pinned: because FTB is a per-program
  master stage after the DSKs, [ADR-W021](ADR-W021.md)'s verb addresses it via
  the M/E that owns the program canvas (one M/E in the MVP) — so
  `…/mix-effects/{id}/ftb` is the *address*, not the stage location. An
  **`audio_follow: bool`** flag fades program audio over
  the same frame window via the master gain stage — mechanism in
  [ADR-0059](ADR-0059.md). The engine **keeps composing beneath an engaged
  FTB** so the clean/monitoring taps and ISO paths stay live; MVP implements
  FTB as a solid full-canvas cover tile whose weight rides the ramp — the
  cover color is configurable (default black), defined in canvas space per
  [ADR-C007](ADR-C007.md)'s dip/FTB target rule — and the
  existing `Tile` + linear `over` already compute exactly a linear fade to
  the cover (`out = (1−w)·under` for black), zero new kernel code (perceptual
  ramp option per [ADR-C007](ADR-C007.md)).

### 2. The on-clock / off-thread split (explicit)

Production graphics (USK/DSK fills, multi-box compositions) move **on-clock**
into the engine render plan. The existing **off-thread egress bake stays** for
*monitoring-style* overlays — tile labels, meters, badges, safe-area, captions,
and the licence-enforcement watermark — exactly where it is today
(`pipeline.rs:2118-2129` taps first, bake after). Rationale: monitoring
overlays are input-decoupled by design and tolerate drop-oldest; switcher
graphics must land on the same tick as the switcher state that turned them on,
and must be visible to the program taps. The brief budgets this split; the bake
path is otherwise untouched.

### 3. Keyer model

```rust
KeySource { fill: SourceRef, key: Option<SourceRef>, premultiplied: bool }
KeyType   { Luma { clip, gain, invert },          // MVP
            Linear,                               // MVP (fill+key / alpha)
            DvePip { rect, … },                   // MVP (scaled box)
            Chroma { … },                         // post-MVP (domain pinned, ADR-C007)
            Pattern { … } }                       // post-MVP (ADR-0055 wave-2 SDF masks)
// every keyer: explicit integer z (key priority), rectangular garbage matte,
// on-air flag, tie/cut/auto behaviour, auto rate in integer frames.
```

- **`fill` + optional `key`**: when `key` is `None`, the keyer **self-keys**
  (luma type cuts alpha from the fill's own Y; linear type reads the fill's own
  alpha plane when the fill is NV12+A per ADR-0058). When `key` names a second
  source, that source's post-range-expand Y **is** the alpha (the classic
  fill+key pair). **Verified substrate:** the GPU already binds per-tile
  texture-array layers (`composite.wgsl:45-46`), so a key source is one more
  layer index in `TileParams`; on the CPU it is a second sampled image inside
  `fold_tile_into_band`.
- **`premultiplied: bool`**: a shaped (premultiplied) fill is consumed as
  already-multiplied — the kernel uses `(fill_lin, a)` directly as the
  premultiplied operand instead of computing `(fill_lin·a, a)`, preventing the
  classic double-multiply edge darkening. Default **`false`** — for raw
  fill+key pairs *and* for [ADR-0058](ADR-0058.md) alpha media alike: that
  ADR's payload carries **straight** alpha, and the kernel premultiplies
  exactly once, in linear light, at the existing premultiply step (skipping
  the multiply on a straight-alpha fill would render un-multiplied edges —
  bright halos, the inverse error). `true` is reserved for externally
  delivered **shaped** (premultiplied) fills that the source or import
  explicitly declares as such; it is never set for library media
  (import normalizes premultiplied-authored assets per ADR-0058).
- **Clip / gain / invert** (luma, and trim on linear):
  `a = clamp((y_norm − clip) × gain, 0, 1)`, `invert ⇒ a := 1 − a`, with
  `clip ∈ [0,1]` on post-range-expand normalized Y and `gain ≥ 0` — the
  de-facto industry clip/gain ramp. Threshold-domain rationale is pinned in
  ADR-C007.
- **Rectangular garbage matte** (MVP): an integer-pixel rectangle in the key's
  destination space; alpha is forced to 0 outside it (a cheap rect clip in both
  kernels). Soft-edged/SDF mattes arrive with the ADR-0055 wave-2 wipe masks.
- **Key priority = explicit z-order**, the same `z` model as `Cell.z`
  (`crates/multiview-core/src/layout.rs:122-166`): USK z orders layers within
  its M/E scene; DSK z orders the canvas-level chain. No implicit "keyer 2 is
  always above keyer 1" rule.
- **MVP keyer types:** **linear/alpha + luma + dve_pip** (dve_pip is a scaled
  box — exactly today's `Tile::scaled` integer-rect placement; sub-pixel
  placement, box styling — borders, drop shadow — and crop/rotation wiring
  land with ADR-0055 wave 2 on the Layout cell model).
  **Chroma and pattern are post-MVP**, but their math domains are pinned now
  (ADR-C007) so the kernels are not re-pinned later.

### 4. Key-math insertion points (inv #8 order is preserved)

Key alpha enters the **existing** step structure without reordering it — the
verified seams in both kernels:

| Key type | Alpha computed from | Insertion point (CPU / GPU) |
|---|---|---|
| Luma (and fill+key Y) | **post-range-expand normalized Y** (code-value space, step 1) | after `range::expand_luma` (`crates/multiview-compositor/src/range.rs:49`; `pipeline.rs:101-104`) / after `composite.wgsl:113` |
| Chroma (post-MVP) | **linear RGB after the YUV→RGB matrix + EOTF** (tile gamut, between steps 3 and 4) | after the EOTF triple (`pipeline.rs:110-114`) / after `composite.wgsl:120-126` |
| Linear / NV12+A | the key plane (R8) or fill alpha plane, normalized | sampled alongside the fill (one more texture layer / plane walk) |

The computed key alpha is then **multiplied into the tile's opacity before the
premultiply + linear-light `over`** — i.e. at exactly the point both kernels
already turn straight alpha into a premultiplied operand
(`pipeline.rs:1139-1147`: `LinearRgba{…, a: opacity}.premultiplied()` then
`over(src, *slot)` per `blend.rs:103-118`; `composite.wgsl:131-135`: step 5).
Alpha is applied **before** the blend, never after OETF; the fixed pipeline
order — detect → range-expand → matrix → linearize → primaries → blend in
linear → OETF → encode → tag — is untouched.

### 5. Multi-box composition source (generic term)

A **multi-box composition** — a background plus overlapping, z-ordered,
sized/cropped boxes, usable as one source on any bus — is **the existing
`Layout` model** (`crates/multiview-core/src/layout.rs:257`; background +
`Vec<Cell>` with z/fit/opacity/crop/rotation) **exposed as a bus-selectable
composition source**, rendered as a **drive-internal pre-pass in the same
tick**: the render-plan resolver composes the named layout first, then feeds
the result to the consuming scene as one tile.

- **NOT a chained ProgramSet program.** Feeding program B's canvas into
  program A's tile store would add one tick of latency, an NV12 4:2:0
  re-quantization per generation (inv #5 makes intermediates NV12 on the CPU
  path), and a second independent clock (ADR-0030 programs deliberately do not
  share ticks). The pre-pass runs under the **same `Tick`**, so a take
  involving a composition is frame-aligned by construction.
- On the GPU the pre-pass can stay in the pooled linear `Rgba16Float` chain
  (the `composite_with_overlays` pass-chaining precedent,
  `gpu/compositor.rs:326` — currently SCAFFOLD/unwired) and avoid the
  intermediate NV12 generation entirely; the CPU reference composes an NV12
  intermediate and accepts the documented generation loss.
- Config declares compositions; document-level validation adds a **cycle
  check** (composition/M/E re-entry) per [ADR-M012](ADR-M012.md). Sources
  inside an on-air composition are PROGRAM-tallied (recursive
  contributes-to-program, [ADR-MV006](ADR-MV006.md)).

### 6. DSK tie / cut / auto, and on-air vs in-next-transition

- **CUT**: the keyer's on-air state flips at the next frame boundary (one
  tick, no ramp).
- **AUTO**: the keyer ramps its alpha over its **own rate** (integer frames);
  the ramp envelope multiplies the key alpha; progress is `f(tick)` evaluated
  by the switcher state machine at the frame-boundary seam
  (`crates/multiview-engine/src/runtime.rs:432-439`, hook widened to receive
  `Tick` per ADR-0054).
- **TIE**: the DSK joins the **next M/E background transition** — its on/off
  ramp rides the transition window (AUTO or T-bar) instead of its own rate.
  USKs achieve the same via ADR-0055 next-transition arming
  (`{background, key[n]}`).
- **On-air** (currently contributing pixels) and **in-next-transition** (armed
  to change state when the next transition runs) are **independent flags**; a
  keyer can be both (on-air and armed to transition off). Tally derives from
  on-air *or* mid-ramp contribution (ADR-MV006); the control surface exposes
  both ([ADR-W021](ADR-W021.md)).
- Keyer state changes are **Class-1** (frame-boundary, no encoder reset) under
  inv #11; classification surfaces through the ADR-M012 plan/take split.

### 7. Invariant posture

- **#1 (output clock):** keyers/FTB are pure per-tick state folded into the
  same synchronous `compose(tick)`; key sources are *sampled* from their
  `TileStore`s like any tile (lock-free `read_at`), never awaited. A missing
  fill/key frame holds last-good (inv #2); persistent loss drops the keyer to
  off-air-safe per [ADR-R011](ADR-R011.md).
- **#10 (isolation):** keyer commands ride the existing bounded command bus and
  frame-boundary drain; the off-thread monitoring bake keeps its drop-oldest
  queue. Nothing new can back-pressure the engine.
- **#5 (NV12-throughout):** fills are NV12 (or NV12+A per ADR-0058, +1 R8
  plane); no per-tile RGBA is ever materialized. The linear `Rgba16Float`
  canvas remains the only float intermediate.
- **#8 (color order):** insertion points above; the order is not reordered and
  range stays handled exactly once.
- **#9 (degradation):** each on-air keyer, the FTB cover, and each multi-box
  pre-pass are admission-accounted composite load (ADR-0055's ladder rung);
  an in-flight keyer AUTO is program-affecting — complete-then-degrade.

### 8. Efficiency budget (standing review)

- **Luma/linear key:** +1 texture-array layer per keyed tile on the GPU (a
  keyed tile consumes 2 of the `MAX_TILES = 64` layers — the budget is halved
  for fully-keyed scenes and is admission-checked at arm time) and a few
  ALU ops/pixel; on the CPU, one extra plane sample + one multiply-clamp per
  covered pixel.
- **DSK chain:** each on-air DSK is one more tile in the final compose; a
  full-canvas 1080p DSK costs roughly one full-canvas tile fold (~2.07 Mpix of
  key math + `over`).
- **FTB:** one full-canvas solid tile while engaged; zero cost inactive.
- **Multi-box pre-pass:** one extra canvas-sized composite per active
  composition per tick — the single biggest cost item; accounted before the
  PVW rung, shed before program tiles.
- **No per-frame allocation:** key parameters ride the `TileParams` extension
  (its `dst_size.zw`/`opacity_transfer.zw` padding, `composite.wgsl:14-34`, is
  the first landing zone) and pooled scratch per the `ComposeScratch`
  discipline; key-source images are pool-held like fills.

## Rationale

- **Two families because their lifecycles differ.** A USK is part of the scene
  a transition mixes between — it must participate in the transition math. A
  DSK is the graphics layer the gallery expects to *survive* the transition.
  One family cannot express both without per-keyer special cases in the
  transition engine.
- **DSK/FTB must be on-clock.** The off-thread bake is asynchronous and
  post-tap by construction (verified above); a DSK there could appear a frame
  late relative to a cut, vanish under drop-oldest shedding, and never reach
  the program taps that feed displays/preview — all unacceptable for
  production graphics. The split keeps the engine's protected core small while
  giving graphics frame-exact semantics.
- **Insertion points fall out of the verified kernel structure.** Luma keying
  on post-range-expand Y is range-independent and operator-familiar; chroma in
  linear tile-gamut RGB is colorimetrically sane (ADR-C007 carries the full
  argument); multiplying key alpha into the existing opacity-premultiply point
  reuses the only blend the product has (one code path, CPU oracle + GPU SSIM
  gates extend, not fork — ADR-C007).
- **Multi-box = the Layout model** because it already *is* one (background,
  overlapping z-ordered cells, per-cell fit/crop/opacity, validated) — a second
  model would duplicate schema, validation, the editor, and the render path.

## Alternatives considered

- **Keyers as overlay layers on the egress bake.** Rejected: the bake is
  input-decoupled and post-tap (verified `pipeline.rs:2118-2129` →
  `consumer_main`/`StreamBaker::bake`), so keys cannot be frame-coordinated
  with on-clock switcher state; `LayerKind` has no video-fill variant and no
  path to sample a `TileStore` (`layer.rs:183-199`); drop-oldest shedding may
  skip graphics frames the program encoded. Monitoring overlays stay there;
  production keying cannot.
- **RGBA-canvas keying** (decode fills to RGBA, key on an RGBA canvas).
  Rejected: violates inv #5/[ADR-E002](ADR-E002.md) (never materialize RGBA per
  tile; 4 B/px vs 1.5–2.5 B/px) and forks the color pipeline. ADR-0058's
  NV12+A payload carries alpha at 2.5 B/px without leaving the NV12 currency.
- **A single keyer family** (all keyers downstream, or all upstream).
  Rejected: all-downstream cannot do key-only transitions or keys that ride a
  wipe (USKs participate in transitions); all-upstream cannot keep a lower
  third steady across a background mix (DSKs survive transitions). Both halves
  are load-bearing.
- **Multi-box composition as a chained ProgramSet program.** Rejected (one
  tick of added latency, NV12 generation loss per hop, independent clocks break
  frame-aligned takes); reserved for feeds that genuinely need their own
  encode per [ADR-0054](ADR-0054.md)/[ADR-R010](ADR-R010.md).
- **FTB as an encoder/egress-side fade.** Rejected: the fade must be visible
  identically on every tap and rendition (encode-once-mux-many, inv #7) and be
  frame-exact against transitions/keyers; a canvas-level cover in the same
  composite is the only point with those properties.

## Consequences

- **Positive:** frame-exact production graphics with one blend law and one
  apply path; clean/dirty/program taps become exact, per-frame cut points
  (recording + aux feeds inherit ADR-0037 vocabulary); the MVP keyer set
  (linear + luma + dve_pip) rides existing primitives — the kernels gain only
  key-alpha sampling and a rect clip; FTB ships with zero kernel changes.
- **Negative / cost:** the GPU layer budget halves for fully-keyed scenes
  (2 layers/keyed tile against `MAX_TILES = 64`); each DSK/composition adds
  real per-tick cost that the admission seam (ADR-0055) must account before
  enabling; `TileParams`/CPU `Tile` grow key fields on both kernels — every
  change lands in the CPU reference first and re-runs the byte-exact oracle +
  GPU SSIM/PSNR gates (ADR-C007); the on-clock/off-thread split adds a second
  graphics vocabulary (production keyers vs monitoring overlays) that docs and
  the UI must keep distinct.
- **Open (deferred to named ADRs):** chroma/pattern key parameters
  (ADR-C007 pins domains now), soft/SDF garbage mattes (ADR-0055 wave 2),
  keyer audio (fills are silent; media-player audio is
  [ADR-0057](ADR-0057.md)/[ADR-0059](ADR-0059.md)).

## Test plan (gates extend, never fork)

1. **CPU reference first:** every keyer/FTB kernel change lands in the CPU
   reference with golden frames; the tile-driven kernel must stay
   byte-identical to the oracle
   (`crates/multiview-compositor/tests/composite_tile_driven.rs:5-8`, property
   test at `:222`).
2. **GPU parity:** keyed-composite test patterns extend the existing gate —
   Y and UV SSIM ≥ 0.98, PSNR ≥ 40 dB vs the CPU oracle
   (`crates/multiview-compositor/tests/gpu_compositor.rs:164-167`), GPU-tagged
   runners only.
3. **Frame coordination:** deterministic `ManualTimeSource` runs assert a DSK
   CUT lands on the same tick as the command's frame boundary, a TIE'd DSK
   tracks the transition window exactly, and the clean tap never contains DSK
   pixels.
4. **Chaos:** wedge the bake consumer and every tap while keyers ramp — the
   engine keeps ticking (the MP-1 pattern, inv #10); kill a fill/key source
   mid-ramp — the keyer holds last-good then drops off-air-safe (ADR-R011).
