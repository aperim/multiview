# ADR-0084: SIP/SIPS + SRTP stack & seam — inbound call = source, outbound = output/return

- **Status:** Proposed
- **Area:** Calling / remote-contribution (new `multiview-sip` leaf crate + `multiview-webrtc` reuse + `multiview-input`/`multiview-output`/`multiview-config`/`multiview-control`)
- **Date:** 2026-06-13
- **Source brief:** [sip-calling.md](../research/sip-calling.md)
- **Builds on:** [ADR-0048](ADR-0048.md) (the shared native ICE/DTLS/SRTP media endpoint — one dual-stack UDP socket, one driver task, session GC), [ADR-T014](ADR-T014.md) (WHIP-ingest source lifecycle/apply precedent), [ADR-0042](ADR-0042.md) (IPv6-first binds/URLs/SDP)
- **Relates to:** [ADR-0085](ADR-0085.md) (the SDP↔media bridge this stack hands sessions to), [ADR-0049](ADR-0049.md) (`Output::WhipPush`, the push-output sibling), [ADR-T013](ADR-T013.md) (RTP-audio rebase), [ADR-T001](ADR-T001.md)/[ADR-RT004](ADR-RT004.md) (invariants #1/#10)

## Context

The operator asked for **full SIP/TLS inbound and outbound calling, with audio and video**. SIP is a
signaling protocol ([RFC 3261](https://www.rfc-editor.org/rfc/rfc3261.html)): it registers, rings,
and negotiates an SDP ([RFC 8866](https://www.rfc-editor.org/rfc/rfc8866.html)) offer/answer — it does
**not** carry media. The media is RTP/SRTP, which Multiview already designs a full native stack for in
[ADR-0048](ADR-0048.md) (str0m: ICE/DTLS/SRTP on one shared `[::]` UDP socket). Standing up a second
media stack for SIP would mean a second socket, a second SRTP implementation, and a second
invariant-#10 surface to audit — for zero benefit.

There is **no SIP in the tree today** (`rg -i '\bsip\b' crates` → no matches), so this is net-new — but
the media seams it must terminate onto are real and verified: the pure `MediaEngine`/`WebRtcProducer`
ingest seam (`crates/multiview-input/src/webrtc/transport.rs:201`/`:547`), `WrapBits::Rtp32`
(`crates/multiview-input/src/normalize.rs:42`), `AudioStore::publish`
(`crates/multiview-audio/src/store.rs:171`), and the encode-once `PacketRouter`/`PacketSink`
(`crates/multiview-output/src/fanout.rs:140`/`:152`).

This ADR pins **the SIP stack, its crate, its transports/security posture, and the source/output
seam**. The media-bridge mechanics (SDP↔str0m, the SRTP keying straddle, mix-minus return) are
[ADR-0085](ADR-0085.md).

## Decision

### 1. A new leaf crate `multiview-sip` (signaling only; media is shared)

`multiview-sip` (lib `multiview_sip`) owns the SIP UA: transaction layer, dialog layer, REGISTER,
digest auth, the offer/answer model, transports, and the SDP it produces/consumes. House native
pattern: `default = []` (a pure-Rust shell so `cargo check --workspace` stays native-free and
deny-clean); `native = ["dep:<sip-stack>", "dep:<srtp>", "multiview-webrtc/native", …]`. Deps:
`multiview-core`, `multiview-webrtc` (the shared media endpoint handle). **No media stack of its
own** — it configures sessions on the [ADR-0048](ADR-0048.md) endpoint. Dependency direction stays
acyclic: `sip → {webrtc, core}`; `cli → sip`. (Crate-vs-`multiview-webrtc`-feature is a reversible
default; pinned as a crate because the SIP state machine is large and independent.)

### 2. The SIP stack — open-standard, Rust, license-gated

The UA implements the [RFC 3261](https://www.rfc-editor.org/rfc/rfc3261.html) transaction/dialog core
+ REGISTER + [RFC 3264](https://www.rfc-editor.org/rfc/rfc3264.html) offer/answer over
[RFC 8866](https://www.rfc-editor.org/rfc/rfc8866.html) SDP. Pinned candidate: the **ezk** Rust crates
(`ezk-sip-core`/`ezk-sip-ua`/`ezk-rtc`, MIT — verified). Its exact `sips:`-TLS + SRTP coverage at the
chosen version is validated **before commit**; any gap is filled in-crate against the RFCs, never
worked around. If no Rust stack cleanly covers SIPS+SRTP, the transaction/dialog core is implemented
in `multiview-sip` (RFC 3261 is well-specified) — the larger-effort path, named honestly. The chosen
dependency + an SRTP crate ride a **`cargo deny` license/advisory gate under the `native` feature
graph**, exactly as str0m is gated for WebRTC. reSIProcate (C++) is a **conceptual** reference only,
not a dependency.

### 3. Transports — SIPS/TLS first, IPv6-first, no new media port

TLS (`sips:`) is the **default** signaling transport ([RFC 3261 §26.2.2](https://www.rfc-editor.org/rfc/rfc3261.html));
TCP and UDP are legacy/allowed-not-default; WS/WSS ([RFC 7118](https://www.rfc-editor.org/rfc/rfc7118.html))
is an optional browser-SIP transport on the existing control-plane HTTP listener. All bind **dual-stack
`[::]`** (`IPV6_V6ONLY=false`); IPv6 URI literals are bracketed; the SDP is IPv6-first (`c=IN IP6`,
no TTL — [ADR-0042](ADR-0042.md)). SIP adds **signaling** ports (5060 UDP/TCP, 5061 TLS by
convention; configurable) only — **media reuses the shared `webrtc.udp_port`** ([ADR-0048](ADR-0048.md)
§4) **when the call can RTP/RTCP-mux on it and demux is unambiguous by 5-tuple + SSRC/PT**; a non-ICE
far end that cannot `rtcp-mux` (or any tuple/SSRC collision) gets a **per-call RTP/RTCP port pair from a
bounded pool** instead — never silently-forced multiplexing ([ADR-0085](ADR-0085.md) §1). NAT: for
outbound media works because we initiate; signaling uses `rport`/symmetric routing
([RFC 3581](https://www.rfc-editor.org/rfc/rfc3581.html)); ICE-in-SDP
([RFC 8839](https://www.rfc-editor.org/rfc/rfc8839.html)) when the far end offers it. For **inbound**
calls without ICE, classic SIP needs **symmetric-RTP/latching** (lock onto the source tuple of the
first received RTP, ignoring the possibly-private address in the offer SDP) plus
`advertised_addresses` rewriting; far ends behind symmetric NAT with no ICE and no relay may be
unreachable — that boundary is stated, not papered over. No embedded STUN/TURN server (boundary,
mirrors [ADR-0048](ADR-0048.md) §11); an external TURN/media relay is an operator option.

### 4. Inbound call = `kind = "sip"` source; outbound = `Output::SipCall`

- **Inbound:** `SourceKind::Sip { … }`, internally tagged `kind = "sip"` (never `untagged` — house
  rule). An accepted INVITE becomes a LIVE source: video → `TileStore` (a routable input stream,
  [ADR-0034](ADR-0034.md)), audio → `AudioStore` → a switcher input strip. Lifecycle mirrors the
  `webrtc` source ([ADR-T014](ADR-T014.md)): configured-but-idle → NO_SIGNAL; active → LIVE;
  `BYE`/timeout → STALE → NO_SIGNAL; **never RECONNECTING for an inbound call** (the far end dials us).
  Inbound INVITEs are admission-gated + rate-limited; auto-answer only for a configured trunk
  allow-list — an INVITE flood sheds at the SIP layer and is admitted **outside** the WebRTC viewer
  pool (the [ADR-0048](ADR-0048.md) §8 pool-split analogue), so it can never starve a media session
  or a tick.
- **Outbound:** `Output::SipCall { id, uri, register?, codec, gpu_pin, audio }` — the SIP sibling of
  `Output::WhipPush` ([ADR-0049](ADR-0049.md)): UAC dials `sip:`/`sips:`, offers program/bus video +
  a **mix-minus** return ([ADR-0085](ADR-0085.md)/[ADR-0078](ADR-0078.md)), supervised redial with
  backoff like the RTMP/SRT/WHIP push clients.

### 5. Security posture (secure by default)

`sips:`/TLS with cert validation + SRTP media are the default. Plaintext SIP and plain RTP are **off
by default**, behind explicit `allow_insecure_signaling`/`allow_insecure_media` flags that log a
standing warning. Digest auth for REGISTER/INVITE; credentials follow the config-secret posture
(rtmp/srt stream-key precedent, [ADR-0049](ADR-0049.md)). The control plane gates who may place/answer
calls (BOLA `authorize_object`). SDES-SRTP keys, when used, **must** ride `sips:`/TLS — never plaintext
signaling ([ADR-0085](ADR-0085.md) §3.3). Note `sips:`/TLS is **hop-by-hop**: it does not give
end-to-end media secrecy, and any SIP proxy/B2BUA on the path can see the SDES key — so SDES is scoped
to **trusted-direct/trusted-proxy** plants and DTLS-SRTP (key never leaves the endpoints) is preferred
wherever the far end supports it.

### 6. Config & apply classification

Additive, serde-tagged (forward-compatible): `SourceKind::Sip` + `Output::SipCall`, plus a top-level
`[sip]` block (registrar, credentials, transport, security flags) mapped by the cli into a plain
`SipConfig` (the crate never depends on `multiview-config`). Add/remove of a call is the same apply
class as other network source/output kinds (invariant #11, recorded in the
[capability matrix](../research/management-capability-matrix.md)); changing a return's **audio-track
set** is Class-2 ([ADR-0079](ADR-0079.md)).

### 7. Efficiency budget

Signaling is I/O-bound and small (transactions; REGISTER refresh at minutes-scale), off the data plane
in `multiview-sip`'s task. Per-call media memory rides the [ADR-0048](ADR-0048.md) §12 per-session
caps. Inbound decode is 1/call at display resolution (inv #6); outbound video is route()+1 (0 extra
encode, inv #7); return audio is 1 encode per **distinct** mix-minus rendition ([ADR-0085](ADR-0085.md)).
Calls shed as an aux/return rung before any program lever (inv #9).

## Consequences

- **One media stack, two signaling front-ends.** WHIP/WHEP (HTTP) and SIP calls share the
  [ADR-0048](ADR-0048.md) endpoint, socket, DTLS cert, driver, and isolation audit. SIP contributes a
  dial-plan + an SDP, not a media stack.
- **Net-new crate + a real interop risk.** `multiview-sip` is new; the SRTP-keying straddle for
  classic SIP (SDES vs DTLS-SRTP) is the largest risk and is carved out to [ADR-0085](ADR-0085.md) §3.
  **SDES-SRTP is a v1 requirement** (classic SIP/TLS plants overwhelmingly use it), so the open question
  is its *mechanism* (reuse str0m vs a vetted SRTP crate), not whether it ships; a DTLS-SRTP-only first
  cut is an early-milestone stopgap that does not yet satisfy "full SIP/TLS calling" — not a silent
  deferral.
- **Secure-by-default may not interop with legacy plaintext SIP** until the operator opts into the
  insecure flags — accepted; the warning is loud.
- **Licensing stays clean:** open IETF standards; Opus/G.711 codecs are royalty-free/LGPL-clean;
  royalty-bearing telephony codecs (G.729) are a flagged escalation, off the default build. The SIP
  stack + SRTP crate are `cargo deny`-gated under `native`. No vendor SDK; far-end quirks are
  clean-room/nominative ([managed-devices.md](../research/managed-devices.md)).
- Downstream surfaces track the new kinds: capability-matrix rows, OpenAPI registration, SPA
  source/output forms + a call-control UI (ring/answer/hangup), in-app docs.

## Alternatives considered

- **A second SIP-owned media stack (e.g. an RTP/SRTP stack inside `multiview-sip`).** Rejected: a
  second socket, a second SRTP implementation, a second invariant-#10 surface — the exact multiplication
  [ADR-0048](ADR-0048.md) created the shared endpoint to avoid. The keying straddle still needs a thin
  SIP-side SRTP path (§3.3 of [ADR-0085](ADR-0085.md)), but on the **shared** socket.
- **A sidecar SIP↔WebRTC gateway (e.g. an external open-source UA + WHIP bridge).** Rejected as *the*
  path: breaks the single-binary product, adds an unmanaged auth/config/lifecycle surface, severs the
  in-process encode-once fan-out and the mix-minus, and forces an extra media hop/transcode. (A sidecar
  remains a valid *operator* interop option, not the product's answer.)
- **Build a full PBX/SBC.** Rejected: out of scope (§1.2 of the brief). Multiview is a UA endpoint +
  media bridge; conferencing is the switcher compositing N calls with per-party mix-minus returns, not
  a SIP conference focus.
- **Plaintext SIP/RTP by default for maximum interop.** Rejected: secure-by-default is non-negotiable;
  insecure transport is an explicit, logged opt-in.
- **Fold SIP into `multiview-webrtc` as a feature now.** Deferred, not rejected: a reversible default;
  the SIP state machine's size argues for a leaf crate, but the decision is not load-bearing.

## How this honors the invariants

- **#1 (output-clock):** SIP signaling and media are entirely off the tick path. Inbound media is
  *sampled* through the bounded, keyframe-gated `MediaEngine`/`AudioStore` path (the WHIP path);
  outbound returns consume the existing fan-out through bounded per-call rings. No SIP code path paces
  or blocks a tick — a call that never connects, drops, or floods us cannot perturb the clock.
- **#10 (isolation):** all SIP work lives in `multiview-sip`'s task + the shared `multiview-webrtc`
  driver, joined to the engine **only** by bounded drop-oldest rings. The engine never awaits the UA
  or a far end; a wedged registrar/UA loses *its* calls' media, never an output tick. The admission
  pool-split keeps an INVITE flood from starving media or the engine, and the chaos-gate shape is the
  same stalled-consumer test the WebRTC/recorder sinks already use.
