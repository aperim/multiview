# ADR-0003: Per-stage HAL with runtime auto-negotiation and zero-copy-island preference

- **Status:** Proposed
- **Area:** Core Engine
- **Date:** 2026-06-02
- **Source brief:** [core-engine.md](../research/core-engine.md)

## Decision

Define backend-agnostic traits per stage (Demuxer/Decoder/Compositor/Encoder/Muxer) with feature-gated impls; select backends per stage at runtime via three-layer capability detection (FFmpeg baseline, vendor deep queries, sandboxed probe) and a scored assignment planner that heavily penalizes cross-vendor stage boundaries.

## Rationale

The project mandates modular per-stage backend selection (e.g. VAAPI decode + NVENC encode + CUDA composite). Per-stage selection is realistic and matches FFmpeg/GStreamer/Norsk practice; the real risk is the cross-backend seam, so encoding boundary copy as a cost and preferring single-device pipelines yields correct, performant plans with graceful degradation.

## Alternatives considered

Single fixed backend per platform (rejected: violates modularity). Greedy per-stage best-pick (rejected: ignores cross-stage copy cost). Runtime-only dlopen of all backends (rejected: heavier binary, weaker static guarantees). Pure-FFmpeg autodetection (rejected: hides session limits and per-stage mixing).

## Consequences

Must build capability detection per vendor (NVML/cuvid caps/oneVPL/VAAPI/VideoToolbox), correlate devices by PCI/UUID/IOKit id, and insert explicit hwtransfer nodes at surviving boundaries. NVML cannot report codec capability — must use SDK caps APIs or probes.
