# ADR-0042 — IPv6-first networking; IPv4 is legacy-only and on a deprecation path

**Status:** Accepted (2026-06-08)
**Area:** cross-cutting — networking & addressing (all network-facing surfaces)
**Drivers:** operator directive — *"all our products are IPv6-first; supporting IPv4 only for legacy
users; IPv4 **will** be deprecated from this product."* The deployment target is IPv6 networks; IPv4
exists solely for a bounded legacy-interop transition.
**Research:** [ipv6-first](../research/ipv6-first.md).
**Related:** [conventions §10](../architecture/conventions.md) (the canonical statement),
[ADR-0040](ADR-0040.md) / [ADR-0041](ADR-0041.md) (multicast + SAP — re-pointed IPv6-first by this
ADR), [ADR-0033](ADR-0033.md) (AES67/ST 2110 — RTP/PTP, multicast), [ADR-0011](ADR-0011.md)
(platforms). Backlog: [ipv6-first-backlog](../development/ipv6-first-backlog.md) (`IPV6-0..`).

## Context

The project had **no documented IPv6 posture** — no convention, ADR, or brief stated that
network-facing surfaces must be IPv6-first or dual-stack. As a result, IPv4-only assumptions had
accreted across the tree and looked intentional rather than accidental. A repo audit (2026-06-08,
against `origin/main`) found the following IPv4-only / IPv4-centric sites (some may already be in
remediation in parallel — where fixed, the matching `IPV6-*` slice is verification-only):

- **SDP parse — IPv4-only:** `multiview-control/src/nmos/is05.rs:340` `parse_sdp_transport` handles
  only `c=IN IP4`; a `c=IN IP6 …` line is silently mangled into `destination_ip = "IP6"`.
- **SDP generation — hardcoded IPv4:** `multiview-preview/src/whep.rs:275/289/306` always emit
  `o=… IN IP4 0.0.0.0` / `c=IN IP4 0.0.0.0`, even when ICE candidates are IPv6.
- **Sockets bound IPv4-only:** telemetry `syslog.rs:392` and `snmp.rs:702` bind `0.0.0.0:0`; the WHEP
  native transport binds `127.0.0.1:0` (`whep/native.rs:237`); the HA transport bind has no IPv6
  handling (`ha/transport.rs:290`).
- **Multicast join — IPv4-only:** `multiview-input/src/st2110/transport.rs:104` exposes only
  `join_multicast_v4`; no IPv6 / `MCAST_JOIN_(SOURCE_)GROUP` counterpart.
- **Config / docs / examples IPv4-centric:** the control `listen` field docs (`multiview-config/src/lib.rs`)
  show only `127.0.0.1`/`0.0.0.0`; `docs/reference/example-streams.md` is all IPv4; and the recently
  merged multicast/SAP docs ([ADR-0040](ADR-0040.md)/[ADR-0041](ADR-0041.md)) led with IPv4 examples,
  put `c=IN IP4` as the primary SDP form, and even described IPv6 multicast SSM as "a follow-up" —
  IPv4-first framing that this ADR reverses.

This is a policy gap plus a remediation backlog, not a single feature.

## Decision

1. **IPv6-first is canonical.** All network-facing surfaces are IPv6-first; **IPv4 is legacy-only and
   on a deprecation path** (slated for removal). New designs, code, config, and docs are **never
   IPv4-only and never IPv4-first**. A network surface that cannot do IPv6 is a **defect**. This is
   recorded in [conventions §10](../architecture/conventions.md) (the source of truth).

2. **Bind dual-stack on `[::]` by default** (`IPV6_V6ONLY=false`), never `0.0.0.0`; loopback `[::1]`,
   never `127.0.0.1`. Applies to the control plane, telemetry (syslog/SNMP), preview/WHEP, RTSP, HLS,
   SRT, and multicast. Dual-stack (a single `[::]` socket accepting both v6 and v4-mapped) is the
   default so legacy IPv4 clients still connect during the deprecation window; an explicit
   IPv6-only or IPv4-only override is allowed but not the default.

3. **Addresses & URLs are IPv6-capable everywhere.** Every address parse/validate path accepts IPv6;
   IPv6 literals are **bracketed** in URLs (`udp://@[ff3e::1]:5004`, `rtp://[…]`, `[::1]:8080`).
   Documentation and config examples **lead with IPv6**; any IPv4 form shown is labelled *legacy*.

4. **SDP handles `IN IP6` as a first-class form.** Parsers and generators handle both `c=IN IP4` and
   `c=IN IP6`. The IPv6 multicast connection line carries **no TTL** — `c=IN IP6 <addr>[/<count>]`
   (the slash is an address *count*); only the IPv4 form takes `/ttl` (RFC 8866 §5.7). The `nmos/is05`
   parser (and the unified SDP model from [ADR-0041](ADR-0041.md)) extract the family correctly;
   WHEP answers emit `IN IP6` when the candidates are IPv6.

5. **Multicast is IPv6-first.** IPv6 multicast `ff00::/8` (flags + scope nibbles) with IPv6 SSM
   `FF3x::/32` via **MLDv2** is the primary path; IPv4 `239/8` (admin) / `232/8` (SSM) + IGMPv3 is the
   legacy peer. A `join_multicast_v6` / protocol-agnostic `MCAST_JOIN_(SOURCE_)GROUP` path is
   **required**, not a follow-up. [ADR-0040](ADR-0040.md) / [ADR-0041](ADR-0041.md) are re-pointed
   IPv6-first (their IPv4-led examples and the "IPv6 SSM is a follow-up" line are corrected).

6. **Remediation is tracked, not deferred-silently.** The audit's IPv4-only sites become the
   `IPV6-0..` backlog ([ipv6-first-backlog](../development/ipv6-first-backlog.md)), each a TDD-first
   slice with an IPv6 (or dual-stack) test. This ADR is the gate: a new network surface lacking IPv6
   support is rejected in review.

7. **Invariants are preserved.** None of this touches the data-plane invariants — IPv6 is an address
   family, not a pacing change. Multicast receive stays a sampled input jitter buffer (inv #1/#2);
   binds/listeners stay bounded and non-blocking (inv #10). Dual-stack bind is purely an addressing
   default.

## Consequences

- The project has an explicit, enforceable IPv6-first guard rail; IPv4-only code now reads as a
  defect, not an intentional choice, and the `IPV6-0..` backlog enumerates the fixes.
- The most user-visible quick wins (dual-stack `[::]` binds, IPv6 example addresses, the `IN IP6` SDP
  parse/generate) are small, pure changes; the broader sweep (every socket, every example, IPv6
  multicast join) is sequenced in the backlog.
- Dual-stack `[::]` defaults keep legacy IPv4 clients working through the deprecation window — the
  posture is IPv6-first, **not** IPv6-only (yet); the eventual IPv4 removal is a future, signposted
  step.
- The merged multicast ([ADR-0040](ADR-0040.md)) and SAP ([ADR-0041](ADR-0041.md)) docs are corrected
  to be IPv6-first in the same change that lands this ADR (no IPv4-first text left on `main`).
- Status **Accepted** (not merely Proposed): this is a standing operator directive, effective
  immediately for all new work.

## Alternatives rejected

- **Leave IPv6 support implicit / per-feature.** Rejected: that is exactly what produced the audit's
  IPv4-only accretion. An explicit canonical convention + ADR is required so the rule is enforceable
  in review and new code doesn't replicate the IPv4-only pattern.
- **Go IPv6-only immediately (drop IPv4 now).** Rejected: legacy IPv4 peers/clients still exist and
  must interop during the transition. The decision is IPv6-**first** with dual-stack defaults and a
  *deprecation path*, not an immediate removal.
- **Default binds to `0.0.0.0` (IPv4) with optional IPv6.** Rejected: that is IPv4-first and the
  violation the operator flagged. Default to `[::]` dual-stack; a v4-mapped path still serves legacy
  clients.
- **Fix only the newly-merged multicast/SAP docs** (the references the operator happened to notice).
  Rejected: the audit shows IPv4-only assumptions are repo-wide (SDP parser, WHEP answer, telemetry
  binds, multicast join). Document the principle once (conventions §10 + this ADR) and backlog the
  whole sweep, rather than patching one corner.
- **Add IPv6-first as a 12th data-plane invariant** (CLAUDE.md §2 table). Rejected: the 11 invariants
  are output/engine data-plane guarantees (output-clock, isolation, …); IPv6-first is an
  addressing/networking *convention* (conventions §10), not a per-frame data-plane invariant. Keeping
  it a convention + ADR is the right altitude.
