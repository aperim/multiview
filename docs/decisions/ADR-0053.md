# ADR-0053 — Conspect support: ticketing, redacted context-pack, data-request local approval, append-only audit

**Status:** Proposed (2026-06-10)
**Area:** Account / Support — the ticketing + diagnostics + audit plane of the Conspect subsystem
**Drivers:** "My machine is misbehaving" must become an **auditable, privacy-respecting** exchange:
the operator can raise a ticket, attach a **redacted** diagnostics bundle, and — crucially — **no data
leaves the machine without explicit local approval**, with every account-side action written to an
**append-only** audit trail. This is the operator's evidence trail and a support precondition.
**Research:** [conspect-account-architecture](../research/conspect-account-architecture.md) §10, §11.
**Related:** [ADR-0052](ADR-0052.md) (the consent-independent local metrics buffer the context-pack is
built from), [ADR-0050](ADR-0050.md) (licence/enforcement state included in the pack),
[ADR-M006](ADR-M006.md) (config-as-code with **reference-only secrets** — the pack carries secret
*references*, never values), [ADR-I003](ADR-I003.md) (the in-memory-default / sqlite-feature
persistence pattern reused for the stores), [ADR-W015](ADR-W015.md) (the existing read-only audit log
the account audit mirrors), [ADR-RT004](ADR-RT004.md) (isolation invariant #10).
**Extends:** `multiview-control` (additive — new account/support routes + stores; existing routes
untouched).

## Context

Three properties make support trustworthy: (1) the diagnostics bundle is **redacted** — diagnostics,
never media, never raw identifiers (brief §8), never secret *values* (only references per
[ADR-M006](ADR-M006.md)); (2) **egress requires local approval** — if support or the licence server
asks for more, the request appears as a **pending action** the operator must approve before anything
leaves; (3) the trail is **append-only** — claim, transfer, lease grant, enforcement change, consent
change, context-pack export, and data-request approve/deny are immutable, timestamped entries.

## Decision

### 1. Tickets as control-plane resources

`/api/v1/account/support/tickets` — create, list, read, comment (append-only thread). Each ticket
carries a status and an immutable comment thread. Standard `multiview-control` contract (RFC 9457,
`ETag`/`If-Match` where mutable, RBAC `Principal`).

### 2. Context-pack — redacted, operator-initiated, `202`

`POST /api/v1/account/support/tickets/{id}/context-pack` (returns `202 + operation_id`) assembles a
bundle from: the **consent-independent local metrics buffer** ([ADR-0052](ADR-0052.md) S5) + the
licence/enforcement state ([ADR-0050](ADR-0050.md)) + config-as-code with **secrets as references, not
values** ([ADR-M006](ADR-M006.md)). The pack contains **diagnostics only** — never media, never raw
serials/MACs/URLs, never secret values (brief §8). It is built and attached to a ticket **only on
explicit operator action**; it is never auto-collected or auto-sent.

### 3. Data-request local approval — no egress without it

If support (or the licence server) requests additional data, the request appears as a **pending
action** at `GET /api/v1/account/support/data-requests`, awaiting **local approval**
(`POST …/{id}/approve`) or denial (`POST …/{id}/deny`). **No data egress occurs without an explicit
local approval — full stop.** The pending-action chrome strip in the SPA surfaces these (research
§13.7). Approval and denial are themselves audited (§4).

### 4. Append-only audit store

`GET /api/v1/account/audit` exposes an **append-only** log of every account-side action: claim,
transfer, lease grant, enforcement-level change, consent change, context-pack export, data-request
approve/deny. **Append-only** = there is **no** update or delete route; entries are immutable and
timestamped. The store is in-memory by default and sqlite-backed under the existing `sqlite` feature
([ADR-I003](ADR-I003.md) pattern); it is rotated by size/age like the existing change-audit log
([ADR-W015](ADR-W015.md)) but never edited in place.

### 5. Isolation (invariant #10)

Every store and the context-pack builder are **control-plane** concerns, off the hot loop. The
context-pack reads the *already-published* local buffer and licence status (wait-free reads); it never
touches the engine, never holds an engine lock, and is never `.await`ed by the engine. Building a large
pack cannot stall a tick.

### 6. Feature ladder + licensing

Additive routes + stores in `multiview-control`; in-memory default, sqlite under the existing `sqlite`
feature. No new licence-escalating dependency.

## Consequences

- **Trustworthy support** — diagnostics are redacted, egress needs local approval, the trail is
  immutable. The operator can prove what was shared, when, and by whom.
- **Privacy by construction** — no media, no raw identifiers, no secret values ever enter a pack
  (reuses [ADR-M006](ADR-M006.md) reference-only secrets and the brief §8 minimisation rule).
- **Append-only evidence** — claim/transfer/lease/enforcement/consent changes are an immutable
  timestamped record (no edit/delete route).
- **Additive + collision-light** — new modules in `multiview-control`; existing routes untouched; one
  coordinated OpenAPI/route registration with the WebUI session (research §15).
- **Isolation preserved** — all support work is off-hot-loop control-plane; invariant #10 holds.

## Alternatives rejected

- **Auto-collect + auto-send diagnostics on error.** Egress without consent; a privacy hazard and a
  directive violation. Rejected — operator-initiated context-pack + explicit local approval for any
  further egress.
- **Mutable audit log (allow edit/delete to "fix mistakes").** Destroys the evidence value of the
  trail. Rejected — append-only, no update/delete route, immutable entries.
- **Ship raw config (secret values) in the context-pack "to help debugging".** Leaks credentials.
  Rejected — reference-only secrets per [ADR-M006](ADR-M006.md).
- **Reuse the existing change-audit log for account actions.** Conflates engine/config changes with
  account/licensing actions and muddies access control. Rejected for a dedicated append-only account
  audit (same persistence pattern, separate store + route).
