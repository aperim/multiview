## Context

The operator asked for "support for the RM-LP350G MIDI (and future MIDI control surfaces)." As a
**design-time author note** (to be confirmed against the tree before implementation, not relied on as a
reviewed fact), no control-surface, MIDI, HID, or fader-ingress code is expected anywhere in the
workspace: the apparent `fader`/`surface`/`HID` matches are the overlay timer, the tally arbiter, and
the HA transport — none a physical-surface ingress. What **does** exist is the substrate a surface must
reuse, not duplicate:

- A **bounded, non-blocking command bus** the control plane already submits onto and the engine drains
  at its leisure: `command_bus(capacity)` (`crates/multiview-control/src/command.rs:476`), the
  `Command` enum (`:174`), `CommandReceiver::try_drain` (`:515`). This is invariant #10 in code.
- The **managed-device** desired/live split and compiled-in driver tier
  ([ADR-M008](ADR-M008.md), [managed-devices.md](../research/managed-devices.md)) with supervised
  reconnect and an alarm ladder.
- The **switcher verb surface** ([ADR-W021](ADR-W021.md)) — bare-verb POSTs, explicit-state idempotent
  commands, the integer-basis-point T-bar setter — whose §13 *external-control contract* was
  deliberately shaped so "an OSC namespace and a MIDI surface adapter are adapters over this surface,
  not new surfaces."
- The **audio mixer** targets ([ADR-0077](ADR-0077.md)): strip faders, gain-preserving mutes, sends.
- The **realtime live-state mirror** + conflated meter lane ([ADR-RT008](ADR-RT008.md),
  `realtime-api.md:173`) — the source for surface feedback (lamps, motor faders, meter bridges).
- The **tally arbiter** (`crates/multiview-engine/src/tally/arbiter.rs:145`) — the proven
  resolve-facts-into-per-lamp-state pattern feedback reuses.
- The **broadcast-cues** rule table ([broadcast-cues.md](../research/broadcast-cues.md),
  [ADR-0072](ADR-0072.md)) — a sibling ingress that maps external *events* onto the same verbs; a
  surface is the same shape with *physical inputs*.

**Verification of the named device:** public sources (B&H, Markertek, Sweetwater, JVC pro pages)
confirm the **RM-LP350G** is a **JVC** desktop **vMix** control surface over **USB** (T-bar, 12-ch
PGM/PVW, PTZ joystick), configured by importing a vMix "shortcut template + activators." **No public
source confirms its wire protocol** — it may be class-compliant MIDI, USB-HID, or vMix-proprietary.
The request assumes MIDI; the evidence does not establish it. This ADR must therefore be honest: build
the generic model, ship a profile *slot* for the RM-LP350G, and resolve its binding on hardware.

## Decision

**Adopt a generic control-surface abstraction in the control plane, with MIDI as the first transport
and the RM-LP350G as a capture-confirmed profile. The surface adds no new command vocabulary.**

1. **A surface is a managed device** ([ADR-M008](ADR-M008.md)). Desired state in config (which
   profile, which mapping table, which transport binding); live state (connected, last input, current
   feedback) engine-/control-plane-owned and mirrored read-only. It reuses the device registry,
   supervised reconnect, and alarm ladder — **no new tier**.

2. **The model is five types** (per [control-surfaces-midi.md](../research/control-surfaces-midi.md)
   §1): `Surface`, `SurfaceProfile` (the physical control list, each control transport-addressed),
   `ControlKind` (internally tagged `Button`/`Encoder`/`Fader`/`Joystick`/`Lamp`/`LedRing`/
   `MotorFader`/`MeterBridge` — `tag = "kind"`, never `untagged`), `SurfaceInput` (`{surface_id,
   control_id, value}`, value ∈ `Press|Release|Delta(i32)|Absolute(u16 basis-points)`), and a
   `MappingTable` of `control_id → Action`. Absolute values use the integer basis-point convention
   ([ADR-W021](ADR-W021.md) §1 T-bar; inv #3 — never float).

3. **Input path:** `decode(raw) → SurfaceInput → lookup(MappingTable) → per-surface ingress lane →
   Command → command_bus` (`command.rs:476`). The driver decodes; the mapping resolves; the **existing**
   command bus carries it; the engine drains at the frame boundary (`:515`). High-rate controls (T-bar,
   faders) are conflated to latest-wins **absolute setters** before the bus — never one command per MIDI
   byte (the W021 §1 T-bar discipline). To stop a single flooding surface from evicting unrelated
   SPA/cue/operator traffic, each surface drains through its **own bounded, conflating ingress lane**
   before the shared bus; on overrun only **that surface's own** events are dropped/conflated (with
   overload telemetry), and the shared bus's drop-oldest is only the last-resort backstop (brief §3).

4. **Mapping targets existing verbs only** (§3 table in the brief): switcher verbs
   ([ADR-W021](ADR-W021.md)), macro run ([ADR-W021](ADR-W021.md) §1 / [ADR-M012](ADR-M012.md) §1),
   audio fader/mute/send ([ADR-0077](ADR-0077.md)), PTZ → managed-device verb. Each action inherits
   its Class-1/Class-2 classification from the verb ([ADR-M012](ADR-M012.md) §9, inv #11); a surface
   cannot change a verb's class.

5. **Feedback path:** `RT008 live-state delta → FeedbackPlan(profile) → SurfaceOutput → output port`,
   reusing the tally-arbiter resolve pattern (`arbiter.rs:145`) and the realtime conflating-sampler
   (`realtime-api.md:173`). Motor-fader feedback is suppressed while a control reports "touched"
   (MCU/HUI touch-sense) to avoid an echo loop. The engine **live state is the single truth**; a
   surface fader is an input, its motor a follower. Because a drop-oldest queue alone does not make the
   physical write non-blocking, the device write runs on a **dedicated per-surface writer worker** fed
   by a bounded latest-value queue with a write deadline; a wedged write trips disconnect-and-reopen on
   that worker and never stalls feedback planning, reconnect handling, or the engine (brief §1.2).

6. **MIDI transport via [`midir`](https://docs.rs/midir)** behind an off-by-default `midi` Cargo
   feature on `multiview-control` (no new crate, per [ADR-0054](ADR-0054.md)). `midir` gives ALSA +
   CoreMIDI (the Linux + macOS platform matrix), virtual ports, and SysEx. USB-MIDI and 5-pin DIN are
   in MVP scope; MIDI 2.0 / UMP is an additive upgrade behind the same abstraction; RTP-MIDI and OSC
   are documented siblings (§5 of the brief) — when added they are **networked and IPv6-first** (bind
   `[::]`, bracket literals; [ADR-0042](ADR-0042.md)). The `midir` callback thread does only the cheap
   decode and hands off over a bounded channel — it never blocks, never touches the engine.

7. **The RM-LP350G is a profile slot, not an asserted binding.** Ship the generic model first, plus an
   `rm-lp350g` `SurfaceProfile` naming its controls by *function* with transport addresses
   **captured on hardware via a learn mode**, never guessed. If the unit turns out to be HID, the HID
   transport sibling carries it with the mapping/feedback model unchanged. Nominative trademark use
   only (mirroring the ZowieTek posture, `managed-devices.md:584`); **no vendor SDK bundled, no
   vendor-published map redistributed.**

8. **Bad inputs are the purpose.** The MIDI decoder is fuzz/property-tested to never panic on
   malformed status bytes, truncated SysEx, or running-status edge cases; it logs-and-drops the
   unknown and keeps the surface alive. A CI chaos test wedges the surface output port and asserts the
   engine keeps ticking (the ADR-0059 audio-consumer chaos pattern).

## Consequences

- **Positive.** Multiview gains physical-surface control as a thin adapter over verbs that already
  exist — the cash-in of the W021 §13 external-control contract. The model is transport-agnostic
  (MIDI now; HID and OSC behind the same abstraction), so "future MIDI control surfaces" and software
  panels are profiles/transports, not new subsystems. Feedback reuses the tally arbiter + realtime
  sampler verbatim.
- **Cost.** `multiview-control` grows a `midi`-gated surface module (model + decoder + feedback
  planner) and a profile loader; the default build stays pure-Rust and GPU-free (feature off).
  `midir`'s platform backends must pass `cargo deny` under the feature. Each surface adds one MIDI
  in/out (or HID/OSC) endpoint with supervised reconnect.
- **Honest gap.** The RM-LP350G binding is **unresolved** until the physical unit is enumerated on a
  bench; this ADR does not claim MIDI support for that specific device, only the generic model + a
  capture-confirmed slot. The open question is carried in the brief, not silently closed.
- **Risk.** A flooding or stuck surface is bounded **first** by its own per-surface conflating ingress
  lane (so it cannot evict other clients' bus traffic) and the dedicated drop-oldest feedback writer
  worker, with the shared command-bus drop-oldest as the last-resort backstop; the chaos gate proves it
  cannot stall the engine.

## Alternatives considered

- **An RM-LP350G-specific driver (no generic model).** Rejected: the request explicitly includes
  *"future MIDI control surfaces,"* and the RM-LP350G's binding is unverified — a device-specific
  driver would be both narrower than asked and possibly built on a wrong protocol assumption. The
  generic model is correct whichever way the device resolves.
- **Per-action WebSocket/MIDI RPC straight into the engine.** Rejected for the same reason W021
  rejected WS RPC: the repo convention is verbs on the bounded command bus (`command.rs:476`); a
  direct engine channel would risk invariant #10 and duplicate classification/idempotency/audit that
  the bus gives for free.
- **A new `multiview-surface` crate now.** Deferred, not adopted: [ADR-0054](ADR-0054.md) pins
  no-new-crates; a `midi`-gated module within `multiview-control` is sufficient until HID + OSC +
  RTP-MIDI grow the code enough to justify a crate (reversible — Open question 2 in the brief).
- **Asserting the RM-LP350G is MIDI and shipping guessed CC/note numbers.** Rejected as dishonest and
  brittle: no source confirms the protocol; a guessed map would silently mis-bind. Capture-and-confirm
  on hardware is the only sound path.
- **A second authority for live state on the surface side** (the panel holds its own truth).
  Rejected: creates a fight between the SPA and the panel; engine live state is the single truth and
  the motor fader follows it (W021's latest-wins idempotent contract).

## How this honors the invariants

- **#1 Output-clock — untouchable.** A surface event (button/fader/T-bar) is an input *sample* placed
  on the bounded command bus and drained by the engine at the frame boundary (`command.rs:515`); it is
  **never** a pacing signal. The engine's cadence is independent of any surface; a dead, flooding, or
  malformed surface cannot slow or stall a single output tick. Absolute values are integer basis
  points / integer frames (inv #3) — no float enters the timing path.
- **#10 Isolation — the engine never awaits the surface.** Input flows surface → per-surface bounded
  conflating ingress lane → shared bounded command bus (drop-oldest as last-resort backstop); feedback
  flows engine-owned live state → conflating sampler → surface output port via a dedicated drop-oldest
  writer worker, the engine never blocking on a USB/MIDI/OSC write. This is the
  [ADR-RT004](ADR-RT004.md) / [realtime-api](../research/realtime-api.md) "physically incapable of
  back-pressuring the engine" rule applied to a physical panel; a CI chaos test (wedge the output
  port, assert the engine keeps ticking) gates the lane.
