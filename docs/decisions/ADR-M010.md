# ADR-M010: Multi-output timing & sync — outbound WallClockRef epoch, link offset, sync groups, published tiers

- **Status:** Proposed
- **Area:** Management
- **Date:** 2026-06-10
- **Source brief:** [display-out.md](../research/display-out.md) (presentation model) + [managed-devices.md](../research/managed-devices.md) (sync groups)

## Decision

The engine computes **one outbound presentation epoch per program**: a `WallClockRef` — the existing exact-affine media↔wall map in `crates/multiview-core/src/wallclock.rs` ([ADR-0038](ADR-0038.md); i128 intermediates, never float) — anchoring the output tick counter to disciplined wall-clock nanoseconds. The wall estimate comes from the existing ST 2059-2 PTP servo (`crates/multiview-engine/src/ptp.rs`, PHC binding behind the off-by-default `ptp` feature) or from chrony-disciplined system time. **PTP never paces the tick loop** (the servo's own contract, invariant #1, [ADR-T001](ADR-T001.md)): it disciplines a reference *estimate* and may at most one-time-seed the clock origin; the cadence stays the exact rational. Distribution is out-of-band-first: (a) publish `{stream_id, WallClockRef, link_offset, clock_source/quality}` on the existing control WebSocket (`/api/v1/ws`, versioned envelope in `multiview-events`, already isolation-safe per [ADR-RT004](ADR-RT004.md)); (b) stamp **RTCP Sender Report** NTP↔RTP mappings from the same epoch on the RTSP output; (c) emit `EXT-X-PROGRAM-DATE-TIME` on HLS; (d) optionally advertise RFC 7273 `a=ts-refclk`/`a=mediaclk` in the RTSP SDP for standards-based receivers. Items (b)–(d) are **new outbound work** — `multiview-output` emits none of them today.

Receivers apply **AES67 link-offset semantics to video**: a fixed per-deployment receiver delay (default ≈ 2× max network jitter + decode time, typically 100–300 ms) added to `wall_at(pts)`; **uniformity across nodes is the goal, not smallness**. Our display nodes use pure frame-choice presentation discipline per [ADR-0045](ADR-0045.md) — decode into a 2–3-frame queue, at each flip present the frame whose `wall_at(pts) + link_offset` is closest to the predicted next vblank, repeat if early, drop if late — nodes are pull-side consumers and **never feed back to the engine**. **Sync groups** are a first-class Devices-domain resource ([managed-devices.md](../research/managed-devices.md)): achieved tier = **weakest member**, displayed immediately and never over-claimed; per-member `offset_ms` trim; a measure/test-pattern action (burnt-in frame counter + binary flash); drift alarm when measured skew exceeds `target_skew_ms` past a dwell.

The published tier table is part of the decision — UI badges and docs state the achieved tier per output class, never a blanket "synchronized" claim: **S** same-node multi-head (frame-accurate, sub-ms flip delta via one atomic commit; CRTCs still free-run); **A** display nodes + PTP (frame-accurate — same frame index — plus 0–1-refresh vsync-phase residual); **B** display nodes + chrony (frame-accurate; occasional ±1-frame decisions at boundaries); **C** vendor decoders such as ZowieBox-class devices (bounded drift ±100–500 ms; optional *scheduled re-align* that blanks **that device** 1–3 s); **D** Cast (none / seconds; never part of a synchronized canvas). We will **never**: attempt scanout-phase (genlock-grade) alignment; pace the engine from any external clock or node; or default to multi-node audible audio — the default is a **single-audio-node policy** per group (two free-running rooms of audio = comb filtering/echo), with disciplined multi-node audio opt-in **on our nodes only** and never two vendor decoders audible in one room.

## Rationale

This is ST 2110's timing model minus uncompressed essence: a shared disciplined clock, a published epoch, and a fixed receiver-side offset — proven semantics (AES67 link offset; GStreamer net-clock/RFC 7273 deployments measure ~50–500 µs skew on LAN) without adopting any of their transports. Carrying the epoch out-of-band on the control WS covers every media transport with one mechanism, reuses an existing type and an existing isolation-safe channel, and degrades gracefully: a node that loses the WS keeps the last epoch and free-runs drift-bounded on its disciplined clock — output never falters. Frame accuracy (identical frame index everywhere) is the honest, industry-real tier achievable on commodity hardware; tier honesty is structural because Tier C/D devices expose no timing knobs we could engineer around.

## Alternatives considered

**GStreamer-style net-clock slaving of the engine** (slave the output pipeline to a shared network clock): rejected — it makes an external clock a pacing authority over the tick loop, violating invariant #1; we discipline the *label* on the timeline, never the cadence. **Per-node free-run + manual offsets**: rejected — unbounded relative drift means perpetual manual re-trim and no measurable sync claim; it is the failure mode we degrade *to*, not a design. **In-band-only timing (RTCP SR alone)**: rejected as sole mechanism — it is per-transport (HLS/SRT/TS have no SR path), depends on receiver opt-in semantics, and is awkward to version/extend; the WS epoch covers all transports uniformly, with SR/PDT/RFC 7273 emitted *in addition* for standards interop. **PTP-mandatory**: rejected — commodity deployments often lack PTP infrastructure; chrony at ~0.5–1 ms LAN error is ~1/30 of a 60 Hz frame period and still yields frame-accurate Tier B, so PTP upgrades the tier rather than gating the feature.

## Consequences

RTCP SR and `EXT-X-PROGRAM-DATE-TIME` emission (plus optional RFC 7273 SDP attributes) is genuinely new code in `multiview-output`, stamped from the engine epoch — verified absent today. The link offset adds a fixed 100–300 ms of presentation latency to every synchronized node; that is the price of uniformity and must be surfaced in the UI. Tier honesty is a UI obligation: sync-group badges show the weakest-member achieved tier, Tier-C re-align visibly blanks the affected device, and Tier-D devices can never join a synchronized canvas. The vsync-phase residual (up to one refresh) is the documented floor — no commodity path to scanout-phase lock exists, and we do not promise one. The frame-accurate claim (Tiers S/A/B) is gated by an acceptance soak before it appears in any badge or doc: burnt-in frame-counter + flash photographed across all displays at ≤1/1000 s (≥99% identical counters, max delta ≤1 frame, automatable via USB camera + OCR), photodiode phase measurement at the 50 µs class, servo offset/ppb exported via `multiview-telemetry` (pass = 99th-percentile |offset| ≤100 µs PTP / ≤1 ms chrony over 24 h), plus the invariant-#1 chaos extension — kill PTP and the WS mid-soak; output cadence must be unaffected and nodes must degrade to drift-bounded free-run.
