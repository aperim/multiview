# ADR-RT006: Document the event API with AsyncAPI 3.0 from shared types, served beside Scalar, with codegen'd typed clients

- **Status:** Proposed
- **Area:** Realtime API
- **Date:** 2026-06-02
- **Source brief:** [realtime-api.md](../research/realtime-api.md)

## Decision

Specify the realtime API in AsyncAPI 3.0 generated by an xtask from the same serde+schemars Rust types in multiview-events that feed utoipa's OpenAPI (single source of truth), with a documented serde_json post-process to inject WS bindings the generator lacks, validated by the AsyncAPI CLI. Serve OpenAPI via Scalar at /docs and AsyncAPI via @asyncapi/react-component at /docs/events from the binary (rust-embed), plus a small in-browser WS 'Try it' console. Generate typed TS (Modelina models + thin hand-written client owning lifecycle/resume) and a Rust client; CI regenerates OpenAPI/AsyncAPI/TS and fails on any git diff. The web app seeds TanStack Query from the snapshot, reconciles deltas via a type→reducer map, routes high-rate meters to a ref store, and confirms optimistic mutations via corr/mutationId.

## Rationale

AsyncAPI is the event-driven analog of OpenAPI; deriving both specs from one set of Rust types makes drift structurally impossible and lets REST and events share schema components. The interactive docs + WS console make the event API exercisable like Swagger's 'Try it'. The generated-types + hand-written-runtime split keeps the conflation/resume semantics under our control while still being typed and CI-guaranteed against the served spec.

## Alternatives considered

Hand-written AsyncAPI/JSON Schema (guaranteed drift); no event-API docs (untestable, hard to integrate); fully generated WS client runtime (Glee/community templates fight conflation/resume); documenting events inside OpenAPI only (no first-class channel/operation/oneOf event model).

## Consequences

asyncapi-rust (v0.2, Nov 2025) is young and lacks WS bindings, requiring a documented, tested post-process step (with a hand-assembly fallback from schema_for!(T)); treat spec generation as its own small tested xtask. Binary meter frames need explicit contentType documentation. The @asyncapi/react-component dependency must be verified to render the exact 3.0 features used (use the UMD bundle + keep the WS console separate so a docs regression can't break live testing).
