# ADR-0064: PTZ control model — one vendor-neutral command model, two driver profiles (ONVIF PTZ + VISCA / VISCA-over-IP)

- **Status:** Proposed
- **Area:** Management (Devices domain) / Control plane
- **Date:** 2026-06-13
- **Source brief:** [onvif-and-ptz.md](../research/onvif-and-ptz.md)
- **Extends:** [ADR-M008](ADR-M008.md) (closed driver enum, fixed capability flags), [ADR-M009](ADR-M009.md) (device facets), [ADR-RT007](ADR-RT007.md) (devices realtime topic), [ADR-W017](ADR-W017.md) (bare-verb routes)
- **Relates to:** [ADR-0062](ADR-0062.md) (ONVIF client — the ONVIF-PTZ profile lives here), [ADR-0063](ADR-0063.md) (ONVIF server — the PTZ-passthrough relay uses this model), [control-surfaces-midi.md](../research/control-surfaces-midi.md) (a joystick/keypad source of PTZ commands), invariants #1/#10

## Context

The operator asked: **"PTZ — is there a standard?"** Yes — **two** widely-used ones:

- **ONVIF PTZ** (the IP-CCTV standard): `AbsoluteMove`, `RelativeMove`, `ContinuousMove`, `Stop`;
  presets (`SetPreset`/`GetPresets`/`GotoPreset`/`RemovePreset`); **PTZ nodes** that declare the
  coordinate **spaces** they support (each a space-URI; ONVIF §5.7 defines the standard generic
  normalized pan-tilt/zoom position+velocity spaces). (web-verified: ONVIF PTZ Service Specification.)
- **VISCA / VISCA-over-IP** (the broadcast PTZ-camera standard): the VISCA byte protocol over serial
  or IP; the common Sony-style VISCA-over-IP profile defaults to UDP/TCP **52381** with an 8-byte
  header carrying a payload type, a **16-bit payload-length field**, and a sequence number, then
  ACK/Completion replies. The exact header layout is profile-specific — the parser is pinned to the
  documented public profile, not a standalone "4th byte". (web-verified: AVer/Sony VISCA-over-IP
  guides — a de-facto vendor standard, no neutral SDO owns the public spec.)

PTZ commands reach a camera from three control-plane origins: the Multiview UI, a control surface
(joystick/MIDI — [control-surfaces-midi.md](../research/control-surfaces-midi.md)), and the
ONVIF-server PTZ-passthrough relay ([ADR-0063](ADR-0063.md)). All must present the operator the **same
controls** whether the bound camera is ONVIF or VISCA. The Devices domain mandates **closed typed
enums + fixed capability flags, no property bags** ([ADR-M008](ADR-M008.md)).

## Decision

Pin **one vendor-neutral internal PTZ command model**, with **two driver profiles** implementing it,
and a **fixed `ptz` capability facet** on the driver capability set.

1. **`PtzCommand` — a closed, typed enum** with **normalized `[-1.0, 1.0]` axes** (pan, tilt, zoom):
   `ContinuousMove { pan, tilt, zoom }` (velocity vector, runs until `Stop`), `Stop`,
   `AbsoluteMove { pan, tilt, zoom, speed }`, `RelativeMove { d_pan, d_tilt, d_zoom, speed }`,
   `GotoPreset { id }`, `SetPreset { id?, name }`, `RemovePreset { id }`, `ListPresets`. Floats are
   fine here — PTZ axes/velocities are genuinely continuous quantities (the "floats for gain/geometry,
   never for time" rule). A **`PtzNode` capability descriptor** (fixed flags: which axes,
   absolute-vs-relative-vs-continuous support, preset support, preset count) is read from the camera at
   adopt and gates the UI (capabilities the camera lacks are greyed out, never colour-alone — WCAG).

2. **Two driver profiles under the one model:**
   - **ONVIF-PTZ** (part of the `onvif` driver, [ADR-0062](ADR-0062.md)): translate `PtzCommand`
     to/from ONVIF PTZ ops, mapping our `[-1,1]` axes through the node's **declared coordinate spaces**
     (`GetNodes`/`GetConfigurations`). Where a camera exposes only device-specific (non-normalized)
     spaces, `ContinuousMove` (velocity) stays robust while `AbsoluteMove` is **approximate** and
     badged as such (UI honesty).
   - **VISCA-IP**: translate `PtzCommand` to/from VISCA byte commands over UDP/TCP (default 52381),
     handling the header + ACK/Completion handshake; pan/tilt drive+stop, zoom, presets, and
     absolute/relative where the camera supports position inquiry. Built from publicly-documented
     command tables, **no vendor SDK**, nominative use only.

3. **Device modelling.** A camera that is both ONVIF and VISCA is **one** `onvif` device with a
   `ptz_profile = "onvif" | "visca-ip"` selector (default `onvif`). A **VISCA-only** camera (no
   ONVIF/RTSP, control-only) is modelled as a thin variant whose only facet is PTZ — **leaning** a new
   `DeviceDriver::Visca` variant to keep the closed-enum/fixed-flags discipline, but the exact choice
   (own variant vs a flag) is an explicit open question (brief §8) resolved when the first VISCA-only
   device is built.

4. **PTZ is control-plane, isolated, and rate-limited.** PTZ verbs are driver-actor calls
   ([ADR-M009](ADR-M009.md) facet model); they are **serialised per device** (the `zowietek`
   per-device serialisation rule) and **rate-limited** so joystick spam cannot DoS a camera or its
   driver task. PTZ position is reported as a **latest-wins** sub-field of `device.status` on
   `Topic::Devices` ([ADR-RT007](ADR-RT007.md)) — never a new topic.

5. **API** (bare-verb routes, [ADR-W017](ADR-W017.md), under `/api/v1`): `GET /devices/{id}/ptz`
   (node capabilities + last-known position); `POST /devices/{id}/ptz/move` (typed `PtzCommand` body)
   → `200` for fast continuous/stop, `202`+op-id only where the camera op is genuinely long-running;
   `POST /devices/{id}/ptz/stop`; `GET/POST /devices/{id}/ptz/presets` + `…/presets/{p}/goto`. The
   same routes serve UI, control-surface, and (via [ADR-0063](ADR-0063.md)) relayed-NVR origins.

## Consequences

- **One operator experience, every camera.** ONVIF and VISCA cameras present identical PTZ controls;
  the relay ([ADR-0063](ADR-0063.md)) and control surfaces speak the one model. **Answer to the
  operator: yes, PTZ is standardised — ONVIF PTZ and VISCA/VISCA-over-IP — and Multiview speaks both.**
- **Honest capability gating.** Absolute-position fidelity varies by camera; the `PtzNode` descriptor
  + approximate-absolute badge make the limits visible rather than silently wrong.
- **A new family/variant always costs a typed addition + ADR** (no plugins, [ADR-M008](ADR-M008.md));
  VISCA-only modelling is a small open decision, not an open registry.
- **FreeD (camera-position telemetry) is *not* built here** but the axis model is designed not to
  preclude a future FreeD-out facet (brief §5.5).
- **VISCA is a de-facto vendor protocol** implemented clean-room from public docs, nominative use only,
  with a non-affiliation disclaimer — deferring to operator/counsel on any vendor's protocol-terms
  stance (the [managed-devices.md §12](../research/managed-devices.md) posture).

## How it honors the invariants

- **#1 (output-clock untouchable):** PTZ is pure control plane — a small UDP/TCP datagram (VISCA) or
  one SOAP call (ONVIF) per command, on the camera's driver task. **No** PTZ path touches the
  output-clock loop, the compositor, or any encode; program output is never contingent on a PTZ
  command completing.
- **#10 (isolation):** PTZ driver actors publish into bounded watch/broadcast channels and are **never
  awaited by the engine**; rate-limiting + per-device serialisation bound the work; a hung camera
  stalls only its own driver task. Position telemetry is latest-wins on the conflated devices topic.

## Alternatives considered

- **Two separate PTZ surfaces (an ONVIF one and a VISCA one).** Rejected: forces the operator and
  every command origin (UI, control surface, relay) to branch on transport; a single normalized
  `PtzCommand` with two driver profiles is the DRY/typed answer.
- **An open PTZ property bag (arbitrary vendor commands).** Rejected: untyped bags cannot gate the UI
  exhaustively and drift from real behaviour ([ADR-M008](ADR-M008.md) fixed-flags discipline).
- **Bundle a vendor VISCA/ONVIF SDK.** Rejected: open-standard/clean-room, no-vendor-SDK,
  `unsafe_code = forbid`, LGPL-clean-default posture.
- **PTZ as raw passthrough only (no internal model).** Rejected: the ONVIF-server relay and control
  surfaces need a normalized model to translate *into*; raw passthrough cannot map a joystick to two
  protocols or badge capabilities.
- **Non-normalized native axes (pass vendor coordinates through).** Rejected: a control surface or
  relay would behave differently per camera; `[-1,1]` normalization with a per-node space mapping is
  the portable contract (absolute-position fidelity honestly badged where the camera lacks normalized
  spaces).
