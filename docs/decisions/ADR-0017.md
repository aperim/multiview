# ADR-0017: GPU/CPU utilisation monitoring + affinity-aware least-loaded multi-GPU work placement

- **Status:** Proposed
- **Area:** Efficiency
- **Date:** 2026-06-03
- **Source brief:** [gpu-monitoring-and-scheduling.md](../research/gpu-monitoring-and-scheduling.md)
- **Extended by:** [ADR-0018](ADR-0018.md) (the full affinity-first placement decision engine + deliberate split + closed-loop re-placement, built on this brief's monitoring + ranking function)

## Decision

Add a feature-gated, runtime-loaded, **off-hot-path live-load probe** in `multiview-hal` that samples per-device GPU/CPU utilisation at ~1-4 Hz — per-device compute-busy %, VRAM used/total (bytes), and **per-engine encoder/decoder utilisation where the vendor exposes it** — and publishes a wait-free `DeviceLoad` snapshot to the scheduler plus per-GPU Prometheus gauges to `multiview-telemetry`. Vendor access is runtime-loaded behind off-by-default features (conventions §7): NVIDIA via **NVML** (`nvml-wrapper`/`libnvidia-ml.so.1`, `cuda` feature); Linux Intel via **i915 PMU + DRM fdinfo** (`vaapi`/`qsv`); Linux AMD via **amdgpu sysfs `gpu_busy_percent` + DRM fdinfo** (`vaapi`); Apple via **IOReport** (private) + `ProcessInfo.thermalState` (`videotoolbox`/macOS).

When the operator has **not** pinned GPUs to tasks, place each pipeline by an **affinity-gated least-loaded** policy: among the GPUs that can host the *whole* `decode -> composite -> encode` pipeline (the zero-copy-island gate, [ADR-0004](ADR-0004.md)) and pass the hard admission gates (capability, per-engine Mpix/s cost budget, free VRAM, and the **per-system NVENC concurrent-session ceiling**), choose the GPU minimising a composite load score over `{free VRAM (primary), encoder-util headroom, decoder-util headroom, NVENC session headroom, compute-busy}`. Splitting a single pipeline across GPUs is a **last-resort, explicitly cost-accounted** degradation step taken only when no single GPU fits. Selection happens at **admission/reconfig time only** (never per-frame); re-balancing a running set is a hysteresis-gated, make-before-break (Class-2) migration ([ADR-R004](ADR-R004.md)). An explicit operator **pin** (by stable device UUID / PCI bus id / Metal registryID) always overrides the auto policy.

## Rationale

The operator's working hypothesis — least-loaded by VRAM + enc/dec utilisation — is the right *ranking function* but an incomplete *algorithm*: it ignores that cross-vendor/cross-GPU on-GPU zero-copy does not exist on desktop ([ADR-0004](ADR-0004.md)), so splitting a pipeline across GPUs costs a PCIe round-trip + latency per frame. Affinity must therefore be a **hard gate before** the least-loaded ranking, not a term in it.

Verification (2026-06-03) confirmed the metric availability is **vendor-asymmetric**, so the policy must degrade gracefully: per-engine encoder/decoder utilisation is cleanly available on NVIDIA (NVML `nvmlDeviceGetEncoderUtilization`/`DecoderUtilization`, 0-100 % + `samplingPeriodUs`) and Intel (i915 PMU VCS/VECS); on AMD it is **per-process only** (DRM fdinfo) and **decode/encode are merged into one "Media engine" figure from VCN4**; on Apple there is **no public per-engine video-utilisation API** at all (VideoToolbox runs on a dedicated Media Engine the public API does not meter). Two NVML traps were corrected: `nvmlDeviceGetUtilizationRates().memory` is the memory-*controller* busy %, **not** bytes used (use `nvmlDeviceGetMemoryInfo` for VRAM pressure), and `nvmlDeviceGetEncoderCapacity` is a **percentage (0-100)**, not a session count. The NVENC concurrent-encode-session limit is a **per-system aggregate** across all consumer cards, enforced at session-create, and a **moving driver-version number** (8/system on Video Codec SDK 13.0; reported as 12 in 2025) — so Multiview tracks its own session count and discovers the ceiling at runtime rather than hard-coding it. NVML loads at runtime via `libloading` (no build-time native dep), matching the conventions §7 vendor-lib posture exactly, and `nvml-wrapper` is dual MIT OR Apache-2.0 (matches the project licence). Polling off the hot path with a wait-free snapshot hand-off (`arc_swap`, [ADR-I001](ADR-I001.md)) and selection only at admission/reconfig keep invariants #1 (output clock never falters) and #10 (isolation) intact.

## Alternatives considered

- **Pure round-robin / spread placement.** Simple and affinity-respecting if a whole pipeline is one round-robin unit, but ignores actual load (VRAM/encoder pressure) and the NVENC session ceiling — over-admits onto a saturated card. Rejected as the primary policy (kept only as a deterministic tie-breaker within an epsilon band).
- **Pure least-loaded ignoring affinity (the unrefined hypothesis).** Optimises instantaneous balance but routinely splits a pipeline's decode/composite/encode across GPUs, paying a per-frame PCIe copy + latency that violates the zero-copy-island design and harms density/latency far more than the imbalance it cures. Rejected — affinity gates least-loaded.
- **Static pinning only (no auto policy).** Predictable and operable, and remains available as the override. But it forces every operator to hand-place every pipeline and cannot react to load, defeating the multi-GPU commodity-hardware goal. Rejected as the default; retained as the explicit override that always wins.
- **`nvidia-smi`/DCGM CLI scraping instead of NVML in-process.** Adds a process dependency and parsing fragility, and the same util% caveats apply. Rejected in favour of the in-process runtime-loaded NVML path.

## Consequences

`multiview-hal` gains a `load` module (`DeviceLoad` snapshot + injectable `LoadProbe` trait, mirroring the existing `DeviceProbe` seam) and a pure, unit-testable `select` module (the affinity-gated least-loaded policy); vendor `LoadProbe` impls are feature-gated like the existing presence detectors and the default pure-Rust build still probes nothing. `multiview-telemetry` gains per-GPU gauges (`multiview_gpu_*` for compute/encoder/decoder util, VRAM used/total, active/ceiling encode sessions, temp/power/clocks); **unknown metrics are not registered** for a blind vendor so dashboards show "n/a", never a false zero. `multiview-engine` owns the probe thread (off the data plane), reads the wait-free snapshot at admission/reconfig, and performs make-before-break migration when re-balancing. `multiview-control` + the WebUI expose the device inventory, the live per-GPU dashboard (via the conflated realtime channel, [ADR-RT004](ADR-RT004.md)), the placement map, and the per-source/per-output pin override with a dry-run live-apply plan. Costs/risks: the composite score blends signals with heterogeneous sampling windows (low-pass to a common ~1 s window; the hard gates are the real safety net); AMD VCN4+ yields one combined media term; Apple is effectively single-island for selection so the per-engine blind spot costs little there; the cross-GPU split cost is a modelled estimate until measured per host; and the NVENC ceiling is learned by attempt-and-handle and cached. Tests assert: selection never splits a pipeline unless no single GPU fits; a blind-vendor probe falls back to VRAM + overall-busy + cost-model without blocking placement; a pin always wins; and the probe can never stall the engine (chaos gate).
