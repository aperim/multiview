# ADR-RT007: Devices realtime topic and event types

- **Status:** Proposed
- **Area:** Realtime API
- **Date:** 2026-06-10
- **Source brief:** [managed-devices.md](../research/managed-devices.md)

## Decision

Add one coarse `Topic::Devices` (wire string `devices`) to the closed `Topic` enum in `crates/multiview-events/src/topic.rs`, scoped finer with the existing `id`/ids filter — never with more topics, per the realtime brief's coarse-topics rule. The topic carries three lanes of event types, all additive minor changes under the existing v1 envelope (ADR-RT002, `{v,t,topic,id,seq,ts,corr,data}`), delivered over the existing WS/SSE stack with snapshot-then-delta semantics (ADR-RT003: `$snapshot` of the device registry plus each device's latest status on subscribe, then deltas). **(1) `device.status`** — a conflated, latest-wins per-device snapshot (envelope `id` = device id): lifecycle state (`DISCOVERED`/`ADOPTING`/`ONLINE`/`DEGRADED`/`AUTH_FAILED`/`UNREACHABLE`), current mode, device-reported active streams (fps/bitrate), temperature, last-seen. **(2) Lossless low-rate lifecycle events**: `device.adopted`, `device.removed`, `device.mode` (mode-convergence start/finish with its declared impact), `device.error`, `device.sync` (sync-group membership/achieved-tier changes and drift-alarm threshold crossings), and `device.discovered` (rows streamed while a `POST /discovery/devices/scan` 202 operation runs, correlated via `corr`). **(3) `timing.status`** — a `TimingStatus` payload for sync telemetry (envelope `id` = program or sync-group id): the outbound presentation epoch `{stream_id, WallClockRef, link_offset, clock_source, clock_quality}` (ADR-M010) plus per-sync-group achieved-skew measurements; latest-wins, because the epoch is an affine map (`multiview-core/src/wallclock.rs`) that remains valid when stale — a node that misses updates keeps the last epoch and free-runs.

Conflation policy follows the isolation doctrine exactly (ADR-RT004, invariant #10). Every producer is a **control-plane driver poller/actor task** publishing into `tokio::watch` (status, timing — latest-wins) and `tokio::broadcast` (lifecycle); the engine never produces, forwards, or awaits a device event — the engine's only contribution is writing its `WallClockRef` estimate into a watch channel, never awaiting a reader. The per-connection session pump conflates `device.status` and `timing.status` and keeps lifecycle events lossless via the bounded replay ring; `device.status`/`timing.status` are **excluded from the ring** (re-snapshot heals them), which extends ADR-RT003's ring-exclusion rule from per-topic granularity (`Topic::is_high_rate`) to per-event-type granularity for this one mixed-cadence topic. All queues stay bounded drop-oldest/latest-wins; nothing on this topic can back-pressure the engine, by construction.

## Rationale

One coarse topic matches the established subscription model (existing topics like `alerts` and `inputs` already carry several event types each) and lets the SPA Devices page subscribe once and switch exhaustively on `t` (ADR-RT002's TypeScript discriminated union), filtering by device id only on detail views. `devices` is, however, the **first** topic to mix a conflated telemetry lane with lossless lifecycle events on one topic — which is exactly why the per-event-type extension of ADR-RT003's ring-exclusion rule above exists and is tested explicitly. The two cadence classes have genuinely different semantics: status and timing are telemetry whose latest value supersedes all prior values — conflation is *correct*, not lossy, and keeps the lane safe at any fleet size (N devices × ~1 Hz vendor polling never grows a queue) — whereas adoption, removal, mode changes, errors, and sync transitions are operator-meaningful discrete facts that are cheap to keep lossless because they are rare. Carrying `TimingStatus` on the same topic puts it where its consumers live: display nodes and the sync-group UI are device-domain subscribers, and the epoch's affine staleness-tolerance makes latest-wins delivery sufficient (the F3 design requires nodes to degrade to free-run on a lost feed, never to stall).

## Alternatives considered

**Per-device topics** (`devices.{id}`) — rejected: explodes the deliberately closed, coarse `Topic` enum, churns subscriptions on every adopt/remove, and duplicates what the existing `id` filter already provides. **Lossless-everything** (put status/timing in the replay ring too) — rejected: per-second snapshots from every device would dominate the bounded ring, evict the lifecycle history the ring exists to protect, and force frequent `$resync`; conflating a snapshot stream loses nothing by definition. **Status-by-polling-only** (no realtime status; clients poll `GET /devices/{id}/status`) — rejected: per-second REST polling per client per device is the wrong shape (latency, N×M load on the control plane *and* on the polled vendor devices, race-prone UI), and it contradicts the repo's push doctrine — the UI subscribes, it never polls live values.

## Consequences

The SPA subscribes once to `devices` and the REST `GET /devices/{id}/status` endpoint becomes a cold-snapshot fallback only — never the live path. Event-schema additions are versioned per ADR-RT002: new device event types and fields are additive minor changes under v1 that old clients ignore. The session pump gains per-event-type ring handling on one topic (conflate status/timing, ring lifecycle) — a small extension of ADR-RT003/RT004 mechanics that must be explicitly tested: resume-after-gap replays lifecycle losslessly while status arrives as a fresh snapshot. A disconnect longer than the ring loses intermediate lifecycle events (`$resync` is a rebuild, per ADR-RT003); the durable adoption/removal record is the audit log, not the WS stream. Honest freshness limits: `device.status` numbers (temperature, decode fps) are vendor-polled at roughly 1 s cadence and delivered best-effort — staleness is surfaced via `last_seen`, never papered over — and `timing.status` reports the *measured* sync tier (down to Tier-C bounded drift of ±100–500 ms for vendor decoders), never an aspirational one.
