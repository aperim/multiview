# Multiview — Recording Storage: Transcoded ISO, Trigger-Gated Capture, S3 Offload & Iceberg/SeaweedFS Catalog

**Area:** Output / Storage / Config / Control / Events / Telemetry / Web — the *durable-storage* plane that sits **downstream** of the recorder.
**Status:** Design brief (Proposed) — docs-only; implementation follows in dependency-ordered waves.
**Drives:** [ADR-0069](../decisions/ADR-0069.md) (transcoded ISO variant + trigger-gated capture, extends ADR-0037), [ADR-0070](../decisions/ADR-0070.md) (object-store / S3 offload + lifecycle — async, bounded, never back-pressures), [ADR-0071](../decisions/ADR-0071.md) (Iceberg / S3-Tables event+media metadata catalog via SeaweedFS).
**Extends:** [iso-program-recording.md](iso-program-recording.md) + [ADR-0037](../decisions/ADR-0037.md) (the raw copy-remux ISO + Program recorder, its bounded drop-oldest write path, dual-axis retention, disk-pressure disarm — **the foundation this brief builds on, never re-litigates**), [ADR-0032](../decisions/ADR-0032.md) (rolling segmenter + atomic publish), [ADR-0035](../decisions/ADR-0035.md) / [self-aware-placement.md](self-aware-placement.md) (the `HealthWarning` surface), [ADR-R003](../decisions/ADR-R003.md) (`backon` capped-jitter back-off), [ADR-R009](../decisions/ADR-R009.md) (`/livez`/`/readyz` semantics — storage faults must not restart-loop the container), [ADR-0042](../decisions/ADR-0042.md) / [ipv6-first.md](ipv6-first.md) (IPv6-first networking).
**Relates to:** [object-detection-ai.md](object-detection-ai.md) + [ADR-0067](../decisions/ADR-0067.md) (record-on-named-object — **already hands clip lifecycle/offload to this brief**, object-detection §5.2), sibling intake briefs [motion-scene-detection.md](motion-scene-detection.md) (the cheap motion gate fronting any trigger), [broadcast-cues.md](broadcast-cues.md) (SCTE-35 / cue-driven start/stop), [online-services-input.md](online-services-input.md), [observability-logging.md](observability-logging.md) + [system-stats-attribution.md](system-stats-attribution.md) (where storage health metrics surface), [conspect-account-architecture.md](conspect-account-architecture.md) + [ADR-0052](../decisions/ADR-0052.md) / [ADR-0053](../decisions/ADR-0053.md) (consent + retention boundary).
**Backlog:** `STORE-*` in [`../development/feature-intake-2026-06-13.md`](../development/feature-intake-2026-06-13.md).

> The operator asked for five things that all live *behind* the recorder: **ISO recording raw** (the accepted foundation — [ADR-0037](../decisions/ADR-0037.md)), **ISO recording transcoded** (a re-encoded archival variant), **S3 storage**, **offload of ISOs/recordings to S3**, and **S3-Tables (Iceberg) buckets** ([github.com/seaweedfs/seaweedfs](https://github.com/seaweedfs/seaweedfs)) to keep **metadata/event data with the video, frame-cap, or ISO-audio**. The raw recorder's design is already accepted in [ADR-0037](../decisions/ADR-0037.md); this brief designs the *storage tail*: a transcoded ISO tier, a trigger-gated capture mode (cue/motion/object → arm/disarm), an **asynchronous, bounded, never-back-pressuring** object-store offload, and an Iceberg catalog that indexes every recording + its sidecar events. Everything here is **proposed** — nothing below is implemented. The genuinely-new bulk is the S3 offload and the Iceberg/SeaweedFS catalog; the recording tiers and triggers *extend* the [ADR-0037](../decisions/ADR-0037.md) recorder design rather than reinventing it.

## 0. Headlines

- **This brief is the storage tail of the recorder, not a second recorder.** The raw copy-remux ISO sink, the Program sink, the rolling segmenter, dual-axis retention, the disk-pressure disarm, and the bounded drop-oldest write path are **already designed** in [ADR-0037](../decisions/ADR-0037.md) / [iso-program-recording.md](iso-program-recording.md) (Status: Proposed — the accepted foundation this brief builds on, not yet implemented). We **reuse all of that design verbatim** and add three downstream capabilities: a *transcoded* ISO tier, *trigger-gated* arm/disarm, and *offload + catalog*.
- **Transcoded ISO is a separate encode, honestly priced.** Raw ISO is `avcodec_parameters_copy` — zero re-encode (invariant #7). A *transcoded* ISO (e.g. a space-efficient H.264/HEVC archival proxy of a single source, decoded near display res per invariant #6) is, by definition, a **separate encode** — invariant #7 sanctions a separate encode only when codec/res/bitrate differ, which is exactly the point here. It is **off by default**, admitted against the GPU/CPU budget like any co-tenant, and shed *before* the program ever moves.
- **Trigger-gated capture only flips arm/disarm — it never touches the data plane.** A cue ([broadcast-cues.md](broadcast-cues.md)), a motion event ([motion-scene-detection.md](motion-scene-detection.md)), or an object detection ([object-detection-ai.md](object-detection-ai.md), [ADR-0067](../decisions/ADR-0067.md)) submits the **same** `Arm/Disarm{Iso,Program}Recording` command [ADR-0037 §F](iso-program-recording.md) already defines, off the clock thread, with pre-roll + hold/cooldown + de-bounce. This brief unifies the trigger *contract* so cue/motion/object all share one rule shape; object-detection already delegates clip lifecycle here ([object-detection-ai.md §5.2](object-detection-ai.md)).
- **The offload path is a best-effort sink that can NEVER stall the engine or the recorder.** It is the **same isolation discipline** as the recorder ([ADR-0037 §E](iso-program-recording.md)): a bounded drop-oldest work queue of *closed, already-on-disk segments*, drained by a dedicated off-hot-path uploader task, `backon` capped-jitter back-off on every S3 error, fail-safe on persistent failure, and **disk-pressure disarm reused unchanged** (a slow object store can only delay deletion of local copies, never grow RAM, never pace a frame). Invariants **#1** and **#10** hold by construction — the uploader operates on files, never on the live packet stream.
- **The object store is the de-facto-open S3 API, vendor-neutral by construction.** We target the S3 HTTP API as implemented by **MinIO, SeaweedFS, Ceph RGW, Garage**, and AWS S3 — nominatively, over the published API, never bundling a vendor SDK. The Rust client is the generic **`object_store`** crate (one trait over S3/GCS/Azure/local — *to verify against a live endpoint before implementation*), so "S3" is a configuration, not a hard dependency on any one vendor.
- **Iceberg/S3-Tables is the metadata-with-the-media answer.** The operator wants event data and metadata kept *with* the video/frame-cap/ISO-audio. SeaweedFS ships a built-in **Iceberg REST Catalog** alongside its S3 gateway (per the SeaweedFS README: default catalog port 8181, S3 port 8333, [github.com/seaweedfs/seaweedfs](https://github.com/seaweedfs/seaweedfs) — the exact port/auth specifics are *to verify against SeaweedFS docs before implementation*). We write an **Apache Iceberg** table of recording + event rows (one row per segment / frame-cap / ISO-audio chunk, keyed to the object key) so any query engine (DuckDB/Trino/Spark) can join "what happened" to "where the bytes are." Iceberg + S3-Tables are **open** — no proprietary lock-in.
- **Consent + retention is a hard boundary, not an afterthought.** Recordings can contain people. Offload + catalog inherit the **Conspect two-pipe discipline** ([ADR-0052](../decisions/ADR-0052.md)): media bytes and their sidecar metadata are the *operator's own data* and **never** leave the machine to Aperim; the catalog carries **no raw identifiers** that data-minimisation forbids; retention/erasure is operator-driven and lifecycle-enforced both locally and in the object store.
- **Honest scope.** v1 should be: raw ISO offloaded to one S3-compatible bucket with lifecycle, an Iceberg sidecar table written via the SeaweedFS REST catalog, and a unified trigger rule. Transcoded ISO is a flagged co-tenant. Multi-destination fan-out, cross-region replication, glacier-tier transitions, and a built-in query UI are noted future axes.

## 1. Recording tiers — raw copy-remux ISO (foundation) vs transcoded ISO (new) vs Program

The operator's "ISO recording (raw)" and "ISO recording (transcoded)" are two tiers of the same per-source archive. The matrix below pins what the accepted foundation already covers vs what this brief adds; the raw column is **[ADR-0037](../decisions/ADR-0037.md), not re-decided here**.

| Tier | What it is | Encode cost | Status | Owner |
|---|---|---|---|---|
| **Raw ISO** | Faithful **copy-remux** of every elementary stream of one source into a rotating `mkv`/`ts` container — no re-encode | **Zero** (`avcodec_parameters_copy`, [iso-program-recording.md §A](iso-program-recording.md)) | **Designed (Proposed)** ([ADR-0037](../decisions/ADR-0037.md)) | ADR-0037 |
| **Transcoded ISO** | A re-encoded archival **proxy** of one source (configurable codec/res/bitrate — e.g. a small H.264 daily) | **One separate encode per transcoded ISO** (inv #7-sanctioned) | **New (Proposed)** | [ADR-0069](../decisions/ADR-0069.md) |
| **Program (PGM)** | A rotating recording of the composited **dirty** program, fed the existing single encode | **Zero extra** (inv #7 fan-out, [iso-program-recording.md §B](iso-program-recording.md)) | **Designed (Proposed)** ([ADR-0037](../decisions/ADR-0037.md)) | ADR-0037 |

### 1.1 Why transcoded ISO is a *separate encode*, and why that is fine

Raw ISO preserves the source bytes (camera-native HEVC/H.264, all audio, all data). But operators frequently want a **small, uniform, long-retention proxy** — a single codec/bitrate/res for every source so a week of 30 cameras fits on a NAS and plays anywhere. That is, by definition, a re-encode. Invariant #7 (encode-once-mux-many) forbids a *redundant* second encode of the *same* pixels at the *same* settings; it **explicitly allows** a separate encode when "codec/res/bitrate differ" — which is the entire purpose of a transcoded proxy. So a transcoded ISO is a **legitimate distinct encode pipeline**, exactly like the deferred *clean* program feed ([iso-program-recording.md §B](iso-program-recording.md)): tap the **already-decoded** source frame, feed a **dedicated `ProgramEncoder`-class encoder** under its own rendition id, and route its packets into the **same** `IsoMuxSink` / rolling-segmenter + bounded-ring write path the raw tier uses ([ADR-0037 §E](iso-program-recording.md)). The *only* new thing is the encoder; segmentation, retention, disk-pressure, offload, and catalog are all shared.

> **Design note — tap the source decode path, not the compositor's last-good tile.** The last-good *tile* store is display-path state: it can be stale (frozen on STALE/NO_SIGNAL), tile/display-resolution, cadence-sampled to the output clock, and it carries no audio or source PTS. Tapping it would silently turn "transcoded ISO" into a *sampled visual proxy of compositor state*, not a faithful per-source archive. So a transcoded ISO **tees from the per-source decoded-frame path** (full source cadence + normalized source PTS, with audio handled on the source's audio path) into a bounded, drop-oldest recorder-encoder queue — reusing the *already-paid* decode (invariant #6, no second decode) but **not** the compositor's sampled tile. Where an operator explicitly wants the cheaper sampled-tile capture, that is a *separate, clearly-named "tile proxy" mode*, not the transcoded ISO. ([ADR-0069](../decisions/ADR-0069.md) `Open question`.)

### 1.2 Cost-control: transcoded ISO is admitted and shed like any co-tenant

A transcoded ISO competes with decode/composite/encode for GPU/CPU. Per [efficiency.md](efficiency.md) and [self-aware-placement.md](self-aware-placement.md), it is **admitted against the per-engine budget at below-program priority** and **shed first** under pressure (invariant #9): under load the engine drops the *transcoded archive*, never a program tile and never a frame. A transcoded ISO that cannot be admitted surfaces a typed capability warning (the `AudioCapability`/`HealthWarning` precedent) — a refused proxy is **loud, not silent**. Default: **off**. Recommended v1 path: a single SW-or-HW H.264 proxy at a capped resolution/bitrate, on a *subset* of sources the operator names — not "transcode everything."

### 1.3 Frame-capture and ISO-audio as first-class storage artefacts

The operator's catalog ask names three media kinds explicitly: **video, frame-cap, ISO-audio**. All three are *recorder outputs* this brief stores + catalogs uniformly:

- **Frame-cap (still/JPEG snapshots).** A periodic or trigger-driven still from the read-only program/tile tap (the same snapshot path preview exposes — [object-detection-ai.md §2.1](object-detection-ai.md) tap points; the `multiview-preview` `snapshot` seam). A frame-cap is a tiny object (one JPEG) with a timestamp + source key — ideal Iceberg-row fodder (thumbnails for an event, evidence stills for a detection).
- **ISO-audio (audio-only ISO).** A source's audio elementary streams stream-copied (or transcoded) to an audio container — the audio analogue of a raw ISO, useful for logging/compliance loudness archives. The raw ISO already copies audio streams; an *audio-only* ISO is a config restriction of the same `IsoMuxSink` (register only audio inventory rows). Catalogued the same way.

All three carry the **same retention/offload/catalog lifecycle**; the catalog row's `media_kind` discriminates `video | frame_cap | iso_audio`.

## 2. Trigger-gated capture — cue / motion / object → arm/disarm (one unified rule)

The operator wants capture that **starts and stops on an event**, not only continuous recording. Three trigger sources already have (or will have) briefs; this section gives them **one common rule contract** so the recorder sees a single arm/disarm semantics regardless of which sensor fired.

### 2.1 The three trigger sources

| Trigger source | Brief | What fires |
|---|---|---|
| **Cue** (SCTE-35 splice, manual cue, schedule) | [broadcast-cues.md](broadcast-cues.md) | A program-timeline cue (e.g. "ad break start/end", "segment in/out") arms/disarms a Program or ISO recorder at the cue's frame boundary |
| **Motion** | [motion-scene-detection.md](motion-scene-detection.md) | A cheap per-region motion delta crosses a threshold (the Frigate-style motion gate) |
| **Object** | [object-detection-ai.md](object-detection-ai.md), [ADR-0067](../decisions/ADR-0067.md) | A named object/zone/confidence match (motion-gated) — **already delegates clip lifecycle to this brief** ([object-detection-ai.md §5.2](object-detection-ai.md)) |

### 2.2 The unified `CaptureTrigger` rule (the contract these share)

A trigger is **a rule that arms the existing recorder** — never a new recording path (the [ADR-0067](../decisions/ADR-0067.md) decision, generalised). The rule shape (config + event, all off the data plane):

- **Source predicate** — cue type / motion zone / object label+zone+confidence (each brief owns its own predicate vocabulary; this brief owns only the *arming action*).
- **Target** — which recorder to arm: an **ISO** on a named input or the **Program** ([iso-program-recording.md §A/§B](iso-program-recording.md)). Reuses `IsoRecording`/`ProgramRecording` **verbatim**.
- **Window** — `pre_roll` (seconds of already-segmented footage to *pin* before the match — satisfiable because the recorder is a rolling segmenter with retention: pre-roll = "do not prune the last N seconds and mark them pinned"), `min_duration`, and `hold`/`cooldown` (keep recording until no match for T seconds, then disarm — kills flapping).
- **De-bounce** — require K matches in a sliding window before arming (kills single-frame false positives). The motion gate ([motion-scene-detection.md](motion-scene-detection.md)) should front the object detector so inference only runs on changed regions (the verified Frigate motion-gating pattern, see [object-detection-ai.md](object-detection-ai.md) Citations).

### 2.3 How a trigger is applied (invariants #1, #10, #11)

- A fired rule submits the **same** `Arm/Disarm{Iso,Program}Recording` command on the existing command bus ([ADR-0037 §F](iso-program-recording.md): `202` + op-id, `Idempotency-Key`, **non-blocking `try_submit` shed-to-`503`**). Arm/disarm is **Class-1 hot** ([ADR-R004](../decisions/ADR-R004.md), invariant #11) — it only flips a best-effort sink's state.
- The trigger runs **off the data plane**, in the cue/detection worker tier. It **never** calls the engine on the clock thread; it enqueues a command the existing drain applies at a frame boundary. A storm of triggers cannot stall output or grow RAM — the recorder's own bounded drop-oldest ring + disk-pressure gating absorbs it ([ADR-0037 §E](iso-program-recording.md)).
- **Pinning.** A triggered clip is an ordinary recorder segment with a `pinned_until` flag (export/expiry). Pinning *exempts* a segment from time/size retention prune ([iso-program-recording.md §C](iso-program-recording.md)) **but never from disk-pressure disarm** — a full disk still wins (it stops *new* segments; existing pinned segments are kept best-effort but the recorder never blocks to keep them). A pinned segment is the offload/catalog's signal to *prioritise* upload (export-before-expiry).
- **Auditability.** Every fire emits its source event (`Event::DetectionTrigger`, a cue event, a motion event) plus the recorder's `Event::RecordingStatus` ([ADR-0037 §F](iso-program-recording.md)). A trigger that *could not* arm (disk pressure, capability) raises a `HealthWarning` — a silently-not-recording trigger is loud.

## 3. S3 / object-store offload — async, multipart, bounded, lifecycle (never back-pressures)

This is the first genuinely-new bulk. The operator wants **closed recordings/ISOs offloaded to S3**. The cardinal rule is the recorder's, restated for the uploader: **a slow/full/unreachable object store can NEVER stall the engine, the recorder, or the live transports.** It operates on **closed files on disk**, asynchronously, and the worst a wedged bucket can do is leave local copies pending (governed by disk-pressure, which already disarms recording before disk fills).

### 3.1 Where the offload taps — closed segments, never live packets

The uploader is **downstream of the rolling segmenter** ([ADR-0032](../decisions/ADR-0032.md), [ADR-0037 §C](iso-program-recording.md)). When the segmenter publishes a closed segment via same-dir `<seg>.tmp → rename(2)` (the `close_current` seam in `multiview-output`'s sink) and the live playlist re-publishes the manifest atomically (`atomic_write` → `sync_all` → `rename`, the `multiview-output` HLS live path) — both **expected seams of the ADR-0037 design, to confirm against the implementation** — the uploader enqueues that **path** (not the bytes) into a **bounded work queue**. The uploader reads a file that already exists and is durable — it is *never* on the packet path. This is the structural reason invariant #1/#10 hold: the offloader and the engine share **no channel, no lock, no buffer** — only a directory of closed files and a bounded queue of filenames.

### 3.2 The bounded, never-back-pressuring uploader (reuse ADR-0037 §E discipline)

| Concern | Design (mirrors [ADR-0037 §E](iso-program-recording.md)) |
|---|---|
| **Ingress** | The segmenter's writer task, after publishing a closed segment, calls a non-blocking `enqueue(path, meta)` on a **bounded** work queue (depth-capped; default ~a few thousand pending objects, sized so the byte footprint is bounded by *local* retention not RAM — the queue holds paths + small metadata, not media bytes). Full queue → **the local file is simply retained** (it is on disk under retention) and a `storage-offload-backlog` warning is raised; nothing blocks, nothing grows in RAM. **The in-memory queue is only a wake-up hint, never the source of truth**: the durable **segment inventory / local index** ([iso-program-recording.md §A](iso-program-recording.md)) carries a per-segment `offloaded` flag, and a periodic **reconciler** scans for closed-but-not-offloaded segments and re-enqueues them. So a path dropped from a full queue (or lost across a restart) is *eventually offloaded by the scan*, not permanently forgotten. |
| **Egress** | A dedicated **off-hot-path uploader task** (own tokio task — never the engine, never the recorder writer) drains the queue and uploads via the S3 client. |
| **Multipart** | Objects above a threshold use **S3 multipart upload** (min part size **5 MiB** except the last part — *verified*); `object_store::put_multipart` streams the file in parts so a large daily ISO never loads wholly into RAM. Bounded in-flight parts. |
| **Back-off** | On any S3 error (network, 5xx, throttle, auth) → **`backon` capped-exponential + full-jitter** back-off (the [ADR-R003](../decisions/ADR-R003.md) mandate; same crate the recorder uses), with a cap (~minutes). Jitter prevents N machines hammering a recovering endpoint. |
| **Fail-safe** | Persistent failure leaves the local copy *in place under retention* and keeps the warning latched; it **never** deletes a local file it has not confirmed durable in the object store. |
| **Idempotency** | Object key = a deterministic function of `(recorder_id, media_kind, segment_start_utc, sequence)`; re-upload after a crash is a safe overwrite (same key, same bytes). Incomplete multipart uploads are reaped by an S3 **lifecycle rule** (`AbortIncompleteMultipartUpload` after N days — *verified standard practice*) so a crash mid-upload never orphans paid parts. |
| **Delete-after-confirm** | A local segment is eligible for **normal-retention** prune **only after** the uploader confirms the object is durable *and* the catalog row is written (§4). "Durable" means an integrity-checked upload (multipart checksum / stored content hash validated against the local file), **not merely HTTP 200 + ETag** — ETag is a commit confirmation, not an integrity proof across all S3-compatible stores. Until then, local retention + disk-pressure govern; the object store is an *addition*, never a precondition for local correctness. (Emergency disk-pressure prune is a separate path — see §3.3.) |

### 3.3 Lifecycle / retention split (local vs object store)

There are now **two** retention domains, and they must not be confused:

- **Local retention** — the recorder's existing dual-axis (time AND size) prune + disk-pressure disarm ([iso-program-recording.md §C/§D](iso-program-recording.md)), **unchanged**. Local disk is a *cache + staging area* once offload is enabled.
- **Object-store lifecycle** — long-term retention is enforced by **S3 lifecycle rules** on the bucket (`ExpirationInDays`, transitions to colder tiers — *verified S3 lifecycle semantics*). Multiview *declares* a recommended lifecycle (and can apply it via the S3 `PutBucketLifecycleConfiguration` API where the endpoint supports it — MinIO/SeaweedFS/Ceph do), but the **bucket owner's lifecycle policy is authoritative**; we document the recommended rules and never silently override an operator's policy.

**Two prune paths, not one (resolving the apparent contradiction).** *Normal* retention prune honours delete-after-confirm: a segment waits for upload+catalog before it is eligible. *Emergency disk-pressure* prune is a **distinct, explicit policy** that **always wins** (the recorder never blocks to keep bytes): when local disk crosses the disk-pressure threshold and the offload backlog is wedged, the recorder first disarms *new* recording, then — if still over threshold — prunes the **oldest not-yet-offloaded local segments**, emitting a `storage-offload-data-loss` warning and recording the dropped object keys, so the loss is **loud and auditable**, never silent. The default policy prefers losing the oldest un-offloaded local copies over stalling; an operator may instead choose "disarm-only, never drop un-offloaded" at the risk of a full disk (a documented config knob, [ADR-0070](../decisions/ADR-0070.md) `Open question`). A wedged *catalog* (upload succeeded, row pending) is not data-loss — the bytes are durable in the bucket and only normal prune is gated.

A `pinned_until` (trigger) clip maps to an object-store **retention/legal-hold** semantics where available, else a documented "do not lifecycle-expire" tag; expiry past the pin is operator-driven.

### 3.4 Vendor-neutrality + the client choice

The S3 HTTP API is **de-facto open**: MinIO, SeaweedFS, Ceph RGW, and Garage all implement it, as does AWS S3. We speak it **nominatively** — no vendor SDK is bundled, no vendor spec redistributed (the [managed-devices.md](managed-devices.md) ZowieTek-driver posture). The Rust client is the generic **`object_store`** crate (one async trait over S3/GCS/Azure/local filesystem, `put_multipart_opts` for streamed multipart, production-proven in InfluxDB IOx — *verified*), so the *same* offload code targets any S3-compatible endpoint, GCS, Azure, or even a local/NFS path for air-gapped sites. (Alternatives `rusty-s3` (sans-IO, MinIO-CI-tested) and the official `aws-sdk-s3` are noted; `object_store` is preferred for vendor-neutrality + the local-filesystem fallback. Final crate selection is an `Open question`.) **IPv6-first** ([ipv6-first.md](ipv6-first.md), [ADR-0042](../decisions/ADR-0042.md)): the endpoint URL accepts bracketed IPv6 literals; the client must dual-stack-resolve and not assume IPv4 — a v1 conformance check, since some S3 SDKs are historically IPv4-centric (flagged as an `Open question` to verify against the chosen crate).

## 4. Iceberg / S3-Tables event+media catalog via SeaweedFS

This is the second genuinely-new bulk and the operator's most specific ask: keep **metadata/event data with the video, frame-cap, or ISO-audio** in an **S3-Tables (Iceberg) bucket** ([github.com/seaweedfs/seaweedfs](https://github.com/seaweedfs/seaweedfs)).

### 4.1 Why Iceberg, and what SeaweedFS provides

A pile of media objects in a bucket has no queryable index — you cannot ask "show me every clip on camera 3 where a person was detected last Tuesday" without scanning. **Apache Iceberg** is an open table format that layers a queryable, ACID, time-travelling table *over* object storage: a catalog maps a table name to a metadata JSON, which points to manifest lists → manifest Avro files → the actual data files (per the Iceberg spec). **SeaweedFS ships a built-in Iceberg REST Catalog** alongside its S3 gateway — no external metastore (Hive/Glue) needed — implementing the Iceberg REST Catalog API on a dedicated port (per its README, default **8181**), with S3 data access on the S3 port (default **8333**), and auth via SigV4 / OAuth2 / unsigned-REST + S3 keys ([github.com/seaweedfs/seaweedfs](https://github.com/seaweedfs/seaweedfs) — the exact port/auth specifics are *to verify against SeaweedFS docs before implementation*). Any query engine (DuckDB, Trino, Spark, Dremio, RisingWave) can then join the catalog.

So the design is: **media bytes → S3 bucket (§3); a row per media artefact + a row per event → an Iceberg table written through SeaweedFS's REST catalog.** The video and its metadata live in the *same* SeaweedFS deployment — "metadata with the media," exactly the ask.

### 4.2 The catalog schema (sidecar tables, keyed to objects)

Multiview should write (at least) two Iceberg tables. The **media artefacts are immutable** once closed and normal operation is **append-only** (corrections are new snapshots); note that **erasure (§5) is the exception** — a row-level delete requires an **Iceberg v2 table with delete-file support** (a tombstone-plus-filter / delete-snapshot model), so the catalog tables are created at the format version and with the writer/query-engine support that erasure needs, not a strictly append-only v1 table:

- **`recordings`** — one row per closed media artefact: `recorder_id`, `media_kind` (`video|frame_cap|iso_audio`), `source_id`, `object_key` (the S3 key from §3), `bucket`, `segment_start_utc`, `segment_end_utc`, `duration_ns`, `byte_size`, `codec`, `container`, `pinned_until` (nullable), `checksum`. Partitioned by `media_kind` + day (`segment_start_utc`).
- **`events`** — one row per realtime event worth persisting, keyed to time + (optionally) a `recorder_id`/`object_key`: the **same** [realtime-api.md](realtime-api.md) event envelope content — `DetectionTrigger`, cue events, `RecordingStatus`, `HealthWarning` — so an analyst joins "what happened at T" to "which object holds the bytes for T." This is the persisted, queryable twin of the live event stream (the live stream is conflated drop-oldest per invariant #10; the catalog is the *durable* record, written off the hot path).

A join on `events.object_key = recordings.object_key` (or on overlapping time ranges) answers "give me the clip for this detection." Frame-caps and ISO-audio are first-class `media_kind`s, so the operator's "with the video, frame-cap, or ISO-audio" is literally one schema.

### 4.3 How the catalog write stays isolated (invariants #1/#10)

The catalog writer is **off the data plane**, the same posture as the uploader (§3.2):

- It runs in a dedicated task. Rows are batched and committed to Iceberg **after** a segment is durably uploaded — an Iceberg commit is an atomic catalog-pointer swap (ACID via atomic catalog update, per the Iceberg spec), so a batch of new rows lands as one snapshot. **Retry idempotency is Multiview's responsibility, not Iceberg's**: Iceberg identifier fields do *not* enforce row uniqueness (the spec assigns dedup to the writing engine), so an append retried after an ambiguous failure can duplicate rows. We make retries idempotent with a durable commit/outbox id and a deterministic data-file path per `(recorder_id, media_kind, object_key)`, so a re-commit of the same batch overwrites the same data file rather than appending duplicates; a failed commit retries with `backon`. *Verify the exact write protocol against the chosen Iceberg writer before implementation.*
- The catalog is **never** read or written on the output-clock or recorder-writer thread. A wedged catalog endpoint loses *queryability of recent rows*, never a frame and never a recording — the media is already in the bucket; the row catches up on retry. This is the same "media-first, index-eventually" ordering the §3 delete-after-confirm rule depends on (delete-local requires *both* upload-confirmed *and* row-written, so a wedged catalog *keeps* local copies, never loses data).
- **No raw identifiers** (data-minimisation, §5): the catalog carries operational keys (source ids, object keys, timestamps, labels) — never credentials, never raw stream URLs with embedded secrets, never PII the operator has not chosen to index.

### 4.4 Honest scope of the catalog

v1 should write the `recordings` table + the `events` table through the SeaweedFS REST catalog and prove a DuckDB/Trino join works end-to-end. The Rust Iceberg writer is **`iceberg-rust`** (the apache/iceberg-rust crate) — *but its write/REST-catalog maturity must be verified at build time* (flagged `Open question` — `iceberg-rust` write support has historically trailed reads; a fallback is to emit Iceberg-compatible Parquet data files + commit via the REST catalog HTTP API directly, or to stage rows and let an external `weed` maintenance job compact). Automated Iceberg table maintenance (compaction, snapshot expiry, orphan removal) is provided by SeaweedFS itself (per its README — *to confirm against SeaweedFS docs/integration before implementation*) — we lean on it rather than reimplementing. Multi-table schemas (per-source tables, a `detections` table with bounding boxes) are future axes.

## 5. Consent / retention boundary (Conspect, ADR-0052)

Recordings can contain people; a searchable event catalog amplifies that. The boundary is **hard** and inherits Conspect's two-pipe discipline ([ADR-0052](../decisions/ADR-0052.md), [conspect-account-architecture.md](conspect-account-architecture.md)):

- **The operator's data never becomes Aperim's.** Media bytes, frame-caps, ISO-audio, and the Iceberg sidecar are the **operator's own data**, written to the **operator's own** object store (their SeaweedFS/MinIO/AWS account). None of it is the telemetry pipe and **none of it phones home** — the [ADR-0052](../decisions/ADR-0052.md) rule that "turning telemetry off never empties the operator's own local diagnostics" extends here: the recording/catalog plane is *categorically* the operator's, separate from the opt-in analytics pipe and the mandatory licensing heartbeat. A test should assert no recording/catalog field ever appears in a telemetry or heartbeat payload (mirroring [ADR-0052](../decisions/ADR-0052.md) §1).
- **Data minimisation in the catalog.** The `events`/`recordings` tables carry operational keys only — never raw serials/MACs/credentials/stream URLs-with-secrets/PII ([ADR-0052](../decisions/ADR-0052.md) §4). Object keys and source ids are operator-meaningful but not secret; detection labels are coarse classes, not identities (re-ID is out of scope, [object-detection-ai.md](object-detection-ai.md)).
- **Retention + erasure are operator-driven and enforceable.** Local dual-axis retention + object-store lifecycle (§3.3) give a concrete, auditable expiry. A consent-revocation / erasure request maps to: stop offload for the source, lifecycle-expire (or hard-delete) the affected object keys, and tombstone the catalog rows (an Iceberg delete snapshot). Because both the bucket and the catalog are the operator's, erasure is *the operator's to enforce* — we provide the levers (a `purge` operation + the lifecycle config) and document them, the [ADR-0053](../decisions/ADR-0053.md) explicit-approval-egress posture.
- **No bundled credentials, secrets by reference.** S3/catalog credentials are **config-as-code reference-only** ([ADR-0052](../decisions/ADR-0052.md) relates to ADR-M006): the config holds an endpoint + a *secret reference* (env/1Password/file), never a literal key in a committed file; the gitleaks/`cargo deny` gates apply.

## 6. Efficiency budget (mem / cpu / gpu / io)

Per the standing efficiency review, the cost of each new piece, and how it stays bounded:

- **Memory.** The offload + catalog queues hold **paths + small metadata, never media bytes** — bounded by queue depth × a few hundred bytes, not by file size. Multipart upload streams parts (bounded in-flight part buffers, default a few × 5–16 MiB), so a multi-GiB daily ISO never lands wholly in RAM. The Iceberg writer batches rows (bounded batch) and commits. **Zero unbounded growth anywhere** — a wedged store grows the *local disk staging area* (governed by the recorder's disk-pressure disarm), never RAM.
- **CPU.** Raw ISO offload + cataloguing is **I/O-bound, near-zero CPU** (copy bytes, sign requests, write Avro/Parquet rows). The one real CPU/GPU cost is **transcoded ISO** (§1) — a full encode per transcoded source — which is precisely why it is **off by default, budgeted, and shed first** (invariant #9). Multipart signing/hashing is amortised over 5 MiB+ parts.
- **GPU.** Only transcoded ISO touches the GPU (decode is already paid — invariant #6 reuse of the decoded frame; encode is the new cost). It is a co-tenant admitted **below program priority** ([self-aware-placement.md](self-aware-placement.md)) and never fragments a pipeline (the GPU-placement affinity rule). Offload + catalog touch **no GPU**.
- **I/O.** Offload doubles read I/O on closed segments (read-to-upload) — sequential, off-peak-schedulable, and on the *staging* disk not the live ingest path. Recommended: a separate spindle/volume for the recording cache so offload reads never contend with ingest writes (a documented deployment note, not a hard requirement). Network egress is bounded by `backon` back-off + the bounded uploader concurrency.

## 7. Config / API / Realtime / UI surface (sketch)

All additive, serde-default, non-breaking (existing configs keep parsing) — pinned in the ADRs, summarised here:

- **Config** ([ADR-0069](../decisions/ADR-0069.md)/[ADR-0070](../decisions/ADR-0070.md)/[ADR-0071](../decisions/ADR-0071.md)): extend `IsoRecording` with an optional `transcode: Option<TranscodeProfile{codec,width,height,bitrate}>` (absent = raw, the [ADR-0037](../decisions/ADR-0037.md) default); a new `[storage]` block — `offload: Option<OffloadPolicy{endpoint, bucket, prefix, region, credentials_ref, multipart_part_mib, lifecycle}>` and `catalog: Option<CatalogPolicy{rest_uri, namespace, tables, auth}>`; a `[capture_triggers]` list of `CaptureTrigger` rules (§2.2). `#[serde(tag=...)]` adjacently-tagged unions, **never `untagged`** (conventions §5). Validation: a transcode profile requires an available encoder (typed capability error); offload requires a reachable-or-deferred endpoint (validated lazily, never blocking start); IPv6 endpoint literals bracketed.
- **API** ([ADR-0070](../decisions/ADR-0070.md)): offload/catalog *config* via the existing `PUT /sources|/outputs` + a `/storage` resource (ETag/`If-Match` → `412`, RFC 9457). Operational: `POST /api/v1/recordings/{id}/offload` (force-offload a clip — `202` + op-id, shed-to-`503`), `GET /api/v1/storage/status` (offload backlog, last-uploaded, catalog last-commit, per-bucket reachability — computed off-engine, read role), `POST /api/v1/recordings/{id}/purge` (erasure lever, §5). Trigger rules via a `/capture-triggers` CRUD ([ADR-0069](../decisions/ADR-0069.md)).
- **Realtime** ([ADR-0070](../decisions/ADR-0070.md)): a re-snapshotable `Event::StorageStatus{offload_backlog, last_uploaded_utc, catalog_last_commit, bucket_reachable, bytes_offloaded}`, conflated drop-oldest (invariant #10), never polled. Reuses the `RecordingStatus` lane shape.
- **Warnings** (additive `#[non_exhaustive]` `WarningCode` variants on the [ADR-0035](../decisions/ADR-0035.md) `HealthWarning` model — the `multiview-events` warning-code enum): `storage-offload-backlog` (queue near cap), `storage-offload-failing` (S3 backing off, attempt #/next-delay/errno class), `catalog-commit-failing` (Iceberg commit backing off), `transcode-iso-shed` (proxy encode shed under load), and `storage-offload-data-loss` (emergency disk-pressure dropped an un-offloaded local segment, §3.3). Raise on bad-state, clear on recovery, coalesced by key — surfaced at `GET /api/v1/health`, **not** `/livez`/`/readyz` ([ADR-R009](../decisions/ADR-R009.md): a wedged bucket must not restart-loop the container).
- **UI**: extend the `RecordingsPage` ([ADR-0037 §F](iso-program-recording.md)) with a per-recording offload-state badge + a storage panel (backlog, last-upload, catalog status, per-bucket reachability bar) and a trigger-rules editor; warnings via the existing `HealthBanner`. API types via `openapi-typescript`.

## Open questions

1. **Rust S3 client.** `object_store` (vendor-neutral, local-fs fallback, `put_multipart`) is the recommended default, but its credential-refresh + per-endpoint quirks (SeaweedFS/Ceph signing edge cases) and **IPv6 dual-stack resolution** must be verified against a live MinIO/SeaweedFS before pinning. `rusty-s3` (sans-IO, MinIO-CI-tested) and `aws-sdk-s3` (official) are fallbacks. Decision deferred to the implementation spike.
2. **`iceberg-rust` write maturity.** The apache/iceberg-rust crate's *write* + *REST-catalog-commit* path has historically trailed its read path. If write support is insufficient at build time, the fallback is to emit Iceberg-compatible Parquet data files + commit via the REST catalog HTTP API directly, or stage rows for SeaweedFS's own maintenance/compaction to absorb. To be settled by a spike against SeaweedFS's REST catalog (port 8181).
3. **Lifecycle authority.** Should Multiview *apply* a recommended S3 lifecycle (`PutBucketLifecycleConfiguration`) on first offload, or only *document* it and defer to the bucket owner? Default proposed: document + offer an opt-in apply; never silently override an operator's existing policy.
4. **Transcoded ISO codec policy.** Default proxy codec/res/bitrate, and whether to allow HEVC/AV1 proxies (HW availability + the `gpl-codecs` licence boundary for x265). Likely default: a capped H.264 proxy on the LGPL-clean path; HEVC/AV1 as HW-gated options.
5. **Catalog schema versioning.** Iceberg supports schema evolution, but the `events` table mirrors the realtime envelope which is itself versioned ([realtime-api.md](realtime-api.md)). How tightly to couple the catalog schema to the envelope version (and how to migrate old snapshots) needs a small design pass.
6. **Multi-destination + tiering.** v1 is one bucket. Fan-out to N destinations, cross-region replication, and cold-tier transitions (Glacier-class) are future — but the `OffloadPolicy` shape should not foreclose them.
7. **Pinned-clip ↔ object-store hold mapping.** S3 object-lock / legal-hold semantics differ across MinIO/SeaweedFS/Ceph/AWS; the `pinned_until` → store-hold mapping needs per-endpoint verification (else fall back to a documented "no-lifecycle-expire" tag the operator's policy must honour).
8. **Transcoded-ISO tap point.** Tee from the per-source decoded-frame path (full source cadence + normalized source PTS + the source's audio) for a faithful archive — vs an explicitly-named, cheaper "tile proxy" mode that samples the compositor's last-good tile (display-res, output-cadence, video-only). v1: faithful source-path tap; the tile-proxy mode is an opt-in, separately-named axis (§1.1 design note).
9. **Emergency disk-pressure prune policy.** Default: when the offload backlog is wedged and disk crosses threshold, disarm new recording then drop the *oldest un-offloaded* local segments with a loud `storage-offload-data-loss` warning + recorded keys. Alternative knob: "disarm-only, never drop un-offloaded" (risks a full disk). Which default, and whether the knob is per-recorder, needs an operator-facing decision (§3.3).
