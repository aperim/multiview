# Multiview — Online-Service URL Ingest: YouTube/Vimeo, ISO-Record, Download-to-VT, S3/Iceberg

**Area:** Input / Config / Control / Output·Storage — the *online-service resolver* plane that fronts the existing HLS ingest, the media library, and the recording-storage tail.
**Status:** Design brief (Proposed) — docs-only; implementation follows in dependency-ordered waves.
**Drives:** [ADR-0087](../decisions/ADR-0087.md) (online-service URL ingest — extends ADR-0015: Vimeo + a general resolver, download-to-VT, record-to-S3/Iceberg).
**Extends:** [ADR-0015](../decisions/ADR-0015.md) + [io/youtube-live.md](../io/youtube-live.md) (the shipped `yt-dlp` YouTube-live resolver — **the foundation this brief generalises, never re-litigates**), [ADR-T004](../decisions/ADR-T004.md) (HLS ingest pacing — PTS-to-wall-clock, not `-re`), [ADR-T003](../decisions/ADR-T003.md) (PTS normalisation).
**Builds on:** [media-playout.md](media-playout.md) + [ADR-0057](../decisions/ADR-0057.md) (media library import pipeline — *download-to-VT* feeds this), [recording-storage-offload.md](recording-storage-offload.md) + [ADR-0037](../decisions/ADR-0037.md)/[ADR-0070](../decisions/ADR-0070.md)/[ADR-0071](../decisions/ADR-0071.md) (the recorder + S3 offload + Iceberg catalog — *record-to-S3* reuses this verbatim), [ffmpeg-strategy.md](ffmpeg-strategy.md) (libav sourcing).
**Relates to:** [ADR-0008](../decisions/ADR-0008.md) (NDI runtime-load precedent — the runtime-discovered-binary posture), [ADR-0012](../decisions/ADR-0012.md) (LGPL-clean default build), [ADR-R003](../decisions/ADR-R003.md) (`backon` capped-jitter back-off), [ADR-R004](../decisions/ADR-R004.md) (Class-1/Class-2 apply, inv #11), [ADR-M006](../decisions/ADR-M006.md) (reference-only secrets), [ADR-0042](../decisions/ADR-0042.md) (IPv6-first); sibling intake briefs [url-input.md](url-input.md), [recording-storage-offload.md](recording-storage-offload.md), [output-metadata-and-orientation.md](output-metadata-and-orientation.md), [webui-operability-gaps.md](webui-operability-gaps.md).
**Backlog:** `ONLINE-*` in [`../development/feature-intake-2026-06-13.md`](../development/feature-intake-2026-06-13.md).

> The operator asked for **native Online-Service URL support — YouTube and Vimeo — with ISO-record, download-to-VT, and S3 / S3-Tables (Iceberg)**. Multiview **already ships** the load-bearing half: a runtime-discovered `yt-dlp` **resolver** that turns a YouTube live URL into an FFmpeg-readable HLS master, fed through the standard HLS ingest path ([ADR-0015](../decisions/ADR-0015.md), `crates/multiview-input/src/youtube/`). This brief **generalises** that one proven seam — it does not invent a new ingest mechanism. The new design surface is three *workflows* layered on the resolver: (1) **live ingest** of any resolver-supported service (Vimeo + a general fallback) reusing the existing HLS-pacing path; (2) **download-to-VT** — fetch a service VOD to a local file and hand it to the **media library** ([ADR-0057](../decisions/ADR-0057.md)) so it becomes an ordinary media-player asset; (3) **record-to-S3/Iceberg** — capture a live online source through the **existing recorder + offload + catalog** ([ADR-0037](../decisions/ADR-0037.md)/[ADR-0070](../decisions/ADR-0070.md)/[ADR-0071](../decisions/ADR-0071.md)). Everything below is **Proposed**; nothing in the Vimeo/general-resolver/download-to-VT paths exists today. The genuinely-new code is small: a resolver generalisation + a download-to-file fetcher. The ISO-record and S3/Iceberg asks are **already-designed machinery reused**, not new subsystems — and we say so explicitly so we do not duplicate shipped work.

---

## 0. Headlines

1. **This extends ONE shipped seam; it is not a new ingest engine.** ADR-0015's resolver is built and tested: a *pure* `yt-dlp -J` info-dict parser (`crates/multiview-input/src/youtube/resolve.rs:1`), a thin subprocess shell that discovers and spawns `yt-dlp` with an argument vector + hard timeout (`crates/multiview-input/src/youtube/process.rs:122`), and a pure re-resolution policy + supervised loop for the ~6 h googlevideo-URL expiry (`crates/multiview-input/src/youtube/reresolve.rs:1`). A `youtube` source kind already exists end-to-end in the schema (`crates/multiview-control/src/openapi_schemas.rs:882`). The proposal **widens** this from `youtube`-only to an **online-service resolver**: same subprocess discipline, same HLS hand-off, same isolation. We reuse the parser, the spawn shell, the timeout, the re-resolve loop, and the LGPL-clean Unlicense posture — only the *extractor surface* and a *download mode* are new.

2. **Vimeo and "the rest" are the same resolver with a wider extractor set.** `yt-dlp` ships a dedicated Vimeo extractor and a large extractor set (reported as 1800+) plus a generic embedded-player fallback (*to verify against the pinned `yt-dlp` version before implementation*). A `vimeo` source kind, and an `online`/`url` general kind, are **the same `yt-dlp -J` → info-dict → format-pick path** the YouTube resolver already runs — only the URL recognition and the format selection generalise. Multiview does **not** hard-code per-service extraction logic (that fragility is exactly what the subprocess boundary exists to externalise, [ADR-0015](../decisions/ADR-0015.md) rationale); it asks `yt-dlp` for the info-dict and consumes a libav-readable URL.

3. **Live ingest is PTS-paced, never `-re` — the resolved transport is just HLS (or a progressive URL).** A resolved live online source becomes an `hls` (or generic-URL) input and flows through the **custom HLS input pacer**: wall-clock-by-PTS release, bounded pre-roll ring, bounded catch-up, re-anchor on discontinuity ([ADR-T004](../decisions/ADR-T004.md), invariant #4). `-re` is for files; live ingest is **never** `-re`. The output clock is untouched — a resolver failure, a 403 at expiry, or a service outage degrades the **tile** (LIVE → STALE → NO_SIGNAL, invariant #2) and never the program (invariants #1/#10).

4. **"ISO-record an online source" is the existing recorder, unchanged.** An online live source, once resolved to an HLS/URL input, is **an ordinary input** to the engine. The operator's "iso record" of it is the **already-designed** per-source ISO recorder ([ADR-0037](../decisions/ADR-0037.md), copy-remux of every elementary stream — zero re-encode, invariant #7) with no new recording code. Note the honest caveat: a resolved online source is often **already-encoded HLS** (not raw camera bytes), so a raw ISO captures the *delivered* rendition faithfully; a uniform archival proxy is the **transcoded-ISO** tier ([ADR-0069](../decisions/ADR-0069.md)), off by default and budgeted/shed like any co-tenant (invariant #9).

5. **"Download-to-VT" is a fetch-to-file bridge into the media library — not a second player.** A *VT* (a media-player channel / video player, [media-playout.md](media-playout.md)) plays **library assets**. For a service VOD the operator wants on a VT, the new piece is a **download-to-file fetcher** (`yt-dlp` in *download* mode, or a libav copy of a progressive/HLS-VOD URL to a local container) that produces a file the **existing import pipeline** ([ADR-0057](../decisions/ADR-0057.md) §4) probes, validates, and admits as an ordinary asset. The VT then cues/plays/loops it like any imported clip — the engine never knows it came from a URL. Download + probe + transcode all happen **at import time, off the hot path** (the [ADR-0057](../decisions/ADR-0057.md) §"import does the work, the hot path does none" rule).

6. **"Record-to-S3/Iceberg" is the recording-storage tail, reused verbatim.** Capturing a live online source to S3-compatible object storage + an Iceberg/S3-Tables sidecar is **exactly** [recording-storage-offload.md](recording-storage-offload.md): the closed ISO/Program segments are tapped downstream of the rolling segmenter, enqueued (path, not bytes) into a bounded drop-oldest work queue, uploaded multipart by an off-hot-path uploader with `backon` back-off, delete-after-confirm, and catalogued in Iceberg via SeaweedFS's REST catalog ([ADR-0070](../decisions/ADR-0070.md)/[ADR-0071](../decisions/ADR-0071.md)). **No new storage code** — an online source is just another recorder input feeding that pipeline.

7. **Licensing & ToS stay clean by construction, stated neutrally.** `yt-dlp` is **public-domain (Unlicense)**, run as an out-of-process subprocess, **runtime-discovered, never vendored or linked** — no copyleft/linking obligation, default build excludes the feature entirely (LGPL-clean, [ADR-0012](../decisions/ADR-0012.md), mirroring the NDI runtime-load posture, [ADR-0008](../decisions/ADR-0008.md)). Lawful use — each service's Terms of Service and the content's rights — is **the operator's responsibility**, surfaced as an explicit UI acknowledgement (the ADR-0015 §7 stance, generalised). Multiview takes no position on, and bundles no circumvention of, any service's access controls; it consumes only `yt-dlp`'s processed output.

8. **Honest scope.** v1 should be: a `vimeo` source kind + a general `online`/`url` resolver kind (live ingest, reusing the YouTube resolver core); download-to-VT for a service VOD into the media library; ISO/record-to-S3 by feeding a resolved live source into the existing recorder/offload/catalog. Per-service auth/cookies (reference-only), DASH-only services, age/geo-gated content, and a built-in "browse the service" UI are **future axes**, named below.

## 1. The resolver pattern — generalise ADR-0015 (`yt-dlp`) from `youtube` to online-service

### 1.1 What is already built (do not rebuild)

The YouTube resolver is a clean three-part split that the online-service resolver should reuse wholesale:

| Part | What it does | Where (verified in-tree) | Reuse |
|---|---|---|---|
| **Pure parse core** | `yt-dlp -J` JSON info-dict → `ResolvedHls { manifest_url, live_status, expire_unix }`; classifies `LiveStatus`; reads the `expire` deadline off the resolved URL. I/O-free, panic-free, fixture- + property-tested. | `crates/multiview-input/src/youtube/resolve.rs:1`, `ResolvedHls` `:54`, `LiveStatus` `:19` | Generalise the *format pick* (live HLS master today) to also pick a VOD/progressive URL; keep the parser shape. |
| **Subprocess shell** | Discover + spawn `yt-dlp` with an **argument vector (no shell)**, a **hard timeout** (a hung process is killed, not awaited — invariant #10), bounded/captured stderr, player-client pinned (`web_safari`). `probe_version` reports availability. | `crates/multiview-input/src/youtube/process.rs:122` (`resolve`), `:87` (`probe_version`), `ResolverConfig` `:31` | Reuse verbatim; the binary, timeout, and no-shell discipline are service-agnostic. |
| **Re-resolution policy + loop** | Pure deadline math (refresh `lead` ahead of `expire`, `ttl_guard` clamp) + an immediate re-resolve on a sustained 403 burst; make-before-break URL swap; supervised, backs off, never blocks the clock. | `crates/multiview-input/src/youtube/reresolve.rs:1`, `ReresolveSchedule`, `run_reresolve_loop` | Reuse for any service whose resolved URL expires (most CDN-signed URLs do). |
| **Source kind in the schema** | `Youtube { url }` adjacently-tagged source variant, OpenAPI-documented. | `crates/multiview-control/src/openapi_schemas.rs:882` | Add sibling variants (§1.3); same shape. |
| **Crate module** | `multiview-input::youtube` behind off-by-default `youtube` feature; requires `ffmpeg`. | `crates/multiview-input/src/lib.rs:82`, `:123` | Promote to a service-agnostic module behind an (off-by-default) feature; `youtube` stays an alias. |

### 1.2 The generalisation: one resolver, an extractor-agnostic info-dict

The resolver's contract becomes: **given a service URL, run `yt-dlp -J`, parse the info-dict, and produce a libav-readable transport** — either a **live HLS/DASH master** (→ HLS ingest path, §2.1) or a **VOD/progressive URL** (→ download-to-VT, §2.2, or a paced VOD-as-live ingest). The only per-kind difference is **format selection policy** and **whether the result is live or VOD**:

- The pure parser already classifies `live_status` (`is_live`/`is_upcoming`/`post_live`/`was_live`/`not_live`, `resolve.rs:19`). For a **live** result it picks the `m3u8_native` master (today); for a **VOD** result it should expose the best progressive/HLS-VOD format URL (or hand the URL to download mode).
- **Vimeo and the general case need no Multiview-side extraction logic.** `yt-dlp` carries a dedicated **Vimeo** extractor and a large extractor set (reported as 1800+) plus a generic embedded-player fallback (*to verify against the pinned `yt-dlp` version before implementation*). Multiview asks `yt-dlp` and consumes the processed URL; it never solves a service's signatures itself. This is the entire point of the subprocess boundary ([ADR-0015](../decisions/ADR-0015.md) rationale): keep the fast-moving scraping surface in an operator-updatable external process, off the bulletproof engine.
- **Capability gating is the same** ([ADR-M007](../decisions/ADR-M007.md), the NDI/`CapabilityReport` model): `yt-dlp` absent → the online-service kinds report **unavailable**, never crash. `probe_version` (`process.rs:87`) already does this; the resolver records the resolver version for staleness alarms.

### 1.3 New source kinds (schema)

Adjacently-tagged variants ([conventions §5](../architecture/conventions.md), **never `untagged`**), siblings of the built `Youtube { url }`:

- **`Vimeo { url }`** — a Vimeo watch/event URL; resolved exactly like `youtube`. Live Vimeo events resolve to a live HLS/DASH master; a Vimeo VOD resolves to a VOD URL (→ §2.2 or VOD-as-live §2.1).
- **`Online { url, mode? }`** (the general resolver) — any `yt-dlp`-supported URL. `mode`: `live` (→ HLS ingest), `vod_as_live` (paced VOD ingest), or `download` (→ media library, §2.2). Default `mode` is inferred from `live_status` (live → `live`, otherwise `download` is the safe default for a VOD that should become an asset rather than a perpetually-re-paced tile).
- `Youtube { url }` is retained unchanged (back-compat); it is the `Online` resolver with a YouTube-pinned player client (`web_safari`, `process.rs:22`). Implementation may make `youtube`/`vimeo` thin presets over `online`.

A resolved **live** kind becomes an `hls` input (existing pacer); a resolved **VOD `download`** becomes a media-library import job (§2.2); a resolved **`vod_as_live`** becomes a paced VOD ingest (§2.1). The engine sees only `hls`/`url` inputs and library assets — never "an online service."

## 2. Three workflows

### 2.1 Live ingest — resolver → HLS pacer (invariants #1/#4)

Identical to the shipped YouTube path, widened to Vimeo/general:

1. Resolver runs `yt-dlp -J` **off the data plane** (control/IO plane), bounded by the hard timeout (`process.rs`). Failure raises an operator alarm and degrades the *tile*, never the clock.
2. The resolved **live HLS master** feeds the **standard HLS ingest path** — libav demux/decode + the **custom HLS input pacer** (wall-clock-by-PTS, bounded pre-roll ring, bounded ≤~1.25× catch-up, re-anchor on `EXT-X-DISCONTINUITY`/large PTS jump, start at live-edge minus hold-back) per [ADR-T004](../decisions/ADR-T004.md) + PTS normalisation per [ADR-T003](../decisions/ADR-T003.md). **`-re` is never used for live ingest** (it is for files; FFmpeg warns it drops packets on live and bursts after a stall — [ADR-T004](../decisions/ADR-T004.md) rationale).
3. The **re-resolution loop** (`reresolve.rs`) refreshes the signed URL ahead of its `expire` deadline (make-before-break swap) and immediately on a 403 burst — so a long run survives URL expiry without the tile blinking. (Honest status from `reresolve.rs:36`: the proactive lead-time loop exists + is unit-tested but the CLI today re-resolves on reconnect, a brief-degrade-at-boundary behaviour; wiring the lead-time loop is the named slice that upgrades it to make-before-break.)
4. **VOD-as-live** (`mode = vod_as_live`): a service VOD ingested as a continuous looping/linear tile. It is still **PTS-paced** (the pacer treats VOD-as-live identically — [ADR-T004](../decisions/ADR-T004.md) covers "Live/VOD-as-live"), **not** `-re`. This is the right mode for "put this Vimeo VOD on a tile forever"; for "play it on a media player with transport," prefer download-to-VT (§2.2).

**Tile state machine** rides as for any input (invariant #2): a resolver outage / service-down / geo-block holds last-good then NO_SIGNAL, with a typed warning — never a stall.

### 2.2 Download-to-VT — fetch a service VOD into the media library

The operator's "download to VT" means: take an online-service **VOD** and make it a **video-player (VT) asset** the operator can cue/play/seek/loop. A VT plays **library assets** ([media-playout.md](media-playout.md) §0: "a media *player* is an ordinary bus-selectable source… loads library assets"), so the bridge is **fetch-to-file → existing import pipeline**:

1. **Fetch.** A new **download fetcher** runs `yt-dlp` in *download* mode (or, when the resolver already yields a directly-libav-readable VOD URL, a libav stream-copy of that URL) to a **local file** in a staging directory. This is a control-plane, long-running operation (`202` + operation id, progress on the realtime stream — the [ADR-0057](../decisions/ADR-0057.md) §4 import-job shape), bounded by a hard timeout + `backon` back-off, **off the data plane**. A wedged/slow download can only delay the *import job*, never a frame (invariants #1/#10).
2. **Import.** The downloaded file enters the **existing media-library import pipeline** ([ADR-0057](../decisions/ADR-0057.md) §4 / [media-playout.md](media-playout.md) §4): probe (container/codec/res/fps/duration/alpha), validate, **reject-or-transcode** (typed RFC 9457 errors — never a half-imported asset). All decode/transcode is **at import time, off the hot path** ([media-playout.md](media-playout.md) §"import does the work; the hot path does none").
3. **Play.** The asset is now an ordinary library clip; a media-player (VT) channel cues/plays/loops/seeks it via the existing transport. The engine samples whatever frame the player's last-good store holds at `tick.pts` — it never knows the asset originated from a URL (invariants #1/#2/#10 by construction, [media-playout.md](media-playout.md) §0).

**Why download (not VOD-as-live) for a VT:** a VT wants **scrub/seek/precise transport** on a *finite, local* asset — exactly what a downloaded file gives. VOD-as-live (§2.1) is the right mode only when the operator wants a perpetually-paced tile, not a transport-controlled player. Both are offered; the resolver `mode` chooses.

**Scope honesty:** the import pipeline + media library are themselves **Proposed** ([ADR-0057](../decisions/ADR-0057.md), not built). Download-to-VT therefore **depends on** that pipeline landing; this brief adds only the **fetch-to-file** stage in front of it and the resolver kind. We do not duplicate the library/import design — we link it.

### 2.3 Record-to-S3 / Iceberg — feed a resolved source into the existing recorder + offload + catalog

"ISO record" and "S3 / S3-Tables (Iceberg)" of an online source are **already-designed machinery**, reused without new storage code. Once an online live source is resolved to an HLS/URL input, it is an ordinary recorder input:

- **ISO record** — the per-source **copy-remux ISO** recorder ([ADR-0037](../decisions/ADR-0037.md), [iso-program-recording.md](iso-program-recording.md)): faithful capture of every elementary stream of the resolved rendition, zero re-encode (invariant #7), rolling-segmented, dual-axis retention, disk-pressure-gated — **never stalls the engine**. (Caveat per Headline #4: a resolved online source is an already-encoded delivery rendition; raw ISO archives *that*. A uniform proxy is the transcoded-ISO tier, [ADR-0069](../decisions/ADR-0069.md).)
- **Record-to-S3** — the **object-store offload** ([ADR-0070](../decisions/ADR-0070.md)): closed segments tapped downstream of the rolling segmenter, enqueued as **paths (not bytes)** into a bounded drop-oldest queue, uploaded **multipart** (≥5 MiB parts, streamed) by an off-hot-path uploader with `backon` back-off, **delete-after-confirm**, vendor-neutral `object_store` client, IPv6-first endpoints. A wedged bucket → local copies pending + a warning, **never a stalled frame** (invariants #1/#10 by construction — the uploader shares no channel/lock/buffer with the engine).
- **Iceberg / S3-Tables catalog** — the **SeaweedFS REST Iceberg catalog** ([ADR-0071](../decisions/ADR-0071.md)): one `recordings` row per closed artefact + an `events` row per persisted realtime event, so "what happened" joins to "where the bytes are." Written off the hot path, after durable upload. **Scope note:** this path targets *generic Iceberg-over-object-storage* via an Iceberg REST catalog; Amazon **S3 Tables** is a distinct managed catalog (its own REST endpoint, maintenance, and namespace semantics), not merely "S3 + any Iceberg REST catalog." Whether to add a dedicated S3 Tables catalog integration (and its compatibility limits) is an Open question (§9, q9), not assumed satisfied by the SeaweedFS path.

The **only** online-specific addition here is **provenance metadata**: the catalog `recordings`/`events` rows should carry a `source_origin` (e.g. `online:vimeo`, the *URL is reference-only and secret-scrubbed* — never the signed CDN URL with embedded tokens, per [ADR-M006](../decisions/ADR-M006.md) + the [ADR-0052](../decisions/ADR-0052.md) data-minimisation rule). That is a one-field schema note on the existing catalog, not a new subsystem.

## 3. Licensing & Terms-of-Service caveats (neutral)

Stated plainly and vendor-neutrally; this is a docs-only design and takes no legal position.

- **`yt-dlp` is public-domain (Unlicense), run as a subprocess, runtime-discovered, never vendored or linked.** No linking/copyleft obligation attaches to the Multiview build; the default build **excludes** the feature entirely and is LGPL-clean ([ADR-0012](../decisions/ADR-0012.md)). This mirrors the NDI runtime-load posture ([ADR-0008](../decisions/ADR-0008.md)) — the capability is simply *reported unavailable* when the binary is absent. (An operator who *redistributes* a PyInstaller-bundled `yt-dlp` should review that bundle's dependency licenses; Multiview ships neither the binary nor a Python runtime.)
- **Terms of Service are the operator's responsibility.** Each online service (YouTube, Vimeo, and any `yt-dlp`-supported site) has its own ToS governing access and downloading, and the *content* has its own rights/licence. Multiview surfaces an **explicit operator acknowledgement** (the [ADR-0015](../decisions/ADR-0015.md) §7 stance, generalised across services and across *download* as well as *ingest*) and records that the operator asserted lawful use. Multiview does not adjudicate or enforce any service's ToS.
- **No circumvention bundled or claimed.** Multiview consumes only `yt-dlp`'s processed output URL/file; it bundles no signature-solver, no DRM circumvention, and no vendor SDK or vendor specification. The clean-room/nominative posture is the [managed-devices.md](managed-devices.md) ZowieTek-driver stance: interoperate over published/community-maintained tooling, name services nominatively only, and **defer to the operator on the legality of any specific source**. DRM-protected content (e.g. Widevine-gated streams) is **out of scope** — `yt-dlp` does not bypass DRM and neither does Multiview.
- **Secrets by reference.** Per-service cookies/auth, where an operator chooses to use them, are **reference-only** ([ADR-M006](../decisions/ADR-M006.md): env/1Password/file), never a literal in committed config; gitleaks/`cargo deny` gates apply. `yt-dlp` is spawned with an argument vector (no shell); stderr is bounded and **must be redacted with tested patterns** before logging. Redaction limits leakage but cannot guarantee every future service's token format is scrubbed — any unknown token reaching the log is treated as a security bug, not an accepted residual.
- **Resolver fragility is expected and handled, not hidden.** Service player surfaces rotate; extraction can break. The subprocess boundary makes this an **operator-updatable** concern (`yt-dlp` is updated independently of Multiview); the resolver records its version, raises a staleness/extraction-failure alarm, and **degrades the tile, never the output** ([ADR-0015](../decisions/ADR-0015.md) consequences).

## 4. Efficiency budget (mem / cpu / gpu / io)

Per the standing efficiency review:

- **Memory.** The resolver holds a bounded info-dict JSON string + a bounded stderr capture (`process.rs:63`) — no media buffering; resolution is off the data plane. The download fetcher streams to disk (no whole-file-in-RAM); the import pipeline's bounds are [ADR-0057](../decisions/ADR-0057.md)'s. Record-to-S3 queues **paths, not bytes** ([ADR-0070](../decisions/ADR-0070.md)) — bounded by queue depth, never file size. **Zero unbounded growth on the data plane.**
- **CPU.** Live ingest adds the existing HLS demux/decode + pacer cost of one more input — **no new per-frame CPU** beyond a normal HLS source. The resolver subprocess is a brief, bounded, off-peak burst (it is `yt-dlp`'s cost, in its own process, killed if it hangs). The one real CPU/GPU cost is an **optional transcoded-ISO proxy** or an **import-transcode** — both off by default, budgeted, and shed first (invariant #9, [ADR-0069](../decisions/ADR-0069.md)/[ADR-0057](../decisions/ADR-0057.md)).
- **GPU.** Live ingest decodes at display resolution like any source (invariant #6); no extra GPU for resolution, download, offload, or catalog. Only an opt-in transcode touches the encoder, as a below-program co-tenant.
- **I/O.** Download-to-VT is a one-shot fetch to staging disk (off-peak, bounded by timeout + back-off). Record-to-S3 doubles read I/O on closed segments (read-to-upload) — sequential, off-peak, on the staging volume, per [ADR-0070](../decisions/ADR-0070.md). Network egress is bounded by `backon` back-off + bounded uploader concurrency. **IPv6-first** throughout ([ADR-0042](../decisions/ADR-0042.md)): resolved/endpoint URLs accept bracketed IPv6 literals and must dual-stack-resolve.

## 5. Config / API / Realtime / UI surface (sketch)

All additive, serde-default, non-breaking (existing configs keep parsing); pinned in [ADR-0087](../decisions/ADR-0087.md), summarised here:

- **Config.** New adjacently-tagged source variants `Vimeo { url }` and `Online { url, mode? }` alongside the built `Youtube { url }` (`openapi_schemas.rs:882`). A `[resolver]` block: `yt_dlp_path` (default `"yt-dlp"`, discovered on `PATH`), `timeout`, optional `cookies_ref`/`player_client` per service (reference-only). A `download` target for VOD → media-library asset id ([ADR-0057](../decisions/ADR-0057.md) import config). Validation: a resolver source requires `yt-dlp` present-or-deferred (typed capability error, never blocking start); IPv6 literals bracketed.
- **API.** Source CRUD via the existing `/api/v1/sources` (the variant rides the existing source union). `POST /api/v1/library/import-from-url { url, mode }` for download-to-VT (`202` + op-id; result on the stream — the [ADR-0057](../decisions/ADR-0057.md) import-job pattern; shed-to-`503`). Online recording/offload reuses [ADR-0070](../decisions/ADR-0070.md)'s `/storage` + `/recordings/{id}/offload` unchanged. RFC 9457 errors; `ETag`/`If-Match`.
- **Realtime.** Resolver health rides a re-snapshotable `Event` (resolver version, last-resolve-utc, extraction-failure count) on the conflated drop-oldest lane (invariant #10) — never polled, never back-pressuring. Import-job progress reuses the [ADR-0057](../decisions/ADR-0057.md) job-progress event; offload status reuses [ADR-0070](../decisions/ADR-0070.md)'s `Event::StorageStatus`.
- **Warnings.** Additive `#[non_exhaustive]` `WarningCode` variants: `resolver-unavailable` (binary absent), `resolver-extraction-failing` (service rotated / anti-bot block — attempt #/next-delay), `resolver-stale` (version older than a recommended floor). Surfaced at `GET /api/v1/health`, **not** `/livez`/`/readyz` (a broken resolver must not restart-loop the container).
- **UI.** A source-add flow accepting an online URL with a **ToS acknowledgement checkbox** (§3); a download-to-VT action in the media-library UI; resolver health in the existing `HealthBanner`. API types via `openapi-typescript`.

## Open questions

1. **Feature/module naming.** Promote `multiview-input::youtube` to a service-agnostic `online`/`resolver` module (keeping `youtube` as an alias + the YouTube-pinned player client), or add `vimeo`/`online` as siblings? A reversible default: a shared `resolver` core with thin `youtube`/`vimeo`/`online` presets — settled at implementation, no invariant impact.
2. **DASH-only services.** The YouTube path pins HLS (`m3u8_native`) and avoids DASH. Some services expose **only** DASH for live; libav can demux DASH but the pacing/discontinuity behaviour differs. Whether to ingest DASH directly or require an HLS rendition needs a spike against a real DASH-only live source (Vimeo's live transport must be verified — *the Vimeo extractor is reported to handle both live and VOD, but the live transport type Multiview receives must be confirmed before implementation*). **v1 acceptance is therefore conditional for live Vimeo:** Vimeo VOD is supported via download-to-VT (§2.2) / VOD-as-live; live Vimeo is supported only after confirming an HLS rendition (or defining DASH pacing/discontinuity behaviour) against a real live event.
3. **VOD-as-live vs download default.** For a `vimeo`/`online` VOD with no explicit `mode`, default to `download` (asset) or `vod_as_live` (tile)? Proposed default: `download` (the safer, transport-rich choice); `vod_as_live` is explicit. Reversible.
4. **Re-resolution generality.** The `expire`-deadline parsing (`resolve.rs` `parse_expire`) is googlevideo-URL-shaped. Other services' signed-URL expiry encodings differ; the re-resolve loop must learn each service's TTL signal (or fall back to the `ttl_guard` upper bound + 403-burst trigger, which already works service-agnostically). Spike per service.
5. **Download fetcher: `yt-dlp` download mode vs libav copy.** When the resolver already yields a libav-readable VOD URL, a libav stream-copy avoids re-spawning `yt-dlp` for the bytes; when it does not (merged formats, fragmented DASH), `yt-dlp` download mode is needed. Policy: prefer libav-copy when the URL is directly readable, else `yt-dlp` download. Verify per service.
6. **Catalog provenance privacy.** The `source_origin` field must never carry a signed URL with embedded tokens ([ADR-M006](../decisions/ADR-M006.md)/[ADR-0052](../decisions/ADR-0052.md)). What granularity is safe (`online:vimeo` + a stable opaque source id) vs operator-meaningful? A small data-minimisation pass.
7. **Auth/cookies & age/geo-gated content.** Reference-only cookies for member-only/age-gated content are a future axis; their handling (per-service secret refs, rotation) and the ToS implications are operator-driven and out of v1 scope — named, not silently dropped.
8. **Dependency on unbuilt media library.** Download-to-VT depends on [ADR-0057](../decisions/ADR-0057.md)'s import pipeline (itself Proposed, unbuilt). Sequencing: the resolver generalisation + live ingest + record-to-S3 can ship on already-built/already-designed machinery; download-to-VT lands with (or after) the media-library import pipeline. Stated so the wave order is honest, not parked. **Acceptance boundary:** the operator's "download-to-VT" ask is **not** complete until that dependent import path is available — the resolver + fetch-to-file stage shipping alone does not satisfy it.
9. **Generic Iceberg vs Amazon S3 Tables.** The reused catalog tail ([ADR-0071](../decisions/ADR-0071.md)) is generic Iceberg-over-object-storage via an Iceberg REST catalog. If the operator specifically requires Amazon **S3 Tables** (its managed catalog endpoint + maintenance semantics), that is a separate catalog-client integration with its own compatibility limits — scope it explicitly before claiming the S3-Tables ask is met. Reversible default: ship generic-Iceberg first; add an S3 Tables catalog adapter only if required.
