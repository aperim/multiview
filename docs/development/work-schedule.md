### 1c. Recommended fanout тАФ 1 serial integrator + 6тАУ7 parallel lanes

Partition by **file territory** (not by stream), so each lane owns a disjoint set of files and runs
as its own worktree; items *within* a lane are serial in `deps:` order.

- **LANEтАСCORE (serial тАФ the long pole, one "dataтАСplane integrator").** Owns `pipeline.rs`,
  `sink.rs`, `run.rs`, `control.rs`, `events/event.rs`, `command.rs`, `config/schema.rs`.
  Order: **GPUтАС1** (encodeтАСonceтАСmuxтАСmany тАФ *do first; everything muxes through it*) тЖТ **ENGтАС1**
  (teardown join) тЖТ **OUTтАС1/OUTтАС3** (output decision + schema) тЖТ **CTLтАС1 тЖТ CTLтАС3 тЖТ CTLтАС2 тЖТ CTLтАС5**
  (controlтЖТengine apply) тЖТ **AUDтАС2 тЖТ AUDтАС3 тЖТ AUDтАС4 тЖТ AUDтАС5 тЖТ AUDтАС7** (audio spine) тЖТ **INтАС3, INтАС5**
  (NDI/YouTube ingest wiring) тЖТ **OUTтАС2/OUTтАС4** (RTSP/NDI sinks) тЖТ **GPUтАС2** (software clock) тЖТ
  **ENGтАС2** (PTS normalizer) тЖТ **CTLтАС6** (ClassтАС2). *This lane is the critical path for
  parallelism тАФ keep it staffed.*
- **LANEтАСIN (parallel).** `crates/multiview-input/src/{st2110,webrtc,youtube}/` тАФ **INтАС1 тЖТ INтАС2 тЖТ
  INтАС6**, **INтАС4**, then **INтАС7** (CI). Only INтАС3/INтАС5 (which touch `pipeline.rs`) hand off to CORE.
- **LANEтАСPRV (parallel).** `crates/multiview-preview/*` тАФ **PRVтАС1 тЖТ PRVтАС3 тЖТ PRVтАС4 тЖТ PRVтАС5**.
  *Coordinate PRVтАС2* (adds routes to `control/routes/mod.rs` + `state.rs`) with LANEтАСAPI/CORE.
- **LANEтАСENG (parallel).** `engine/src/{ptp,ha}.rs`, `hal/src/load.rs`, `cli/wallclock.rs` тАФ
  **ENGтАС6**, **ENGтАС5 тЖТ ENGтАС3**, **ENGтАС4** *(shares `hal/load.rs` with GPUтАС5 тАФ merge with LANEтАСGPU
  or let one owner hold `load.rs`)*.
- **LANEтАСGPU (parallel).** `compositor/*`, `ffmpeg/hwframe.rs`, `hal/select.rs` тАФ **GPUтАС3 тЖТ GPUтАС4**,
  **GPUтАС5**, **GPUтАС6**. (GPUтАС1/GPUтАС2 are in CORE.)
- **LANEтАСBCAST (parallel).** `control/src/{nmos,is07}.rs` тАФ **SURтАС1**, **SURтАС2**; plus **SURтАС3**
  (captions, `ffmpeg`/`input` тАФ independent).
- **LANEтАСAPI (parallel, but the *crossтАСlane coordination hub*).** Owns `control/openapi.rs` +
  `routes/mod.rs` route registration + the codegen тАФ **SURтАС4 тЖТ SURтАС5**, **SURтАС6**, and **CTLтАС4**
  (ApplyLayout route). Because PRVтАС2, CTLтАС4, SURтАС4, SURтАС6 all touch `openapi.rs`/`routes/mod.rs`,
  **one owner registers all new routes + OpenAPI annotations** to avoid churn; other lanes file
  their handler bodies and let LANEтАСAPI wire the router.
- **LANEтАСWEB (parallel).** `web/src/` тАФ **AUDтАС8** (audio matrix UI) after AUDтАС7; **SURтАС5** (generated
  client) after SURтАС4.

**Three coordination points to watch in any fanout:** (i) `control/routes/mod.rs` + `openapi.rs`
(CTLтАС4 ┬╖ PRVтАС2 ┬╖ SURтАС4 ┬╖ SURтАС6) тЖТ LANEтАСAPI owns route/spec registration; (ii) `hal/load.rs`
(ENGтАС4 ┬╖ GPUтАС5) тЖТ one owner; (iii) `control/state.rs` (CTLтАС3 ┬╖ PRVтАС2) тЖТ agree the `AppState` shape
first.

**So a "complete fanout" = ~7 concurrent worktrees** (CORE + IN + PRV + ENG + GPU + BCAST + API/WEB),
with CORE as the staffed critical path and the three coordination points preтАСagreed. Expect CORE to
dominate wallтАСclock; the six parallel lanes finish well before it.

- **Touches:** `StreamItem` (pipeline.rs:1397) gains an `audio: ProgramAudio` field (program-bus `AudioBlock` + per-discrete-track blocks); the hot-loop projection `state_of` (pipeline.rs:979) samples the audio stores and runs `Mixer::mix_program` (mixer.rs:148); `StreamEgress`/`StreamingFrameSource` (pipeline.rs:1570, run_one_output:1729) carry the audio to sinks.
- **Approach:**
  1. Construct one `Mixer` (mixer.rs:60) per run at the canonical 48k/stereo format; `add_input` + `route_to_program` per source from the routing config (AUD-7).
  2. In `state_of`, after sampling video, pull `samples_per_tick` from each audio store, `Mixer::submit` each, call `mix_program()` тЖТ the program `AudioBlock`; attach to `StreamItem`. The mix is cheap and pure; it runs on the hot loop but does no I/O and cannot block (the stores are non-blocking) тАФ assert this is within tick budget.
  3. Extend the egress channel payload from `Arc<Nv12Image>` to a `(Arc<Nv12Image>, ProgramAudio)` (or a small struct) so each sink receives matched A/V per tick. Honour the same drop-on-overload/block-for-exact `SendPolicy` (pipeline.rs:122) тАФ audio rides the same bounded queue, dropped *with* its frame so A/V stay locked.
- **Acceptance (done when):** test drives N ticks and asserts `total_audio_samples тЙИ ticks ┬╖ samples_per_tick` (continuity тАФ the audio analogue of "output-clock never stalls"); a dropped-source test asserts the program bus is gap-free silence, never absent. Invariant #1 re-assert: audio sample count is a pure function of tick count; #10: mixing never back-pressures the engine. ffprobe check deferred to AUD-4.
- **Risks/notes:** `samples_per_tick` is fractional for 1001-denominator rates (e.g. 30000/1001 тЖТ 1601.6 samples/tick) тАФ accumulate the remainder across ticks (1601/1602 alternation) so the long-run sample count stays exact (this is the audio side of the drift invariant). ## TLS / ACME (DNS-01) for the control plane тАФ phased backlog

Subsystem: `multiview-control` (+ `multiview-config`, `multiview-cli`). Design: ADR-0029, brief
[acme-tls](../research/acme-tls.md). Off-by-default Cargo features `tls` (тЖТ `axum-server/tls-rustls`)
and `acme` (тЖТ `instant-acme` + `reqwest/rustls-tls`), wired into `multiview-cli` aggregates. Cloudflare
is the first and (for now) only DNS provider. Dependency-ordered; each phase commits failing tests first
(TDD), keeps default `cargo check` plain-HTTP + LGPL-clean + `cargo deny check` green.

### TLS-0 тАФ static-cert rustls termination (the floor; no ACME)
- New `tls` feature; `serve_tls()` sibling of `serve()` using `axum_server::from_tcp_rustls` over the
  existing bound listener (`tokio::TcpListener` тЖТ `.into_std()`), `RustlsConfig::from_pem_file`,
  `axum_server::Handle` for graceful shutdown. Config `[control.tls] mode="static"` +
  `cert_file`/`key_file`. Pure serving glue, no network.
- **Acceptance:** with `mode="static"` and an operator cert, the SPA/REST/WS are served over HTTPS on
  the bound listener; `mode` absent тЗТ unchanged plain HTTP; default build pulls no openssl
  (`cargo deny check` green).

### TLS-1 тАФ ACME core: account + order + DNS-01 flow (provider stubbed)
- New `acme` feature; integrate `instant-acme` 0.8.5: `Account::create`/`from_credentials` with
  `AccountCredentials` persisted `0600` under `/var/lib/multiview/acme/account.<env>.json`; `new_order`
  тЖТ walk authorizations тЖТ `Dns01` тЖТ `key_authorization().dns_value()`; bounded-backoff order polling +
  finalize (`rcgen` CSR) + `certificate()`. DNS publish behind a `DnsProvider` trait whose test impl is
  an in-memory fake. Directory URL config-driven, default **staging**; CI uses Pebble.
- **Acceptance:** against Pebble/staging with the fake provider, a full issue cycle yields a valid cert
  chain + key on disk (`0600`); account creds persist and are reused on a second run (no re-register).

### TLS-2 тАФ `DnsProvider` trait + Cloudflare implementation
- Finalise the object-safe `async_trait` `DnsProvider` (`create_txt_record` тЖТ `TxtRecordHandle` newtype,
  idempotent `delete_txt_record`, `wait_for_propagation(deadline)`). Cloudflare impl over
  `reqwest` (rustls-tls, no default features): `POST`/`DELETE /zones/{zone_id}/dns_records`, Bearer
  token, `ttl:60`, optional `GET /zones?name=` when `zone_id` unset. `wait_for_propagation` polls the
  zone's **authoritative** NS for the TXT until present or deadline. Mandatory TXT cleanup on success
  and on failed orders.
- **Acceptance:** an end-to-end DNS-01 issuance against staging using a real single-zone
  **Zone.DNS:Edit** token publishes then deletes the `_acme-challenge` TXT (no residue) and installs a
  trusted leaf with the requested SAN/wildcard.

### TLS-3 тАФ config schema + hot-reload renewal task
- Extend `ControlConfig` (`#[non_exhaustive]`): `Option<TlsConfig>` with adjacently-tagged `mode`
  (`static`|`acme`) and `provider` enums (never `untagged`); `[control.tls.acme]` (`directory`,
  `contact_email`, `domains`, `state_dir`, optional `renewal_lead`) + `[control.tls.acme.cloudflare]`
  (`zone_id`, `api_token_ref = "env:тАж"|"file:тАж"|"op://тАж"` тАФ never the literal token). Validation: exactly
  one of static/acme per `mode`, non-empty `domains`, parseable directory URL, known provider, token ref
  present; reject a literal-looking token. Detached `tokio::spawn` renewal task: renew at
  тЕУ-remaining-lifetime + jitter, `RustlsConfig::reload_from_pem` hot-swap, **fail-soft** (warn +
  telemetry + bounded backoff, keep serving existing cert). Metrics
  `multiview_tls_cert_expiry_seconds`, `multiview_tls_renewal_failures_total`.
- **Acceptance:** a config-driven `mode="acme"` run issues at boot and hot-swaps a renewed cert with no
  listener restart and no dropped connections; a forced renewal failure leaves the existing cert serving
  and emits the failure metric; the token is sourced from env/1Password, never logged.

### TLS-4 тАФ hardening, isolation soak, and tests
- Chaos/isolation gate assertion: a stalled/failing ACME order or killed renewal task leaves **program
  output and the existing TLS listener unaffected** (invariant #10 soak). Property/idempotency tests on
  `delete_txt_record` and the renewal scheduler (no duplicate orders on transient failure). Atomic cert
  install (temp+fsync+mode+rename) test; `0600`/uid-10001 permission test. Operator docs: CAA + RFC 8657
  `accounturi`, single-zone token + IP filter, CT-log monitoring, stagingтЖТprod promotion runbook,
  account-key/token rotation runbook. Confirm `reqwest` resolves to rustls in `Cargo.lock`.
- **Acceptance:** isolation soak passes (output never stalls under ACME fault injection); mutation/property
  tests green on the renewal + cleanup logic; `cargo deny check` green with `--features tls,acme`; default
  build still plain-HTTP + openssl-free.
